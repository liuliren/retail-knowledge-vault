## §3 任务状态机(v2.0 §6 采纳·对齐环协议)

```
取项(INIT) → 摸底(DISCOVERY·只读不改) → 方案(PLAN·重活先落方案件)
→ 授权门(操作四档:A/B直行·C清单·D签字) → 执行(EXECUTION·每大步更新RESUME)
→ 验证(VALIDATION·脚本/实跑) → 验收(REVIEW·抽验≥3处落盘) → 落账(COMMIT·队列+Ledger+git) → DONE
```

- 不许跳步;跳步=该轮无效。轻活(A档单文件)允许 摸底/方案 合并为一念,但验收落账不可省。
- DONE 判据(v2.0 §16 采纳):实现✅+验证✅+文档✅+状态文件✅+git✅+RESUME 已更新指向下一步。

## §4 上下文与用量自动降级(v2.0 §8/§9 吸收·阈值本地化)

| 信号 | 动作 |
|---|---|
| 上下文 <50% | 正常执行 |
| 50–70% | 上下文卫生:禁"以防万一"读文件(Read Gate 严格执行)/禁重复解释/长结论落盘代替留在对话 |
| 70–85% | **先更 RESUME+队列 → 再 /compact**(PreCompact hook 自动出摘要卡,已在位) |
| >85% | 停止取新项;当前项收口到最近可落账点→落盘→开新会话按 RESUME 接 |
| 代理撞限额 | 限额自保协议(已生效):落盘→回灌队列→主线程串行小步→探针试恢复→重派(2026-07-07 实战验证全流程) |
| 主线程撞限额 | 落盘(队列+RESUME+commit)→ScheduleWakeup 链式滚动(≤1h/次)至重置点→自动接跑 |

## §5 BLOCKED 机制(v2.0 §15 采纳·新增)

卡住时**禁止连猜三次以上**。第 3 次同类失败 → 在队列该项下挂 `⛔BLOCKED` 行:问题/证据/可能原因/需要谁的什么决定;属六哥判断相的→转⏸六哥并继续取下一项,不空转。

## §6 夜跑规程(v2.0 §13 采纳·护栏加严)

- 触发:六哥说"挂夜跑"或 cron/ScheduleWakeup 到点;逐轮=环协议 §3 全状态机。
- **夜跑白名单**:A/B 档(可逆增量/批量)+L0/L1 机械活+lint消债+压测+草稿起草。
- **夜跑黑名单(醒后人裁)**:C/D 档一律不做;对外发布/删除/T2T3 三硬红线;首次跑的新脚本对客户数据目录的写操作;任何"标准/铁律"类文件升级。
- 每晨产物:夜跑报告一页(做了什么/验收证据/BLOCKED 清单/建议今日 3 撬动)——并入开机播报。

## §7 与 v2.0 的三处显式分歧(审议时重点碰)

1. **模型路由**:v2.0 "Opus 默认做架构+复审" ✗ → 本库 L4 Opus 过双门(结构必要+不可分解)才用,主线程=Fable(L3)。已签机制优先。
2. **文件系统**:v2.0 `.agent/` 七件 ✗ → 全部映射既有 SSOT(§2),零新建。
3. **APPROVAL 一刀切** ✗ → 用操作四档分级(A/B 免批直行是扩权令成果,回退到"每任务都审批"是倒退)。
