## §3 任务状态机(v2.0 §6 采纳·对齐环协议)

```
取项(INIT) → 摸底(DISCOVERY·只读不改) → 方案(PLAN·重活先落方案件)
→ 授权门(操作四档:A/B直行·C清单·D签字) → 执行(EXECUTION·每大步更新RESUME)
→ 验证(VALIDATION·脚本/实跑) → 验收(REVIEW·抽验≥3处落盘) → 落账(COMMIT·队列+Ledger+git) → DONE
```

- 不许跳步;跳步=该轮无效。摸底/方案合并权按档位划(2026-07-07 v2.2 修订·应修-7):**A/B 档均可合并**摸底/方案为一念(B 档合并时须在队列行内一句话写明批量范围,如"lint消债·涉N文件·范围X");**C/D 档、或预计执行>30分钟的任务,强制独立方案件**,不得合并。验收落账不可省,任何档位均不可省。
- DONE 判据(v2.0 §16 采纳):实现✅+验证✅+文档✅+状态文件✅+git✅+RESUME 已更新指向下一步。
- **验收执行(2026-07-07 v2.2 修订·应修-6)**:验收=按 `SYS-LOOP-001` 验收铁律执行,本协议不复述抽验数字(旧"≥3处"对单文件/批量两端规模都失真,单文件凑3处无意义、批量千字段抽3处覆盖率趋近于0)。**建议**(不越权改队列文件,仅记录于此供未来采纳):验收铁律侧可补规模分档——单件全验、批量按 √N 或 ≥5% 抽样,该建议只在本协议内提出,是否采纳由 `SYS-LOOP-001` 自行裁定。
- **扩权令作用域裁定(六哥 2026-07-07 裁·P8阻断解除)**:扩权令(迭代免签直行)**仅在 SYS-LOOP-001 当前迭代窗口内、有主线程实时在场时有效**;C 档动作(移动/重命名/覆盖等不可逆但非删除/非对外/非T2T3的操作)在**夜跑或任何无人值守场景下一律回落操作四档默认**(列清单→下次六哥在场时确认→执行),不得免签直行。此裁定解除第2轮审议阻断-1。

## §4 上下文与用量自动降级(v2.0 §8/§9 吸收·2026-07-07 v2.2 修订:阈值改可观测信号·应修-3)

> 旧版四行阈值表按"上下文百分比"划档,但百分比对模型/代理自身不可观测(无法自报"我在70%"),且对"单任务本身超85%上下文"(如一次性通读大文件)无解——故本版删阈值表,改用两条可观测规则:

1. **INIT 阶段预飞估算(新增·治单任务超限)**:取项时先估读量。预计本任务需读的文件/内容量**>本会话预算的一半**(以"已读文件数骤增""需通读的单文件超常规篇幅"等可感知信号判断,不要求精确百分比)→ **先切分任务**(拆成可独立收口的子任务)**或派隔离上下文的子代理**处理该子任务;切不动(任务本身不可分且必须整体读完)→ 挂 `⛔BLOCKED-CONTEXT` 转六哥裁(是否值得为此单任务开一整个会话)。
2. **执行中可观测信号触发降级(替代原百分比表)**:以下任一信号出现即触发"先落盘再压缩"——① PreCompact hook 触发(已在位,自动出摘要卡);② 单次会话已读文件数/ 修改文件数明显超出任务原定范围(如原估3-5个文件、实读超10个);③ 连续多轮(如≥15轮)未到收口点。触发后动作统一:**先更 RESUME+队列 → 再 /compact**;若 hook 未触发但已到自然收口点,亦可主动落盘开新会话接续,不必等硬信号。
3. **代理/主线程撞限额(纯链接,不复述·应修-3)**:撞限额后的处理流程,统一按**限额自保协议**(已生效·2026-07-07 实战验证全流程)执行,本协议不重复其步骤细节,仅在此建立映射链接,防止两处文本各自漂移。

## §5 BLOCKED 机制(v2.0 §15 采纳·2026-07-07 v2.2 修订:判据从"次数"改"信息增量"·应修-4)

> 旧版"第3次同类失败挂 BLOCKED"用固定次数判据,对两类场景都不准:① 健康调试(如 posclean 七坑逐坑排除,本是靠"排除法"收敛,每次失败都在缩小范围,不该被误杀);② 不可逆操作(等到第3次失败可能已造成不可逆损失,太晚)。改判据为**信息增量**:

- **"同类"定义**:同一假设下的重试 = 同类尝试;一旦更换假设/方法 = 新序列,重新计数。
- **可逆/可调试操作(如脚本调试、数据清洗排坑)**:判据不看次数,看**是否还在产生新证据或新假设**——连续尝试**无新证据、无假设更新**(即在重复同一件已知会失败的事,而非排除新的可能)→ 挂 `⛔BLOCKED`。这通常在 2–3 次尝试后自然触发,但触发条件是"信息停滞"而非"数到3"。
- **不可逆操作(删除/覆盖/对外发布/写客户数据等)**:**首次失败或首次遇到不确定性即停**,不得因为"还没到第3次"而继续试探。
- 挂起格式不变:问题/证据/可能原因/需要谁的什么决定;属六哥判断相的→转⏸六哥并继续取下一项,不空转。

## §6 夜跑规程(v2.0 §13 采纳·护栏加严)

- 触发:六哥说"挂夜跑"或 cron/ScheduleWakeup 到点;逐轮=环协议 §3 全状态机。
- **夜跑白名单**(2026-07-07 v2.2 修订·应修-5,补两处缝):A/B 档(可逆增量/批量)+L0/L1 机械活+lint消债+压测(**仅限 P7-1/P7-2/P7-4 类**,见下)+草稿起草(**标准/铁律类起草物受限**,见下)。
  - **草稿起草的限定**:凡起草物内容涉及"标准/铁律/规范/红线"字样(不论是全新起草还是改写既有 draft 标准件的中间态),一律:① 只许落 `Claude输出区/`,不得落正式目录;② 强制标 `status: draft` 且 `审议轮次: 0`;③ 列入次日晨报"待裁清单"首位,不因夜跑"做后报"档位而免于晨报置顶。此规则同时补上"改写既有 draft 标准件"这个中间态的缝(旧版只覆盖"新起草",遗漏了"接着改一份已有 draft 标准件"的情况)。
  - **压测的限定**:白名单"压测"仅指 P7-1(常规功能压测)/P7-2(性能压测)/P7-4(其余非客户安全类压测);**P7-3 客户安全红队测试(读 raw 客户数据 + 攻击性测试手法)移入黑名单**,因其与"读客户原始数据"及"攻击性操作"两条黑线冲突,必须醒后有人在场跑,不得夜跑无人值守执行。
- **夜跑黑名单(醒后人裁)**:C/D 档一律不做(2026-07-07 六哥裁·C档在夜跑场景不适用扩权令,回落四档默认须确认);对外发布/删除/T2T3 三硬红线;首次跑的新脚本对客户数据目录的写操作;任何"标准/铁律"类文件升级(即便按上条白名单限定起草,起草物本身在审议完成前不得升级为生效标准);**P7-3 客户安全红队测试**(2026-07-07 v2.2 新增,见上)。
- 每晨产物:夜跑报告一页(做了什么/验收证据/BLOCKED 清单/建议今日 3 撬动)——并入开机播报。

## §9 第3轮外部对标(2026-07-07 · fact_layer: inferred · 轻量版·基于已有知识,未联网检索)

> 对照对象:业界公开的"长跑 agent / 多阶段任务恢复"常见范式(如 AutoGPT/BabyAGI 一类任务分解-执行-复盘循环、CI/CD 里的 checkpoint-resume 模式、SRE 领域的 incident escalation 分级)。以下给出 steelman 式对照,只挑对本协议有实际启发的点,没价值的不硬凑。

1. **上下文预算:本协议(v2.2 §4 预飞估算+可观测信号)已优于常见做法,但可借鉴"预算硬顶+软预警"双层设计**。多数长跑 agent 实践(如 AutoGPT 类框架)只做"软预警"(接近上限时提示),真正硬顶留给平台层(如 API 的 max_tokens 报错)兜底,中间缺"主动预飞"这一层——这恰是本协议已经做到、且做得比常见实践更早的部分(预飞估算发生在 INIT 阶段,而非执行中才发现超限)。**一条可采纳的启发**:常见做法里"预警阈值"和"硬顶阈值"是两条独立线,本协议目前把"预飞估算"和"执行中信号"合并在一节,可在后续迭代(非本次)中考虑显式区分"软预警(建议切分)"与"硬顶(强制BLOCKED)"两条线,但当前信息量下不足以判断是否值得为此增加复杂度,故本轮不改,留待试运行反例回填。
2. **BLOCKED 升级路由:本协议(v2.2 §5 信息增量判据)已优于 SRE 常见的"次数计数器"范式,与 incident escalation 的"严重度分级"范式方向一致**。SRE 领域早已放弃"重试N次自动升级"的简单计数器(因误报率高),转向"错误类型+影响面"分级升级——本协议这次把"次数判据"改为"信息增量判据"+"可逆性首败即停",本质上是同一方向的改进,且已经针对本库场景(posclean 排坑类调试)做了具体化,**判断:现有设计已达到或优于常见做法,无需再采纳外部方案**。
3. **夜跑/无人值守白名单分级**:CI/CD 领域对"无人值守可执行动作"的通行做法是"环境隔离+权限最小化"(如生产环境部署强制人工 gate,预发布环境可自动跑),与本协议 v2.2 §6"C/D档回落四档默认""P7-3移黑名单"的思路一致(按操作的不可逆性/敏感度分级,而非按任务类型笼统开白名单)。**判断:现有设计方向正确,无需采纳外部方案**,唯一提醒:CI/CD 常见做法会给每类白名单动作配一条自动化回滚路径,本协议目前白名单动作(A/B档)本身就限定为"可逆",已隐含满足这一诉求,故不需额外机制。

**结论**:本轮外部对标未发现需要吸收的新增机制;§4/§5/§6 三处修订方向与业界常见范式一致或更严格,维持现状即可。
