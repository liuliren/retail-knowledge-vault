# 第一部分 · 方法论体系（01_科学零售方法论）

## 0. 一张图看懂结构

```
科学零售总纲（顶层 / quadrant I）
├── 三大业务支柱（"三力"，纵向）
│   ├── 商品力总纲 ── 卖什么/卖给谁/怎么卖
│   │   ├── 品类管理（结构层）
│   │   ├── 单品管理（SKU层）
│   │   ├── 商品配置理论（空间维度）
│   │   └── 52周MD（时间维度）
│   ├── 组织力总纲 ── 谁来做/怎么做/如何保障
│   └── 运营力总纲 ── 如何持续/如何改善
├── 经营控制链（横向穿透三力）
│   └── 金额管理（目标层）→ 数量管理（执行层）→ 品质管理（校验层）→ 经营复盘（反馈，待建）
└── 数据与系统能力底座（基础设施，明确"不是第四力"）
    └── 陈列资源命名治理（区域代号机制 + 命名规范，已升 v1.0/active）
```

> 正交关系是核心：**三力是纵向业务支柱，金额/数量/品质是横向穿透三力的控制链，数据底座是基础设施**——三者维度不同、不可混淆。多数方法论文件为 v0.1 draft / confidence medium；唯有两个命名治理文件因经花厅坊实战验证升到 v1.0-v1.1 / active / high。

## 1. 科学零售总纲（v0.3）

- **定义**：以数据和规律为基础，通过商品力、组织力、运营力三大支柱协同，实现可持续经营目标的零售管理体系。
- **三力核心问题**：商品力=卖什么/卖给谁/怎么卖；组织力=谁来做/怎么做/如何保障；运营力=如何持续/如何改善。
- **经营控制链**：金额管理(定目标)→数量管理(拆动作)→品质管理(控质量)→经营复盘(反馈修正)，闭环。
- **关键判断**：数据与系统能力底座 **不是第四力**，是支撑三力的基础设施，作用是让经验判断升级为"可观察、可分析、可执行、可复盘、可复制"。
- **v0.3 修订**：商品力下游四模块(品类/单品/商品配置/52周MD)由"层级"改为 **并列**，消除口径冲突。
- **独有误区**：把科学零售=数字化零售；把金额/数量/品质视为并列独立指标而非上下游。

## 2. 三力总纲

### 2.1 商品力总纲（v0.2）
- **定义**：门店持续满足目标顾客购买需求的商品配置与管理能力，本质是 **结构性能力** 而非单点商品好坏。
- **4 维度**：正确的商品(What) / 正确的价格(Price) / 正确的陈列(Where·How) / 正确的数量(How much)。
- **下游分工**：品类管理=品类维度结构；单品管理=SKU维度；商品配置理论=空间维度；52周MD=时间维度。
- **独有判据**：SKU 数量多 ≠ 商品力强(精准选品＞堆 SKU)；不能单凭销量判断单品(受活动/陈列/缺货/价格干扰)；陈列是商品力组成而非"美化"；商品力 ≠ 毛利率。

### 2.2 组织力总纲（v0.1）
- **定义**：持续把"正确的人、正确的事、正确的标准"落地到日常经营动作的能力，本质是 **执行落地能力**。
- **7 大构成**：岗位职责 / 流程标准 / 任务闭环 / 培训机制 / 复盘机制 / 激励机制 / 组织文化。
- **强弱判据**：强=店长能把规划落地到每个货架、复盘能识别根因；弱=有方案无落地、有会议无结论、依赖老板一人盯、标准因人而异。
- **独有强调**：系统是底座不是能力；社区超市组织力核心常是 **店长文化和团队习惯**。

### 2.3 运营力总纲（v0.1）
- **定义**：把商品策略和组织执行转化为持续经营结果的能力，核心是 **持续运转、动态调整、闭环改善**。
- **7 大构成**：陈列优化 / 营销活动 / 门店日常运营 / 异常管理 / 库存与缺货处理 / 周月复盘 / 经营改善闭环。
- **三力口诀**：商品力=卖什么；组织力=谁来做、怎么做；运营力=如何持续跑起来、如何改善。
- **与 52周MD 关系**：运营力是 52周MD 的 **下游执行层**(引用而非拥有)，把品类策略/季节波段转为年月周运营动作。

## 3. 经营控制链（金额→数量→品质）

三者均 **不归属任一单独支柱**，横向穿透三力。

### 3.1 金额管理（v0.1）— 目标层 / "经营结果的量化语言"
- **指标**：销售额(规模)、毛利额(成果)、毛利率(结构)、库存金额(占用)、损耗金额(质量)、费用金额(待确认归属)、坪效/人效(待确认)。
- **社区超市追踪优先序**：销售额→毛利额→毛利率→库存金额→损耗金额。坪效/人效作横向对比基准。
- **判据**：毛利额比毛利率更直接反映成果(毛利额=销售额×毛利率)；只追销售额不看毛利额=经营虚胖；目标须拆到品类/单品；金额管理 ≠ 对账。

### 3.2 数量管理（v0.1）— 执行层 / "经营动作的量化语言"
- **5 类指标**：商品侧(SKU数/动销SKU数/销售数量)、库存侧(库存数量/缺货SKU数)、陈列侧(陈列面数/层板数/端架数)、活动侧(堆头数/活动商品数)、执行侧(补货/巡检次数，待确认)。
- **边界**：商品/库存/陈列/活动数量归数量管理；人员任务/会议/培训数量归组织力。
- **判据**：排面是稀缺资源，须按贡献度和顾客决策路径分配，不能"摆满就行"。

### 3.3 品质管理（v0.1）— 校验层 / "结果真实性与可持续性校验机制"
- **8 维度**：商品/结构/价格/陈列/库存/数据/执行/顾客体验品质(后者待确认)。
- **异常信号("数字好看但不健康")**：销售涨但毛利额降；SKU 数增但动销 SKU 未同步；排面增但销量未改善；库存金额升但周转变慢；活动期销售增但活动后积压。
- **判据**：品质管理 ≠ 商品质量检查；先做数据品质验证再下业务结论；数据发现异常、现场解释异常。

## 4. 商品力四大下游模块

### 4.1 品类管理（v0.1）
- **定义**：以顾客购买任务和经营目标为基础，把品类作为 **战略业务单元** 规划。品类 ≠ 系统里的商品分类。
- **八环节闭环**：①品类定义(边界+购买任务+CDT)→②品类战略→③品类角色→④品类评估→⑤品类策略→⑥品类战术→⑦品类执行→⑧品类复盘。
- **品类角色 6 类**：目标性/常规性/季节便利性/形象/流量/利润品类。
- **社区超市第一阶段优先解决**：品类边界不清 / SKU结构失衡 / 价格带断层 / 陈列资源错配 / 复盘机制缺失。
- **实战验证(花厅坊5/6)**：M-DEC-001 被老板原话验证("太多条码了，需要细化和筛选")；"8环节直接给一线员工不可行"(员工多初中学历) → 触发顾问层+一线员工层双轨。

### 4.2 单品管理（v0.1）
- **定义**：以 SKU(以条码为唯一识别)为最小经营单元的持续评估/分类/决策/动作管理，本质是 **SKU级经营控制系统**；依赖 **经过清洗的商品档案**。
- **与品类管理**：**并列、不同颗粒度协同**(非上下级)。
- **ABCD 分类**：A 高贡献(保货源/保面/稳价)；B 中等(维持观察)；C 低贡献(收缩观察)；D 极低/零(淘汰候选)。(维度/权重/阈值/周期均待确认)
- **ABC-Z 模型**：在 ABCD 上补 Z 类异常(零动销/数据异常/已停售)。ABCD=当期评估，ABC-Z=全生命周期。
- **4 类动作**：引进/淘汰/替换/调整。
- **反幻觉硬规则**：滞销淘汰必须 **先排除活动、陈列、缺货、价格异常**；缺货期零销售 ≠ 无需求，判断须排除缺货区间。
- **候选商品池**：淘汰 SKU 替补来源，保证淘汰不造成品类空洞。
- **实战验证**：60天滞销 439 SKU、库存成本 46,696.6 元(与老板"具体没统计"对齐，证明数据先行价值)。

### 4.3 商品配置理论（v0.1）— 体系中最深方法论
- **定义**：在明确购买任务/品类战略/货架资源前提下，决定"卖哪些商品、各价格带放多少 SKU、哪些品种必配、哪些可精简"的系统方法，核心输出是 **商品结构图**(非 SKU 清单)。
- **来源**：日本渥美俊一《商品構成》(术语标"待核验")；分 **P域(大众品)与 M域(趋势品)**。
- **三大核心工具**：CDT 消费者决策树(顾客视角·怎么选) / HS矩阵 品种结构表(需求×属性·配齐没) / IPC 商品配置图(经营×价格·价格结构合理否)。
- **IPC + P/M 域参考值**：P域≈月薪1%–30%、SKU占比 60%–80%(现金流/走量)；M域≈月薪30%–360%、SKU占比 20%–40%(利润流/走价)。IPC="现金流心电图"；调整=抬高/左移 P域顶点；理想=右下倾斜结构。
- **HS矩阵**：Hard(规格/口味/包装/品牌/容量)×Soft(场景/任务/人群/价格敏感度)。**投入产出差异系数 = 毛利额占比 ÷ SKU数占比**(<1 高投入低产出查删减；≈1 匹配；>1 评估扩充)。**限制**：承担购买完整性/价格形象/连带购买的低毛利 SKU 不能仅凭系数删减，须结合 CDT 和品类角色。
- **"六分仪"**：每个"价格带×需求类型"交叉格观察 6 指标(SKU数占比/陈列面占比/销售额占比/毛利额占比/毛利率/CTM)。
- **四类配置**：标配/选配/季节配/活动配。
- **两种作业顺序(核心判据)**：新店规划 CDT→HS→IPC；老店诊断 IPC→HS→CDT。
- **经营数学接口(均待建独立定义)**：GMROI=毛利额÷平均库存金额=毛利率×周转率；TF函数、TL/TH、CTM、Si值、对数转换。v0.1 不写固定公式阈值。

### 4.4 52周MD（v0.1）
- **定义**：以 52 周为周期，把季节/节日/生活任务/商品波段/陈列资源/活动节奏/库存准备统一的 **商品经营计划**(时间维度，不是促销日历)。
- **四维度**：时间(以周为最小单元) / 商品波段(何时引进清退/配哪价格带) / 陈列资源(端架堆头主题时序) / 库存准备(提前备货避免波段滞销)。
- **归属**：52周MD=规划层；营销活动策划(07_)=执行方案层；DisplayMap(06_)=现场落地层。向下只传"时间节点+主题方向+品类资源优先级"。
- **实战验证(花厅坊5/6读图)**：端午、母亲节临近无主题陈列；9.90 元锚点过密缺三段结构 → 月度活动需"低引流-中走量-高形象"三段轮换。

## 5. 数据与系统能力底座（v0.1）

- **定位**：**不是第四力**(以"是否独立产生经营价值/有独立目标/可独立考核"三维论证均为否)，核心是 **支撑放大** 三力。
- **8 大构成**：数据采集/数据清洗/指标体系/分析模型/管理系统/AI工具(Claude Code)/报告输出/执行闭环。
- **判据**：数据清洗和字段标准化先于分析模型；只建报表不建执行闭环=没用；照搬大连锁系统会超社区超市承载力。

## 6. 陈列资源命名治理（数据底座落地标准 / v1.0-v1.1·active·已验证）

体系中唯一升到 active/high 的方法论。

- **公司级 11 项区域代号(lock)**：XS休食 / JS酒水 / TW调味 / LY粮油 / RH日化家清 / SX生鲜 / LD冷冻冷藏(待确认) / SY收银 / RK入口 / TD主通道 / TC特陈区。
- **alias 机制**：HS→XS、MLY→LY 标 deprecated_alias(历史可读、新文件禁用、不批量回改)。
- **四层命名体系**：公司通用标准层 / 门店适配层 / 执行采集层(快采+入库升级) / 数据应用层。
- **完整编码公式**：`{客户/门店}-{区域代号}-{资源类型}{编号}-{面位}-{子组}-{层位}-{排面位}-L{品类}`，示例 `LYG-HTF-XS-G3-B-04-L03-P02-L方便速食`。
- **成熟度 L0-L5**：未命名→快采→标准命名→映射完成→数据可分析→系统化；整店等级=各区最低值。
- **反幻觉边界**：G1-G4/T01-T08 等是单店取值，**不进公司级固定值**；公司级标准只保留"方法"。
- **变更治理 7 步**：提原因→grep 扫影响→判公司/项目级→建 alias→只改主+活跃文件→历史保留→建审计+commit。禁止全 vault `sed -i`、禁止重写 git history、禁止删历史文件。

## 7. 贯穿全体系的"科学零售"独有思想

1. 三力(纵向) + 经营控制链(横向) + 数据底座(基础设施)正交，反复强调"数据≠第四力"。
2. 量化语言三层递进：金额=结果，数量=动作，品质=真实性/可持续性校验。
3. 反"虚胖经营"：必须区分自然动销 vs 促销拉动、区分活动前中后、先排除缺货/陈列/价格/活动干扰再下结论。
4. 结构性思维：商品力=结构性能力；商品配置=结构图非清单；SKU 多 ≠ 商品力强。
5. 本土化纪律：强制区分大连锁与社区超市，参考值一律标"待样板验证"。
6. 强反幻觉/工程化：术语来源、数学公式均标"待核验"，v0.1 不写固定阈值。
7. 实战驱动反向回写：每文件有"实战登记"段，客户原话一字不改，≥1 次验证升 v0.X、≥2 次跨战役升 v1.0。

## ⑥ 工程操作纪律 6 大类（§17.11-16 / 手册 §3-8）

1. **目录与路径**：绝不 `cd /tmp` 切走 cwd；git 用 `git -C <repo>`；可疑命令前 `pwd`。
2. **Git**：破坏性命令须用户确认；commit 首行 ≤72 字符；不 `--no-verify`；不 `--amend` 已 push。
3. **文件**：Read 先于 Edit/Write；优先 Edit；绝不擅自创建 .md/README；不删重要文件(mv 99_归档)。
4. **数据**(与 §13.20 协同)：每个数字带 evidence；不一致 ≥3 解释+数学验证。
5. **进程**：长任务(>30秒)用 background；失败 ≥3 次必停排查根因。
6. **工具**：优先 dedicated tools(Read 非 cat/Edit 非 sed)；不 `find /` 用 Glob/Grep。

- **L1 操作契约** `pre-flight_check_checklist_v1.0`：8 大场景"不通过不执行"。
