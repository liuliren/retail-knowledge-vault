## 6. RetailOS：门店决策操作系统（封装层 · 最新主力迭代）

**RetailOS = 把 L2 骨架 + L3 武器封装成"店员可照做、可打印、不依赖理论"的门店决策 SOP**。它纵向贯穿 L2-L4，是体系的"可执行/可售卖"出口，**不是新层，而是把现有层压成动作句的操作系统外壳**。

> ⚠ 重要区分：RetailOS 是**真实门店业务系统**，与 98_AI协作中枢（Obsidian+Claude+Codex 三脑管道）是两回事，勿混淆。

**架构（A-G 六层 + 减法/加法双向）**：
```
v1.0 减法层（汰换）：A库存→B SKU结构→C价格带→D陈列→E ABC裁决→F误判→G门店SOP
v1.1 三项补深：     E层毛利交叉ABC / A层库龄+售罄+水位 / B层商品角色
v2.0 加法层（选品/引进）：找缺口→门槛4闸→合理SKU数→放哪定价→观察期
合成 = "汰旧上新"攻防完整系统
```

**v2.0 选品层 4 闸（本会话头号成果）**：
1. **真需求** — 该缺口是不是真顾客需求；
2. **有位置** — 上新必腾位（置换式，一对一挂清退SKU）；
3. **算得过来** — PSD（单品日均销）+ 毛利双信号；
4. **供得上** — 用"可得性代理法"（在购状态+近90天进货活跃+采购规模+是否本地）判断，**不依赖采购周期**（客户管理水平不高，过程字段不可靠）。

**三轮验证链（已跑通）**：
- 单小类闭环：花厅坊 T03 饼干（M-DEC-009~012）；
- 全门槛回测：G4-A 麦片货架（74 SKU，独立重现专家"压18补7置换式"决策）；
- **跨店泛化**：沙埔大道店休食（1218 SKU，重现第二店专家选品决策→证明非花厅坊过拟合）；
- 供得上闸数据闭合：供应商能力库 v0.2（63 供应商 / 100% 覆盖 / 采购规模分级）。

**商业交付**：`10_咨询交付模板/02_交付SOP/Retail_OS_v1.0 + v2.0_商业交付版`（脱个案、可售卖）。

**关键结论**：RetailOS 选品层是体系当前**最成熟、最有现金价值、验证最扎实**的资产——4 闸全数据驱动，跨小类+跨店双重稳健，已有商业交付版。
