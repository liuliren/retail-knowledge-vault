### 1.2 四大病灶（按严重度·均已校验）

**🔴 P0-2 链接断裂（可寻性根本）**
- **236 个断链目标**（被 ≥2 文件引用却不存在），其中 **40 个被 10+ 文件引用**。
- 重灾：`[[M-DEC-007]]`(77 引)、`[[feedback_*]]` 系列(14 类笔记池根本没建)、5 月旧版工作文档(42 条过期链)、目录级链接无 README(54+)。

**🔴 P0-3 数据安全漏条码（客户合规·硬红线）**
- **15 个 git 跟踪的 md 含真实 EAN-13 条码**（69 开头中国码，已抽样确认非误报）。违反 §6 "条码不入 git" + 工作边界"0 条码"。
- 注：原始 xls/csv 明细本身未入 git（.gitignore 达标）；漏的是**正文里嵌的条码**（效果报告/字段规范示例/数据资产表）。

**🟡 P1 版本与重复治理**
- 文件名版本 ≠ frontmatter 版本（M-DEC-008/011 等 FM 已 v0.2 但文件名仍 v0.1 → 改名即批量断链）。
- 外部源版本号（零售老刘 ClusterA_v5.0）混进文件名，与内部 v0.1 打架。
- 竞争定义：科学零售总纲 ×3、RetailOS ×5-7（咨询版/案例版/SaaS规划版混目录）、SOP（通用 01-03 vs 花厅坊 M-DEC-005~008）、陈列定义散 3 处。
- 97 孤儿文件（18%），56 在 98_（半正常），但 09_/16_ 正式件有 22 个该有入链却孤立。

### 2.1 世界一流咨询公司标准（McKinsey / BCG / Bain）
| 标准 | 含义 | 落到本库 |
|---|---|---|
| **金字塔原理**（Minto） | 结论先行→论据 MECE→数据支撑 | 每份交付件首段必"结论先行"；已是写作铁律，升级为硬验收 |

| **80/20** | 抓撬动性少数 | 少而深；每日 3-5 撬动任务 |
| **质量门 / 复核** | 交付前独立 review | 引入"交付前硬化检查"门（自解释/可组合/去黑盒）|
| **一页纸高管摘要** | 老板 30 秒读懂 | E 层一页简报（0 术语 0 条码）|

## 第三部分 · 卡帕西式闭环自动化知识系统

### 3.1 知识生命周期闭环（7 环·工程化成回路，不靠人自律）
```
① INGEST 进料 → ② COMPILE 编译 → ③ LINK 互联 → ④ LINT 体检
      ↑                                                    ↓
⑦ FEEDBACK 回流 ← ⑥ PUBLISH 发布(签字门) ← ⑤ DISTILL 蒸馏
```
| 环 | 动作 | 自动化抓手 | 已有/待建 |
|---|---|---|---|
| ① Ingest | 原始源进库→一句话摘要+标签→**冻结** | G01_Ingest | 有骨架 |
| ② Compile | LLM 把原料编译进方法论页 | Claude/Codex | 有 |
| ③ Link | 自动补 wikilink + 回填 MOC | lint 脚本 | **待建** |
| ④ Lint | 孤儿/断链/状态/schema/条码 五查 | **G03_Lint v2** | **待建·本蓝图核心** |
| ⑤ Distill | 多案例稳定→蒸馏 M-DEC | M-DEC 回路 | 有(今天实跑) |
| ⑥ Publish | 升 active=**签字门** | 状态机+签字 | 机制需修 |
| ⑦ Feedback | 实跑校准回写规则/注册表 | SOP-03 R1-R4 | 有(本会话建) |

### 3.2 G03_Lint v2 —— 自动体检（闭环的红绿灯）
一条命令扫全库，输出健康度仪表盘 + 阻断清单：
1. **断链查**：所有 `[[x]]` 比对实际文件 → 报断链。
2. **孤儿查**：正式目录(01/04/09/10/16)无入链文件 → 报孤儿。
3. **状态查**：`active` 无 `signoff` → 报违规。
4. **Schema 查**：缺必填 frontmatter 字段 → 报。
5. **条码/敏感查**：正文含 EAN-13/进价裸值 → 报红线。
6. **版本查**：文件名版本 ≠ frontmatter 版本 → 报。
> 产出 `00_入口与总索引/05_审计与档案/lint_仪表盘_最新.md`（孤儿率/断链数/签字率/红线数），月度跑、趋势可见。可派 Codex 写脚本（纯文本扫描，不碰客户数据）。

### 3.3 状态机（写死流转，每态有门）
```
draft ──(内容完整+自检过)──> candidate ──(≥3案例/2客户验证)──> active ──(过时)──> deprecated
                                              ↑签字门(六哥)            ↓移 99_归档(签字)
```

## 第五部分 · Obsidian 看板 + Wiki 知识树（答你最新问题）

> **能做到，且是 Obsidian 原生强项。** 三层方案，从轻到重：

### 5.1 知识树 / Wiki 全景（核心诉求）—— MOC + Dataview + 关系图
- **① 知识树枢纽（MOC）**：建一个 `科学零售知识树_MOC.md`，按 **L1理念→L2骨架→L3方法武器(M1-M8)→L4工具/SOP→案例** 层级，用 wikilink 把所有概念/定义/方法论挂成树。这就是你说的"看到整个科学零售所有概念的知识树"。**本蓝图随附已建一版起步稿**（见下）。
- **② Dataview 自动列**：装 Dataview 插件，知识树里用查询自动列出"所有 active 方法论""所有 M-DEC""某支柱下全部件"，**新增文件自动进树**（治孤儿）。示例：
  ````
  ```dataview
  TABLE status, version FROM "01_科学零售方法论" WHERE source_type="method" SORT file.name
  ```
  ````
- **③ 关系图谱（原生 Graph）**：Obsidian 自带 Graph view + 局部图(local graph)，按 tag/文件夹着色，**就是活的 wiki 知识网**。链接修好后，图谱即知识树的可视化。

### 5.2 看板（项目/状态管理）—— Kanban 插件 或 Dataview 看板
- **Kanban 插件**：装社区插件 `Obsidian Kanban`，建 `科学零售_看板.md`，列 = 状态（draft/candidate/active/deprecated）或阶段（进行中/待签字/已交付）。卡片可双链到知识件。
- **Dataview 看板（推荐·自动）**：用 frontmatter 的 `status` 字段，Dataview 按状态分组列出所有件——**改文件 status，看板自动流动**，无需手搬卡片。与状态机天然咬合。

### 5.3 落地顺序
1. 先修断链（否则知识树/图谱全是断点）→ 2. 建知识树 MOC + 装 Dataview → 3. 建 Dataview 看板（绑 status）→ 4. Graph view 当全景图。
