# Retail OS v2.0｜选品/引进层 立项卡

> 立项理由（一句话）：**RetailOS v1.0 只会做减法（汰换），答不了门店最常问的"该上什么新品"。**
> 本模块补"加法"，是 v2.0 头号模块。

---

## 1. 要解决的问题
- v1.0 决策链 A-F 全部针对**已在货架上的 SKU** 做体检/裁决；
- 门店真实场景一半是"引进"：补品类缺口、上新、换季、对标竞品；
- 数据层已有「新商品引进表」，但**没有决策逻辑去驱动它**——表是空的方法论。

## 2. 模块范围（封装 Clippings 选品配置 75-225）
| 子能力 | Clippings 来源 | 产出动作句方向 |
|---|---|---|
| 缺口识别 | 宽类窄品 204-214 | "类要全、品要精"——先看缺哪个需求单元/价格带 |
| 合理 SKU 数 | 197-201 | 单品数约束：A品数×系数；不是越多越好 |
| 可得性 | 212-213 | 强单品必须常在、不断货 |
| 选品标准 | 75-78 PSD、144 硬折扣 | 新品进场门槛：效率/毛利/差异 |
| 供货模式 | 139-146 JIT/供应链 | 引进与供货节奏匹配 |

## 3. 与 v1.0 的对接
- 选品层 = v1.0 的**镜像加法层**：v1.0 答"砍谁"，选品层答"上谁"；
- 共用同一套 ABC / 价格带 / 商品角色语言（v1.1 已铺好双维ABC与角色）；
- 对接现有「新商品引进表」作为输出落地表。

## 4. 治理与验证路径
1. 需**扩 A-G**：新增"选品 Cluster"（编号待六哥分配）+ 对应 M-DEC 候选；
2. 走 candidate → T03/同类单案例 → T01 跨案例 → active；
3. 与 v1.1 一致：保持"店员/采购可照做"的动作句定位，建模部分留方法论层。

## 5. 里程碑（建议）
| 阶段 | 产出 |
|---|---|
| M1 立项（本卡） | 范围/对接/治理确认 |
| M2 规则草案 | 选品 5 步 + 引进门槛 + 缺口识别卡（动作句） |
| M3 单案例落地 | 花厅坊一个小类试跑（如饼干/薯片缺口补品） |
| M4 跨案例 | T01 验证后升通用，并入商业交付版 |
