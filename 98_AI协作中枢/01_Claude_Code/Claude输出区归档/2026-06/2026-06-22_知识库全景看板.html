🧭 体系骨架

三力(纵向业务支柱) + 经营控制链(横向穿透) + 数据底座(基础设施)，三者正交、维度不同、不可混淆。
纵向 · 三力

商品力 / 组织力 / 运营力

  • 商品力：卖什么/卖给谁/怎么卖
  • 组织力：谁来做/怎么做/如何保障
  • 运营力：如何持续/如何改善
横向 · 经营控制链

金额 → 数量 → 品质

  • 金额=结果量化语言(目标层)
  • 数量=动作量化语言(执行层)
  • 品质=真实性/可持续校验(校验层)
  • 经营复盘=反馈(待建)
基础设施

数据与系统能力底座

  • 明确"不是第四力"
  • 支撑放大三力，非独立驱动
  • 含命名治理(唯一已升 v1.0)
⚠️ 全体系反复强调："数据≠第四力"；判断商品力不能单凭销量；参考值一律标"待样板验证"。

商品配置理论(渥美俊一《商品構成》, 术语待核验)是体系最深方法论：HS 差异系数 = 毛利额占比÷SKU数占比(<1 查删减/>1 评估扩充)，但承担购买完整性/价格形象的低毛利 SKU 不能仅凭系数删减。经营数学工具(GMROI/TF/TL-TH/Si)均待建独立定义。

任何元素必须归一象限，frontmatter 强制 quadrant。双向防火墙：防 SOP 杀创造性 + 防创造性瓦解标准。

不变 + 可复制

不变 + 权变

老板访谈话术/现场危机判断/表达色调/"这次客户特殊"通道 — 权变后必复盘

变化 + 权变

🧭 三回归(§20 · 反假忙碌)

回归到人·回归到价值·回归到结果

回归到人

  • 本期几个真人因我们的工作变好？

回归到价值

  • 客户能"自己说出来"的价值是什么？
  • 说不出来=还没产生价值

回归到结果

  • 离回款/样板可讲/新客户各近几步？
三个永远拷问：①跟交付/签约/保留有何直接关系？②哪个真人因此变好？③属哪个象限？— 答得清,干；答不清,停。 实测 5/18 W20-Day7 判定"已触红线"(14 commit+393 照片但离目标 0 步)→强制 5/19 切回主线。

⚙️ 工程操作纪律 6 大类 + 操作三公民

类1 目录路径
  • 绝不 cd /tmp 切走 cwd
  • git 用 git -C
类2 Git
  • 破坏性命令须确认
  • 不 --no-verify / --amend 已 push
类3 文件
  • Read 先于 Edit/Write
  • 不擅自创建 .md/README
类4 数据
  • 每数字带 evidence
  • 不一致≥3 解释+数学验证
类5 进程
  • 长任务 background
  • 失败≥3 次必停查根因
类6 工具
  • 优先 dedicated(Read 非 cat)
  • 不 find / 用 Glob/Grep
三个一等公民(§18)→ Ingest(G01 进库) · Query(G02 调用) · Lint(G03 每周巡检)
第 7 支柱 §17.21：一个 prompt 默认只完成一个主任务闭环；主任务完成必须先回报不得自动启动下一个；跨象限任务必须单独 prompt+单独 commit+单独审计。
