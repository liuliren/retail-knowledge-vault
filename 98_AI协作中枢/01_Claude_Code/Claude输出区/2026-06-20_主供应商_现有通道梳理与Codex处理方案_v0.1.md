# 主供应商 现有通道梳理与 Codex 处理方案

> 回答两问：① 现有通道是什么样？ ② Codex 那边怎么处理？
> 核心结论：**通道已成型，清洗脚本已产出带"主供应商"空列的合并表——Codex 扩脚本填列即可，不是新建管道。**

---

## 一、现有是"两条通道"叠加

| 通道 | 作用 | 关键节点 |
|------|------|---------|
| **A. 业务数据管道** | 真实数据进→清洗→出 | 启明拉数 → 原始素材 → 清洗脚本 → 清洗输出 → 诊断/选品 |
| **B. AI协作通道** | 给 Codex 派活、收结果 | Codex收件箱 → 任务分发器 → Codex输出区 → 执行日志 |

> A 是"数据怎么流"，B 是"活怎么派给 Codex"。主供应商两条通道都要用：B 派活、A 跑数据。

---

## 二、业务数据管道全景（A 通道）

```
① 启明拉数(现场/ERP导出)
   └─ ERP可导出供应商+进价字段(六哥5/10 confirm) ★主供应商在这步进来
        ▼
② 99_原始素材/01_门店数据材料/*.xls  (原始, 全程只读)
        ▼
③ _excel_converter (xls→xlsx)
        ▼
④ 花厅坊POS清洗脚本_v0.1.2 (Codex工具)
   loaders → normalizers → mappers → anomaly_rules → writer
   纪律: 原始只读+SHA256校验 / dry-run默认 / execute需授权 / 异常率守门(<5%过,>15%阻断)
        ▼
⑤ 03_商品诊断/01_清洗输出/
   销售统计汇总_合并版.xlsx ★含"主供应商"列(schema在/数据空) ← 待填
   商品档案_合并版 / 库存即时快照 / 异常清单 / 资料索引表 / 清洗日志
        ▼
⑥ 03_商品诊断 → RetailOS分析 / 选品层"供得上"闸
```

> **主供应商的补入点 = ①拉数带出 + ④清洗脚本join填列 + ⑤合并表那个空列。** 管道现成，缺的是把供应商字段拉出来+脚本加一段。

---

## 三、Codex 那边怎么处理（B 通道派活 + A 通道执行）

Codex 的活 = **扩现有清洗脚本**，分 4 段，全部复用现有纪律（只读/SHA256/dry-run）：

| 段 | Codex 做什么 | 改哪个文件 |
|----|------------|-----------|
| **C1 读供应商列** | loaders 增加读取新拉数里的"供应商/进价"列 | `loaders.py` |
| **C2 主商判定** | 一品多商按 采购额>频次>最近窗口 定主商；其余入"全部供应商" | `mappers.py` |
| **C3 写回合并表** | writer 把主供应商 join 回销售统计汇总的空列（条码主键） | `writer.py` |
| **C4 聚合能力库** | 新增 supplier_capability：按供应商聚合 品类×价格带×本地 → 输出表B | 新增 `supplier_agg.py` |

**执行纪律（沿用脚本现有边界）**：
1. 原始拉数文件进 `99_原始素材/`，全程只读 + SHA256；
2. **先 dry-run**（写 `_dryrun_preview/`，出主供应商填充率/一品多商占比/异常）→ 六哥/David 审阅；
3. 审阅通过**才 execute**，写 `01_清洗输出/`；
4. 异常率守门：>15% 阻断；
5. 采购额绝对值**不入 md**，能力库只留分级（高/中/低）。

## 五、"其它品类逐步推进"在通道里怎么走
- 清洗脚本 v0.2 起**已通用化**（README §2），不绑死饼干/麦片；
- **滚动推进**：启明每拉一个品类的供应商数据 → Codex 同一套脚本跑该品类 → 能力库增量补；
- 选品层引进方向扩到哪个品类，就让启明拉哪个品类——**数据跟着选品需求走，不一次性全店拉**；
- 生鲜类进价系统缺口单列（不阻塞非生鲜主线）。

---

## 六、Claude / Codex / 现场 边界
- **现场(启明)**：ERP 拉数（供应商字段）+ 可得性三项（MOQ/配送/稳定性）；
- **Codex**：扩脚本 C1-C4 + dry-run/execute + 月更；
- **Claude**：字段口径、主商判定规则、与选品层闸对接逻辑；
- **六哥/David**：口径 confirm + dry-run 审阅授权。
