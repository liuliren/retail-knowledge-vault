# 主供应商数据建设方案 v0.1

> 目标：补齐选品层「供得上」闸的主供应商数据残口。
> **核心判断**：供应商字段 **ERP 系统可导出**（六哥 5/10 confirm），不是"没有"，是"没并进分析底座"。
> → 这是一个 **导出 + 并表 + 前瞻能力库** 工程，**不是凭空建**。

---

## 一、先厘清：选品层要的是"两类"供应商数据

很多人把"建供应商数据"理解成"给每个商品填个供应商名"。但选品层「供得上」闸真正要回答的是
**"我想引进的方向，有没有稳定货源"**——这需要两类数据，分清才不会建错：

| 类型 | 是什么 | 用途 | 粒度 |
|------|--------|------|------|
| **① SKU 级主供应商**（存量·地基） | 现有每个商品由谁供货 | 对账 / 补货 / 汰换 | 条码→主供应商 |
| **② 品类级供应商能力库**（前瞻·闸真正用的） | 每个供应商**能供哪些品类/价格带/能否本地/起订量** | 选品「供得上」闸：引进方向→查货源 | 供应商→品类×能力 |

> ①是地基（从 ERP 导），②是选品层真正查的表（从①聚合 + 现场补）。**只建①不够，必须升到②。**

---

## 二、数据结构设计（两张表）

### 表 A｜SKU 主供应商表（存量·条码主键）
| 字段 | 说明 | 来源 |
|------|------|------|
| 商品条码 | 主键（对接现有底座） | ERP |
| 主供应商 | 见§三判定规则 | ERP导出 |
| 全部供应商 | 一品多商时全列 | ERP |
| 供货方式 | 集团统配DC / 供应商直送DSD / 本地采购 | ERP+现场 |
| 进价 | 已确认可导出 | ERP |
| 最近采购日 | 判定主商+活跃度 | ERP |
| 状态 | 在供 / 停供 / 待核验 | 清洗标记 |

### 表 B｜供应商能力库（前瞻·供应商主键）
| 字段 | 说明 |
|------|------|
| 供应商名 | 主键 |
| 可供品类(L3/L4) | 能供哪些小类（从表A聚合 + 现场补） |
| 覆盖价格带 | 低/中/高（决定能否补高端空档） |
| 是否本地 | 本地货源（决定本地口味/快速补货） |
| 起订量MOQ | 最小起订（决定能否小批试销） |
| 配送周期 | 下单到货天数（决定可得性） |
| 供货稳定性 | 高/中/低（小众品风险标记） |
| 合作状态 | 现合作 / 潜在 / 黑名单 |

---

## 三、主供应商判定规则（一品多商时）
> 一个商品常有多个供货来源，须定"主供应商"，否则闸无法判。建议优先级：
1. **采购额优先**：近 90 天采购金额最高者为主商；
2. **采购频次次之**：金额相近时取下单频次高者；
3. **最近窗口兜底**：仍相近时取最近一次采购来源；
4. **标记多商**：主商之外的全列入"全部供应商"，供得上闸视为"有备选=可得性更高"。

---

## 四、分阶段建设路径

| 阶段 | 动作 | 负责 | 产出 |
|------|------|------|------|
| **P0 定义** | 字段口径 + 主商判定规则确认 | Claude 拟 + 六哥 confirm | 表A/B 结构定稿 |
| **P1 存量导出并表** | ERP 导出供应商字段 → 条码 join 进销售统计汇总（填那个空列） | 启明拉数 + Codex 清洗join | 表A 填实 |
| **P2 聚合能力库** | 表A 按供应商聚合 → 品类×价格带×本地 | Codex 聚合 | 表B 初版 |
| **P3 可得性补全** | MOQ / 配送周期 / 稳定性 现场补 | 店长/采购 + 启明 | 表B 完整 |
| **P4 对接闸+维护** | 供得上闸读表B；定更新频率 | Claude 对接 + Codex 月更 | 闸闭环 |

> P1 是**关键解锁步**——系统已确认可导出，只差启明拉数 + Codex 并表。P1 完成后「供得上」闸即可半闭环（凭存量供应商判断）；P2-P3 完成后全闭环（能判前瞻引进方向）。

---

## 五、分工（三脑 + 现场）
- **启明（现场拉数）**：ERP 导出供应商/进价字段（5/12 已有拉数动作，复用通道）；
- **Codex（工具脑）**：清洗 + 条码 join + 一品多商主商判定 + 聚合能力库（复用花厅坊POS清洗脚本）；
- **店长/采购（现场）**：可得性三项（MOQ/配送/稳定性）+ 潜在供应商；
- **Claude（方法脑）**：字段口径、判定规则、与选品层闸对接逻辑；
- **六哥**：口径 confirm + 主商规则拍板。

---

## 六、按需驱动（不必全店一次建全）
> 全店全建成本高、且选品层只在"引进"时查。建议**按选品需求驱动、分批建**：
1. **先建当前引进方向覆盖的品类**：麦片冲调（G4-A）+ 休食主线（饼干/膨化/坚果等）；
2. 生鲜类进价缺口单列（已知仅生鲜缺，不阻塞非生鲜主线）；
3. 引进方向扩到哪个品类，能力库就补到哪个品类——**滚动建设，避免一次性大工程**。

---

## 七、与选品层「供得上」闸对接
```
引进方向(如"高端进口燕麦/中高价")
   │ 查 表B 供应商能力库
   ▼
该品类×该价格带 有可用供应商?
   ├─ 有(现合作/本地/MOQ适配) → ✅过闸
   ├─ 仅潜在/小众/高MOQ → ⚠风险标记(像Z5白咖啡/养生品)
   └─ 无货源 → ❌该方向暂缓
```
> 这样「供得上」闸从"拍脑袋"变成"查能力库"，与选品层 v1.0 的 PSD/毛利双信号一起，4 闸全部数据驱动。

## 治理
- 供应商/采购额属敏感商业数据：只入结构化能力库，**采购额绝对值不入 md**，能力库只留分级（高/中/低）；
- 一品多商主商判定留 audit（保留全部供应商）；
- 生鲜进价缺口单独标记，不阻塞非生鲜主线。
