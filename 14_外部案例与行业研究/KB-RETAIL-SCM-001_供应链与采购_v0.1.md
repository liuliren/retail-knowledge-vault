# 供应链与采购

> 零售供应链管理的核心不是算法精度，而是数据纪律：SKU 主数据的质量决定全链执行质量，净需求在每个流程节点逐层丧失，而历史销量中隐藏的"缺货扭曲"是最常见的订货陷阱。

---

## 六哥当前可用场景

> 场景：花厅坊供应链诊断——采购节点如何管控？订货失准根因在哪？供应商评估用什么结构？

**具体用法（3条可执行动作）**：

1. **订货失准诊断**：当门店出现"畅销品长期缺货 + 滞销品积压并存"时，优先检查订货数据来源——历史销数是否包含大量缺货期，若是，则订货量被系统性低估，需先还原真实需求（满陈列期间销量），再加安全库存。
2. **库存结构体检**：不要只看整体周转天数，拆解三层：①整体结构（商品存货占总存货比，目标 20–30%）→②商品结构（A 类品缺货同时 C 类积压是典型错配）→③成本结构（高毛利品是否获得足够库存资金分配）。
3. **前置条件优先于建模**：在推进订货系统化前，先核查三项前置数据是否就位——供应商商品主数据准确性、每个 SKU 的订货周期设置、满陈列量标准（这是"内仓为零"库存管理的基础）；三项不齐，任何算法都会失效。

---

## 核心命题（来自零售老刘实践体系）

### 1. 真实需求还原原则：缺货期销量不可作为订货基础

历史销量隐藏着"缺货扭曲"——商品缺货 9 天，实际销量仅 1 个，传统订货系统会因低销量继续削减订货，形成恶性循环。正确订货公式是：**订货量 = 真实需求 + 安全库存**，而非历史销量 + 补充。还原真实需求的方法：取满陈列期间的实际销量，或通过同类商品/历史同期预测。在社区生鲜中，这一原则对高频生鲜（鲜蔬、散称）尤为关键，缺货一天即损失一天需求数据。

### 2. 净需求漏斗：每一个流程节点都在丧失需求

MRP 逻辑揭示的核心问题：顾客需求 20 → 报货仅 15（丧失 5）→ 采购订单 10（再丧失 5）→ 实收 5（再丧失 5）→ 入库 0。净需求在报货、订单、收货、入库四个节点逐层消耗，最终表现为慢性缺货，而非单一环节的失误。治理要点：①单据间时间与数量需互相校验；②每一层级须有责任方追踪净需求缺口；③系统须能自动回写缺货预警，避免逐层隐藏。

### 3. 库存评估的三维结构：单指标管理必然失策

周转天数是单指标，看似合理可能掩盖结构性错配。正确的库存管理需同时评估三个维度：①**整体结构**（商品存货 / 总存货，健康区间 20–30%，低于 10% 或超 50% 均有风险）；②**商品结构**（不同品类、小类的库存配置是否与销售贡献匹配，A 类缺货同时 C 类积压是典型错配信号）；③**成本结构**（不同毛利率商品的资金分配，需按毛利目标分解，而非按件数均摊）。三个维度须同时监控，单独优化任意一个都可能恶化另外两个。

### 4. 前置条件优先于算法：生鲜订货的真正卡点

许多企业建了复杂的订货模型，最终回归人工拍脑袋——根本原因不在算法，而在前置信息缺失：供应商主数据不准、订货周期未按供应商错开设置、满陈列量无标准化。没有这三项，再精密的算法也输入垃圾数据。务实路径：先投入数据治理，把前置信息做准；再用相对简单的规则模型，让一线人员可执行；最后才是系统化建模。

### 5. 参数动态性困境：精细模型与基层可操作性的张力

理论上，安全库存应随需求波动（标准差）、供应提前期、服务水平系数动态调整；预算系数应按季节、增长率变化实时调参。但现实中大多企业仍用固定天数安全库存、固定订货周期，因为动态调参需要：历史数据积累、正态分布验证、参数维护人力。务实的中间方案是**分层管理**：A 类关键品做精细动态参数（值得投入），B/C 类长尾品用简化固定参数（降低维护成本），而非对所有商品强求同等精度。

### 6. SKU 主数据纪律即供应链纪律

SKU 从导入到退出经历完整生命周期（导入期→成长期→成熟期→衰退期），每个状态变化触发采购、仓储、配送、门店、财务的连锁单据流转。SKU 新增涉及 7 类单据，SKU 退出涉及 6 类单据，任何一项数据错误（条码、属性、状态标签）都会导致全链执行失准。**结论：零售供应链管理的质量，本质上是对 SKU 主数据纪律的执行力。** 系统中 SKU 生命状态标签的准确性、新旧 SKU 并行期管理、财务核算自动生成，都依赖主数据的精准维护。

## 关键框架与模型

### MRP/DRP 净需求算法

```
净需求 = 真实需求 + 安全库存 - 期初库存 - 在途收货预期
```

关键：真实需求须先还原（排除缺货期），不可直接使用历史销量。

### 安全库存三因素量化

```
安全库存 = Z × σ_d × √(LT)
```

- `Z`：服务水平系数（95% 服务水平 → Z = 1.65）
- `σ_d`：日需求标准差（需历史数据积累）
- `LT`：供应提前期（天）

注意：生鲜品需额外校验是否符合正态分布，非正态分布需用泊松分布处理。

### 订货净需求漏斗模型

```
顾客需求
    ↓ [报货层：需求丧失]
报货数量
    ↓ [订单层：需求丧失]
采购订单
    ↓ [收货层：需求丧失]
实收数量
    ↓ [入库层：需求丧失]
入库数量（≤ 顾客原始需求）
```

每层均有潜在缺口，须逐层设置校验与责任追踪。

### 库存三维评估结构

| 维度 | 评估指标 | 健康区间 | 典型风险信号 |
|---|---|---|---|
| 整体结构 | 商品存货 / 总存货 | 20–30% | <10% 或 >50% |
| 商品结构 | 各品类库存 vs 销售贡献比 | 与销售结构匹配 | A 类缺货 + C 类积压并存 |
| 成本结构 | 高毛利品资金占比 | 按毛利目标分解 | 低毛利品占用过多资金 |

### 前置数据三项必检（生鲜订货前置条件）

1. 供应商商品主数据准确性（条码、规格、起订量、交货期）
2. 每个 SKU 的订货周期设置（不同供应商须错开，避免集中收货）
3. 满陈列量标准化（"内仓为零"库存管理的基础单位）

## 实践要点（零售老刘反复强调的操作细节）

- **缺货期须标注**：系统应自动记录每个 SKU 的缺货天数，订货参数计算时排除缺货期数据，否则历史销量天然被低估。
- **单据间必须互相校验**：报货单、采购订单、收货单、入库单的数量与时间需四联比对，任何缺口自动触发预警，不能依赖人工发现。
- **A 类品优先精细化**：有限的数据治理与人力资源，优先投入销售贡献前 20% 的 A 类商品，长尾 C 类用简化规则即可。
- **订货周期错开设置**：同一供应商的多个 SKU 订货日期不宜完全重叠，避免单日收货量峰值造成仓储压力和验收失准。
- **满陈列量是订货起点**：每个 SKU 须有明确的满陈列量标准（货架容量），订货补货以此为基准，而非凭感觉估算。
- **SKU 生命状态实时同步**：商品进入衰退期须立即在系统中标注"停购"状态，防止采购自动补货继续下单滞销品。

## 矛盾与张力（供批判性使用）

| 张力 | 说明 |
|---|---|
| 伪需求陷阱 | 历史销量因缺货被系统性低估，基于历史追加式订货会持续恶化缺货 |
| 单指标陷阱 | 周转天数合理 ≠ 库存结构合理，三维结构可能同时劣化 |
| 精细化落地困境 | 动态安全库存模型科学但前置成本高，大多数企业实际回归固定参数 |
| 漏斗性耗损 | 净需求不是一环丧失，而是四个流程节点逐层消耗，归因困难 |
| 算法 vs 数据纪律 | 生鲜订货的痛点从不在算法复杂度，而在前置主数据是否干净 |

## 待深入研究

1. **满陈列量的动态调整**：季节性品类（如夏季西瓜、冬季火锅食材）满陈列量如何随需求波动动态调整，零售老刘原文缺乏具体案例，需补充实践验证。
2. **生鲜品需求分布的非正态处理**：安全库存三因素假设正态分布，但生鲜高峰日（周末、节假日）需求分布高度偏态，泊松分布处理方式需要更多量化案例。
3. **净需求漏斗的量化追踪工具**：漏斗模型的概念清晰，但各节点缺口的自动化追踪在没有完整 ERP 的中小超市中如何低成本实现，原文未给出具体方案。
4. **供应商评估标准**：本页侧重采购流程与库存计算，供应商准入、评级、淘汰的量化标准在零售老刘体系中尚未形成专项命题，需补充。
5. **DRP 区域配送场景**：MRP/DRP 逻辑在总部→门店的配送层如何落地（区别于单店采购），多门店连锁场景下的净需求汇总机制值得单独研究。
