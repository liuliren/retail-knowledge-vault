# 品类表与 POS 清洗程序 · 最终推荐方案 · draft

> **本文全部为 Claude 工程建议，建议不等于六哥签字**；引用的三版品类表与三版程序均为 mock 实验件；**V6.0 xlsx 仍是唯一品类主数据 SSOT**。

## 1. 推荐组合

- **品类表：v3（22 列，诊断参数化版）**——含 v2 全部映射知识字段 + 诊断开关；前提是立治理边界：`business_role` 等为品类默认建议值，不是分类维度，不覆盖 SKU 级 is_traffic/is_margin 权威（2026-06-29 L4 裁决）。若六哥裁"品类表须纯分类"→ 回退 v2 + 诊断参数另表 JOIN。
- **程序：category_pos_cleaner v0.3**——评分/置信三档/多轮双重循环（先严后宽、外层可加 3–5 次宽松循环、零新增自动收敛，响应六哥 2026-07-08 补充要求）/批处理/质量报告。sample 实跑：96 行 mock → 映射 96.9%、未匹配 3、复核 25、已知陷阱 0 误入 high/mid；单测 8/8 过。

## 2. 两者如何配合

```
V6.0 xlsx (SSOT·六哥编辑权威)
   └─(授权派生脚本)→ category_table_v3.csv (机器投影·带版本号)
                          ↓
POS原始导出 →[POS清洗库 probe/sales/archive]→ sku_registry.csv
                          ↓
              [cleaner v0.3 多轮循环映射]
                          ↓
   mapping(high/mid 直用) + review(六哥/店长人审) + unmatched(品类表反哺)
                          ↓
              /trio · /diagnose 按 v3 诊断开关自动配诊断项
```

闭环点：质量报告"反哺建议"→ 品类表加词/加节点 → 下轮映射率上升——表和程序互相迭代。

## 4. 如何接入现有 posclean 链

- 字段契约：cleaner 表头候选已覆盖 sku_registry.csv 的 条码/商品名称 命名；正式接入时加一条契约单测。
- 调用位：/posclean 完成 archive 后追加可选步骤"品类映射"；/trio 前置检查"是否已有 mapping.csv"。
- 不重叠：格式坑归 POS清洗库，品类坑归 cleaner；两者都不改原始数据。

## 5. 服务花厅坊 / 富城 / 好家源

- **花厅坊**：即刻价值最大——全店 SKU 品类标注的增量维护（新品进档自动映射，只审 review）；诊断开关直接驱动 T03 式品类诊断。
- **富城 / 好家源**：结构层（v3 骨架+keywords）共享，各店只建 alias 方言覆写文件；好家源三店历史数据汇总与反复交叉验证（六哥 2026-07-08 指示）列为接真实数据后第一个批处理战役——**本轮未动任何真实数据，该战役需六哥授权后在主线程执行**。
