## 3. 匹配算法

### 3.1 算法说明

**关键词 + N-gram 滑窗匹配** / 按 L4 → L3 → 品牌 → L4 定义关键词四层加分：

| 命中类型 | 加分 |
|---|:---:|
| L4 名整体在商品名（如 L4="棒棒糖" / 商品名含"棒棒糖"）| **+12** |
| L4 名 N-gram 子串（2-4 字滑窗 / 长度越长加分越多）| +5 起步 + 长度奖励 |
| L3 名整体在商品名 | +8 |
| L3 名 N-gram 子串 | +3 起步 + 长度奖励 |
| 品牌精确匹配（V4.0 品牌示例 ⊂ 商品品牌 或 反之）| +5 |
| L4 定义关键词（3 字以上中文）在商品名 | +3 |

### 3.2 置信度门槛

| 置信度 | 分数门槛 | fact_layer |
|:---:|:---:|:---:|
| **high** | ≥ 14 | observed |
| **medium** | 8-13 | inferred |
| **low** | 1-7 | pending |
| **未匹配** | 0 | pending |

### 3.3 反幻觉边界

- ✅ 匹配仅基于商品名 + 品牌字段 / 不假设商品规格 / 单位与品类关系
- ✅ 中低置信度 + 未匹配的 SKU 必须**人工核**才能入库
- ❌ 不自动将 medium / low 当 final 归属（参 [[2026-05-19_商品基础库分类争议治理规则_v0.1]]）
- ❌ 不修改 V4.0 品类表（V4.0 是 single source / 本匹配仅对应不更新）
- ❌ 不修改源文件 `商品食品基础表_V1.0.xlsx`

## 5. 已知错误模式

| # | 错误模式 | 示例 | 应人工核纠正 |

## 8. 与 vault 商品库 v0.1 的关系

### 8.1 三层归属（参 [[商品库_v0.1/01_字段字典]] v1.1）

| 层 | 内容 | 本次匹配输出去向 |
|:---:|---|---|
| **B 品类骨架** | `B_category_skeleton.csv` / 品类级稳定属性 | 本匹配引用 V4.0 / 不直接更新 B 层 |
| **A 核心 SKU 种子** | `A_core_sku_seed.csv` / 商品级稳定属性 | 本匹配结果 → W22 整理后可作 A 层候选 |
| **C 客户特定** | `C_client_specific/*_mapped.csv` | ✅ **本匹配输出位置**（mapped.csv / git ignore）|

### 8.2 字段对齐建议（W22+）

| 商品基础表字段 | 商品库 A 层字段 | 处理 |
|---|---|---|
| 条码 | barcode | 直接映射 |
| 商品名称 | standard_product_name | 直接映射 |
| 规格 | spec_value + spec_unit | 拆分（如 70g → 70 + g）|
| 单位 | spec_unit（包装单位）| 字段含义略有歧义 / 需 confirm |
| 品牌 | brand_name | 直接映射 |
| 匹配 L4 | category_l4 | 直接映射 / 但需 category_l4_code |
| L4 → L4_code | category_l4_code | **缺失字段**（V4.0 表无 L4 编码 / 需补）|

⚠️ **关键 gap**：V4.0 表无 `category_l4_code`（数字 5 位）/ 而商品库 A/B 层用 code 作主键。W22 需要建立 V4.0 L4 名 → code 映射表。
