# vault 商品库使用规则 v1.1

> **目的**：定义商品库**何时用 / 如何用 / 边界** / 防止误用 / 与 vault 治理纪律一致。

---

## §1. 何时使用商品库

| 场景 | 用途 | 调用层 |
|---|---|---|
| 客户数据进 vault | 客户 SKU 文本 → 标准品名 + 品类 | 4 层匹配算法（W26+）|
| 客户经营诊断 | 标准 SKU → ABCD 分类 / 价格带 / 滞销判定 | 客户战役 |
| 跨客户诊断对比 | 同一标准 SKU 在不同客户的表现 | Year 2 数据飞轮 |
| 顾问经验沉淀 | 新规则 / 新别名 / 新关键词 | keyword_rule + brand_alias |
| 陈列方案建议 | 商品级 + 门店级经营属性 | DisplayMap 关联 |
| 报告生成 | 标准品类 + 经营判断 | 客户交付 |

## §2. 如何使用商品库

### 2.1 客户 SKU 接入流程（W25 起步示范）

```
Step 1：客户 POS / ERP 数据导入
  → 直接拷贝原始字段进 C_client_specific/<客户>_SKU_raw.csv
  → raw 层不修改 / 不覆盖 / 仅追加

Step 2：清洗标准化
  → 去空格 / 去乱码 / 字段分离
  → 写入 mapped 层 clean_product_name 字段

Step 3：初步匹配（4 层算法 W26+）
  → Layer 1 规则（keyword_rule + brand_alias + package_unit）
  → Layer 2 模糊（Levenshtein / TF-IDF）
  → Layer 3 LLM 语义（W30+）
  → 写入 mapped 层 matched_sku_id / matched_skeleton_id / match_confidence

Step 4：人工审核（启明 / 顾问）
  → 写入 manual_review_status / manual_reviewer

Step 5：门店级经营判断
  → is_traffic / is_margin / is_image / suitable_end_cap / business_action

Step 6：反哺商品库
  → feedback_to_library = true 的标记后 / 入主库 A 或 B
  → 新规则沉淀 keyword_rule
```

### 2.2 字段调用层级

```
B 层（品类骨架）：
  - 品类匹配 / 品类角色 / 价格带 / 民生刚需判断
  - 优先级最高 / 早期主战场

A 层（标准 SKU）：
  - 精确 SKU 识别 / 商品级稳定属性
  - 可空（早期允许 matched_sku_id = NULL）

C 层（客户独有）：
  - raw 永不覆盖 / mapped 工作流主入口
  - 门店级经营判断
```

## §3. 使用边界（v1.1 强制）

### 3.1 数据使用边界

```
✅ 可以做：
  - 跨客户复用「标准 SKU + 品类骨架 + 规则」
  - 跨客户对比「品类结构 / 价格带 / 业绩占比」
  - 反哺 keyword_rule + brand_alias

❌ 不可以做：
  - 跨客户共享「销售数据 / 毛利数据 / 库存数据」（脱敏后才可统计）
  - 跨客户共享「客户内部码」（client_internal_code 仅本客户用）
  - 跨客户共享「门店级经营判断」（is_traffic 等门店级属性不可借用）
```

### 3.2 字段填写边界

```
P0 必填：缺失则不入库（脏数据拦截）
P1 推荐：有则填 / 没有可空 / 不强制
P2 未来：占位 / 不纳入 v1.1 验收
```

### 3.3 三层归属严格遵守

```
品类级属性 → 仅放 B 表
商品级属性 → 仅放 A 表
门店级属性 → 仅放 C mapped 表

⛔ 禁止跨层堆字段（如把 is_traffic 放 A 表）
```

### 3.4 客户真实数据接入前置检查（W23+ 必查 / P0）⭐

> **触发**：每次导入客户真实 raw/mapped 数据前。

**5 项强制检查**：

```bash
# 检查 1：raw/mapped 是否仍被 .gitignore 忽略
git -C <vault> check-ignore -v 13_*/商品库_v0.1/C_client_specific/<客户>_SKU_raw.csv
git -C <vault> check-ignore -v 13_*/商品库_v0.1/C_client_specific/<客户>_SKU_mapped.csv
# 期望：均命中 **/*.csv 全局忽略规则

# 检查 2：导入后 git status
git -C <vault> status -s
# 期望：raw/mapped 不出现在未跟踪 / 不出现在 staging

# 检查 3：staging 区无敏感数据
git -C <vault> diff --cached --name-only | grep -E "_raw\.csv|_mapped\.csv"
# 期望：空输出
```

**4 项强制规则**：

| # | 规则 | 违规处理 |
|---|---|---|
| 1 | raw/mapped 不得进入 staging | 立即 `git reset HEAD <file>` + 复查 .gitignore |
| 2 | 误加入客户敏感数据 / 立即 git reset | 沉淀 [[工程操作错误案例库]] + 通知 6 哥 |
| 3 | 不得用 `git add -f` 强制添加客户经营数据 | 视为重大违规 / 启明 sign off 后处理 |
| 4 | 跨设备同步客户数据走加密通道 / 不依赖 git | 详见 [[03_商品库维护SOP]] §7 |

### 3.5 结构层 CSV vs 客户数据 CSV 边界（数据契约）

| 类别 | 文件示例 | git 跟踪 | 跨客户复用 | 跨设备同步 |
|---|---|---|---|---|
| **结构资产** | A_core_sku_seed / B_category_skeleton / _maintenance/* / _客户SKU模板 | ✅ 上 git | ✅ 允许 | git |
| **客户原始数据** | C_client_specific/*_raw.csv | ❌ 不上 git | ❌ 禁止 | 加密备份 / 私有云 |
| **客户匹配数据** | C_client_specific/*_mapped.csv | ❌ 不上 git | ❌ 禁止（除非脱敏后入 _anonymized/）| 加密备份 / 私有云 |

## §5. 与 4 层匹配算法的协同

```
v1.1 起步：商品库 + Layer B 骨架（W24-W25）
v0.3：规则匹配 + 模糊匹配 + keyword_rule 沉淀（W26-W27）
v0.4：LLM 语义匹配 / sentence-transformers（W30+）
v1.0：咨询诊断系统正式接入（Year 1 末）
```

→ 商品库不是 4 层算法的"前置一次性工程" / 是"持续学习底座"。

## §6. 反幻觉硬约束（与 §13.16-20 协同）

| 约束 | 落地 |
|---|---|
| 不编造 SKU 数据 | source_type 必填 / 标 manual 时备注来源 |
| 客户视角防火墙 | C 层 raw_xxx 字段为客户原始 / mapped 层为 vault 推断 |
| 推断标注 | 算法匹配结果带 match_confidence / < 0.7 必人工审核 |
| 用户订正优先 | 启明 sign off 后的 mapped 优先于算法结果 |
| 数据一致性追溯 | feedback_to_library 标记前必查重复 + FK 完整性 |
