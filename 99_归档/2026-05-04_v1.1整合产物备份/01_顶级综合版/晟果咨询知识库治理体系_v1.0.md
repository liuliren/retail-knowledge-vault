# 第 1 部分 · 总章

## 第 1 章 治理体系的目的、范围、原则

### 1.1 目的

本治理体系解决以下 5 个问题：

1. **vault 长得像什么** — 顶层结构 / 子目录规则 / 命名标准
2. **文档怎么写** — 工程化标准 / 知识结晶结构 / frontmatter
3. **多版本怎么处理** — 综合分辨 / 权威裁定 / 老版本归档
4. **变更怎么管** — 影响检查 / 双向同构 / 反幻觉与一致性守门
5. **治理怎么执行** — 仪式 / 角色 / RACI

### 1.2 范围

**适用于**：
- vault 内所有 .md / .pdf / .docx / .xlsx / .csv / .py 等文件
- 19 个一级目录及其子目录
- vault ↔ Claude Code（`.claude/`）的同步
- vault ↔ 客户交付物的边界

**不适用于**：
- 客户私有数据（这部分受 `client_safety: confidential` 单独治理）
- 个人备忘 / 临时笔记（应放 14_刻意练习与成长/日记/）

### 1.3 三大原则

| 原则 | 含义 | 实操指引 |
|---|---|---|
| **服务实战** | 一切沉淀为实战服务 / 落地有结果是目标 | 不为知识完整而沉淀；每份重要文档必须能在实战中调用 |
| **教科书严谨** | 借鉴教科书体系：目录 / 章节 / 术语 / 索引 / 引用 / 版本 | 每份核心文档有 ToC / Executive Summary / 章节编号 / 关联引用 / 版本记录 |
| **顶级咨询品质** | 全球顶级咨询公司的工程化标准 | 命名一致 / 结构清晰 / 多版本权威裁定 / 不允许"临时垃圾"前缀 |

## 第 3 章 三个核心命题

### 3.1 命题 A · vault 是工程化资产

**含义**：vault 不是个人笔记本 / 不是文档堆 / 是**有版本、有引用、有反幻觉机制、有逻辑一致性检查**的工程产物。

**判据**：
- 每份重要文档有 frontmatter（含 version / status / owner / quadrant / client_safety）
- 每份重要文档有版本记录段
- 每份重要文档有关联段（上游 / 下游 / 案例）
- 每份重要文档过反幻觉 + 一致性两关

**反例**：
- ❌ "随手写的灵感" → 应该放 14_/03_临时灵感/（如有此目录）/ 不进核心模块
- ❌ "客户对话记录原文" → 应放 90_素材原始档案/00_客户原始材料/（client_safety: confidential）

### 3.2 命题 B · 教科书严谨 + 实战导向

**教科书严谨的具体表现**：
- 完整 ToC（目录索引）
- 章节编号统一（一级 / 二级 / 三级）
- 术语统一（参 v3.2 00 术语与对象定义）
- 引用格式标准（wikilink + 路径 + 行号）
- 参考文献段（说明来源）
- 版本号约定（v0.x / v1.x / v2.x 含义清晰）

**实战导向的具体表现**：
- Executive Summary 一页就懂
- 实战调用入口（5 分钟内查到规则）
- 实证段（关联具体战役 / 案例）
- 待迭代清单（实战回流的钩子）

### 3.3 命题 C · 质量先于数量

**判据**：
- 一份 "知识结晶版" 综合多源 + 实证 + 矛盾裁定 / 胜过十份 "占位 README"
- 一份 v1.2 综合版（综合 4 份原版 + 5 份实证 + 实战）/ 胜过 4 份 v3.2 第三轮主版堆叠

**反例**：
- ❌ 用脚本批量生成 12 个 README（模板化 / 内容空洞）
- ❌ 把多版本文件简单 mv 到子目录（没真正综合）

# 第 2 部分 · 工程化标准

## 第 4 章 目录结构规范

### 4.1 一级目录命名规范

| 编号区间 | 含义 | 例 |
|---|---|---|
| 00_ | 治理层 | 00_治理与导航 |
| 01-14_ | 业务层（14 个一级业务模块）| 01_方法论核心 |
| 90_ | Raw 元层 | 90_素材原始档案 |
| 99_ | 归档层 | 99_归档 |
| 中文名（无前缀编号）| 元层（特殊）| 晟果咨询知识底座_v1.1 |

**禁止**：
- ❌ "_暂存_" "_原 xx" "_v3.2 xx 专项" 等临时性 / 修饰性前缀
- ❌ "test_" "draft_" "todo_" 等英文临时标记
- ❌ 半角符号 / 特殊字符（除了 `_` 和数字）
- ❌ 数字 + 字母混编（如 1.A / 1.B → 应用 1.10 / 1.11）

### 4.2 二级目录（子目录）命名规范

格式：`<2位数字编号>_<中文名>`

```
01_方法论核心/
├── 1.0_统一语言/
├── 1.1_总入口诊断/
├── 1.2_品类体系/
...
└── 1.11_组织力运营力总纲/
```

**例外**：
- `_附件_<名>` / `_早期素材/` 等子目录可用 `_` 前缀（明确标记为附属/历史）
- 但**一级目录绝不用 `_` 前缀**

### 4.3 文件命名规范

#### 4.3.1 综合版 / 知识结晶文档

格式：`<主题>综合版_v<版本>.md`

例：
- `品类体系综合版_v1.2.md`
- `陈列与空间效率综合版_v1.0.md`

**禁止**：
- ❌ `_品类体系综合版_v1.1.md`（前导 `_` / 显得"次要"）
- ❌ `品类体系综合版_最新版.md`（"最新版"会过时）

#### 4.3.2 v3.2 体系原版

保留原命名（如 `02_品类体系_第三轮回灌式精修版_v3.2.md`）/ 不动。

#### 4.3.3 实战工具

格式：`<日期>_<主题>_v<版本>.md`

例：
- `2026-05-04_G4-A冲调首战军师简报_v0.3.md`
- `2026-05-05_06_战役1_G-A首战作战手册_v0.1.md`

#### 4.3.4 客户验证日志

格式：`<日期>_<客户代号>_<场景>.md`

例：
- `2026-05-05_花厅坊_二访.md`
- `2026-05-06_花厅坊_G4-A调改首战.md`

## 第 5 章 命名规范（深度细则）

### 5.1 中英文混排规则

| 场景 | 规则 | 例 |
|---|---|---|
| 一级目录 | 中文为主 | 01_方法论核心 |
| 系统术语 | 保留英文（不翻译）| L1 / L2 / L3 / L4 / SKU / DisplayMap / CDT / TPOS |
| 工具方法编号 | 英文字母 + 数字 | A1 / A2 / B2 / DX-STORE-01 / SL-V-01 |
| 客户代号 | 中文为主 | 花厅坊 / 沙埔大道店 |
| 字段名（YAML / 表格）| 中文（除非系统强制英文）| `分类` `品牌` `规格` |

### 5.2 版本号规则

| 版本号 | 含义 | 何时升级 |
|---|---|---|
| **v0.1** | 草稿 / 初稿 | 首次创建 |
| **v0.5** | 可讨论版 | 内部审阅后 |
| **v1.0** | 首个正式版 | 实战验证后 |
| **v1.x** | 小修订（实战回流 / 增量优化）| 每次实战回流后 |
| **v2.0** | 结构性升级 | 完整咨询项目复盘 / 跨客户复用 |

### 5.3 日期格式

**必须 YYYY-MM-DD**（ISO 8601）/ 不允许 MM/DD/YYYY 或 中文"X 月 X 日"。

例：
- ✅ `2026-05-04`
- ❌ `2026/5/4` `5月4日` `5.4`

### 5.4 标点符号

| 符号 | 用法 | 禁忌 |
|---|---|---|
| `_` | 编号分隔 / 子目录附属标记 | 不用于一级目录前缀 |
| `-` | 工具方法编号（如 `A1` `DX-L3-01`）| 不用于通用文件名 |
| `/` | 目录分隔 / 表格分隔 | 不在文件名中用 |
| `·` | 标题装饰（如 "晟果咨询 · 知识结晶")| 慎用 / 仅顶级文档 |


# 第 3 部分 · 内容质量

## 第 8 章 反幻觉规则（15 条硬约束）

来源：[[CLAUDE.md]] §13。在本治理体系中**强制约束所有 vault 文档**。

| # | 规则 |
|---|---|
| 1 | 真实门店 / 商品 / 数据相关内容必须区分**事实 / 推断 / 建议 / 待确认** |
| 2 | 不得把推测写成事实 |
| 3 | 不得编造门店数据、销售数据、库存数据、顾客反馈、竞品信息 |
| 4 | 不得只凭销量判断商品力 |
| 5 | 商品诊断必须同时检查活动、陈列、缺货、价格、库存、客群影响 |
| 6 | 陈列方案必须考虑包装尺寸、层板高度、货架结构、顾客拿取便利性、真实门店条件 |
| 7 | 活动分析必须区分自然动销和促销拉动 |
| 8 | 外部案例必须经过本土化判断（不能直接套用到社区超市）|
| 9 | 不确定信息必须标注 "待确认" |
| 10 | 生成正式结论前应列依据文件或依据事实 |
| 11 | 现场照片 / Excel / 销售系统数据 / 门店反馈之间矛盾必须标注 / 不能强行合并 |
| 12 | DisplayMap / 陈列方案必须考虑真实货架结构 / 商品包装 / 品类边界 / 动线 |
| 13 | SKU 淘汰 / 保留 / 新增建议必须同时考虑销售额、毛利额、库存、动销、陈列、活动、客群 |
| 14 | 营销活动效果判断必须区分活动期间 / 前 / 后（不能只看单点销量）|
| 15 | 系统 PRD / 工具规则必须标注数据字段依赖 + 待确认项 |

## 第 10 章 综合分辨与权威裁定流程

> **本章是治理体系的关键** — 处理"多版本并存"问题。

### 10.1 综合分辨 5 步

```
Step 1: 列出所有版本（含时间 / 出处 / 字段）
   ↓
Step 2: 对照差异点（同一规则在不同版本怎么说）
   ↓
Step 3: 推断意图（为什么会有版本差异：升级 / 错填 / 场景不同 / 临时实验）
   ↓
Step 4: 裁定权威（用三个判据）
   ① 最新版本优先（时间）
   ② 实战验证优先（实证）
   ③ 客户认可优先（结果）
   ↓
Step 5: 影响范围登记（哪些下游文件需要同步更新）
```

### 10.2 三个判据的优先级

```
客户认可（最高）> 实战验证 > 最新版本（最低）
```

**为什么客户认可最高**：晟果咨询的核心命题是"为客户创造价值" / 客户认可才是终极判据。

**实战例**（5/6 G4-A 6 组货架分组矛盾）：
- 48 组位编码 V4.1（最早）/ 按 L4 类聚
- GPT 04 母版（中期）/ 按 L4 类聚 + 端架
- **六哥版本 B 实图（最新 / 实战版）/ 按 SKU 角色 / CDT** ← 裁定权威

### 10.3 矛盾点必须集中处理

所有矛盾点 → 集中到 [[晟果咨询知识底座_v1.1/99_矛盾与待裁定]] / 不分散在各文档。

每个矛盾点条目应包含：
1. 矛盾描述（多版本对比表）
2. 综合判断
3. 权威裁定
4. 影响范围
5. 下游文件同步状态

## 第 11 章 知识结晶文档标准结构

### 11.1 知识结晶定义

**知识结晶 = 综合多源 + 实战实证 + 矛盾裁定 + 可调用** 的方法论文档。

**与"导航 README"区别**：
- 导航 README = 指向其他文件（占位 / 索引）
- 知识结晶 = 自身就是方法论（综合内容 / 不指向）

### 11.2 知识结晶强制结构（17 段标准）

```markdown
---

<最基础的概念定义 / 不可省略>

## 二、决策矩阵 / 判据
<具体可调用的决策规则>

## 三 ~ N、详细方法论
<按业务逻辑展开>

## N+1、跨模块承接关系
<上下游关系图>

## N+2、矛盾点裁定
<引用知识底座 99_矛盾与待裁定>

## N+3、实战调用入口
<5 分钟内查到规则>

## N+6、待迭代

## N+7、关联
- 上游 / 平行 / 下游 / 实战

## N+8、版本记录

## N+9（可选）、本文件作为"知识结晶"的特点
<与原版对比表>
```

### 11.3 知识结晶质量验收（10 项）

| # | 验收项 | 判据 |
|---|---|---|
