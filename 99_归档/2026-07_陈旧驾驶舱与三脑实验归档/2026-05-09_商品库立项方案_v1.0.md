# vault 商品库 v0.1 立项方案 v1.1

> **战略地位**：vault 数据飞轮 + 咨询业务核心壁垒 / Year 2-3 形成跨客户复用护城河。
>
> **立项触发**：5/9 用户提议 + Q1-Q5 决策 + Q2-B' 折中 / 综合 ChatGPT 8 项升级建议。
>
> **核心定位**：商品库 = "咨询诊断系统 + LMWiki 知识库"中间层基础设施 / 不只是 SKU 数据库。

## §3. 三层架构（v1.1 升级 / 加经营属性层）

### 3.1 三层 SKU 结构（横向）

```
Layer A: 核心 SKU 库（精确 / Year 2 50,000+ 目标）
  - W26 起步：500-1,000 SKU 种子（OFF 食品类）/ 不追求规模
  - 早期重点：字段标准 + 品名标准 + 品牌别名 + 规格单位

Layer B: 品类骨架库（覆盖广 / 长尾兜底）⭐ 起步主战场
  - W24 起步：V4.0 308 L4 全覆盖
  - 早期重点：作为 4 层匹配算法的 target reference

Layer C: 客户独有 SKU 库（长尾 / 人工标注）
  - W25 起步：花厅坊真实 SKU 接入（raw + mapped 双层）
  - 早期重点：反哺机制 / 越用越准
```

### 3.2 四层商品知识结构（纵向 / v1.1 新增 ⭐⭐⭐）

```
Level 1: 品类骨架层
  来自 V4.0 / 解决"商品归到哪个标准品类节点？"

Level 2: 商品识别层
  品名+品牌+规格+包装 / 解决"这个商品是什么？"

Level 3: 经营属性层 ⭐⭐⭐ 晟果咨询核心壁垒
  品类角色 / 价格带 / 形象/流量/毛利/民生 / 端架/黄金层适合性
  解决"商品在门店经营中应该怎么看？"
  → 普通商品库不会做 / 这是晟果区别于 GS1/京东商品库的关键

Level 4: 客户反哺层
  客户名 / 销量 / 毛利 / ABCD / 处理建议 / 复盘结果
  解决"客户项目里这个商品实际表现如何？"
```

## §4. 商品库 vs 4 层匹配算法的关系（v1.1 重大订正 ⭐⭐）

### 4.1 错误关系（v1.0 旧）

```
❌ 商品库是 4 层算法的"前置一次性工程"
   商品库 v0.1（W19-W26）→ 完成 → 4 层算法 v0.1（W27+）
   
问题：等商品库完美后再做算法 → 错位
```

### 4.2 正确关系（v1.1 新 / 采纳 ChatGPT）

```
✅ 商品库是 4 层算法的"持续学习底座"
   边建库 / 边匹配 / 边反哺 / 持续迭代

W19    商品库骨架（字段+目录）⭐ 已完成
W24    Layer B 品类骨架（V4.0 308 L4）
W25    Layer C 花厅坊 SKU 接入（raw 层）
W26    规则匹配 + 模糊匹配（Layer 1+2 同时启动）⭐
W27    人工修正 + 反哺商品库
W28+   LLM 语义匹配（Layer 3）+ 持续反哺
W30+   Layer A OFF 种子（不追求规模 / 字段稳后才上）
```

### 4.3 关键洞察

> **客户 SKU 不一定马上能映射到标准 SKU / 但必须先映射到品类骨架**

允许：
- `matched_sku_id = NULL`（暂未映射到标准 SKU）
- `matched_skeleton_id = 有值`（已映射到品类骨架）

→ 这是早期商品库的实用逻辑 / 不要追求一步到位。

## §7. 两套系统视角（v1.1 新增 ⭐）

商品库在两套系统中作用不同 / 但是同一底座：

### 7.1 咨询诊断系统视角

```
客户原始商品数据 → 标准化商品识别 → 品类归属 → 商品诊断 → 咨询建议

商品库价值：让客户数据可以被稳定识别和分析
```

### 7.2 LMWiki 知识库视角

```
回答：
  - 这个商品属于哪个品类？
  - 这个品类在社区超市里承担什么角色？
  - 这个商品常见规格 / 适合陈列位置？
  - 这个品类有哪些常见品牌 / 价格带划分？
  - 这个商品在历史客户中表现如何？

商品库价值：让每次客户项目都反哺知识库 / 知识库越来越有行业判断力
```

→ **商品库 = 两套系统的中间层基础设施**（不是孤立模块）。

## §8. 5 大原则（v1.1 新增 / 采纳 ChatGPT）

| # | 原则 | 含义 |
|---|---|---|
| 1 | **先品类，后单品** | 早期不执着每个 SKU 精准识别 / 先保证客户 SKU → 正确 L4 品类 |
| 2 | **先白盒，后智能** | 先做字段规则 + 关键词 + 品牌别名 + 规格标准化 + 人工审核 / 后做 embedding + LLM |
| 3 | **先反哺，后规模** | 商品库不靠一次性导入做大 / 靠客户项目做准 / 每客户战役沉淀新商品 + 新别名 + 新规则 |
| 4 | **必须绑定 V4.0 品类树** | 所有标准商品必须指向 category_l4_code（V4.0）/ 不能自长一套分类 |
| 5 | **业务数据资产 / 不是治理支柱** | 不是 vault 第 7 大治理支柱 / 是数据与系统能力底座下的业务数据资产 |

## §12. 风险与权衡（v1.1 升级）

| # | 风险 | 严重度 | 缓解 |
|---|---|---|---|
| R-01 | 字段标准不稳就堆数据 ⚠️ ChatGPT 警告 | **高** | v0.1 骨架期不堆数据 / 字段稳后再 v0.2 反哺 |

| R-04 | 跨客户数据隐私 / 法律风险 | 中 | 商品库默认匿名化 / 仅 SKU + 品类 / 不保留客户销售数据 |
| R-05 | 数据飞轮启动慢 | 中 | 第 2 客户战役（破晓肇庆）开始受益 / 可接受 |
| R-06 | 经营属性层主观判断重 | 中 | 起步用品类共识（米油盐=民生 / 高端礼盒=形象）/ G11 月度修正 |
