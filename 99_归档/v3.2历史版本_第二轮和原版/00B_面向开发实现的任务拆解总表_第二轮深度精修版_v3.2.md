## 一、第二轮深度精修的结论

第一轮 `00B` 已经完成了：

- 阶段划分
- 模块拆解
- MVP / P1 / P2 边界
- 依赖、交付物、验收方向

但那一版更偏：

> **系统级开发总控表**

而不是：

> **可直接进入排期、分工、联调、验收的实施清单**

所以第二轮深度精修，重点只做 5 件事：

1. **任务颗粒度下沉**
2. **依赖关系硬化**
3. **验收标准量化**
4. **MVP 边界再压实**
5. **补齐角色、工作量、风险、联调关系**

这一轮之后，`00B` 的定位变成：

> **整套系统开发实施与排期的主控底稿。**

## 二、使用方式

这份文件建议作为以下 4 类工作的共同底稿：

1. **产品排期会**
2. **前后端分工会**
3. **规则 / 数据 / 接口联调会**
4. **阶段验收会**

也就是说，它不再只是读给自己看的说明稿，而是：

> **项目执行层的总表。**

## 三、任务拆解的统一标准

## 3.1 角色代码

| 代码 | 角色 |
|---|---|
| `PM` | 产品 / 项目负责人 |
| `ARCH` | 系统架构 |
| `BE` | 后端开发 |
| `FE` | 前端开发 |
| `DATA` | 数据建模 / 数据工程 |
| `RULE` | 规则引擎 / 任务流开发 |
| `QA` | 测试 / 联调 / 验收 |
| `OPS` | 运维 / 部署 / 环境支持 |

---

## 3.2 工作量等级

| 等级 | 含义 |
|---|---|
| `XS` | 很小，通常 0.5–1 天内可完成 |
| `S` | 小，通常 1–2 天 |
| `M` | 中，通常 2–4 天 |
| `L` | 大，通常 4–7 天 |
| `XL` | 很大，通常 7 天以上，建议拆分 |

> 说明：这里是任务颗粒度管理口径，不是正式工时承诺。

---

## 3.3 依赖关系代码

| 代码 | 含义 |
|---|---|
| `HARD` | 硬依赖，前置不完成不能开始 |
| `SOFT` | 软依赖，前置最好完成，但可并行推进 |
| `PARA` | 可并行 |
| `LATE` | 可后置，不阻塞主链路 |

---

## 3.4 验收等级

| 等级 | 含义 |
|---|---|
| `A` | 必须可上线 / 可演示 / 可联调 |
| `B` | 必须可用，但允许内部能力先跑通 |
| `C` | 先具备基础能力，页面或自动化可延后 |

## 五、MVP 范围（第二轮压实版）

## 5.1 MVP 必做功能（必须上线或必须演示）
以下项目列为 **MVP 必做**：

1. 核心对象与主表建模
2. 主树结构与基础经营维度落地
3. 门店总览页
4. L1 板块分析页
5. L3 分析页
6. SKU 详情页
7. 六层数学引擎基础版
8. 规则命中服务基础版
9. 任务生成与流转基础版

---

## 5.2 MVP 可只做接口、不做完整页面的功能
以下项目在 MVP 可只做接口和内部对象，不要求完整前端体验：

1. 上下文增强维度对象
2. 复杂类 L4 决策组对比
3. 上下文数学增强结果对象
4. 规则复盘对象
5. 数据审计日志对象

---

## 5.3 MVP 暂不做复杂化的功能
以下内容明确不建议抢跑：

1. 全量自动化复盘引擎
2. 复杂类全量专题页
3. 高级可视化对比看板
4. 大量后台配置中心细化
5. 过细的人群画像系统

### 一句话原则
> **MVP 的目标不是“做全”，而是“跑通主链路并可解释”。**
