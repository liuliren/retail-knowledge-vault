# 零售知识系统 V2 迭代方案

> 依据:Karpathy LLM Wiki(原型)→ Rohit V2 Gist(架构)→ agentmemory(实现)+ 两份 Gist 评论区的生产派批评
> 适用对象:六哥零售 retail-knowledge-vault / M-DEC 长期记忆系统
> 立场:采「最简正确版」,把生产派批评当设计约束,拒绝过度工程
> 日期:2026-06-22

---

## 0. 判断先行:我们要的不是 V2 全套,是「最简正确版」

V2 是一份**菜单,不是规格书**。它列了一堆工程能力(置信度、遗忘曲线、知识图谱、混合检索、自动化 hooks、质量门),但 Gist 评论区里两位生产用户(gnusupport、一位匿名长期运行者)和 V3 作者的反驳,才是真正的设计约束:

| V2 提出的 | 生产派的修正 | 我们采用 |
|---|---|---|
| confidence score(0.85 这种数字) | 假精确;真信号是证据链 | **provenance(溯源),不打分** |
| 遗忘曲线 / decay | 旧≠过时,衰减错误=重犯错误 | **supersession(显式取代),不衰减** |
| 事件驱动自动写入 | 假设 LLM 可靠,会悄悄污染 | **人工写入闸门** |
| 一堆后处理机器 | schema 干了约 90% 的活 | **先把 schema 做好,机器后置** |
| 知识图谱 / 向量库 | 几百节点不算图谱 | **暂不上,用命名边代替** |

> 一句话:V2 的价值是**词汇表和问题清单**,不是施工蓝图。我们抄思想,不抄计划。
> 这也呼应前两轮的「少即是多」与「模型之外都是 Harness」——**知识系统就是零售诊断这匹马的记忆型马具**,做厚的是判断,不是机器。

## 1. 三层映射:把 Karpathy 三层翻译成零售知识库

| Karpathy 层 | 零售落地 | 谁维护 |
|---|---|---|
| **raw sources(只读)** | 门店现场记录、诊断原始数据、客户资料、外部素材(万维钢/idoubi 蒸馏) | 只读,永不改写 |
| **wiki(LLM 维护)** | 方法论资产层:7D、品类诊断、选品、SOP、案例页、KB | 人审 + AI 写 |
| **schema(规则书)** | 你的 vault CLAUDE.md ← **真正的产品** | 你和 AI 共同演进 |

三个操作的零售版:
- **ingest**:一次现场决策 / 一份诊断 → 写进对应页 + 更新 index + 追加 log
- **query**:做新店诊断时,先查库里同业态/同品类/同 IssueTag 的既有结论
- **lint**:周度健康检查(挂靠花厅坊周节奏)——查矛盾、查没回填结果的孤儿决策、查该晋升的 IssueTag

## 2. 五条核心设计原则(每条带零售落地约定)

### 原则一:四级固化 = M-DEC 的引擎

agentmemory 的 `working→episodic→semantic→procedural` 正是 M-DEC 的零售版:

| 固化层 | 零售对应 | 例 |
|---|---|---|
| working | 现场原始观察 | 「花厅坊烧腊区 17:00 后排面塌陷」 |
| episodic | 单次诊断决策(完整 M-DEC) | 「砍掉 X 品类,因为…,结果回填…」 |
| semantic | 跨店提炼的事实 | 「社区住宅店烧腊是高毛利锚点」 |
| procedural | 可复用 SOP | 「烧腊区 7D 诊断 + 调整 SOP」 |

**晋升靠证据累积,不靠时间。** 越往上越压缩、越可信、越长寿。

### 原则二:supersession 取代 decay

零售决策**不过期,只被取代**。半年前某店的定价判断仍是有效背景,新判断出现时:
- 旧页**保留**,加头部标记 `被取代:: [[新决策]]`
- git 自然成为审计轨迹
- 好处:你永远能回看「当时为什么那么决定」,避免重犯

### 原则三:provenance 取代 confidence score

每条方法论结论必须挂**证据链**,不是浮点数:
- ❌「该砍此品类(置信度 0.85)」
- ✅「该砍此品类 ← 来源:: [[好家源-3号店案例]]、[[花厅坊-烧腊诊断]]」
- 可信度 = 挂着几个独立案例,看得见、可核

### 原则四:人工写入闸门

AI 可生成初稿,但**入库必须经你 sign-off**(你已设的 self-optimization triple set 正是这个,生产经验直接验证它是对的,别取消)。说不清推理的结论,退回,不签字、不入库——这是你 10% 护城河的底线。

### 原则五:schema 承载 90%,机器后置

先把 vault 的规则书写清楚(见第 3 节),其余自动化等「手动跑过几十轮、模式稳定」后再上。

## 3. 零售域 Schema 雏形(实体 + 命名边)

不上图数据库。用 Obsidian 风格 wikilink + **命名边**(参考评论区 ChristopherA 的 `derived_from::` 思路),用纯文本 + 纪律拿到 80% 的图谱价值。

**实体类型**

| 实体 | 关键字段 |
|---|---|
| 门店 Store | id、业态(社区住宅/菜场邻近)、面积、客群、状态 |
| 品类 Category | 名称、H-Score、CDT |
| SKU | H-Score |
| IssueTag | 问题码、严重度 |
| 决策 Decision(M-DEC) | 现象/决策/推理/结果、来源店、状态(live/superseded) |
| 案例 Case | 项目、门店、交付记录 |
| 方法论 KB | 晋升来源、适用条件 |

**命名边(关系)**

```
诊断出::    门店 → IssueTag
触发::/导致:: 现象 → 决策
被取代::     旧决策 → 新决策
晋升为::     ≥3 店决策 → 方法论
适用于::     方法论 → 业态/品类
来源::       任何结论 → 案例/门店(provenance)
```

> 写入纪律(参考评论区 watsonrm):**按 provenance 身份去重,不按文字相似度**;同一现象在不同店重复出现 → 是新证据,不是重复;真冲突进「待审」,不自动解决。

## 4. M-DEC 修复 = 本方案的心脏(最高优先级)

这是知识库审计早就标出的关键缺口。修复 = 让现场决策**回流并晋升**。

**最小记录格式(30 秒可口述)**

```markdown
## [日期] 决策 | 门店 | 一句话标题
- 现象:
- 决策:
- 推理(一句话):
- 来源:: [[门店]] / [[IssueTag]]
- 结果:(留空,2–4 周后回填)
- 状态: live
```

**晋升路径**
1. 现场拍板 → 30 秒记 working/episodic
2. 决策后 2–4 周 → **回填结果**(这步才把「记录」升级成「记忆」)
3. 同类现象/决策在 **≥3 家独立门店**重复且结果一致 → `晋升为::` 一条 semantic/procedural 方法论(你已设的 sample gate)
4. 晋升前,你 sign-off(人工闸门)

**回填闭环是关键**:没结果的决策是孤儿,lint 每周把它们揪出来提醒。

## 5. 分阶段迭代路线(套用 V2 implementation spectrum,但重排序)

> 每阶段有明确的「触发下一阶段的条件」,不到不做——防止提前过度工程。

### Phase 0 — Minimal Viable(本周,半天)
- **动作**:写 vault schema 文件(CLAUDE.md)+ 定 M-DEC 最小格式 + 确认 index.md/log.md 约定
- **产出**:一份规则书 + 一个决策模板
- **验收**:新决策能照模板独立记录,不需问你

### Phase 1 — Add Lifecycle(5–8 月,挂靠花厅坊)
- **动作**:跑 ingest;启用 supersession(取代标记)+ provenance(来源边)+ 人工闸门
- **产出**:花厅坊 4 周内 ≥8 条 M-DEC、≥1 条晋升方法论
- **验收**:每条原始记录都查得到「被提炼进了哪条上层结论」,无孤儿
- **触发 Phase 2 的条件**:vault 页数稳定增长、跨店复用开始发生

### Phase 2 — Add Structure(好家源 / 秋季)
- **动作**:轻结构化——为高频实体建独立页 + 补全命名边 + lint 固定进周节奏
- **产出**:好家源旗舰店沉淀后,第 2 家店诊断耗时下降 ≥40%(存算分离的复利)
- **验收**:7D → 品类诊断 → 选品能像积木无缝串起
- **触发 Phase 3 的条件**:**页数逼近 100–200**(index.md 一次读不完)

### Phase 3 — Add Scale / Automation(达到规模后才做)
- **动作**:此时才考虑混合检索(BM25+向量)/ 自动化 hooks
- **明确不提前做**:没到 100–200 页这条线,`index.md` 足够,不上向量库

### (暂缓)Add Collaboration
- 多人/多 Agent 协作是团队场景,你单兵阶段不需要

## 6. 与现有工作流的嵌合(防注意力分散)

这是最重要的执行约束。知识系统**不能变成又一个平行大工程**(你的 attention 红线)。

- **不新开战场**:Phase 1 的 ingest 直接绑到花厅坊周时间块,现场决策顺手记,周末批量回填
- **借力 community-sop**:B+D 完成后接 A/C/E 时,顺带把 schema 的实体定义/IssueTag→SOP 映射补全(本来就要做的事)
- **资产刷新搭车**:每刷新一个老资产(7D/品类诊断/选品/科学零售库/花厅坊产出),顺手按新 schema 补上 provenance 边

---

## 7. 风险与反模式(明确「不做什么」)

1. ❌ 不上 confidence score(假精确)
2. ❌ 不上 decay/遗忘(零售决策不过期,只被取代)
3. ❌ 不让 AI 未经 sign-off 自动写入(会悄悄污染)
4. ❌ 不提前上向量库/图数据库(没到规模线)
5. ❌ 不让知识系统抢占花厅坊主线注意力(5–8 月主线只有一条)
6. ❌ 不追求完美记录,追求**不断流**(断了比简陋更糟)
