## §13 用人话沟通 / 不讲玄乎话（v1.0 新增 / 5/16 P0）

### 13.1 强制规则

vault 任何输出（对话 / 文档 / 任务清单 / 报告）一律:
- ✅ 用具体数字 + 文件名 + 行号 + 直白描述
- ❌ 禁止行业术语堆砌 / 含糊代词 / "感觉玄乎"

### 13.2 反例 → 正例对照

| ❌ 玄乎话（禁用）| ✅ 人话（替代）|
|---|---|
| sign off | **确认 / Confirm** |
| 启明 sign off 5+2 件 | 启明要确认的 5 件 |
| 积压 SKU 子集 / 不包含全量在售 SKU | 5/8 这份表只列了 53 个有积压问题的商品 / 没列所有正在卖的方便速食 |
| 0206% 口径与方便速食业务定义不完全对齐 | 系统里 0206% 是"方便食品大类" / 业务上叫"方便速食" / 可能多/少几个品类 / 待确认 |
| 数据底座 fact_layer.client_told | 你刚告诉我的事实 |
| BUS-DATA-008 前置闸门 12/12 未满足 | 全自动清洗数据这条路 / 12 个准备工作还有 6 个没做 / 暂走不动 |
| 字段完整性判定 | 这份表里有没有我们需要的字段 |
| 调改 sub-loop 第一轮 +24h 客户验证 | 5/6 改完货架的第二天 / 看店里反应 |
| confirmed by client_told | 你已告诉我 / 已记 |

### 13.3 自检清单（每段输出前过）

- [ ] 用了具体数字吗?（不是"很多 / 一些"）

- [ ] 范围明确吗?（不是"基本 / 大概"）

## §15 V4.0 + ETL + 主字段 = 内部数据接入唯一标准（v1.0 新增 / 5/16 P0）

### 15.1 强制规则

| 项 | 内容 |
|---|---|
| **单一标准** | 大湾区品类表 V4.0 = 咨询方法论唯一标准 |
| **POS 数据** | 只抓核心字段 + 参考分类方式 / **不作决策依据** |
| **数据接入机制** | 匹配表（SKU 级）+ 映射表（品类码级）/ ETL 思想 |
| **主字段** | 必须有一个主字段（如 SKU 商品编码 / category_l4_code）作唯一锚点 |

### 15.2 ETL pipeline 标准步骤

```
Step 1: 各店 POS 导出原始数据
Step 2: 用匹配表 / POS SKU 编码 → 我们的 SKU 标准编码
Step 3: 用映射表 / POS 品类码 → 大湾区品类表 V4.0 品类
Step 4: 数据进入 "商品档案 / 商品基础表" 标准格式
Step 5: 后续分析全部按我们的标准做
```

### 15.3 深层意义（3 个）

| # | 意义 | 含义 |
|---|---|---|
| 1 | 方法论主权 | 咨询师定义品类边界 / 不被 POS 系统绑架 |
| 2 | 可复用性 | 同套标准跨 3 店（花厅坊+沙浦大道+沙浦市场）/ 跨客户复用 |
| 3 | 跨店可比 | 不同店 POS 编码差异 / 但内部标准统一 → 4 店共性命题可算 |
