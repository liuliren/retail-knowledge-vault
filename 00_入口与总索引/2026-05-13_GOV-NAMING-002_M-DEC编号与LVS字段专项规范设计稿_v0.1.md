**总计**：7 文件存在 / 1 缺号 / 0 aliases / **134 处短码 vs 82 处完整引用**。

### 3.2 短码断链根因

- **Obsidian 默认按文件名 basename 解析双链**
- 7 个 M-DEC 文件名都带后缀（`_<主题>_v<版本>_<状态>`）/ 短码 `M-DEC-005` 无后缀 / **basename 不匹配**
- Obsidian 把无 basename 匹配的 wikilink 显示为"未创建文件"占位空白
- **解决方案**：frontmatter 加 `aliases: ["M-DEC-005"]` / Obsidian 按 alias 查找匹配 / 短码重新可跳转

### 3.3 短码 vs 完整引用分布特点

- **M-DEC-007 短码 82 处**（最严重 / 最常被引用 / 最先要修）
- **M-DEC-006 短码 26 处**（次严重）
- **M-DEC-008 短码 0 处 / 完整 13 处**（用户引用习惯已自然向完整名靠）
- 修复优先级：**007 > 006 > 005 > 002 > 001 ≈ 003 > 008**（按短码引用数）

## 4. M-DEC 命名规则建议（草案）

### 4.1 编号短码

- **M-DEC-001**（M + 短横 + DEC + 短横 + 三位数字）= 决策编号 / 唯一标识
- M-DEC-001 可作为 frontmatter `aliases` 字段值
- `[[M-DEC-001]]` 必须能跳转到真实文件（**通过 alias 实现**）
- 短码只用于引用 / 不作为完整文件名唯一依据 / 文件名仍带主题 + 版本 + 状态

### 4.2 完整文件名

建议格式：

```text
M-DEC-<NNN>_<主题>_v<版本>_<状态>.md
```

字段说明：

| 字段 | 含义 | 例 |
|---|---|---|
| M-DEC-NNN | 决策编号（三位数字 / 首位补 0）| M-DEC-001 / M-DEC-008 |
| 主题 | 中文业务主题 | 6组角色拆分判据 / 错品类货架重建SOP |
| v 版本 | 语义版本 | v0.1 / v0.5 / v1.0 |
| 状态 | 候选 / 验证 / 正式（晋级前不变 / 晋级后改）| 候选 / 验证 / 正式 |

### 4.3 下划线 vs 短横使用规则

- **M-DEC 编号内部用 `-`**：`M-DEC-008`（不写 M_DEC_008 / 不写 MDEC008）
- **完整文件名大字段间用 `_`**：`M-DEC-008_门店多商品调改SOP_v0.1_候选.md`
- **alias 用编号短码**：`M-DEC-008`
- **不把版本号或状态写进 alias**（因为 alias 应稳定 / 版本会升）

### 4.4 aliases 建议

frontmatter 示例：

```yaml
aliases:
  - M-DEC-008
```

如需多个 alias：

```yaml
aliases:
  - M-DEC-008
  - 门店多商品调改SOP
```

> **第一阶段建议**：**只加编号短码 alias**（避免 alias 污染 / 减少冲突风险 / 简化 lint）。第二阶段再视情况加主题 alias。

### 4.5 短码使用边界

| 场景 | 是否允许短码 |
|---|---|
| 项目文件 frontmatter `related` 字段 | ✅ 允许（前提：目标文件已加 alias）|
| 项目文件正文双链 | ✅ 允许（同上）|
| 主定义文件之间相互引用 | ⚠️ 推荐用完整名 / 避免依赖 alias 链 |
| 索引 / 看板 / 进度看板 | ✅ 允许（短码视觉清爽）|

## 7. LVS / L-V-S 字段规范现状

### 7.1 关键发现：vault 中无"LVS"主定义

**全库 grep**（5 个变体 / `LVS` / `L-V-S` / `L_V_S` / `L/V/S` / 等）→ **0 命中**。

**结论**：vault 中**不存在**"LVS"作为统一字段规范的主定义。用户提及的 "LVS / L-V-S" 可能：
- 是用户外部使用的术语 / 待 vault 化
- 是误词 / 实际指 P+L+G（命名规范 v0.3 §2 三轴）
- 是分散字段（L 品类 + V 版本 + S 空间/SKU）的合并提议

**本节不自创 LVS 定义**（参 §13.6 反幻觉硬约束）。仅做**审计 + 候选解释 + 待确认问题**。
