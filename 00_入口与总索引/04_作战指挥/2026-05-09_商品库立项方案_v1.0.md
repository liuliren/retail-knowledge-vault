# vault 商品库 v0.1 立项方案 v1.0

> **战略地位**：vault 数据飞轮 + 咨询业务核心壁垒 / Year 2-3 形成跨客户复用护城河。
>
> **立项触发**：5/9 用户提议 + Q1-Q5 决策 + Q2-B' 折中（采纳）。
>
> **核心定位**：晟果新零售咨询专属商品库 / 服务跨客户咨询交付。

---

## §1. 战略意义

```
vault Year 1（2026）：内核稳固
  - 6 大治理支柱建立
  - 1 客户战役（花厅坊）
  - V4.0 品类表（308 L4）

vault Year 2（2027）：数据飞轮 ⭐ 商品库核心
  - 商品库累计 50,000+ SKU
  - 4 层匹配算法 95%+ 准确度
  - 3-5 客户战役 / 标注反哺
  - 跨客户复用形成壁垒

vault Year 3（2028）：核心壁垒
  - 商品库 100,000+ SKU
  - 跨业态扩展（便利店 / 折扣店 / 生鲜店）
  - SaaS 化候选
```

→ **没商品库 = 每个客户从零开始 / 4 层算法天花板 70-80% / 无核心壁垒**
→ **建商品库 = 跨客户标注复用 / 数据飞轮 / Year 2-3 护城河**

## §3. 三层架构

```
Layer A: 核心 SKU 库（精确 / Year 2 50,000+ 目标）
  - 颗粒度：标准 SKU（条码 + 品名 + 品牌 + 容量 + 包装）
  - 起步（W26）：Open Food Facts 食品类 ~1,000-2,000 SKU
  - 长期（Year 2）：50,000+ SKU（采购 + 客户反哺）

Layer B: 品类骨架库（覆盖广 / 长尾兜底）⭐ 起步重点
  - 颗粒度：品类 + 品牌 + 关键参数
  - 起步（W24）：V4.0 308 L4 全覆盖（约 3,000-5,000 条目）

Layer C: 客户独有 SKU 库（长尾 / 人工标注）
  - 颗粒度：客户独有标识（条码 / 品名）
  - 起步（W25）：花厅坊 7,139 滞销 SKU + 在售 SKU 标注
  - 跨客户：每个战役 +500-2,000
```

## §8. 与 4 层匹配算法的关系（W27+ 立项）

```
商品库 v0.1（基础设施 / W19+W24-W26）
    ↓
4 层匹配算法（应用层 / W27+）
    ├── Layer 1 规则层（关键字 / brand_alias + package_unit）
    ├── Layer 2 模糊匹配（Levenshtein / TF-IDF）
    ├── Layer 3 LLM 语义（embedding / W30+ 启动）
    └── Layer 4 人工标注（反哺 Layer C）
    ↓
客户数据清洗模块（前端 / W30+）
    ↓
诊断 / 分析 / 咨询交付（后端）
```

→ 商品库是 4 层算法的物理基础 / 必须先建商品库 / 后开工算法。

## §9. 风险与权衡

### 9.1 风险登记

| # | 风险 | 严重度 | 缓解 |
|---|---|---|---|
| R-01 | OFF 中国食品 SKU 覆盖不足 | 中 | 起步 1,000-2,000 SKU 即可 / Layer C 反哺补充 |
| R-02 | Layer C 花厅坊客户独有 SKU 难标准化 | 中 | 仅整理 / 不强制标准化 / W27+ 算法时再清洗 |
| R-03 | 跨客户数据隐私 / 法律风险 | 中 | 商品库默认匿名化 / 仅保留 SKU + 品类信息 / 不保留客户销售数据 |
| R-04 | 数据飞轮启动慢（需 3-5 客户后显著）| 中 | 第 2 客户战役（破晓肇庆）开始受益 / 可接受 |
| R-05 | embedding 升级 W30+ 时 .csv 格式可能需重构 | 低 | W30+ 时再评估 / 起步阶段 .csv 兼容 |

### 9.2 权衡

```
不建：
  - 4 层算法天花板 70-80%
  - 跨客户无法复用
  - 每客户重头来
  - 无核心壁垒
  
建（Q2-B' 折中）：
  - 起步 4 天投入（不重）
  - W22+ 渐进 / 不冲突主战场
  - Year 2 数据飞轮启动
  - Year 3 形成核心壁垒
```

→ **强烈推荐建** / **Q2-B' 折中是最优解**。
