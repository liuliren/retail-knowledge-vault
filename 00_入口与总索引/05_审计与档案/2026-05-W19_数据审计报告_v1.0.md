## §2. 结论 1（客流 +38.3%）重推 / 重大订正 ⭐⭐

### 2.1 v2.2 原始推算

```
30T 客流 145,490 / 32 天 = 4,547 客户/天（v2.2.1 称"日均 4,850" 实际计算后 4,547）
60T 推算客流 ≈ 210,340 / 60 天 = 3,506 客户/天
变化 = (4,547 - 3,506) / 3,506 = +29.7%（v2.2 报告 +38.3% / 计算口径略不同）

[v2.2 错误]：当成"客流"（成交客户数）/ 推断"客流大盘 +30-40%"
```

### 2.2 重推（forward / 用户 5/9 第 2 次订正后）

```
145,490 = 32 天 客单数（订单数 / 不是客流）
[推断] 双倍统计 → 真实 32 天客单数 ≈ 72,745
日均客单数 ≈ 2,273 单/天

数学验证：
  日均销售 96,723 元 / 日均客单数 2,273 = 42.55 元/单
  ≈ 报表客单价 42.14 元（差 0.96%）⭐⭐ 完美闭合！

→ "客流 +38.3%" 实际是 "客单数 +30-40%"（不是客流）
→ 真实客流变化（进店人次 / 待启明告知计算路径）/ 暂不可知
```

### 2.3 倒推（reverse / 找出错在哪个环节）

```
出错环节定位：

1. 字段定义错认（最严重）⚠️
   - POS 报表里"客流"字段 ≠ 真实客流
   - 实际是"客单数"（订单数）/ 但报表叫"客流"
   - vault 没追溯字段定义 / 直接当成"客流"

2. 时段不一致（次严重）⚠️
   - 客流报表 4/7-5/8（32 天）
   - 销售报表 4/8-5/8（31 天）
   - 错开 1 天 / 差 3.2%
   - vault 当成"30 天"（实际 32 天 + 31 天混用）

3. 双倍统计未识别（中等）⚠️
   - [推断] 145,490 实际是 18 时段累加（每个客单可能被多个时段重复计数）
   - 或 60T 与 30T 计算口径不同
   - vault 没列 ≥ 3 种解释 + 数学验证

4. 客流 vs 客单数 vs 客单价 三层口径混淆（基础）⚠️
   - 客流 = 进店人次（独立客户）
   - 客单数 = POS 订单数（一笔 = 一张单）
   - 客单价 = 销售 / 客单数（POS 笔均）
   - vault 三层混用 / 算出"客单 × 客流 = 613 万 vs 实际 300 万"
```

### 关键教训

1. **字段定义优先**：报表"客流"字段 ≠ 真实客流（必须追溯字段语义）
2. **时段必须精确**：32 天 vs 31 天 vs 30 天 / 错 1 天差 3.2%
3. **行数 vs SKU 数**：去 header / 去小计
4. **客流 / 客单数 / 客单价 三层口径不可混用**
5. **数学验证必须**：客单 × 客流 = 销售 / 不通过则必有错
