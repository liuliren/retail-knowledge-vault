## ① 三批关系判断

**这不是三个平行模块，而是同一套「商品经营系统 v3.2 设计文档族」（00~13 编号：术语/诊断/分析/数学/字典/表结构/规则/页面/API）的两轮串联式「回灌精修」+ 一个空壳打包目录：**

1. **迭代链条（从文件头「基于：」字段实证）**：
   `v3.2 基线（第N轮精修版）` → **+地域风土时令增强**（另一批次 `_地域风土时令增强/`，4篇，不在本批）→ **+陈列与空间效率增强**（本批13篇，多份文件基于「地域风土时令增强回灌版」）→ **+活动与营销机制增强**（本批13篇，每份都基于「陈列与空间效率增强回灌版」）。
   即：**陈列批是活动批的直接底稿，活动批是整条链的最末端（最新版）**。
2. **两批结构完全同构**：同样的 00A/00B/00/01/05/06/08/09/10/11/12/13 编号，同样的「回灌结论→新增对象→新增字段→新增表→结论」八股结构。陈列批篇幅约为活动批 3 倍（陈列批 09/10 有完整字段口径与建表设计，活动批只有骨架清单）。
3. **`_v3.2回灌版资产/` 为空壳**（`02_03_04_07_最新回灌版_v3.2/`、`品类体系总包_最新回灌版_v3.2/` 两个子目录各只有 .DS_Store）。同名内容疑似存在于 `90_素材_全量/_1.1零售实践/1.1.3零售系统模型/` 下的 V-陈列回灌版 与 V_活动与营销板块回灌版 目录及两个 zip 包中（未拆读，属 `_1.1零售实践` 批次范围）。**建议进度表将此批标注为「空壳目录，0 实体文件」**，避免后续批次重复排队。
4. **性质判定**：这 26 篇是**面向软件开发的系统设计文档**（对象字典/字段口径/建表/API/前端清单/开发排期），不是零售方法论正文。零售判断密度低，绝大多数段落是「建议新增 campaign_xxx / display_xxx 对象」的命名工程。回灌来源方案正文（`活动与营销机制增强方案_v3.2.md`、`陈列与空间效率增强方案_3.2.md`）在 `90_素材_全量/` 根层，不在本批目录内，属大批次待读。

**整体星级：⭐⭐**（体系化程度高但已被现有 KB 在方法论层全面超越；残值集中在数据模型层，见③）。

## ③ ⭐回填候选（现有体系确有遗漏或未成型的部分）

> 按边界约定：只记候选，不下「应建新文件」结论。

### 候选1 ⭐⭐⭐ 陈列数据模型七表（陈列批 10 + 09，本批最完整的独有资产）
`fixture_master / display_slot_master / sku_display_mapping / display_snapshot / display_scoring_result / display_rule_hit_result / display_task_log` 七表设计，含完整字段口径（09 有 4 层字段分层：基础对象/经营维度/结果/动作，逐字段给类型、值域、归属表）与三条建表原则（主表与快照分开/映射与结果分开/陈列任务不脱离统一任务体系）。
**现有体系遗漏点**：06_陈列体系全部是方法论 + Excel 手工工具（DisplayMap手工绘制工具包），**没有任何正式的陈列数据 schema**。若未来把 DisplayMap/陈列诊断产品化（对接 /report-export、诊断卡系统、或给客户建陈列台账），这套七表 + 「display_snapshot 时间切片支持整改前后对比」的设计可直接作起点——尤其与现有《陈列整改前后对比卡》的对比需求天然契合。
**建议标注**：作为 KB-DISPLAY-PRODUCT-001（装配框架）或未来陈列工具开发的 reference 挂指针即可，不必现在编译。

### 候选2 ⭐⭐⭐ 「顾客任务路径断裂」作为独立陈列问题类型（陈列批 01/03/06）
`display_task_break`：早餐链/烹饪链/汤饮链/即食链没有通过陈列形成完整路径——含配套对象 `display_path_tag`（商品属于哪条任务路径）、`mission_display_role`（lead/support/bundle/reminder 任务表达角色）、四类任务模板的陈列增强判词（「家庭补货不只要齐，还要顺手拿齐」「早餐要让顾客10秒内想到并拿完」「即食最怕的不是缺货是看不见」）。
**现有体系遗漏点**：现行失真类型库（4类→v2.0 6类：结构过密/价格带遮蔽/SKU冗余/动销错配/行为驱动/品牌遮蔽）**全部从「分层结构投影」出发，没有「顾客任务路径完整性」这一维**；KB-CAT-CDT-001 管单个货架内的子格顺序，不管跨货架的任务路径连贯。这是 v3.2 稿里真正的视角增量。
**建议标注**：候选回填进失真类型库（作第7类「任务路径断裂型」）或分层到陈列一致性模型的开放问题区；`mission_display_role` 角色四分可作 M4 货位簇的补充视角。

### 候选3 ⭐⭐ 单位面产出指标族 + 陈列错配损失估算（陈列批 05/09）
`sales_per_face / gp_per_face / units_per_face / sales_per_display_slot / display_efficiency_index / display_misallocation_loss（错配损失金额估算）`。
**现有体系遗漏点**：KB-CAT-DX-MISMATCH-001 用「销售占比/陈列占比」**比值**定位错配（相对量），KB-SPACE-ARCH-001 是坪效/格效双分母（面积级）；「以面（facing）为分母的绝对产出指标」和「把错配折成损失金额」在现有指标字典（KB-METRIC-DICT v0.3 五组33项）中未见对应条目。对客户沟通时「这个错配一年损失约 X 元」比比值更有说服力。
**建议标注**：候选提交 KB-METRIC-DICT-001 v0.3 的 5 项候补区评估；损失估算法需先解决「应得面数基准」的标定问题（v3.2 稿自己也没给算法，只有字段名）。

### 候选4 ⭐⭐ 活动—陈列承接的量化接口（活动批 00/05/12）
`campaign_display_support`（活动是否/如何被陈列承接）+ `campaign_display_support_index`（承接指数）+ 页面「活动与陈列承接区」（活动流量是否被端架/堆头接住、承接是否断层）。判词「活动与陈列不是两张皮，活动要起量必须有陈列承接动作」。
**现有体系遗漏点**：KB-EVENT-ACTIVATION-001 §3 要素⑤只有定性要求（「陈列没跟上效果折损50%+」），**没有活动后复盘时验收「陈列承接是否到位」的检查项**——ROI<1 归因第2类「陈列/传播缺位」有名目无检法。
**建议标注**：候选补进 KB-EVENT-ACTIVATION-001 §4 归因2 的检查清单（活动期目标SKU是否实际占据端架/堆头/高价值位，可巡店拍照核验），无需引入指数计算。

### 候选5 ⭐ 活动复盘表与负面清单的表结构预留（活动批 10）
`campaign_master → campaign_result_snapshot → campaign_review_log` 三表链。KB-EVENT §8 第7项「活动 ROI 追踪表模板（待建）」正好缺一个字段起点，v3.2 的字段清单（09 字段组C/D）可作该模板的初稿参考。价值有限（字段名堆砌），仅作指针。
