## §2. 4 个错误的失败模式分析

### 2.1 共同根因（IBM 5 Whys 分析）

```
为什么 vault 会出错？
  → 把推断/假设/类比 → 写成事实

为什么会写成事实？
  → 缺乏强制标注机制（frontmatter fact_layer 之前不存在）

为什么没有强制标注机制？
  → §13 反幻觉规则只是文字声明 / 不是数据契约 / 不是 lint 强制

为什么只是文字声明？
  → 之前 vault 是 reactive 防错（用户纠偏 → memory 沉淀）/ 不是 proactive 拦截

为什么是 reactive？
  → vault v1.0-v2.0 早期 LLM 协作模式 / 没系统化设计 Constitutional 自检
```

→ **根本原因**：缺乏**事实 vs 推断**的强制标注 + **客户视角防火墙** + **Constitutional 自检**。

### 2.2 4 个错误的具体失败模式

| 错误 | 失败模式 | 错误链长度 | 用户纠偏方式 |
|---|---|---|---|
| #1 启明=店长股东 | 用户最早表述 → vault 推断 → 写成事实 | 1 步 | 直接订正 |
| #2 家族压制 | 推断 → 套西方剧本 → 写成事实 | 2 步 | 实质纠偏（不是单纯订正信息 / 还纠偏框架）|
| #3 Sam→Rob 夺权 | 类比 → 戏剧化 → 套客户身上 | 3 步 | 同 #2 / 框架级纠偏 |
| #4 启明导出 89 .xls | 文件 mtime → 推断操作人 → 写成事实 | 2 步 | 视角级纠偏（"我是六哥 / 启明是我的客户"）|

错误链越长 / 纠偏越难 / vault 越不容易自检。

### 2.3 为什么用户纠偏后 vault 还会再犯？（错误 #4）

错误 #1-3 用户在同一个 thread 里纠偏后 / vault 知道"启明 = 老板（小股东）"。但 #4 仍然犯了"启明导出"的错误。原因：

[推断] vault 在不同**抽象层级**用同一个名字 / 没区分：
- "启明"作为**客户身份**（老板 / 小股东 / 二代 / 启明本人）
- "启明侧"作为**客户方主体**（区别于"顾问方"）
- "启明门店"作为**客户企业实体**（花厅坊店）

§13.16 客户视角防火墙的真意 = vault 永远不假设客户内部具体操作 / 即使谈话主体就是客户老板本人。

## §3. 4 层治理底盘的设计推导

### 3.1 为什么需要 4 层而不是 1 层？

单层规则容易被绕过。4 层互相强化 = **纵深防御**（IBM Defense in Depth）：

```
Layer 1: 数据契约（frontmatter fact_layer）        ← 写文档时 enforce
Layer 2: 规则升级（§13.16-19）                     ← 知识层 / 顾问培训
Layer 3: lint 自动扫（G03 v0.2）                  ← 周一定期 enforce
Layer 4: Constitutional 自检（每次 vault 写入前）  ← 实时 enforce
```

任何一层失守 / 其他三层兜底。

### 3.2 为什么 frontmatter fact_layer 是关键？

LLM 自然倾向于"流畅叙述"而不是"分层标注"。frontmatter 强制 4 元字段（observed / client_told / inferred / pending）= **数据契约**：每个事实陈述必须分配到 4 个子字段之一 / 不允许空挂。

→ 这是从 Oracle/IBM 的"schema 强制" 借鉴 / 把数据库治理思想引入 vault 文档治理。

### 3.3 为什么客户视角防火墙独立成 §13.16？

中国家族企业咨询场景下 / "客户内部具体谁操作"是**反复出错的 #1 模式**。把它抽象为独立硬约束 / 让 vault 在每次输出前自动 self-check"我是不是在假定客户内部分工？"。

→ 这是从 Anthropic Constitutional AI 借鉴 / 让 LLM 自己 monitor 自己。

### 3.4 为什么用户订正优先（§13.19）单列？

LLM 长会话有 context decay / 偏向使用早期表述（最早的更容易被记住）。§13.19 强制"订正优先"= 防止 vault 用过时事实 / 即使最早的表述印象更深。

→ 这是从软件开发"latest commit wins"借鉴 / 但保留 audit-log 不删（IBM Audit Trail）。

## §4. 升级后的预期效果

### 4.1 量化预期（[推断]）

- 重复同类错误率：估计降 60-80%
- 错误检测时间：从"用户事后纠偏"提前到"vault 写入前自检"
- 用户纠偏次数：估计每会话从 4-5 次降到 0-1 次（同类问题）

### 4.2 不可解决的部分

- LLM 仍会犯**新类型**错误（4 层治理是针对**已识别**的失败模式）
- 用户纠偏循环不可消除（这是 LLM 协作的本质 / Anthropic HHH "Helpful" 内含此 trade-off）
- 4 层治理本身需要**持续维护**（G03 lint 规则要随新错误模式扩展）

→ vault v3.0 候选：基于这次事件 / 进一步升级到 Karpathy "self-improving" 模式（vault 自动发现错误模式 / 自动 propose 新硬约束）。

## §6. 给未来 vault 维护者的话

如果你（未来的 vault 维护者 / 不一定是 LLM 也可能是新接手的人）读到这个文件：

1. **§13.16-19 是用真实错误换来的硬约束**，不是凭空写的规则。**不要轻易放松**。
2. 错误链越长 / 越难发现。**始终从最短链路自检开始**：写下任何客户事实陈述 / 先问"这是 observed / client_told / inferred / pending 的哪一种？"
3. **客户视角防火墙永远优先**：vault 不在客户内部 / 永远只能写门店主体 / 除非客户亲口告知。
4. **类比是双刃剑**：保留结构同构有价值 / 戏剧化情节有害 / 用类比时永远明示这是类比。
5. **用户最早表述 < 后续订正**。如果你在长会话里发现 vault 在用旧版事实 / 立刻订正 / 不要等用户提醒。
6. **本复盘本身就是 vault 自我学习的范本**：下次发生类似事件 / 也写一份 AAR / 让 vault 持续进化。
