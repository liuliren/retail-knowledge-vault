# 晟果新零售知识库全量资产与自动化成熟度审计 v0.1

## 0. 执行结论

当前知识库已经具备较强的目录治理、方法论主干、客户数据安全规则和单点自动化工具，但尚未形成稳定的“知识编译流水线”。

核心问题不是简单的文件数量过多，而是以下四个断层：

1. **输入很多，编译不足**：Clippings、AI 输出、执行日志、案例过程件持续增加，但没有统一的归位时限和晋级规则。
2. **规则很多，运行不足**：已有 7 环生命周期、状态机、G03 Lint、任务分发器等设计，但多数仍依赖人工触发。
3. **索引存在，更新不同步**：主定义索引、知识库总索引、知识树 MOC 分工已明确，但总索引陈旧，MOC 的 Dataview 自动列件未实际启用。
4. **治理能力强于可观测能力**：规范和审计文档较完整，但缺少机器可读资产台账、增量差异报告、自动告警和闭环回写。

综合成熟度评估为 **5.9/10，处于 L2“可重复执行、局部半自动”阶段**。下一阶段不应大规模重构目录，而应优先把现有规则变成可持续运行的自动化闭环。

### 3.2 与 G03 Lint 的差异

G03 Lint 报告当前显示：

- 断链目标：364
- 孤儿文件：11
- active 无签字：230
- 缺字段文件：196
- 版本不一致：60
- 敏感红线：0

G03 的“孤儿 11”使用正式目录和入链规则，属于治理口径；本报告的“完全独立 78”使用全活跃区入链或出链规则，属于资产口径。两者不冲突：

- **11 个**是应优先治理的正式知识孤儿。
- **78 个**反映 AI 输出区、运行时文档和原料区的系统接入程度。

### 3.3 根因判断

独立文件的主要原因不是缺少 wikilink 技巧，而是缺少“文件出生时必须选择父级”的硬规则。当前大量文件先生成，再等待人工决定是否进入主链，导致 AI 输出区逐渐成为第二个 Inbox。

## 4. 元数据与生命周期审计

### 4.1 元数据现状

活跃扫描区 616 个 Markdown 中：

- 93 个缺少明确 `status`。
- 175 个缺少 `source_type`。
- 188 个缺少 `client_safety`。
- `source_type` 实际出现超过 80 种写法，枚举已经失控。
- `client_safety` 同时存在 `internal_only`、`internal`、`confidential` 等近义口径。

核心方法论和治理区的 frontmatter 完整度较高；主要缺口集中在 AI 协作中枢、运行时文档、旧模板和部分案例过程件。

### 4.2 状态分布

| 状态 | 数量 | 判断 |
|---|---:|---|
| active | 266 | 其中 230 个未满足现行签字口径 |
| draft | 193 | 草稿池较大，缺少 aging 管理 |
| review/candidate 等 | 48 | 状态名称分散，难以自动流转 |
| 缺失 | 93 | 无法进入状态机 |

### 4.3 生命周期问题

- 状态值过多，自动化无法可靠识别“下一步是什么”。
- `active` 历史使用口径与新签字机制不一致，造成大量治理债。
- AI 输出、执行日志、审计报告和正式方法论共用相似 frontmatter，但生命周期不同。
- 存在 60 个文件名版本与 frontmatter 版本不一致，说明“内容版本”和“文件路径稳定性”尚未完全解耦。

## 5. 导航与单一真相源审计

### 5.1 已有正确设计

当前三导航分工是合理的：

1. `核心概念主定义索引`：概念 SSOT。
2. `知识库总索引`：全库重要文件台账。
3. `科学零售知识树 MOC`：方法论学习路径和全景树。

现有 L0-L8 层级和 Ingest→Compile→Link→Lint→Distill→Publish→Feedback 七环也应继续作为主框架，不建议再建立第四套知识架构。

### 5.2 当前缺口

- `知识库总索引` 更新时间仍停留在 2026-05-04，多个目录标记“待补充”，已经不能代表现有资产。
- MOC 中的 Dataview 查询已写入，但当前社区插件仅安装 `agent-dashboard`，未安装 Dataview，因此“新增文件自动进树”尚未生效。
- 23 个文件密集目录没有本级 README，其中包括花厅坊项目总览、商品诊断、AI 输出区、执行日志和审计档案。
- 断链 364 个目标中，既有真正失效链接，也有前瞻链接、历史别名和归档映射，尚未形成机器可识别的例外清单。

## 6. 重复、归档与存储审计

### 6.1 精确重复

公开区精确哈希扫描发现：

| 指标 | 数量 |
|---|---:|
| 精确重复组 | 507 |
| 多余副本数 | 596 |
| 理论重复容量 | 约 129.3 MB |

重复主要集中在：

- 多轮历史归档快照之间。
- 归档与活跃门店原始材料之间。
- 早期素材包与专题归档包之间。

### 6.2 裁决

当前重复量只占总容量约 4%，不是首要瓶颈。由于大量重复属于原始证据、封存快照或客户项目材料，本轮不建议做物理去重。

正确做法是先建立“归档清单 + 哈希 + 保留理由”，未来只处理已确认的归档内部重复，不触碰活跃项目和原始数据。

**综合评分：5.9/10，L2。**

### 7.3 已经真实可运行的自动化

- G03 Lint v2：可人工一键扫描并生成仪表盘。
- POS 清洗和零售工具：具备 dry-run/execute 分离和部分单测。
- Codex 每日收件箱检查：在 Vault 外部自动化中运行。
- Git 客户数据保密闸：已有明确提交前检查规则。

### 7.4 仍属于“文档自动化”的部分

- `SYSTEM_RUNTIME/auto_scanner.md`、`dispatcher.md`、`event_router.md`、`evolution_engine.md` 是架构说明，不是持续运行的程序。
- “任何任务写入即触发执行”当前没有稳定的文件监听、队列状态和幂等控制。
- 失败重试、任务锁、重复执行防护、超时告警和状态回写没有统一实现。
- Vault 健康仪表部分指标仍依赖人工清点，且最后主要基线停留在 5 月。

## 8. 当前主要风险

### P0：必须先控住

1. **工作区变更卫生风险**：审计时 Git 工作区存在约 238 个修改、1 个删除、12 个未跟踪项。自动化若直接写入或提交，极易混入非本轮文件。
2. **AI 输出区堆积风险**：63 个完全独立文件集中在 AI 协作中枢，输出区正在承担临时区、日志区、候选区和正式成果区四种职责。
3. **自动化事实与文档状态不一致**：CLAUDE.md 仍把部分已实现能力标记为 in_progress/planned，运行事实与治理状态可能漂移。

### P1：影响调用效率

1. 总索引陈旧，找文件依赖搜索而非台账。
2. 断链目标 364 个，真实错误、前瞻链接和历史映射混在一起。
3. `source_type` 和 `client_safety` 枚举不收敛，自动分类困难。
4. 23 个高密度目录无 README，目录级上下文缺失。
5. Dataview 自动列件未启用，MOC 仍需人工维护。

### P2：中长期成本

1. 归档快照重复较多，但暂未建立可安全去重的保留策略。
2. 工具脚本缺统一 `pyproject`、测试入口、运行清单和结果登记。
3. 业务目录内容严重不均衡，部分目录是规划结构而非真实能力。
4. 健康仪表、任务队列、治理债务队列存在多处重复状态维护。

## 9. 建议采用的管理方式

## 9.1 总原则：一个架构，三套控制面

保留现有 L0-L8 和七环生命周期，不再新造知识分类体系。在此基础上建立三套控制面：

### A. 语义控制面

负责“知识是什么、权威在哪里”：

- 概念唯一主定义：`核心概念主定义索引`。
- 方法论学习路径：`科学零售知识树 MOC`。
- 文件资产台账：`知识库总索引` + 自动生成资产注册表。
- 案例只验证方法，不重定义方法。
- AI 输出不能直接成为主定义。

### B. 运行控制面

负责“任务怎么流转”：

- Inbox：只收任务，不存成果。
- Output：只存待裁决成果，最长保留 7 天未归位状态。
- Queue：只记录状态和责任人，不重复存任务正文。
- Log：只记录执行事实，不承载新规则。
- Runtime：必须区分“架构说明”和“真实可执行程序”。

### C. 质量与安全控制面

负责“什么能进入正式库和 Git”：

- G03 Lint：链接、孤儿、Schema、状态、版本、敏感、来源和替代关系。
- SENSITIVE-GOV：客户数据和派生结论隔离。
- Git 提交前闸：只允许精确 staged 文件通过。
- 发布签字门：candidate/draft 不能自动升 active。

## 9.2 六类资产区

| 资产区 | 对应位置 | 管理原则 |
|---|---|---|
| 原始源料区 | `99_原始素材`、项目 `99_原始素材` | 不可变、只读、带哈希和来源 |
| 外部输入区 | `Clippings` | 只作为来源，不直接进入主链 |
| 待编译区 | AI 输出区、90 暂存 | 必须有归位状态和到期时间 |
| 正式知识区 | `00`、`01`、`02-08` | SSOT、Schema、链接、签字受控 |
| 执行资产区 | `10`、`13` | 模板/工具需版本、测试、调用说明 |
| 证据与档案区 | `09`、`15`、`16`、`99_归档` | 案例证据与历史冻结，不反向污染主定义 |

## 9.3 文件关联硬规则

所有新建正式 Markdown 必须同时满足：

1. 有且只有一个父级入口：README、MOC、主定义索引或项目总览之一。
2. 至少有一个结构性上游链接和一个明确下游/应用关系；纯导航页可豁免。
3. frontmatter 使用收敛后的必填字段。
4. AI 输出在 7 天内完成四选一裁决：`promote`、`merge`、`hold`、`archive_candidate`。
5. 未裁决文件不得进入正式方法论目录。

## 9.4 最小 Schema

建议自动化只强制以下 10 个字段：

```yaml
title:
version:
status:
owner:
created:
updated:
module:
source_type:
client_safety:
summary:
```

状态仅保留：

`draft → candidate → active → deprecated → archived`

特殊运行状态 `blocked/failed` 作为 `execution_status`，不再混入知识生命周期 `status`。

`source_type` 建议收敛为：

`methodology / decision_rule / case_study / product_definition / template / tool / index / governance / audit / execution_log / source / archive`

旧值通过别名映射兼容，不做一次性批量重写。

## 10. 自动化目标架构

```text
文件进入
  ↓
资产扫描器：路径/类型/哈希/敏感级/父级
  ↓
知识编译器：Schema校验/链接解析/状态识别
  ↓
G03质量门：断链/孤儿/版本/签字/敏感/重复
  ↓
裁决队列：promote / merge / hold / archive_candidate
  ↓
索引生成器：更新MOC/资产台账/增量报告
  ↓
人工签字门
  ↓
正式知识或执行资产
  ↓
案例反馈与M-DEC蒸馏
```

自动化必须遵守：

- 自动发现，不自动改正文。
- 自动生成建议，不自动移动或删除。
- 自动阻断敏感内容，不自动脱敏。
- 自动更新机器生成区，不覆盖人工主定义。
- 所有自动动作可重跑、幂等、可追溯。

## 11. 分阶段优化路线

### Phase 0：止血与建立真实基线（1 周）

1. 将当前大量 Git 工作区变更先做归属裁决，恢复“一个任务一个 staged 集合”。
2. 刷新 CLAUDE.md 中 G03 等能力的实际状态，消除“已实现但仍写 planned”的事实漂移。

4. 建立 AI 输出区 7 天归位 SLA，但本阶段不批量移动历史文件。
5. 将本报告设为本轮审计基线。

### Phase 1：机器可读资产台账（2-3 周）

1. 扩展 G03，额外输出 JSON/CSV 资产清单和增量差异。
2. 统一统计所有 Markdown 的父级、状态、类型、密级、入链、出链、更新时间。
3. 将 364 个断链目标分为：真实断链、前瞻链接、历史别名、归档映射、外部引用。
4. 收敛 `status/source_type/client_safety` 枚举，先对新文件强制，旧文件按触发编辑逐步修正。
5. 刷新 `知识库总索引`，避免继续人工维护全量文件清单。

### Phase 2：自动导航与质量门（3-6 周）

1. 经确认后启用 Dataview，激活 MOC 自动列件和状态看板。
2. 增加 pre-commit 只读检查：Schema、敏感、版本、未审查文件和 staged 范围。
3. 将每日自动化从“看收件箱”升级为“看收件箱 + 看资产增量 + 看 G03 新增问题”。
4. 建立每周增量仪表，只报告本周新增问题，不重复输出全部历史债务。
5. 为工具目录建立统一测试入口、依赖清单和执行器注册。

### Phase 3：任务与知识闭环（6-12 周）

1. 将 SYSTEM_RUNTIME 从 Markdown 规格升级为真实队列状态机。
2. 每个任务拥有唯一 ID、输入、输出、状态、责任人、最后更新时间和幂等键。
3. AI 输出完成后自动进入裁决队列，但正式晋级必须人工签字。
4. 将案例反馈、客户验证日志和 M-DEC 晋级条件接入仪表。
5. 第二客户接入后，再评估跨客户知识蒸馏和语义检索，不提前建设复杂向量系统。

## 14. 最终建议

优先顺序应是：

1. **先建立机器可读资产台账和增量差异，不先整理目录。**
2. **先治理 AI 输出区和任务闭环，不先清理 99_归档。**
3. **先收敛 Schema 和状态枚举，不批量改旧文件正文。**
4. **先把 G03 接入每日/每周自动化，不立即建设复杂知识图谱。**
5. **先解决 staged/工作区卫生，再扩大自动写入范围。**

这套知识库已经具备成为晟果新零售“方法论、交付、案例、工具、AI 协作”统一操作系统的基础。下一阶段的关键不再是继续增加规范，而是把现有规范编译成稳定、可观察、可阻断、可回写的运行系统。
