## 2. 顶层设计原则（7 条 / 已拍板）

1. **知识库模块与项目实施模块逻辑分层**：00_/01_-08_/10_-15_ 是知识库（跨客户 / 静态 / 方法论 + 模板 + 内容 + 工具 + 外研 + 个人）；09_/16_/90_ 是项目实施（客户级 / 动态 / 业务 + 治理 + 缓冲）；99_ 归档。
2. **09 管业务实施 / 16 管战役治理**（Q1=A）：09_ = 客户、门店、诊断、调改、复盘、交付物过程；16_ = 章程、M-DEC、W 周期推进、阶段纪要、风险边界、战役决策。**不合并、不重叠**。
3. **SOP 主定义必须独立沉淀**：抽 10_/02_交付SOP/ 主库 + 双链反向引用。day-file 不承载 SOP 主定义。
4. **项目文件记录事实过程，不承载方法论主定义**：09_/16_ 文件可引用方法论，但方法论本体住 01_-08_ + 10_。
5. **M-DEC 先战役级验证，再跨客户晋级**（Q4=A）：v0.1-v0.5 候选 → 16_/<战役>/_M-DEC决策库/；v1.0 跨战役验证 → 04_中层决策库/ 或 10_/02_交付SOP/。
6. **先逻辑重构，后物理迁移**：本设计稿先把规则定好；小范围验证（方便速食 + SOP 第一批）跑通后；W22+ 再分阶段批量迁移。
7. **5/30 G05 前不做全库大迁移**（Q6=B）：花厅坊战役进行中 / 5/13 现场临近 / 5/30 G05 是合理治理时点 / 不抢业务节奏。

**双向规则**：
- **知识库 → 项目实施**：项目文件**引用**方法论 / SOP / 模板（不得复制粘贴 / 不得改主定义）。
- **项目实施 → 知识库**：业务实战经验通过 M-DEC v0.5 → v1.0 跨战役验证后**晋级**到 04_/中层决策库 或 10_/02_交付SOP。
- **三大禁止**：① 方法论主定义不得住 09_/16_ ② SOP 主定义不得住 day-file ③ 跨客户内容不得住客户子目录。

### 3.3 文件归属判断树（5 步 / v0.5 新增）

> **使用场景**：写新文件前 / 整理旧文件归位前 / 决定 mv 路径前 / Lint 扫描可疑文件时。
>
> **顺序执行 / 不跳步**：从 Step 1 开始 / 第一个 Yes 就锁定目标目录 / 不再进下一步。

```
Step 1：是临时素材 / 待整理 / 未判定？
   YES → 90_素材暂存与待整理/
   NO  → 进 Step 2

Step 2：是已失效 / 旧版本 / 历史归档？
   YES → 当前模块下 _历史/  或  99_归档/（跨模块整体归档时）
   NO  → 进 Step 3

Step 3：是跨客户内容（不含具体客户/门店/SKU/客户原话/客户数据）？
   YES → 看内容性质：
         ├─ 科学零售理论方法（总纲 / 商品力 / 品类管理 / 52周MD / 等） → 01_-08_
         ├─ SOP / 标准模板 / 可复用作业流程（含未跨客户验证 v0.5 候选） → 10_咨询交付模板/02_交付SOP/
         ├─ 系统 PRD / 字段字典 / 实体关系                            → 11_系统产品与PRD/
         ├─ 内容 / 文案 / 视频选题 / 自媒体素材                        → 12_自媒体内容与表达转化/
         ├─ 数据脚本 / 清洗规则 / 工具配置                            → 13_数据分析与工具脚本/
         ├─ 行业研究 / 外部案例 / 竞品研究 / 业态研究                  → 14_外部案例与行业研究/
         └─ 个人成长 / 客户验证日志 / 刻意练习                        → 15_刻意练习与成长/
   NO  → 进 Step 4

Step 4（客户级）：是业务事实 / 过程 / 还是战役治理？
   业务事实/过程（数据 / 诊断 / 调改 / 复盘 / 交付物 / 现场操作 / SKU 清单 / 客户原话） → 09_门店案例与项目复盘/
   战役治理（章程 / 周报 / M-DEC 候选 / 风险登记 / 决策记录 / 阶段纪要 / 战役决策）   → 16_客户与战役档案/

Step 5（已确定 09 或 16）：往下细分到子模块
   09_：客户 → 门店 → 6 大阶段（00 总览 / 01 诊断 / 02 调改 / 03 后期 / 04 交付 / 05 复盘）→ 子组（子组_X / 6 步）
   16_：战役 → 8 子目录（00 章程 / 01 周报 / 02 风险 / 03 决策 / _M-DEC决策库 / _战役实战 / _阶段纪要 / _历史）
```

#### 3.3.1 边界与歧义判定（4 条硬规则）

1. **业务事实 vs 战役治理判别**：含"具体 SKU 调整"/"具体陈列动作"/"客户原话录入"/"现场拍摄数据"= **业务事实** → 09_。含"周计划"/"风险登记"/"决策依据"/"M-DEC 候选"/"战役章程修订"= **治理** → 16_。**两者都含 = 拆 2 文件 / 不混存**。
2. **跨客户 vs 客户级判别**：文件正文含具体客户名 / 门店名 / SKU / 客户原话 = **客户级** → 09/16；正文不含任何客户信息 / 可直接复制给第二客户用 = **跨客户** → 01-08/10-15。**含但脱敏可去 = 跨客户**。
3. **SOP 候选 vs 主定义**：未跨客户验证的 SOP（v0.5）→ 10_/02_交付SOP/_候选/；跨战役验证后（v1.0）→ 10_/02_交付SOP/。**不混存**。
4. **M-DEC 候选 vs 晋级版**：v0.1-v0.5 候选 → 16_/<战役>/_M-DEC决策库/；v1.0 跨战役验证 → 04_商品诊断与商品力提升/中层决策库/。**不跨级**。

### 3.4 不重复建文件 / 6 步查找规则（v0.5 新增）

> **触发场景**：写新文件前先 grep + 走判断树 / 防止重复发明 / 防止跨目录重复沉淀。

```
Step 1：找入口
   ↓ 进度看板（如 [[00_花厅坊调改进度看板]]）/ vault README / 子组 README / 战役 README
Step 2：找客户
   ↓ 09_/<客户>/  或  16_/<战役>/
Step 3：找门店（仅 09_）
   ↓ 09_/<客户>/<门店>/
Step 4：找子组
   ↓ 09_/<客户>/<门店>/<阶段>/子组_<X>/
Step 5：找阶段
   ↓ 子组_<X>/01_数据分析 / 02_TPOS_CDT / 03_商品分析与去留判断 / 04_表格与清单输出 / 05_执行动作 / 06_复盘/
Step 6：找不到才新建
   ↓ 用模板 + frontmatter 6 强制字段 + 双链反向引用主定义
```

#### 3.4.1 4 条新建前硬约束

1. **grep 强制**：写新文件前必先 `grep -r "<关键词>"` 全 vault / 命中相似文件 → 评估是否升级 / 而非新建。
2. **判断树先走**：先走 §3.3 判断树定 / 路径 / 再决定文件名 / 再决定是否新建。
3. **同一概念主定义只能一份**：参 [[CLAUDE.md]] §14.8 / 如发现已有主定义 / 加双链引用 / 不复制粘贴。
4. **新建必带 6 强制 frontmatter**：quadrant / project / client / store / phase / business_module（参 §8.1）/ 缺一不准 commit。
