## 3. 为什么不全量改历史文件

### 3.1 5 条原因

1. **历史文件是历史事实**：5/13-5/19 期间生成的命名规范 / 复盘报告 / 客户验证日志 = 当时口径 / 不应回溯篡改
2. **commit history 是 audit trail**：`CLAUDE.md` §17.X §17.12 工程操作纪律 / 不破坏历史 / Git 内 IBM Audit Trail 纪律
3. **全量 sed 风险**：38 文档 / 大量交叉引用 / 全 replace 易引入语义错位（如某些"HS"实际是其他缩写）
4. **历史可读性 > 历史一致性**：历史文档可读懂 / 通过 alias 映射表理解即可 / 不需完全一致
5. **新代号从本轮起生效**：未来所有新文件用 XS / LY / 老文件渐进式过期

### 3.2 反例（不做）

- ❌ 全 vault `sed -i 's/HS-/XS-/g'`
- ❌ 重写 git history 把所有 commit 中的 HS 改为 XS
- ❌ 把 5/13 命名规范冲突清单中的 HS 也改为 XS（那是当时讨论 HS 的清单 / 不能改）

| **MLY** | **LY** | 米面粮油 | deprecated_alias | 历史文件可读 / **新文件禁用** |

### 4.2 含义说明

- **deprecated_alias**：已废弃别名 / 保留只读权 / 写入禁用

- **新文件禁用**：5/19（含 5/19）起生成的所有新文件 / **必须**使用 XS / LY / 不得继续用 HS / MLY

## 6. 用户本地照片处理建议（vault 端 rename map v0.2 草案）

### 6.1 处理原则

- ✅ vault 端**不物理改名** / 仅生成 rename map v0.2 建议段
- ✅ 用户本地手动改 / 或写 shell 脚本批量改
- ✅ 改完后告知 vault 端 / W22 入库升级时一并 audit
- ❌ vault 端不触达用户本地照片（vault git 排除 99_原始素材）

### 6.2 用户本地照片批量改名建议（shell 脚本 / 用户参考）

```bash
# 假设照片在 /用户本地/2026-05-19/
cd /用户本地/2026-05-19/

# HS- → XS-（86 张主货架 + 10 张端架）
for f in HS-*; do
  mv "$f" "${f/HS-/XS-}"
done

# 验证
ls XS-* | wc -l   # 应为 ~96
```

⚠️ **强烈建议**先用 `for f in HS-*; do echo "${f/HS-/XS-}"; done` dry-run 验证 / 再执行真改。

### 6.3 照片改完后

- 告知 vault 端（微信 / 直接 commit）
- vault 端在 rename map 草案中标 ✅ 实际改名完成日期
- W22 入库升级（5 步 SOP §19）时一并 audit trail 落档

## 8. 历史文件读取规则

### 8.1 历史文件标识

| 标识 | 含义 |
|---|---|
| 文件名含 `HS-T06` / `HS-G4-A` / `HS-A03` 等 | 历史口径 / 实际是 XS-T06 / XS-G4-A / XS-A03 |
| 文件名含 `MLY` | 历史口径 / 实际是 LY |
| 文件内含 `[[HS-T06-...]]` 等双链 | 引用历史命名 / 按 alias 映射理解 |

### 8.2 读取流程

1. 看到 HS / MLY 不要立即认为是错
2. 查 [[2026-05-19_晟果新零售陈列资源命名治理规范_v1.0]] §6.3 alias 映射表
3. 按 HS→XS / MLY→LY 理解
4. 如需引用 / 在新文件中用新代号 + 标"原文档使用历史代号 HS / MLY / 现已订正为 XS / LY"

### 8.3 历史文件引用格式（标准）

```
本文件引用 [[2026-05-23_5月19日86张照片入库audit_v0.1]]（原文使用历史代号 HS- / 现已订正为 XS- / 参 [[2026-05-19_区域代号HS到XS_MLY到LY变更审计_v0.1]]）
```
