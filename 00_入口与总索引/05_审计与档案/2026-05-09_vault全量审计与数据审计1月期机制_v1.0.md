## §3. 数据审计方法（POS+ERP 100% 准确假设）⭐⭐⭐

### 3.1 核心假设

```
✅ POS+ERP 数据 100% 准确（用户 5/9 告知 / 撇开品类管理机制问题不谈）
❌ 我们的分析问题在我们这一侧（算法 / 时段口径 / 字段定义 / 行数 vs SKU 数）
```

### 3.2 数据审计方法 = 重推 + 倒推 + 链路追溯

#### A. 重推（forward-tracing）

```
原始数据（POS .xls / ERP .xlsx）
    ↓
[重推每一步]
    ↓
我们的分析结论（v2.2 报告 / 算法 / 6 表）

每一步检查：
  - 字段定义是否正确？
  - 时段是否一致？
  - 行数 vs SKU 数？
  - 计算公式是否正确？
  - 单位是否混淆？
```

#### B. 倒推（reverse-tracing）

```
我们的分析结论（v2.2 报告 / 算法）
    ↓
[倒推每一步]
    ↓
原始数据（POS .xls / ERP .xlsx）

每个数字检查：
  - 来自哪个文件？
  - 哪个 sheet？
  - 哪几行？
  - 时段定义？
  - 与其他独立数据交叉验证？
```

#### C. 链路追溯（end-to-end audit trail）

```
每个数字 = 数据来源 + 计算公式 + 时段 + 字段定义 + evidence
不允许"我记得是 X"或"应该差不多"
```


```
  1. 全量扫描 vault 核心文件（70+）
```

### 5.2 G08：vault 数据治理 SOP（Oracle ACID 纪律）

```
触发：每次新数据进 vault（POS / ERP / 客户提供）
流程：
  1. 数据来源 evidence 标注（文件 + sheet + 行号 + 时段 + 字段）
  2. frontmatter `data_source` 字段强制
  3. 4 层治理过 §13.16-20（推断标注 + 客户视角 + 类比 + 订正 + 一致性）
  4. 数学验证（重推 + 倒推 + 交叉验证）
  5. 不一致时列 ≥ 3 种解释 + 用户 confirm
  6. 沉淀至 [[数据一致性追溯案例库]]

落地：90_素材暂存 → Ingest（G01）→ 正式目录
```
