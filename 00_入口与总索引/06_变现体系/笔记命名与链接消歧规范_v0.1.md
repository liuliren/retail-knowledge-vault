# 笔记命名与链接消歧规范 v0.1

> **结论先行**：Obsidian 裸 `[[基名]]` 在重名时会**链到错的页**——这是变现链路最隐蔽的断点。
> 本规范立 5 条命名/链接红线 + 3 类同名处置法，目标：**重名可控、链接不歧义、不再新增断链**。
> 与既有命名规范**互补不冲突**（[[2026-05-13_命名规范总纲_v0.1]] 管编号/版本/ID；本页专管「重名与链接消歧」这一窄而高频的痛点）。
>
> 上联枢纽：[[晟果新零售变现体系总控_MOC]]｜配合：[[链接债治理总表]]（C 级 stub / B 级路径修复）

## 1. 命名红线（5 条）

1. **README / SKILL 可重名，但跨目录链接必须路径化**：库内有大量 `README.md`，禁止用裸 `[[README]]`；引用某目录的 README 一律写 `[[完整或部分路径/README|可读别名]]`，或直接用代码体路径 `目录/README.md`。
2. **总纲不许重名**：每个「XX总纲」全库唯一（如 [[商品力总纲]] / [[科学零售总纲]] 已唯一）；不得在子目录再起同名总纲，避免 SSOT 竞争（§10.2 单一真相源）。
3. **MOC 文件名必须含领域**：MOC 不许叫裸 `MOC.md`，须 `<领域>_MOC`（如 [[科学零售知识树_MOC]]、[[晟果新零售变现体系总控_MOC]]），保证全库基名唯一、可裸链。
4. **客户项目件带「客户/门店 + 日期」**：如 `2026-05-09_花厅坊综合诊断分析报告_v3.0`，避免跨客户/跨批次混淆；对外件再加 `_对外版`。
5. **产品包件带「产品名 + 版本」**：如 `社区超市30天商品提效包_交付物总清单_v0.1`，同一产品多版本靠 `_vX.Y` 区分，不靠目录隐含。

## 2. 链接规则（写法）

- **基名唯一 → 可裸链**：`[[商品力总纲]]`。
- **基名重复 → 必须路径化或带别名**：`[[10_咨询交付模板/README|交付模板说明]]`。
- **嵌入附件 → 用相对/完整路径**，不靠基名猜，降 A 级断链（见 [[链接债治理总表]]）。
- **指向目录件（非 md，如脚本目录）**：用代码体路径而非 wikilink，例如 `13_数据分析与工具脚本/花厅坊POS清洗脚本_v0.1/`。

## 3. 同名笔记三类处置

| 类型 | 判定 | 处置 | 是否需签字 |
|---|---|---|---|
| **合法同名（结构件）** | README / SKILL / 各目录索引等结构性重名，本就该每目录一份 | **保留**，全部改为**路径化链接**，禁裸链 | 否（只改链接写法） |
| **历史重复（归档残留）** | 同一内容在 `99_归档` 与活跃区各存一份 | **保留两份 + 在活跃区页头加一行说明**指明归档副本路径与冻结理由（§6 源头不可变） | 否（只加说明行） |
| **核心重名（SSOT 冲突）** | 两页都自称某概念的权威定义（如两个「总纲/主定义」） | **人工合并**：定一处为 SSOT，另一处降「导航/引用视图」或归档 | **是**（合并=动正式文件，§3④/§10.2） |

## 4. 已知重名样本（供消歧参考，待 lint 补全）
- `README.md`：分布于多目录（如 `10_咨询交付模板/`、`13_数据分析与工具脚本/`、各案例目录）——属**合法同名**，一律路径化。
- `晟果新零售科学零售术语口径统一表_v1.0`：同名件存在于 `00_入口与总索引/02_方法论索引/` 与 `00_入口与总索引/03_治理规范/`——疑似**历史重复/SSOT 冲突**，待人工判定后按 §3 处置（标「待判定」）。

## 5. 执行纪律
- 本规范只立**新增/链接写法**的红线；**不批量重命名存量文件**（动正式文件须签字，§3④；避免 v1.1 式重构翻腾，§8）。
- 新建件时即按本规范命名 + 路径化链接（建即入链、不留歧义）。
- 本页 candidate，升 active 需六哥签字。
