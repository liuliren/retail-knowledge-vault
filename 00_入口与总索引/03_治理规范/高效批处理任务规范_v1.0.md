# 高效批处理任务规范 v1.0

> 把"多任务合批、但不越界"固化为 Claude / Codex / Vault 治理任务的通用执行规则。配合 CLAUDE.md §3 协作契约、§11.3 状态机使用。**最小写入：本规范独立成文，不改 CLAUDE.md 正文（避免再走治理总纲签字）。**

## 1. 合批原则
同一轮可合并多个任务，但**必须属于相同或相近风险等级、共享同一边界**。跨风险等级不得合批。

## 2. 可合批任务（低风险·可同轮）
- 只读审阅 / 预审；
- 索引 / README / MOC 最小补链；
- 术语表 / 命名声明 / 分工表等治理预览或登记；
- 输出区资产分级；
- lint / 仪表盘 / 注册表**路径登记类**提交；
- 执行日志追加；
- 不触碰真实数据的预审报告；
- frontmatter 最小补字段（alias / signoff，按授权）。

## 3. 不可合批任务（必须单独开卡 + 单独授权）
- **真实数据 execute / 写回**；
- **脚本逻辑修改**；
- **M-DEC candidate → active**；
- **active signoff 批量补签**；
- **客户版 Word / PDF 正式交付**；
- **批量改全库术语**；
- **批量移动 / 删除 / 重命名文件**；
- **真实 SKU / 条码 / 销售额处理**；
- **.gitignore 修改**；
- **外部 dirty 清理**。

## 4. 默认推进顺序
`预览 → 最小写入 → 复查 → commit → 完成报告`。不确定的只出表/预览，不擅自落定。

## 5. git 纪律
- **禁止 `git add .` / `git add -A`**；只 add 本轮授权文件（显式路径）。
- 外部 dirty（如 linter 改动 / `未命名.*`）**默认不碰、不提交**。
- 多方协作时**区分作者**：Claude 文件与 Codex 产出分开判定、分开提交。
- commit body **必须写明"未触碰范围"**。
- 提交前安全闸：grep 暂存区确认无越界/敏感/外部 dirty 混入。

## 6. 每轮必出
- 执行结论 ｜ 实际修改文件 ｜ 未触碰范围 ｜ 风险与未决问题 ｜ 下一步候选。

## 7. 与既有治理的关系
- 不替代 CLAUDE.md §3（签字契约）/ §6（权限分层）/ §11.3（状态机）；本规范是其**执行层操作细则**。
- 如本规范与 CLAUDE.md 冲突，以 CLAUDE.md 为准。
