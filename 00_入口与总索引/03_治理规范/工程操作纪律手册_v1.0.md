## §3. 类 1 — 目录与路径纪律（5/9 触发）⭐

> **5/9 触发事件**：`cd /tmp && python3 ...` 切走 Bash cwd / 后续 git 报"致命错误：不是 Git 仓库"

### 3.1 5 条硬约束

1. **绝不在 Bash 层用 `cd /tmp`、`cd <other>` 等切走 vault cwd**
2. **Python / Shell 脚本读外部目录** → 在脚本**内部** `os.chdir()` 或用**绝对路径**
3. **git 命令必要时用 `git -C <repo> ...`** / 而不是 `cd <repo> && git ...`
4. **如必须 cd**（极少数 / 用户明确要求）→ 命令末尾必须 `cd -` 或 `cd <vault>` 切回
5. **执行可疑的目录敏感命令前**（git / Read / Write）→ 先 `pwd` 验证 cwd

### 3.2 反例 + 正例

```bash
# ❌ 反例（5/9 W19 实战错误）
cd /tmp && python3 << 'EOF'
import openpyxl; ...
EOF
# 后续 git add -A → "致命错误：不是 Git 仓库"

# ✅ 正例 1：脚本内 os.chdir
python3 << 'EOF'
import os; os.chdir("/tmp/01_门店数据材料_xlsx")
import openpyxl; wb = openpyxl.load_workbook("滞销商品.xlsx")
EOF

# ✅ 正例 2：绝对路径
python3 << 'EOF'
import openpyxl
wb = openpyxl.load_workbook("/tmp/01_门店数据材料_xlsx/滞销商品.xlsx")
EOF

# ✅ 正例 3：git -C
git -C /Users/davidliu/KnowledgeBase/retail-knowledge-vault add -A
```

## §4. 类 2 — Git 操作纪律

### 4.1 6 条硬约束

1. **不破坏性命令需用户明确确认**：`git push --force` / `git reset --hard` / `git branch -D` / `git clean -f` / `git checkout --` / 绝不擅自执行
2. **commit message 格式**：第 1 行 ≤ 72 字符 / 简明动词开头（chore / feat / fix / docs / refactor）/ 详细说明在 body
3. **commit 前必查 `git status`**：避免遗漏文件 / 避免误 commit `.env` 等敏感文件
4. **优先 `git add <specific>`** / 而非 `git add -A` / `git add .`（除非确认无敏感文件）
5. **绝不 `--no-verify` 跳过 hooks** / 除非用户明确要求
6. **绝不 `--amend` 已 push 的 commit**（除非用户明确要求 / 因为可能改变其他人本地历史）

### 4.2 默认动作

- 遇 hook 失败 → 修复根因 / 创建新 commit / 不 amend
- 不确定是否 commit → 先列 `git status` + `git diff` / 让用户决定
- push 远程前 → 必须用户明确指令

## §5. 类 3 — 文件操作纪律

### 5.1 5 条硬约束

1. **Read 先于 Edit / Write**：必须先 Read 才能 Edit（系统强制）/ Write 之前应 Read 检查现状
2. **优先 Edit 而非 Write**：Edit 是增量 / Write 会全文覆盖（已有文件用 Edit / 新文件用 Write）
3. **绝不擅自创建文档**（特别是 `.md` README 等）：除非用户明确要求
4. **绝不在 user 未确认下大范围修改文件**：每次只处理一个小任务包 / 范围内文件
5. **文件命名遵守 [[文件命名规范]]**：中文为主 / 必要时加版本号 v0.1/v1.0 / 不删除重要文件（mv 99_归档/）

### 5.2 与现有 §15.2 文件工程化协同

本类规则覆盖**操作层** / [[CLAUDE.md]] §15.2 文件工程化覆盖**内容层**（标题 / 版本 / 状态 / frontmatter 等）/ 二者协同。

## §7. 类 5 — 进程与执行纪律

### 7.1 6 条硬约束

1. **长任务用 background**：超过 30 秒 → `run_in_background: true` / 不 sleep 等待
2. **不用 sleep 轮询**：用 Monitor 工具 / 完成后系统通知
3. **超时设置合理**：默认 2 分钟 / 长任务用 timeout 参数（最大 10 分钟）
4. **错误时不重试**：3 次失败必停 / 排查根因 / 不死循环
5. **不擅自杀进程**：`kill` / `pkill` 命令需用户确认
6. **背景任务必须 track**：用 BashOutput 检查 / 不丢任务

### 7.2 反例

```bash
# ❌ 反例：sleep 轮询长任务
while ! grep "完成" log.txt; do sleep 5; done  # 浪费缓存 + cwd 漂移

# ✅ 正例：background + Monitor
# Bash run_in_background: true → 系统通知完成
```

## §8. 类 6 — 工具调用纪律

### 8.1 5 条硬约束

1. **优先 dedicated tools 而非 Bash**：Read 而非 cat / Edit 而非 sed / Write 而非 echo>
2. **并行无依赖工具调用**：多个独立 Read / Bash 同 message 内并行
3. **绝不 search 主目录大型文件**：用 Glob / Grep 工具（不 `find /`）
4. **subagent 用于明确分工**：`general-purpose` 跨域研究 / `Explore` 文件搜索 / 不滥用
5. **MCP 工具优先官方**：Figma / Notion / Slack 等 / 不用第三方未审计

### 8.2 性能优化

- 每次 message 尽量并行多 tool call（不串行）
- 长上下文工作用 subagent（保护主 context）
- 缓存敏感操作不重复（不 re-read 刚 Edit 的文件）
