# 外部资料编译规则 v0.2（已签字生效）

> ✅ 三轮审议（六哥视角→批判视角→专业/应用验证视角）已全部完成，六哥已于2026-07-14签字批准，记录见 [[外部资料编译规则_v0.2三轮审议记录_v0.1]]。**本规则现已生效（status: active）**，适用范围见frontmatter `effective_scope`；`not_applicable`范围（stable方法论修改、客户正式交付文件）不受本规则约束，仍走各自既有治理流程。

## 1. 三层结构

### Layer 1 · Fact（事实层）

**允许**：原始事实、企业动作、来源信息

**禁止**：客户建议、经营判断

### Layer 2 · Inferred（推断层）

**允许**：机制总结、行业规律假设

**必须标注**：`fact_layer: inferred`

### Layer 3 · Application Pending（应用待验证层）

四段式格式：

```markdown
- **应用假设**：[如果...可能...句式]
- **验证条件**：[需要观察什么/满足什么前提]
- **所需数据**：[需要采集的客户自己的数据]
- **当前状态**：[未验证 / 验证中 / 已被客户实证否决 / 已被客户实证支持]
```

**硬性禁用词**：应该 / 必须 / 立即（含立刻/马上/现在就等变体）

### Fact / Inferred / Application Pending 操作化判断标准（Round1六哥确认）

| 层 | 判断标准 |
|---|---|
| Fact | **单信源可验证**——能直接追溯到某一份来源（某财报/某篇报道/某次访谈）直接写出来的内容 |
| Inferred | **跨信源归纳**——从多个案例/多篇分析中提炼出的模式或规律，脱离了单一具体来源 |
| Application Pending | **应用假设未验证**——任何"如果这个机制迁移到客户身上会怎样"的推测，不管多有道理，只要没有客户自己的证据支撑，一律停在这一层 |

## 2. 知识资产状态机（Round1已确认延伸至Active，新增Rejected/Archived退出状态）

```
Fact → Inferred → Application Pending → Customer Evidence → Methodology Candidate → Active
                                              │
                                              ├─ evidence_status: rejected → Archived（默认退出态，非删除）
                                              │        │
                                              │        └─ 六哥可手动重开 → 退回 Application Pending
                                              │
                                              └─ evidence_status: validated → 正常推进 Methodology Candidate
```

**状态说明**：
- **Active** 不是本规则新定义的平行状态，是**指向** retail vault CLAUDE.md §11.3 既有状态机的 active 状态——本规则的 Methodology Candidate 等同于 retail vault 既有的 candidate，两者是同一件事的两种称呼，从 Methodology Candidate 到 Active 仍然走既有的 checklist 和签字流程，本规则不重复定义
- **Archived** 是外部资料假设条目专用的退出状态，不等同于 retail vault 既有的 `status: deprecated`（deprecated专指方法论正文降级，Archived专指本规则里被拒绝的假设条目），两个概念互不影响对方判定
- Archived状态不是终局删除，**六哥可以随时手动把某个Archived条目重新打开，退回Application Pending**，用于"当时条件不成熟，现在环境变了想再试"的场景

### Customer Evidence 精确定义

**= 已获得客户证据。不等于**：验证成功 / 方法有效 / 可推广。

### Agent 禁止事项（硬性红线）

Agent不能：
1. 自行把 Application Pending 推进到 Customer Evidence
2. 自行把 Customer Evidence 推进到 Methodology Candidate
3. 判断客户成功
4. 修改stable

### 状态推进的两个合法来源

**六哥确认**——覆盖所有场景，包括清单外的情况。

**授权诊断流程**（Round1确认，正式清单）：
1. **7D诊断框架**（`KB-DX-FULLSTORE-001`）
2. **品类诊断题库/品类诊断框架**（`KB-DX-QUESTION-001`、`KB-DX-FRAMEWORK-001`）
3. **现场门店/POS数据交叉验证**（现场走访 + 客户自有经营数据交叉对照）

**普通AI推演不得进入Customer Evidence**——即使一个假设推理得再严谨、逻辑再自洽，只要不满足下方"Authorized Analysis四要素"且不属于三类授权流程之一、也没有六哥手动确认，就只能停留在Application Pending，不能自称已经"获得客户证据"。**外部案例类比、纯理论分析同样只能停留在Application Pending**，不因为分析本身写得再详细就自动升级。

### Authorized Analysis 四要素（Round1最终确认，正式生效）

**只有同时满足"四要素记录完整"和"属于三类授权流程之一"，才允许把假设推进到 Customer Evidence。**

进入 Customer Evidence 前必须记录：

```yaml
authorized_by: [授权人，默认六哥本人]

analysis_scope: [具体客户/任务/诊断范围，如"花厅坊·7月品类调整诊断"]

data_source:  # 三选一
  - field_visit           # 现场巡店/走访
  - customer_data          # 客户自有经营数据（POS/库存系统等）
  - authorized_system_data # 授权诊断系统产出的数据（如7D诊断框架跑出的结果）

output_purpose:  # 四选一
  - diagnosis         # 诊断用途
  - verification       # 验证某个具体假设
  - AAR               # 复盘/事后回顾
  - decision_support   # 支持某项具体决策
```

**四要素中任何一项缺失，该条目不能标记为Customer Evidence，只能停留在Application Pending。**

### Customer Evidence 字段设计（最终版）

```yaml
evidence_source:  # 三选一
  - field_visit
  - customer_data
  - authorized_analysis

evidence_status:  # 三选一
  - collected
  - validated
  - rejected

# 当 evidence_source: authorized_analysis 时，必须同时附带以下四要素（见上方定义）：
authorized_by: [...]
analysis_scope: [...]
data_source: [...]
output_purpose: [...]
```

## 3. 边界规则

外部案例编译页**不能直接产生**：客户经营结论 / 方法论升级 / stable修改

外部案例编译页**只能产生**：学习材料（Layer1+2） / 验证假设（Layer3） / 候选输入

## 4. Summary治理

frontmatter的`summary`字段属于知识资产正文，必须接受与正文同等的三项检查：Fact检查 / 越界检查 / 客户应用检查。

## 5. 高风险主题清单（Round1确认新增）

以下三类主题编译时**要求增强审查**——编译流程必须额外加入越界提醒，编译完成后逐条自查是否出现"XX该抄/XX可迁移/XX宜学"这类直接判断句：

1. **组织力**（企业管理方法论、经营分析框架、组织建设案例）
2. **供应链/自有品牌**（供应商关系、自有品牌开发、采购策略案例）
3. **硬折扣/标杆企业**（ALDI/山姆/Costco/唐吉诃德/胖东来等对标案例）

其余主题沿用标准编译前检查清单，不强制额外加强。

## 6. 编译前检查清单

1. 是否区分了事实和判断？
2. 是否存在对客户的直接建议？
3. 是否存在未经验证的经营数字？
4. 是否把案例经验升级成了规则？
5. 是否需要客户数据验证？
6. 如果主题属于第5节高风险清单三类之一，是否已额外做过越界自查？
