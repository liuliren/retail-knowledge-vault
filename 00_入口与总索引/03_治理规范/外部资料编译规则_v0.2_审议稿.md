# 外部资料编译规则 v0.2 审议稿

> ⚠️ 本文件涉及"规则/必须/禁止"字样，按全局宪法"标准审议铁律"，签字前均为draft，不具约束力。本稿是**审议准备材料**，不是审议结果——六哥视角/批判视角/专业视角三轮尚未走。

## 1. 三层结构（沿用v0.1，未改动核心定义）

### Layer 1 · Fact（事实层）

**允许**：原始事实、企业动作、来源信息（谁说的/哪篇文章/什么时间/什么数据）

**禁止**：客户建议、经营判断——即使一句话里混了事实和判断，判断部分也不能留在这一层

### Layer 2 · Inferred（推断层）

**允许**：机制总结（从事实中提炼的规律）、行业规律假设（可能在其他场景也成立的通用模式）

**必须标注**：`fact_layer: inferred`

### Layer 3 · Application Pending（应用待验证层）

**所有客户相关内容必须进入此层**，格式四段式（v0.2审议稿新增"当前状态"字段，v0.1只有三段）：

```markdown
- **应用假设**：[如果...可能...句式]
- **验证条件**：[需要观察什么/满足什么前提]
- **所需数据**：[需要采集的客户自己的数据]
- **当前状态**：[未验证 / 验证中 / 已被客户实证否决 / 已被客户实证支持——待新增，追踪假设的生命周期，避免同一个假设被反复提出却从不推进]
```

**硬性禁用词**（Layer 3 及以上全部禁止；v0.2审议稿收紧措辞，把"建议立即"改为更泛化的"立即"，覆盖更多变体如"立刻/马上/现在就"）：
- "应该"
- "必须"
- "立即"（含"立刻/马上/现在就"等同义变体）

## 2. 知识资产状态机（v0.2审议稿新增）

```
Fact → Inferred → Application Pending → Customer Evidence → Methodology Candidate
```

| 状态 | 定义 | 进入条件 | 谁能推进到下一态 |
|---|---|---|---|
| Fact | 原始案例事实 | 编译时直接产生 | Agent可写 |
| Inferred | 从事实提炼的机制/规律 | 编译时直接产生，标`fact_layer: inferred` | Agent可写 |
| Application Pending | 对客户的应用假设，未验证 | 编译时把"if...then"式假设放入这一层 | Agent可写（提出假设），但假设内容本身不构成结论 |
| Customer Evidence | 假设在真实客户（花厅坊/好家源等）身上被验证或证伪 | **必须有客户现场数据支撑**，不能靠外部案例类比 | 只能六哥或六哥授权的诊断流程产出，Agent不能自行判定"已验证" |
| Methodology Candidate | 多个客户案例反复验证后，具备升级为方法论候选的资格 | ≥3案例实证 OR ≥2家客户门店验证（对齐retail vault §11.3现有active门槛） | 走既有"→candidate""→active"签字门，本规则不新增或替代既有状态机 |

**这条状态机只补充"外部案例编译"这一条进入通道，不改变、不替代 retail vault CLAUDE.md §11.3 已有的 draft→candidate→active→deprecated 状态机**——Application Pending 和 Customer Evidence 是插在"外部资料被读入"和"够格参选candidate"之间的两级过渡态，避免外部案例跳过客户验证直接冒充方法论。

## 3. 边界规则（v0.2审议稿新增，回应"外部案例不能干什么"）

**外部案例编译页不能直接产生：**
1. 客户经营结论（"花厅坊应该/花厅坊的问题是XX"）
2. 方法论升级（不能自己宣称"这构成一个新框架"）
3. stable修改（不能以任何形式建议改动已签字的active/stable页面）

**外部案例编译页只能产生：**
1. 学习材料（Layer 1 Fact + Layer 2 Inferred）
2. 验证假设（Layer 3 Application Pending，格式化的"如果...可能..."）
3. 候选输入（供六哥或诊断流程未来参考，不代表已经是决策依据）

## 4. 编译前检查清单（v0.2审议稿新增，供未来编译时前置自查）

每篇外部案例进入KB前，编译者（无论Agent还是人）应逐条自问：

1. **是否区分了事实和判断？**（Layer1/2和Layer3是否物理分离，而不是揉在一句话里）
2. **是否存在对客户的直接建议？**（搜索"花厅坊应该/六哥应该/客户应该"及其隐含变体）
3. **是否存在未经验证的经营数字？**（如"预计提升XX%毛利"这类具体数字，如果来自外部案例类比而非客户自己的数据，必须标注为假设而非结论）
4. **是否把案例经验升级成了规则？**（一个案例≠一条规律，规则化需要多案例交叉验证，见状态机的Methodology Candidate门槛）
5. **是否需要客户数据验证？**（如果内容涉及"这对客户有用"，是否已经诚实地标成Application Pending而非默认为真）

## 5. 特别案例复核

### 案例1：供应链与自有品牌A——"通用启发式 vs 特定客户判断"边界

**背景**：diff演练时，agent把这句话判定为Layer 2 Inferred保留原位、未移入pending区：
> "如果有客户想通过'绑定一家供应商'建立护城河，先问这家供应商的东西是否真的'同行做不到/买不起'，否则只是普通合作关系，不构成壁垒。"

**审议稿裁定标准（草案）**：区分"通用启发式"和"特定客户判断"，看是否满足以下**全部三条**：
1. 主语是否泛化（"任何客户/有客户"而非"花厅坊/六哥"）——本例满足
2. 结论是否可证伪且不依赖特定客户的现场数据（"是否同行做不到"这个问题本身是分析框架，不是对某个具体供应商的判断）——本例满足
3. 是否仍然是"提出一个可以拿去问的问题"而非"给出一个现成答案"——本例满足（"先问...是否..."是问题句式，不是"某供应商就是/不是护城河"的结论句）

**裁定**：这条满足全部三条，维持Layer 2 Inferred不移动的判断是合理的。**建议把这三条写进正式规则**，作为编译时判断"通用启发式"边界的操作化标准，而不是靠感觉判断。

### 案例2：硬折扣E——frontmatter summary字段是否需要同样治理

**背景**：diff演练发现 `2026-07-13_硬折扣对标企业E_市场观察_v0.1.md` 的frontmatter `summary:` 字段本身含类似应用判断措辞，但原任务范围限定在"正文结论先行+so-what"，未处理。

**审议稿裁定（草案）**：**应该同样治理，理由**：
- summary字段是G02"编译层优先铁律"里检索的第一遍读取对象——如果summary本身混了判断和事实，查询时even不读正文也会被误导
- 现有模板要求summary"≤40字一句话"，篇幅极短，更容易把判断压缩成看似客观的陈述句（比如"花厅坊该抄XX"压缩成"XX可复用"，语气弱化但本质仍是判断）

**建议动作**：本规则正式生效后，编译检查清单第2条（"是否存在对客户的直接建议"）**应同时适用于frontmatter的summary字段，不只是正文**。这是v0.2审议稿相对v0.1的一个范围扩展，需要六哥确认是否认可。
