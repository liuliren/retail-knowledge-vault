# 模型合理使用完善计划 v1.0（✅六哥 2026-07-05 授权定版·全库 Claude 模型使用的固定方法）

> 现行规则:retail CLAUDE.md §7"脚本>Sonnet(机械相)>Opus(判断相)"——**已过时**:现在主线程是 Fable(高于 Opus),子代理默认继承 Fable,且有 Haiku 批处理层。本计划是 §7 的 v2 提案,**改治理=D档,签字前不生效**。

## 一、五层调度表 v2(提案)

| 层 | 引擎 | 适用 | 典型任务 | token成本 |
|---|---|---|---|---|
| **L0 脚本** | python/bash | 确定性校验/统计/批量改名 | lint、checker 门禁(验收框架T1)、字段统计 | **零** |
| **L1 批处理** | Haiku(batch-processor) | 零判断批量文本操作 | frontmatter 批量补齐(现存1450缺字段)、版本号归位(373)、给定清单的机械标注 | 极低 |
| **L2 机械代理** | Sonnet(Agent 调用时 model: sonnet) | 给定骨架的单件起草、对账核查、分类清点 | 孤儿分类、文件收编、给定裁决的写回 | 低 |
| **L3 主线程** | Fable | **判断相**:裁决建议、批判审议判定、审计合成、路线图、与六哥对话、编排 | 本日审计合成、签字包起草 | 高·省着用 |
| **L4 深推理** | Opus(complex-task-handler) | 过双门(结构必要+不可分解)才用 | 架构/本体/多约束冲突设计 | 高 |

**红线不变**:判断相(裁决/签字背书/调改取舍)不下放 L2 以下;新增一条——**批判审议的"判定"与"必修项定级"必须 L3+**,L2 只许收集证据。

## 二、限额管理纪律(今日两次卡点的直接产物)

1. **并发满级代理 ≤3**;第 4 个起要么排队要么降层(L1/L2)。
2. **大 fan-out 前先 checkpoint**:commit + RESUME 断点行,限额死后可一键断点重跑(本日已实践两轮,固化为规)。
3. **会话节奏**:判断相排前段(限额新鲜时),机械批量排后段或下放 L1;限额将近不开新满级代理。
4. **卡点协议**:限额死 → RESUME 记"⚡限额卡点"行(死了什么/重跑指令/护栏教训)→ 重置后一键重跑。

## 三、代理护栏(prompt 之外的第二道)

- 机械代理任务一律写**动作白名单**("只允许追加行"/"禁 git 写"/"禁改 stable 正文"),violation=作废重跑。
- 涉 stable/active 签字件的正文修改,**只允许主线程亲手做**,代理只能提建议清单。
- 长期解(承接 GOV-001 批判 R1/验收框架 T1):机械门禁全部脚本化成 L0 checker,代理产出过 checker 再入库——把护栏从 prompt 层挪到执行层(keys not prompts)。

## 四、token 经济补丁(既有铁律之上)

- 代理 prompt 强制带"编译层优先+读量上限"条款(本日已实践:只读 README/frontmatter/summary)。
- 债务批量修复(1450 缺字段/373 版本不一致/174 status)一律 L0 脚本先算清单 → L1 批量执行 → L0 复验,**全程不碰 L3**。
- lint 排除 `.claude/worktrees/`(当前 104 处污染统计),残留 worktree 清理后统计才可信。
