# 外部资料编译规则 v0.2 三轮审议记录（预演稿）

## ⚠️ 重要说明：这不是正式的标准审议铁律三轮审议

全局宪法"标准审议铁律"规定的三轮是：**第1轮六哥(用户视角)初稿确认 → 第2轮批判角色(逆向/魔鬼代言人) → 第3轮专业角色(领域视角)steelman → 六哥签字**。

**第1轮"六哥视角初稿确认"本质上必须由六哥本人做出判断，不能由Agent代为完成或模拟。** 本文件的"Round 1"是**起草方（Agent）的完整性自查**，不等同于六哥视角确认——这是准备材料，不是审议本身。Round 2（魔鬼代言人）和Round 3（应用验证）是Agent可以合理承担的分析性工作，本文件里这两轮是真实执行的分析，但仍需六哥最终认可其结论是否成立。

**换句话说：本文件跑完，规则仍然是draft，仍然不具约束力，仍然需要六哥亲自过一遍第1轮才能真正启动审议流程。**

## Round 1（起草方自查·非六哥视角）：状态机完整性 / Agent权限 / summary治理

### 状态机完整性自查

- [x] 五个状态（Fact→Inferred→Application Pending→Customer Evidence→Methodology Candidate）之间的转移条件是否都写清楚了？—— 是，但**Application Pending→Customer Evidence**这一步的"六哥确认"具体怎么操作（是六哥手动改字段，还是有个命令/流程）没有写清楚，属于遗漏
- [x] 是否和retail vault既有的draft→candidate→active→deprecated状态机冲突？—— 不冲突，本规则明确是"插在外部资料读入和参选candidate之间的过渡态"，衔接点是Methodology Candidate对应既有的candidate
- [ ] **未覆盖**：如果一个Application Pending假设被证据**拒绝**（evidence_status: rejected），它应该停留在哪个状态？是留在原地标记rejected，还是需要"降级"或"归档"？规则目前没写明rejected之后的去向

### Agent权限自查

- [x] 四项禁止是否清楚？—— 是
- [x] 两个合法推进来源（六哥确认/授权诊断流程）是否清楚？—— 是
- [ ] **未覆盖**："授权诊断流程"本身的授权范围谁来定义、怎么变更？如果六哥今天授权了7D诊断框架，明天想撤销授权，流程是什么？规则没写

### summary治理自查

- [x] 三项检查（Fact/越界/客户应用）是否清楚？—— 是，且今天的35页审计已经实际按这三项标准跑过一遍，可执行性验证通过（见Round 3）
- [ ] **未覆盖**：规则说"summary必须接受同等检查"，但没有说**谁**来执行这个检查——是每次编译时Agent自查，还是需要事后审计（像今天这样）？如果是自查，怎么保证Agent不会在写summary时又犯同样的错（这正是本次35页出问题的原因——当时的编译提示词本身就没有这条规则）

### Round 1 自查结论

## Round 2（魔鬼代言人·真实批判视角）

### 质疑1：Customer Evidence 是否过严？

**批判观点**：把"Application Pending → Customer Evidence"这道关卡设成必须"六哥确认或授权诊断流程"，意味着**任何一个外部案例假设，如果没有六哥亲自看过或走既有诊断流程，就永远停留在"未验证假设"状态**。这在实践中可能导致：
- 六哥的时间成为唯一瓶颈——如果Application Pending积累了几十上百条假设，六哥不可能逐条确认，规则形同虚设（要么假设永远堆积不动，要么六哥被迫囫囵吞枣快速签字，失去把关意义）
- 与"90/10护城河"原则（AI做可商品化的90%，六哥做不可替代的10%判断）有潜在张力——如果"是否算验证"这件事全部收归六哥，AI在这个环节的杠杆作用被削得很薄

**反驳/回应**：这个担忧是合理的，但规则本身已经提供了缓解机制——"授权诊断流程"这个通道本身就是六哥把一部分判断权**批量授权**出去的方式（比如"7D诊断框架跑出来的都算数"），不是每次都要六哥手动点头。真正的问题不是"是否过严"，而是"**授权诊断流程"这个概念目前定义得太模糊**（这也是Round 1自查发现的遗漏之一）——如果六哥能明确定义几类"自动算Customer Evidence"的场景（比如"花厅坊现场巡店记录+POS数据交叉验证的，自动进Customer Evidence，不需要额外手动确认"），瓶颈问题就能缓解。**结论：不是规则过严，是"授权诊断流程"这个概念需要六哥进一步具体化，否则确实会在实践中卡住。**

### 质疑2：授权诊断流程边界不清

**批判观点**：规则说"授权诊断流程"可以把假设推进到Customer Evidence，但没有定义：
- 什么算"诊断流程"？7D诊断框架算，那品类诊断题库、门店全景诊断框架算不算？
- "六哥授权"这个授权动作本身怎么留痕？口头说一句"这个流程我认了"算不算数，还是需要写进某个文件？
- 如果诊断流程本身产出的结论有争议（比如两次诊断结果矛盾），谁来裁定？

**这条质疑成立，且没有很好的反驳**——规则目前对"授权诊断流程"这四个字的定义确实是空的，只是提了一个名词。**建议在正式三轮审议时，把这一条列为必须补全的事项，而不是留白**。

### 质疑3：Fact/Inferred 边界

**批判观点**：Layer 1（Fact，允许原始事实/企业动作/来源信息，禁止客户建议/经营判断）和Layer 2（Inferred，允许机制总结/行业规律假设）之间的边界，在实践中很容易模糊——"ALDI的护城河是采购规模+成本纪律"这句话，到底是Fact（ALDI真的这么做了，这是可验证的企业行为总结）还是Inferred（这是从ALDI案例里提炼出的一条"护城河机制"，本质是一个归纳判断）？

**举证**：今天的35页审计里，"库存价格组织力合集"的summary "库存转利润靠费用纪律，损耗靠现场SOP..." 被判定为"把Inferred层写成了确定性结论"——这恰恰说明**Fact和Inferred之间没有一条清晰的操作化标准，只能靠"读起来像不像结论"这种主观判断**。

**回应**：这条质疑也成立。规则里目前唯一给出操作化标准的是"供应链A"案例的三条判断（主语泛化/结论可证伪不依赖客户数据/问题句式而非答案句式），但那三条是用来区分"通用启发式该不该移入Layer3"的，不是用来区分"Layer1还是Layer2"的。**Fact/Inferred这条边界目前没有类似的操作化标准，这是本轮审议发现的一个真实缺口**。

**建议**：借用"供应链A"三条标准的思路，为Fact/Inferred也补一套操作化判断——初步建议（未经六哥确认，仅供参考）：
- 如果一句话可以直接追溯到"某企业/某人说了/做了什么，有可查证的来源"→ Fact
- 如果一句话是"从多个事实归纳出的模式/规律，脱离了具体的单一来源"→ Inferred
- 判断标准：**能不能只靠一个信源验证？能→Fact；需要跨信源/跨案例才能验证的抽象总结→Inferred**

### Round 2 结论

三条质疑里，质疑2（授权诊断流程边界不清）和质疑3（Fact/Inferred边界模糊）**都成立且缺乏操作化标准**，需要在正式审议时补全。质疑1（Customer Evidence过严）**不完全成立**，真正的问题也是授权诊断流程定义不清，不是关卡本身设得太高。

## Round 3（应用验证·以35页为验证样本）

### 验证方法

用今天完成的《2026-07-14_35页外部资料summary审计报告_v0.1》作为实测数据——这份报告是**在规则最终审议稿写完之后**、用规则里定义的"越界/部分/正常"三分类标准，对34个真实summary字段做的实际判断。这相当于一次小规模的规则可用性测试。

### 验证发现

1. **规则可执行**：34页summary都能被明确归入"是/部分/否"三类之一，没有出现"无法判断"的情况，说明规则的判断标准（点名客户+祈使句式判断 vs 通用陈述）在实践中是可操作的
2. **6条明确越界+8条部分混淆，合计14/34（41%）需要不同程度的关注**——说明规则如果正式生效，工作量不小，不是走个形式
3. **意外发现**（Round1自查时没预料到的）：明确越界的6条summary，全部集中在"硬折扣对标企业""组织力与经营分析""供应链与自有品牌"三个主题簇里，和门店调改、陈列、生鲜经营几个主题簇的summary几乎没有明确越界（"部分"里门店调改占了2条，但都是轻微的）。**这提示越界程度可能和主题内容特性相关**——硬折扣/组织力/供应链这几个主题的原始素材本身更容易诱导"这个企业这么做，所以我们客户也该这么做"的类比式判断，而陈列/生鲜经营这类操作细节类内容天然更偏Fact/SOP描述，不容易越界。这个发现没有被写进规则草案，**建议正式审议时补充为"高风险主题清单"，对硬折扣/组织力/供应链类内容的编译提示词做额外的越界提醒**
4. **规则的"部分"档位在实践中占比不低（23.5%）**，说明"点名客户但语气中性"这种中间态很常见，规则需要明确六哥对这个中间态的容忍策略（严格执行还是暂缓观察），否则每次审计都会积累一堆"部分"级别的待办，长期不处理

### Round 3 结论

规则本身可执行、可操作，但验证过程中发现规则遗漏了"高风险主题"的识别机制——如果能在编译提示词阶段就针对硬折扣/组织力/供应链类内容加强越界提醒，可能比事后审计更有效率。
