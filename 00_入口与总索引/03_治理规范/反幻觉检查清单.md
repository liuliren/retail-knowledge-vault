# 反幻觉检查清单

本文件是 [[反幻觉规则总库]] 的**可操作版本**，用于在生成重要内容前快速自查。按场景分类，逐条核对。

---

## A. 通用基础检查（每次输出前必查）

- [ ] 内容中是否区分了：**事实 / 分析判断 / 建议 / 待确认**？
- [ ] 是否存在把推测写成事实的表述？
- [ ] 不确定信息是否已标注「待确认」？
- [ ] 正式结论是否列出了所依据的文件或事实？

---

## B. 真实门店数据检查

- [ ] 门店数据（销售额、毛利率、库存等）是否来自真实可查的来源？
- [ ] 是否存在自行编造的门店数据？
- [ ] 现场照片、Excel 数据、门店反馈之间是否有矛盾？如有，是否已标注冲突？

---

## C. 商品诊断检查

- [ ] 商品判断是否**仅凭销量**得出？（禁止）
- [ ] 商品判断是否同时考虑了：活动影响、陈列影响、缺货影响、价格影响、库存影响、客群影响？
- [ ] SKU 的淘汰/保留/新增建议是否基于多维度综合判断（销售额、毛利额、库存、动销、陈列、活动、客群）？

---

## D. 陈列方案检查

- [ ] 陈列建议是否考虑了真实货架结构（层数、层板高度、宽度）？
- [ ] 是否考虑了商品包装尺寸与层位的匹配？
- [ ] 是否考虑了品类边界和门店动线？
- [ ] DisplayMap 建议是否基于真实门店条件，而不是理想状态？

---

## E. 活动效果分析检查

- [ ] 活动效果判断是否区分了：**活动期间 / 活动前 / 活动后**？
- [ ] 销量增长是否区分了：**自然动销 vs. 促销拉动**？
- [ ] 是否避免了只看单点销量的简单判断？

---

## F. 外部案例引用检查

- [ ] 外部案例（胖东来、奥乐齐、山姆等）是否经过**本土化判断**？
- [ ] 是否直接套用外部案例的结论到社区超市场景？（禁止）
- [ ] 引用的外部信息是否注明了来源？

---

## G. 系统 PRD 与数据字段检查

- [ ] PRD 中的字段是否与字段字典（待建，归属 11_系统产品与PRD）保持一致？
- [ ] 是否标注了数据字段的依赖关系？
- [ ] 是否标注了待确认的字段或规则？

---

## H. 方法论内容检查

- [ ] 是否只引用了 [[核心概念主定义索引]] 中登记的主定义？
- [ ] 是否在文件中重新发明了已有主定义的概念（禁止）？
- [ ] 新方法论内容是否与已有方法论存在矛盾？

---

## I. 客户视角防火墙（§13.16 / 2026-05-09 v1.1 新增）⭐

> 来源：2026-05-09 vault 连环错误事件 / "启明导出 89 .xls"等表述把推断写成事实 / 视角错位

- [ ] 文档中"客户做了 X"的表述，**X 是客户亲口告知**还是 **vault 推断**？
- [ ] 客户内部具体操作人（启明 / 高彬 / 张力 / 佳妮 / 文员 / 等）是否未经客户告知就被写成事实？
- [ ] 是否使用门店主体表述（"客户方提供 / 门店 POS 导出 / 客户协调"）替代"启明做了 X"？
- [ ] 文档 frontmatter `client_internal_ops: true` 时，对应内容是否在 `client_told` 段（不是 `observed` / `inferred`）？

**通用替换映射**：
| ❌ 推断写成事实 | ✅ 中性主体 |
|---|---|
| "启明导出" | "客户方提供 / 门店 POS 导出" |
| "佳妮采购" / "文员录入" | "客户内部分工 / 客户协调" |
| "店长拒绝" | "门店反馈待确认" |

---

## J. 推断标注强制（§13.17 / 2026-05-09 v1.1 新增）⭐

- [ ] 任何"推断 / 假设 / 估计"的表述是否前缀 `[推断]` / `[假设]` / `[待确认]`？
- [ ] 含"应该 / 可能 / 推测 / 估计 / 大概 / 大约 / 约"等词的段落，是否明示推断状态？
- [ ] 数据估算（如差值法）是否标注精度限制（"±X%"）？
- [ ] 推断 vs 客户告知 vs 观察事实是否在 frontmatter `fact_layer` 字段中分别记录？

---

## K. 类比边界（§13.18 / 2026-05-09 v1.1 新增）⭐

> 来源：2026-05-09 vault 把"Sam → Rob 接班"沃尔玛类比推到"启明二代崛起夺权"戏码

- [ ] 使用类比框架（Walton / 海底捞 / Apple / 等）是否**明示"这是类比"**？
- [ ] 类比中**保留的部分**（结构同构）是否清单化？
- [ ] 类比中**删除的部分**（戏剧化情节 / 文化预设 / 时代背景错位）是否清单化？
- [ ] 中国家族企业类比时，是否默认协同+信任（参 [[feedback_客户家族关系协同信任]]）？

**类比示范模板**：
```markdown
## 沃尔玛早期 vs 陈氏家族类比（修正版）
**保留**（结构同构）：家族联合 / 多店分管 / 兄弟联手 / 二代参股
**删除**（戏剧化错位）：Sam → Rob 接班张力 / Succession 剧 / 父子斗
**应用边界**：仅用于结构理解 / 不推导具体动作 / 客户视角防火墙优先
```

---

## M. 数据一致性追溯（§13.20 / 2026-05-09 v1.2 新增）⭐⭐

> 来源：2026-05-09 vault v2.2 报告中 4 个数字（滞销 7140 / 库存 3300 / 客流 145490 / 销售 300 万）/ 被用户追溯发现：滞销实际 7,139 + 库存 3,293 + 客流时段 32 天非 30 天 + 销售时段 31 天非 30 天 / **多个细节差异叠加 = 系统性错误**

- [ ] 数字是否有 evidence？（文件名 + sheet + 行号 + 时段 + 字段定义）
- [ ] 时段是否一致？（不同文件时段错开 1-2 天会导致显著差异）
- [ ] 是否区分"原始行数 vs 实际数据行数"（去 header / 小计行）？
- [ ] 是否区分"过店流量 vs 成交客户数 vs 销售单数"？
- [ ] 是否区分"客单价 = 销售/客流 vs 客单价 = POS 客单价中位数"？
- [ ] 数字之间不一致时 / 是否列 ≥ 3 种解释假设 + 数学验证？
- [ ] 数字之间不一致时 / 是否找到内部一致性最高的解释？
- [ ] 是否标 [推断]？是否让用户 confirm？
- [ ] 是否沉淀为案例（[[数据一致性追溯案例库]]）？

**触发场景**：vault 在不同数据源 / 不同时段 / 不同维度遇到同一指标的数字不一致。

**反例（v2.2 错误做法）**：
- ❌ 凭"差 2 倍"猜测"双倍统计"作为结论
- ❌ 用"30 天"代替实际"32 天"
- ❌ "7,140 行 = 7,140 SKU"（实际去 header 后 7,139）

**正例（用户驱动 / 5/9）**：
- ✅ 用户问"7139 怎么来"→ 重读文件 → 实际 7,139 SKU
- ✅ 时段 4/7-5/8（32 天）vs 4/8-5/8（31 天）/ 时段错开 1 天
- ✅ 客流定义 = 成交客户数（用户 confirm）/ 但客单仍不一致 / 标推断 + 待确认

---

## L. 用户订正优先（§13.19 / 2026-05-09 v1.1 新增）⭐

- [ ] 当用户最早表述与后续订正冲突时，**优先订正版**（不要"我记得用户最早说过 X"）
- [ ] 旧版表述作 audit-log 留存（不删 / 标 ❌ 或 ~~~~~~ / 加版本说明）
- [ ] 是否检查了用户在本会话 / 跨会话的最新订正？
- [ ] feedback memory 沉淀是否覆盖此订正（让 vault 跨会话不再犯）？

---

## 使用说明

- 生成**正式方法论文件**前，检查 A + H 类。
- 生成**商品诊断相关内容**前，检查 A + B + C 类。
- 生成**陈列方案**前，检查 A + B + D 类。
- 生成**活动分析**前，检查 A + B + E 类。
- 引用**外部案例**时，检查 F + K 类。
- 生成**系统 PRD**时，检查 G 类。
- **凡涉及客户内部具体操作 / 客户家族关系 / 类比框架 / 订正历史的内容，I + J + K + L 全部必查 ⭐**（v1.1 新增 / 2026-05-09 起强制）
