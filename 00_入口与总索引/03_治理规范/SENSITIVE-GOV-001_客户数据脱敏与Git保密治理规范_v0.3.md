## 〇、结论先行

客户真实数据/裸值/未脱敏客户名/客户派生经营结论**不进新的普通 Git 提交**；已 tracked 的 226 件历史诊断件按 private 仓既成事实**存量豁免**（文件名含裸值者除外，单独裁决）；把关三层：.gitignore 白名单（机械）→ 提交前 checklist+留痕（人工过渡态）→ pre-commit hook（目标态·W31 前落地）。**本仓绝不设 public。**

## 一、红线（增量适用：管新增提交，不溯及存量）

R1 客户原始数据（xls/csv/db/POS导出/dry-run结果）｜R2 裸值（EAN-13/进价/供应商/SKU明细）｜R3 未脱敏客户名（判据=客户代号注册表，见§一之三）｜R4 客户派生经营结论｜R5 客户诊断/项目日志。
fail-safe 不变：查不到授权=不暴露；无裸值≠安全。

### 一之二 · 存量裁决与归口（R2 清偿·新增）

1. **归口二分**：本 SSOT 只裁"**Git 边界**"（能否 tracked/能否 push）；retail CLAUDE.md §6 裁"**内容纪律**"（正文能写什么，如 C 层"只汇总不输出裸值"）。同一文件两问分答，不再对撞。
2. **存量豁免区**：截至 2026-07-05 已 tracked 的客户诊断/复盘件（约 226 件·09_/16_ 为主），基于 private 远端 + P1-0 清洗裁决 + §6 C层运行事实，**签字接受为存量豁免**——lint-privacy 不再对其报"红线违规"，仅纳入"内容纪律"抽查。
3. **豁免的例外**：文件名本身含客户名+裸值经营数字者（已知：`2026-05-09_HTF_滞销与库存挤压_4级重判_v1.0.md`；lint 补一条定向扫描）→ 出**重命名清单**走 C 档，改为代号+去数字命名。
4. **增量不豁免**：本 v0.3 生效后新提交一律过红线；存量文件被实质性重写时视同新增。

### 一之三 · 客户代号注册表（多客户时代 R3 判据·新增）

- 建 `.claude/客户代号注册表_LOCAL.txt`（**gitignored·本地 only**）：每行"真名=代号"，由六哥从敏感区映射表**单向摘录**（只摘真名清单供匹配，不含反向解密语境）。
- R3 判据从"花厅坊/好家源等"枚举式改为"**注册表内任一真名**"；新客户签约当日补行。hook/lint 从此文件读词表，隐私铁律（映射表不出敏感区）不破。

## 二、绿线（不变）

方法论/模板/SOP/治理文档；工程与治理执行日志；已脱敏案例；结构资产 CSV 白名单（§五）。

## 三、提交前检查（人工件的闸·过渡态）

十条 checklist 沿用（v0.1 §11 normative·见§七之二），双红线不变：**禁 `git add .`**；**push 前须过本 checklist**。

### 三之二 · 过闸留痕与 push 定性（R4 清偿·新增）

1. **留痕**：每次过闸的 commit，message 尾行加 **`gate: pass`**。无此行=未过闸=审计红旗。lint 抽查最近 20 条 commit 的留痕率。
2. **push 定性**：2026-07-05 解冻为**常设授权**（不必每次再问），但每次 push 前仍须过 §三 checklist；含新增客户相关文件的 push，在 commit message 注明豁免依据或脱敏说明。
3. **退役条款**：pre-commit hook（§六）上线后，本节 checklist 减至 3 条（扩展名/路径/新增文件含注册表真名），仅对白名单目录外**新增**文件生效。

## 四、登记与留痕

SignoffLedger（删除/升级/发布/脱敏签字痕）；Codex 前置登记表（含客户项目状态者移出 tracked 或脱敏）；本 SSOT 为保密口径唯一来源。（原"§9"悬空引用删除——R3 清偿）

## 五、.gitignore 白名单（机械件的闸·不变）

默认忽略 `**/*.{xls,xlsx,csv,db,ppt,key,pdf,dwg}`；仅白名单放行 A_*/B_*/_maintenance/*.csv 结构资产；C_client_specific 永不放行；白名单增删=改保密边界=过本 SSOT 复核。

## 六、执行层 keys not prompts（机械判据并入·R1 清偿）

pre-commit hook 承接以下**客观判据**（硬停）：① staged 含 §五默认忽略扩展名；② staged 路径含 `99_原始素材/`、`C_client_specific`、`_DO_NOT_INGEST`；③ 新增文件名/正文含客户代号注册表真名（审查触发·非硬停，关键词硬停误报率已实证过高）。
**落地时限：W31（2026-08-02）前另立工程任务交付**；上线前 §三 checklist 为过渡态。

## 七、被收敛文件处置（已执行完毕）

GATE-001、HISTORY-PLAN-001、v0.1 均已 superseded（Ledger 有痕）。

### 七之二 · v0.1 条款级 normative 标注（R3 清偿·新增）

v0.1 整体 superseded，但以下五节**条款级仍具约束力**（normative），本 SSOT 引用即生效：§4 五级敏感分级、§11 十条 checklist、§13 脱敏改写规则、§14 历史残留处理、§18 反模式清单。v0.1 其余章节作废。（待 v0.4 或 hook 落地时将五节全文并入，彻底摘除依赖。）

## 八、公开升级预案（R1 清偿·承接 HISTORY-PLAN §8·新增）

1. **红线：本仓（retail-knowledge-vault）绝不设 public**——历史含客户敏感信息，公开=不可逆扩散。
2. 方法论对外开源/公开构建的唯一路径：**另建 orphan 干净新仓**，逐件人工审核搬运（每件过 §一红线+授权链），与本仓无 git 历史血缘。
3. 任何"设 public/改可见性/添加协作者"动作 = D 档签字门。
