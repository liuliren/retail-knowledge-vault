## 2. 风险条款

| 条款 | 风险描述 | 严重度 |
|---|---|---|
| **状态机闭环性** | 现有状态机止步于"Methodology Candidate"，本轮任务要求检查是否应该延伸到"Active"、以及是否需要"Rejected/Archived"状态——**现状不闭环**：一个假设被证据拒绝（evidence_status: rejected）之后没有下一步去向，状态机在"坏结果"分支上是断头的，只在"好结果"分支（一路推进到Methodology Candidate）上是完整的 | 高——这是此前Round1自查已发现、但尚未处理的遗漏，本轮任务再次点出，说明这确实是需要解决的真问题，不是次要细节 |
| **Authorized Analysis定义不够严格** | 此前六哥已确认3类授权诊断流程清单（7D诊断框架/品类诊断题库/现场巡店+POS交叉验证），但只是"流程名单"，没有规定每次使用授权分析时**必须同时明确**：谁授权的、这次任务是什么、数据来源是什么、用途是什么——如果只认"用了这三个流程之一"就自动算Customer Evidence，存在被滥用的空间（比如随便找个理由套用"品类诊断框架"这个名字，但实际没有真正跑完整套诊断） | 高——直接影响Customer Evidence状态的可信度，是这次审议新发现的问题，之前的版本没有考虑到"每次使用"层面的把关，只做了"流程名单"层面的把关 |
| **Methodology Candidate → Active 衔接是否需要重新说明** | 规则里写的是"走既有draft→candidate→active→deprecated签字门，不改变retail vault §11.3现有机制"，但没有明确说清楚：从"外部资料编译规则的Methodology Candidate"到"retail vault既有状态机的candidate"，中间是不是同一个状态、需不需要额外的转换动作 | 中——概念上说得通，但操作上如果六哥或未来的Agent真遇到一个假设走到这一步，可能会疑惑"到底现在算candidate了吗，还是要重新走一遍" |
| **Rejected状态之后如果六哥想"重新验证"怎么办** | 如果一个假设被判定rejected，但六哥后来觉得"当时条件不成熟，现在环境变了，想再试一次"，规则目前没有"重新打开"这条路径 | 中——不是高频场景，但完全没设计过，一旦真遇到会卡住 |

## 3. 建议修改

1. **状态机延伸建议**（供六哥裁决，非既定结论）：

```
Fact → Inferred → Application Pending → Customer Evidence → Methodology Candidate → Active
                                              │
                                              ├─ rejected → Archived（默认终态，六哥可手动重开→回到Application Pending）
                                              └─ validated → Methodology Candidate（原有路径不变）
```

   - 新增 **Archived** 作为rejected之后的默认落点，不是"删除"，是"存档但不再主动出现在查询结果里"（类似retail vault既有的deprecated概念，但deprecated目前专指方法论正文，Archived这里专指外部资料假设条目，两者不是同一个字段，避免混淆）
   - Archived状态允许六哥手动"重开"退回Application Pending，解决"环境变了想重试"的问题
   - "Active"作为状态机终点只是**指向**retail vault既有的active状态，不是本规则新定义一个平行的active——即Methodology Candidate升级到Active，走的还是retail vault CLAUDE.md §11.3原有的checklist和签字流程，本规则不重复定义

2. **Authorized Analysis四要素**（六哥本轮任务已直接给出，建议原样写入规则）：

   每次使用授权诊断流程把假设推进到Customer Evidence，必须同时明确记录：
   - **授权人**：谁批准了这次分析（默认六哥本人，如有授权代理人需写明）
   - **任务**：这次分析具体在验证哪个假设、任务边界是什么
   - **数据来源**：用的是花厅坊哪个门店/哪个时间段/哪类数据
   - **用途**：这次分析结果打算用在哪（比如"验证XX假设，供本季度品类调整参考"）

   这四要素建议作为 `evidence_source: authorized_analysis` 的**子字段**，而不是新增独立字段，格式建议：
   ```yaml
   evidence_source: authorized_analysis
   authorized_analysis_detail:
     authorizer: 六哥
     task: [具体任务描述]
     data_source: [具体数据来源]
     purpose: [具体用途]
   ```

3. **Methodology Candidate 衔接澄清**（建议文字表述层面补充，不改变逻辑）：明确写一句"本规则的Methodology Candidate状态，等同于retail vault既有状态机的candidate状态，不是独立的第二套candidate，两者是同一件事的两种称呼"，消除歧义。
