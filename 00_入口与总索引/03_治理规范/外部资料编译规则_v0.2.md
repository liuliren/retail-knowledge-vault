# 外部资料编译规则 v0.2（三轮审议已完成，待六哥签字生效）

> 三轮审议记录见 [[外部资料编译规则_v0.2三轮审议记录]]。Round 1 已由六哥本人于2026-07-14做出真实判断（非Agent代答），四项决定已并入本文件正文。**本文件仍是draft，需六哥在文末签字后才升级为active，具备约束力。**

## 1. 三层结构

### Layer 1 · Fact（事实层）

**允许**：原始事实、企业动作、来源信息

**禁止**：客户建议、经营判断

**判断标准（Round2审议确认，六哥已认可写入正式规则）**：
> **能不能只靠一个信源验证？能 → Fact；需要跨信源/跨案例才能验证的抽象总结 → Inferred。**
> 例：「ALDI 2026年报显示自有品牌占比65%」是Fact（单一信源可查证）；「ALDI的护城河是采购规模+成本纪律」是Inferred（这是跨多个案例/多篇分析归纳出的判断，不是某一份财报直接写出来的）。

### Layer 2 · Inferred（推断层）

**允许**：机制总结、行业规律假设

**必须标注**：`fact_layer: inferred`

**判断标准**：同上，跨信源归纳 = Inferred。

### Layer 3 · Application Pending（应用待验证层）

所有客户相关内容进入此层，四段式格式：

```markdown
- **应用假设**：[如果...可能...句式]
- **验证条件**：[需要观察什么/满足什么前提]
- **所需数据**：[需要采集的客户自己的数据]
- **当前状态**：[未验证 / 验证中 / 已被客户实证否决 / 已被客户实证支持]
```

**硬性禁用词**：应该 / 必须 / 立即（含立刻/马上/现在就等变体）

## 2. 知识资产状态机

```
Fact → Inferred → Application Pending → Customer Evidence → Methodology Candidate
```

### Customer Evidence 精确定义

**= 已获得客户证据。不等于**：验证成功 / 方法有效 / 可推广。

### Agent 禁止事项（硬性红线）

Agent不能：
1. 自行把 Application Pending 推进到 Customer Evidence
2. 自行把 Customer Evidence 推进到 Methodology Candidate
3. 判断客户成功
4. 修改stable

### 状态推进的两个合法来源

**六哥确认**——六哥本人看过证据后拍板，覆盖所有未列入下方授权清单的场景。

**授权诊断流程**（2026-07-14六哥Round1确认，正式清单，非draft待定）：
1. **7D诊断框架**（全店7D诊断框架2.0，`KB-DX-FULLSTORE-001`）
2. **品类诊断题库/品类诊断框架**（`KB-DX-QUESTION-001`诊断题库、`KB-DX-FRAMEWORK-001`社区生鲜超市品类诊断框架）
3. **现场巡店/POS数据交叉验证**（六哥或授权人员现场走访 + 客户POS/经营数据交叉对照，不依赖特定命名工具，只要满足"现场观察+客户自有数据"两个条件同时成立）

> 以上三类流程产出的结论，**自动视为Customer Evidence状态**，不需要每次额外找六哥手动确认。**清单外的场景**（比如一份纯外部案例类比、没有任何花厅坊自己数据支撑的判断），仍必须走"六哥确认"这条路径，不能自称已进入Customer Evidence。
> 清单的增删变更权在六哥，本文件不预设自动扩展机制。

### Customer Evidence 字段设计

```yaml
evidence_source:  # 三选一
  - field_visit
  - customer_data
  - authorized_analysis

evidence_status:  # 三选一
  - collected
  - validated
  - rejected
```

## 3. 边界规则

外部案例编译页**不能直接产生**：客户经营结论 / 方法论升级 / stable修改

外部案例编译页**只能产生**：学习材料（Layer1+2） / 验证假设（Layer3） / 候选输入

## 4. Summary治理

frontmatter的`summary`字段属于知识资产正文，必须接受与正文同等的三项检查：Fact检查 / 越界检查 / 客户应用检查。

## 5. 高风险主题清单（2026-07-14六哥Round1确认，新增）

Round3应用验证（对35页summary实测审计）发现：**明确越界的表达集中出现在硬折扣对标企业、组织力与经营分析、供应链与自有品牌这三类主题**，陈列/生鲜经营等操作细节类内容几乎不越界。

**规则要求**：编译以下三类主题的外部资料时，编译提示词/编译流程必须额外加入越界提醒，编译完成后必须逐条自查一遍是否出现"XX该抄/XX可迁移/XX宜学"这类直接判断句：

1. **硬折扣对标企业类**（ALDI/山姆/Costco/唐吉诃德/胖东来等对标案例）
2. **组织力与经营分析类**（企业管理方法论、经营分析框架、组织建设案例）
3. **供应链与自有品牌类**（供应商关系、自有品牌开发、采购策略案例）

其余主题（品类管理、生鲜经营、陈列、门店调改、商品规划、库存与订货等）沿用标准编译前检查清单即可，不强制额外加强。

## 6. 编译前检查清单

1. 是否区分了事实和判断？
2. 是否存在对客户的直接建议？
3. 是否存在未经验证的经营数字？
4. 是否把案例经验升级成了规则？
5. 是否需要客户数据验证？
6. **（新增）如果主题属于第5节高风险清单三类之一，是否已额外做过越界自查？**
