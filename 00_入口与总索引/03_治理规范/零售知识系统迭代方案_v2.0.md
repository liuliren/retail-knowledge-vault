## 1. 判断先行:最简正确版

V2 是**菜单,不是规格书**。真正的设计约束来自 Gist 评论区生产派 + GBrain 的取舍:

| 诱惑 | 我们的选择 | 理由 |
|---|---|---|
| confidence score(0.85) | **provenance 溯源**,不打分 | 零售判断不适合伪精确;真信号是证据链 |
| 遗忘曲线 / decay | **supersession 显式取代** | 旧≠过时;衰减错误=重犯错误 |
| 事件驱动自动写入 / dream cycle | **signoff 人工闸门** | LLM 会悄悄污染;判断是 10% 护城河 |
| 一堆后处理机器 | **schema 承载主纪律,机器后置** | 跑稳几十轮再自动化 |
| 知识图谱 / 向量库 | **markdown + 命名边** | 连 10 万页的 GBrain 都以 markdown+git 为真相源 |

> V2 的价值是**词汇表和问题清单**,不是施工蓝图。抄思想,不抄计划。
> 呼应「少即是多」与「模型之外都是 Harness」——**知识系统是零售诊断这匹马的记忆型马具**,做厚的是判断,不是机器。

## 2. 三层映射:Karpathy 三层 → 零售知识库

| Karpathy 层 | 零售落地 | 谁维护 |
|---|---|---|
| raw sources(只读) | 现场记录、诊断原始数据、客户资料、外部素材(万维钢/idoubi) | 只读,永不改写 |
| wiki(LLM 维护) | 方法论资产层:7D、品类诊断、选品、SOP、案例页、KB | 人审 + AI 写 |
| schema(规则书) | **独立治理文件**(M-DEC Schema v0.1.1)+ Wiki 治理方案 | 你和 AI 共同演进 |

**三操作的零售版**(词翻译到现有体系,不另起脚手架):
- **ingest** → 现场决策/诊断写进对应层 + 更新 README/MOC + 追加执行日志
- **query** → 新店诊断前先查库,且**诚实标注未知**(最近一次记录何时/有无矛盾/有无空洞,借 GBrain gap-analysis)
- **lint** → 周度体检由 **G03_Lint v2** 执行(查孤儿决策/矛盾/逼近晋升门槛的规律)

## 3. 核心设计原则(原则级;操作细节见 Schema v0.1.1)

1. **四级固化 = M-DEC 引擎**:现场观察 → 单店决策 → 跨店规律 → 标准动作。晋升靠证据累积,不靠时间;**晋升=上层新建页回指,不改写原记录**。
2. **两轴分清**:层级(生来固定)⊥ 状态(draft→active→superseded/failed/archived)。
3. **provenance 取代 confidence**:结论挂证据链,禁止打分。
4. **supersession 取代 decay**:旧判断不删,标 `被取代::` + 取代原因(自由文本,不强制枚举)。
5. **failed 记录是资产**:决策被证伪 → 标 `failed` 永久保留,进 lint 的「教训」视图,严禁删/淡化。
6. **signoff 人工闸门**:进 active/规律/标准动作必须签字;**不抄 GBrain dream cycle 自动富集**(与闸门冲突,会污染判断)。
7. **捕获优先**:现场 5 秒 → 30 秒落 inbox → 周末 triage 三出口(升为决策/并入已有/丢弃记原因)。schema 再好,料喂不进就空转。

## 4. M-DEC 修复 = 本方案的心脏(最高优先级)

知识库审计早就标出的关键缺口。修复 = 让现场决策**回流 → 回填 → 晋升 → 复用 → 再验证**形成闭环。

最小格式、命名约定、晋升路径细节 → 见 Schema v0.1.1 §4/§5/§11。这里只锚定三条不可动摇的:
- **回填闭环**:无结果的 active 决策 = 孤儿,lint 每周揪出;结果为负 → `failed` 保留。
- **晋升门槛**:默认 **≥3 个独立场景 + 排除共因**(同供应商/季节/区域/口径不算独立);强证据 + signoff 走例外,例外理由须溯源。
- **人工闸门**:晋升前你 sign-off。

## 5. 分阶段路线(触发驱动,不到不做)

> GBrain = 这条路线的**天花板参照**(Phase 3+ 的样子),不是现在要建的东西。

### Phase 0 — 立规则与试运行【进行中】
- 0A｜**已完成**:Schema v0.1.1 预览稿落盘 `03_治理规范/`
- 0B｜**当前**:用 M-DEC-013(膨化小类长尾)在花厅坊真跑一遍(捕获→单店决策→回填占位→是否够格升规律)
- 0C｜试填后回答三问(字段够不够/捕获顺不顺/门槛清不清)→ 决定升 v0.2

### Phase 1 — Add Lifecycle(5–8 月,挂靠花厅坊)
- 动作:跑 ingest + supersession + provenance + 人工闸门 + 周 lint
- 软目标:4 周跑通 **3–5 条高质量 M-DEC**、**至少 1 条回填**;**不强求晋升、不设硬条数**
- 触发 Phase 2:vault 页数稳定增长 + 跨店复用真实发生

### Phase 2 — Add Structure(触发条件,非时间锁)
- **触发条件**:第二个真实客户项目启动 **且** 出现跨店复用需求(预计好家源,但不锁日历)
- 动作:高频实体建独立页 + 补全命名边 + lint 固定进周节奏
- 产出:旗舰店沉淀后,第 2 家店诊断耗时下降 ≥40%(存算分离复利)
- 触发 Phase 3:**页数逼近 100–200**(index 一次读不完)

### Phase 3 — Add Scale / Automation(达到规模才做)
- 此时才考虑混合检索(BM25+向量)/ 自动化 hooks
- 明确不提前:没到 100–200 页线,现有 README/MOC 足够

### (暂缓)Add Collaboration
- 多人/多 Agent 是团队场景,单兵阶段不需要

## 6. 嵌合现有工作流(防注意力分散)

最重要的执行约束:知识系统**不能变成又一个平行大工程**(attention 红线)。

- **不新开战场**:ingest 绑花厅坊周时间块,现场顺手记,周末批量回填
- **借力 community-sop**:B+D 完成后接 A/C/E 时,顺带补 schema 实体定义 / IssueTag→SOP 映射(本就要做)
- **资产刷新搭车**:每刷新一个老资产(7D/品类诊断/选品/科学零售库/花厅坊产出),顺手补 provenance 边
- **存量 grandfather**:M-DEC 候选池存量不回填改造,新候选起按 Schema 走

## 7. 风险与反模式(明确「不做什么」)

1. ❌ schema 不写进 CLAUDE.md(只引用)
2. ❌ 不打 confidence score
3. ❌ 不上 decay / 遗忘
4. ❌ 不上向量库 / 图数据库 / 自动写入 / dream cycle(未到规模线 + 冲突闸门)
5. ❌ 不晋升时改写原记录(晋升=上层新建)
6. ❌ 不删 / 不淡化 failed 记录
7. ❌ 不做 IssueTag 大标签工程——先只 5 个:`长尾过宽/价格带过密/陈列错位/缺货误判/负毛利清仓`
8. ❌ 不设硬条数指标(造假);只求**不断流**
9. ❌ 不让知识系统抢占花厅坊主线注意力(5–8 月主线只有一条)
