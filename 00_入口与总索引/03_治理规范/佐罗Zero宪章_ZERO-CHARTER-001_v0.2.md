# 佐罗 Zero 宪章 v0.2

> 本文是「六哥个人操作系统 佐罗 Zero」的**内核 doctrine**。命名见 [[佐罗Zero_Z系统命名档案_ZERO-NAMING-001_v0.1]]。
> **它不复制已有宪法**——[[CLAUDE]] 全局/项目双宪法即 Zero 的内核源码,本文只把它们**组织进 OS 视角**。
> 状态:`draft`,待六哥签字升 `active`。**提升至全局层另设门槛**:待 Zero 跑通一个完整回路(调度器 boot→flush)后再提(见 §4)。

## 0. 给 Zero 的最高指令(boot 第一行)
> 把可商品化的 90% 持续卸载给 AI,让六哥只运行在不可替代的 10%——那个签名 **Z**。每次运行都复利沉淀为资产。

## 1. Zero 是什么
- **定位**:运行「六哥这个人」的操作系统(非知识库)。它是 §1 三层视角里**驭马人层的放大器**——会增值、不贬值的那一层。
- **不是什么**:不是方法论内容(那是 RetailOS),不是又一个治理文档集,不是对外产品。
- **服务对象**:六哥自己(向内 operate the operator),最终服务于 2026 三目标(30万/5客户/30诊断)与花厅坊 150 天窗口。

## 2. 内核 Kernel(Ring 0 · 不可协商)
最高权限层,任何进程不得覆盖:
1. **90/10 护城河(方向性,非硬比例)**:可商品化的活尽量卸给 AI,六哥的精力**收敛到判断与签字**。90/10 是方向不是配额——别抠数字,抠的是"我是否还在做该买回的低价值事"(每周自问)。
2. **产出收敛为一个 Z**:六哥的最终产出**收敛成一个 Z(判断 + 签字背书)**——可解释=可负责;讲不清推理的结论**不签字、退回**。Z 是最小动作、最大责任(关键创造仍归六哥,只是不外溢到可委托的 90%)。
3. **8 条第一性原则**:引用 [[CLAUDE]] §2,不重抄。主体性优先 · AI 是批判伙伴 · 判断归我执行归 AI · 自动化深嵌业务流 · 一切复利为资产 · 源头不可变编译可重来 · 少而深 · 公开构建。
4. **签字门**(引用项目宪法 §3):改本宪章 / 升 active/stable / 对外交付 / 删改正式文件 / 动客户数据——必须六哥签字。

## 3. OS 架构(七部件 · 含落地现状)
| 部件 | 对应 | 现状 |
|---|---|---|
| 内核 Kernel | 8 原则 + 90/10 + 本宪章 | ✅ |
| 文件系统 FS | vault(raw→wiki→agent_rules)+ 检索层 summary | ✅ 核心检索层刚建成 |
| 进程 Processes | G 回路(Ingest/Query/Lint/M-DEC) | 🟡 部分跑通 |
| **调度器 Scheduler** | 每日 3–5 撬动任务回路 | 🔴 **第一块落地(§5)** |
| 内存 Memory | M-DEC 长期决策记忆 | 🟡 随调度器自然沉淀 |
| 系统调用 Syscalls | 你↔AI 接口(skills/命令) | 🟡 lint-kb 等 |
| 驱动 Drivers | 接外部(客户/公众号/数据) | 🔴 按交付需要逐个长 |

> 判断:Zero 已悄然存在(近期 lint/SSOT/审计/检索层皆其基建);当前最缺**调度器 + 内存 + syscall**——即让 OS 真正"跑起来管时间和判断"那部分。

## 4. Zero 边界(OS 与其上 App,分层非平行)
> **Zero 是 OS;它之上跑多个 App。** 当前两个旗舰 App:
> - **[[RetailOS]]**:交付给**客户**的零售方法论产品(马具/90%)。
> - **媒体程序(Media App)**:产出给**公众**的公众号/视频号/小红书内容(§8 下游)——与 RetailOS **并列**,非其子集。

- **判据(三分)**:交到**客户**手里 → RetailOS;发给**公众** → 媒体程序;运行**你自己**时间·判断·聚焦 → Zero(OS 本体,驭马人/10%)。
- Zero 是 **meta**:retail vault 与个人 vault(§9)都是它挂载的文件系统。
- **提升至全局层的门槛**:Zero 跨双 vault,理应归全局;但**不在签字时立即提全局**,而是**待 Zero 跑通一个完整回路(调度器 boot→flush)、证明它真在运行后再提**(避免过早升格成又一份悬空宪法)。
- 纪律:逐步把"操作层(Zero)"与"内容层(RetailOS/媒体)"在心智与目录上分清,不再糊在一起。

## 5. 第一块落地承诺:调度器先行
- **方案 1(调度器·每日撬动回路)先建**:晨起 boot syscall(AI 读 vault+目标+花厅坊+未关回路 → 提 3–5 撬动任务)+ 晚间 flush syscall(回流产出为 source/决策 → 喂 M-DEC → 备明天)。
- **方案 2(M-DEC 内存)随后自然长**:flush 那一步即决策捕获入口;**不先建空教堂**,待真实决策有量再正式化。
- 理由:调度器最轻最快、强制少而深、直接服务花厅坊交付而非抢注意力。

## 6. 反教堂纪律(铁律)
1. **Zero 必须作为交付的副产品长出来,不能是独立战线。** 每个部件须在跑花厅坊时被真实需要才建。
2. **WOS/OS 本身就是终极"马具",会贬值**(§1)——禁止在"把 OS 做漂亮"上过度投入。
3. **别为 OS 而 OS**:刚经历"治理件膨胀"教训(见 [[2026-06-25_双系统事故后全面审计与收口_v0.1]]),Zero 同样警惕。少而深也适用于本宪章——抵制膨胀。
