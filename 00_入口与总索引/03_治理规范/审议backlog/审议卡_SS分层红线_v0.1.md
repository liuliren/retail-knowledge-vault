# 审议卡 · SS 分层红线

- **提案一句话**：安全库存须按动销分层——A 类稳定快消用正态 SS，B/慢动用泊松/粗式，**C 类长尾禁自动 SS**（给 C 类设 SS = 发永久补货通行证 = 压货），直接服务花厅坊"C类−25%清呆滞"。
- **载体**：[[KB-RETAIL-INV-001_库存管理_v0.1]] §命题「【草案·未激活】SS分层红线」（约 L147）
- **现状 status**：draft · 审议轮次 1/3 · 不作执行依据
- **花厅坊相关度**：**高** — 直接护栏花厅坊"C类−25%清呆滞"目标、方便食品48/52慢动禁套 Z×σ
- **触发词**：红线 · SS · 阈值 · 不得

## 三轮
- **第 1 轮 · 六哥（用户视角）**：☐ 待 — 三问：分层口径对不对？A/B/C 边界够不够？与"C类−25%清呆滞"是否一致？
- **第 2 轮 · 批判角色（魔鬼代言人）**：☐ 待（AI 预置弹药）
  - 预置批判焦点（原）：① C 类"禁自动 SS"会不会误伤 C 类里的季节性/引流长尾（断货掉客流）？② A/B/C 的动销分档阈值本身是不是又一条待审的 draft？③ 慢动品改高频小批订货已能降压货，SS 分层是否冗余？
  - **【深化·三条最锋利反例】**
    - **反例①（什么场景出错）·"C 类禁 SS"会砍掉刚性引流长尾**：花厅坊社区客群里，酱油/食用碱/儿童辅食/特定地方调料这类**周动销 <1 但一断就掉客流**的品，按 ABC 掉进 C 类。若红线"C 类一律禁自动 SS"照字面执行 → 系统不给补货信号 → 缺货 → 主力家庭客一次买不齐愤而转他店。**失效根因**：ABC 按销量分档，但"是否可断货"取决于**可替代性/客流依附度**，与销量不同维。红线若不引入"C 类里手动挂保护标（引流/形象/刚需长尾）豁免"，就是用一维刀切二维问题。
    - **反例②（与哪条规则冲突）·与 §2.7「少而深」及"高频小批订货"自相冲突**：本红线要求先建 A/B/C 动销分档阈值，而该阈值本身又是一条未过审 draft（见反例焦点②）——**红线 A 依赖红线 B**，B 未立则 A 悬空，违反"未过三轮不得被引为执行依据"。同时若花厅坊已推**高频小批+日配/次日达**（社区店主流补货形态），慢动品的压货本就被订货频率压掉了，再叠一层 SS 分层 = 重复造轮子，与"少而深、只关不增回路"冲突。**更轻替代**：不设"SS 分层红线"这条独立标准，改为在订货规则里加一句"C 类默认手动订货、不进自动 SS 引擎"——把红线降级为订货参数默认值，省掉一套分层维护。
    - **反例③（五年后是否成立/成本）**：维护成本 = 每个 SKU 要持续打 A/B/C 标 + 维护三套 SS 参数 + 定期重算动销档，社区店往往店长一人扛，**大概率退化成"建了不更新"的僵尸分层**，反而给出过期补货信号（比统一 SS 更危险）。五年后若店铺上了带需求预测的补货系统（社区店 SaaS 已普及），"手工三层 SS"会被自动服务水平优化取代——这条红线是**过渡期权宜，不宜钉成永久红线**，建议定"复审触发条件（上补货系统即废止）"。
- **第 3 轮 · 专业角色（行业 steelman）**：☐ 待（AI 预置弹药）
  - 预置对标（原）：分角色安全库存（引流/利润/形象/长尾）是零售通行做法（vs 老刘全店统一 SS）；核对是否有成熟的"服务水平差异化 SS"框架可直接采。
  - **【深化·行业成熟对标】**
    - **对标①·差异化服务水平（Differentiated Service Level）是教科书标准解**：现代库存管理（如 Silver-Pyke-Thomas《Inventory Management》、供应链 CSL 理论）的通行做法不是"C 类禁 SS"，而是**给每个 ABC 档位设不同目标服务水平**：A 类 CSL 98–99%、B 类 95%、C 类 90% 甚至更低——SS 由 `Z(CSL)×σ×√LT` 自动算出，C 类因目标服务水平低、SS 自然趋近于 0，但**不为 0、也不"禁"**，从而保留刚需长尾的最低保护。这比"一刀禁 C 类 SS"更平滑、且是同一条公式驱动（维护更省）。建议把红线从"C 类禁自动 SS"改写为"C 类目标服务水平下调至 85–90%、并允许手动上调保护标"——**保留老刘公式骨架、只改参数**，最稳妥。
    - **对标②·生鲜/社区落地形态**：社区生鲜真正省事的替代是**"订货周期缩短 > 拉高 SS"**——盒马/钱大妈系普遍用日配+按前 N 日移动均销订货，SS 概念被"订货频率"吸收。故本红线**进库最稳的形态**是：不作独立"红线"，而是并入《订货规则 SOP》作为一条"C 类默认手动/低服务水平"参数，避免新增一个需三轮审议+长期维护的顶格标准。
