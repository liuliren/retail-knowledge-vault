# 审议卡 · 诊断行动指导体系（KB-ACTION-GUIDE-001）

- **提案一句话**：把诊断结论映射到动作清单的指导体系（映射工具，不替代现场判断）。含「铁律/必须/不得」类表述。
- **载体**：[[KB-ACTION-GUIDE-001_诊断行动指导体系_v0.1]]
- **现状 status**：draft · 审议轮次 1
- **花厅坊相关度**：**中** — 诊断→动作的映射工具
- **触发词**：铁律 · 必须 · 不得

## 三轮
- **第 1 轮 · 六哥（用户视角）**：☐ 待 — 三问：映射覆盖是否穷尽？动作清单颗粒度对不对？"调改取舍归六哥 10%"边界是否清楚？
- **第 2 轮 · 批判角色（魔鬼代言人）**：☐ 待
  - 预置批判焦点：① 映射表把"诊断→动作"固化，会不会让 Agent 机械套用、越过驭马人 10%？② 动作清单是 inferred 还是有交付实证？③ 与全店诊断编排 SOP、7D 框架的分工边界。
  - **【深化·三条最锋利反例】**
    - **反例①（什么场景出错）·同一诊断信号在不同社区店指向相反动作,固定映射会开错药**：例"某品类动销低"这一信号,在花厅坊(社区家庭客、刚需)可能是"陈列/价格问题→优化",但同样信号在另一店可能是"客群根本不需要→直接砍品"。若映射表写死"动销低→降价促销",Agent 会对该砍的品狂促销、越烧越亏。**失效根因**:动作取决于"信号×店型×客群×品类角色"多维上下文,一维 if-then 无法承载,机械套用正好碰穿驭马人 10%(判断/担责本该归六哥)。红线若不写"映射输出是候选动作集、终选归人"就危险。
    - **反例②（与哪条冲突/成本）·与 #9 编排 SOP、7D 框架三份文件边界重叠**:"诊断→动作"这段职责,7D 框架给方法、编排 SOP 给流程、本体系给动作映射,三者极易在"某 D 亮红后怎么办"上重复且措辞冲突。**成本**:动作清单若 inferred(无交付实证),等于把一份未验证的处方钉成"必须/铁律",一旦客户照做出问题,交付方担责。**更轻替代**:不立独立"指导体系"红线,把动作映射降为 7D 框架每个维度下的"参考处方库(候选清单)",标 inferred,不给"必须"字样。
    - **反例③（五年后是否成立）·处方随渠道/供给结构变,固定映射会过时**:今天"清呆滞→场内促销/员工内购"的动作,几年后社区店普遍接入线上清仓/临期折扣平台后,最优动作变了。把动作清单钉成标准,维护成本高且易过期误导。建议:只把"诊断→动作"的**映射逻辑(如何从信号推动作的思路)**沉淀为方法,具体动作清单保持 draft/参考态,定期随渠道更新,不升 stable 红线。
- **第 3 轮 · 专业角色（行业 steelman）**：☐ 待
  - 预置对标：诊断-处方映射（决策树/if-then playbook）是否有成熟范式可锚。
  - **【深化·行业成熟对标】**
    - **对标①·if-then playbook/决策树是成熟范式,但业界都保留"人工复核关口"**:零售运营(如品类管理 8 步、门店运营 playbook)确有"信号→标准动作"的处方库,ITIL/SRE 的 runbook 也是同构思路。**但成熟 playbook 的通例是"给候选动作+触发条件+需人判断的岔口",不是全自动执行**——尤其涉及砍品、调价、改陈列这类不可逆/担责动作,都设人工 gate。这 steelman 了反例①:本体系定位为"候选动作库+人工终选"完全符合业界,定位为"自动映射必执行"则违背通例。
    - **对标②·最稳妥入库形态**:建议入库为 **7D 框架的"维度→候选处方"附录**,每条动作带三标签:`触发信号 / 前置条件 / 是否需六哥签字(不可逆动作=是)`;整体 status 标"参考处方库(inferred,待交付实证补真)",不带"铁律/必须"。这样 Agent 能调用给建议、又强制把不可逆动作交回人手,既有杠杆又守住 10%。
