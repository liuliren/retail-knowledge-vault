## 三轮
- **第 1 轮 · 六哥（用户视角）**：☐ 待 — 三问：状态定义穷尽吗？打分阈值取值？迁移触发条件对不对？
- **第 2 轮 · 批判角色（魔鬼代言人）**：☐ 待（注：已有"双轮验证"修正，但不等于宪法三轮的第2轮批判角色）
  - 预置批判焦点（原）：① 打分阈值是 inferred 还是实证？② 状态机会不会把连续的品类演变硬切成离散档、丢边界信息？③ 与 KB-SKU-STATE-001（单品状态机）层级如何不打架？
  - **【深化·三条最锋利反例】**
    - **反例①（什么场景出错）·离散状态硬切连续演变，在生鲜被季节噪声打穿**：品类演变是连续的，状态机切成"成长/成熟/衰退"离散档，处在"成熟→衰退"边界的品类会被硬分到一侧、触发错误的资源撤离。**社区生鲜更致命**——番茄/叶菜冬天销量下滑是**季节淡季**不是**品类衰退**，状态机在无季节标注时会把"当季退场"误判成"衰退"，据此撤资撤排面，来年旺季反而缺货。**失效根因**：状态迁移判据吃"跨期同比稳定数据"，而生鲜季节强扰动 + 散装无固定条码，数据前提在社区店本就脆弱。红线须补"生鲜品类迁移须先剔除季节因子，否则只作提示不作撤资开关"。
    - **反例②（治理成本 vs 收益·会不会自造官僚）·给几十个品类的小店套大卖场品类机器**：500–2000㎡ 社区单店品类就几十个，店长凭经验就知道哪个火哪个冷；给每个品类跑状态机 + 维护打分阈值数据，**维护成本大概率 > 洞见增量**。这正是元治理类必须自问的：这套"品类状态机 + 打分标准"是解决了真问题，还是把大卖场的品类管理官僚搬进了单店？**若店长最终仍凭经验决策、状态机只是事后贴标签，它就是纯维护负债**。建议正文钉"本状态机仅用于多店/连锁复盘或方法论沉淀，单店交付默认不强制跑分"。
    - **反例③（与哪条冲突/五年后）·与单品状态机层级判据未定义 → 互相打架**：品类状态=衰退，但该品类内可能藏一个新晋爆款单品（单品状态机判为 `_A` 成长）；若品类状态机的"衰退撤资"字面执行，会连带砍掉里面的明星单品。**两台状态机的优先级/覆盖关系不写清就会对同一 SKU 给相反指令**。**五年后**：模型进步后，离散状态机很可能被"连续评分 + LLM 直接读数据判断"取代，这套硬编码档位是否值得现在固化为红线？
- **第 3 轮 · 专业角色（行业 steelman）**：☐ 待
  - 预置对标（原）：品类生命周期（导入/成长/成熟/衰退）是经典模型；核对状态机相对经典 PLC 的增量。
  - **【深化·行业成熟对标】**
    - **对标①·经典 PLC 之外，品类管理主流用的是"品类角色"而非"生命周期状态"**：现代品类管理（ECR / Nielsen）给品类定的是**角色**（目标性/常规性/便利性/季节性），与"生命周期阶段"是**正交两轴**。状态机若只是把 PLC（导入/成长/成熟/衰退）换皮，增量有限——不如直接采"品类角色 × 生命周期"二维矩阵，角色定资源基线、生命周期定调整方向。**须逼问提案：相对经典 PLC，这台状态机多出的判别力具体是什么？** 说不清就是换皮。
    - **对标②·最稳形态=提示信号，不作自动撤资开关**：成熟品类管理里，生命周期判断**从不自动触发撤资**，而是进人工品类评审（category review）复核——这与提案"评分交机器、调改取舍归六哥"精神一致。**进库最稳形态**：① 评分交机器打分，但状态迁移 → 撤资/扩资的动作**必须人工复核**；② 生鲜品类强制"季节因子标注"，淡季不判衰退；③ 明确与单品状态机的优先级（品类状态仅定资源基线，单品明星单品可豁免品类撤资）。三条并上，比裸状态机稳。
