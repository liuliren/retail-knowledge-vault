# 审议卡 · C1 生鲜/社区 SKU 公式失效边界

- **提案一句话**：老刘 SKU 数约束公式（三大算法）**对生鲜散装从根失效——仅包装标品成立**；逐条画出生鲜/社区失效边界表，并纠偏老刘"SKU 永远过多"断言（社区店可能真过瘦）。含"红线/标准/武器"字样边界条目须过三轮方可升执行依据。

- **第 2 轮 · 批判角色（魔鬼代言人）**：☐ 待（AI 预置弹药）
  - 预置批判焦点（原）：① "公式对生鲜从根失效"是否过度否定——加权/分区后老刘公式还有残值吗？② "社区真过瘦" vs 命题"永远过多"两个断言并存，花厅坊到底偏哪边、用什么数据判？③ 边界表 fact_layer 是 inferred 还是有实证？
  - **【深化·三条最锋利反例】**
    - **反例①（过度否定/什么场景出错）·"从根失效"会把公式里仍有效的部分一起扔掉**：老刘 SKU 公式（陈列容量/周转/坪效约束）在**生鲜的包装标品与恒定单元区（盒装豆制品、瓶装调料、定量包装水果）仍成立**，甚至散装区的"陈列面×排面容量"约束也部分有效（一个菜台能摆的品种数有物理上限）。若红线写死"生鲜从根失效"，会让 Agent 在本该用公式约束的包装生鲜上也放弃量化，反而放任 SKU 膨胀。**更准表述**：不是"从根失效"而是"**散装/非标/高损耗品失效，包装标品/定量单元仍适用**"——按品项形态分区判定，而非按"生鲜/包装"大类。
    - **反例②（两个断言并存/自相矛盾）·"社区真过瘦"与老刘"永远过多"若不给判据就是各说各话**：提案同时留了"社区店可能真过瘦"和纠偏老刘"永远过多"两个反向结论，但**没给花厅坊到底偏哪边的可操作判据**——Agent 拿到会无所适从。须钉一条二选一的观测规则：如"**缺货率/顾客点名要而无货的频次 + 品类完整性下限清单覆盖率**判过瘦；周转天数长+滞销SKU占比高判过多"，让"过瘦/过多"变成可测而非口号。否则这条边界表本身不满足宪法"直接可调用"。
    - **反例③（fact_layer/五年后·成本）**：边界表若多为 `inferred`（老刘文本反推）而非花厅坊实证，则它是"批判假设"不是"验收标准"——**含'红线/标准/武器'字样却无实证支撑，正是宪法标准审议要拦的**。成本上，一张逐条失效边界表需持续对照新数据维护，若只停在推断层，五年后无人回填实证就会腐烂成教条。建议：边界表**先降级为"批判性参考/待实证清单"，标 `fact_layer: inferred`，明确"未经花厅坊数据验证前不得作对客弹药"**，只把已有实证的少数条目升执行依据。
- **第 3 轮 · 专业角色（行业 steelman）**：☐ 待（AI 预置弹药）
  - 预置对标（原）：宽品种窄规格（生鲜）vs 宽 SPU 窄 SKU（包装）是社区生鲜通行分区逻辑；核对是否有成熟的生鲜品种数上限模型。
  - **【深化·行业成熟对标】**
    - **对标①·"宽品种·窄规格"是社区生鲜公认结构，且有落地参数可锚**：钱大妈/谊品/朴朴系社区生鲜的通行结构是**生鲜求"品种齐"（SKU 宽度=一餐买得齐）、每品种只留 1–2 个规格**，与包装区"窄 SPU、多规格/多品牌"相反。这 steelman 了提案"生鲜不能套包装 SKU 压缩公式"。可锚的经验参数：社区生鲜店生鲜活跃 SKU 常在数百级、蔬菜小类 30–60 品种打底——花厅坊可用"品类完整性下限清单"而非老刘压缩公式来定生鲜品种数。
    - **对标②·生鲜品种数上限的成熟约束不是"算法公式"而是"损耗/周转闭环"**：成熟生鲜运营控 SKU 数不靠陈列公式，而靠**"每个新增品种须自负损耗与周转"**——即品种数上限=损耗预算与周转天数能兜住的品种数（宽度受损耗预算约束，不受货架面积主导）。故"社区真过瘦 vs 过多"的成熟判法：**看损耗率与缺货率的对冲点**——损耗低且常缺货=过瘦可加，损耗高且滞销多=过多需砍。**进库最稳形态**：边界表以"损耗×周转×缺货"三量作生鲜品种数裁决口径，替代老刘算法公式，且明标实证层级。
