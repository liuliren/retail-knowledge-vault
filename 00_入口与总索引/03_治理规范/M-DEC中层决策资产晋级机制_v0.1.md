# M-DEC 中层决策资产晋级机制 v0.1

> 任务编号：M-DEC-001（机制定义任务，与同名 M-DEC-001_6组角色拆分判据 候选无关）
> 触发：W21 花厅坊样板战役 7 份 M-DEC 候选已沉淀，需明确"候选 → 入库"的判定标准，防止"为填库而填库"与"现场决策永远停在候选状态"两种失败模式。

## 1. M-DEC 的定义

**M-DEC（Mid-level DECision asset，中层决策资产）** 是从真实客户现场决策中抽象出的**可迁移决策件**。

它**不是**：
- ❌ 普通项目日志 / 现场会议纪要（→ 落 `16_/<战役名>/02_战役进展.md`）
- ❌ 纯方法论文章 / 理论稳态总纲（→ 落 `01_科学零售方法论/`）
- ❌ 一次性 SOP 套件（→ 落 `10_咨询交付模板/00_战役SOP/`）
- ❌ 客户特例的剧本化叙述（→ 落 `09_<门店>/`）

它**是**：
- ✅ 现场遇到的真实决策点 → 抽象出**判据 / 框架 / 标准动作**
- ✅ 服务"下一个不同客户 / 不同门店 / 不同品类"的中层判断
- ✅ 既保留现场来源（可追溯），又满足跨场景复用（可迁移）的桥接资产

**M-DEC 之所以叫"中层"**：它在"现场素材"与"稳态方法论"之间。比方法论更具体（带场景边界、带判断阈值、带反例），比现场日志更抽象（剥离客户专名、留下可复用骨架）。

## 3. 四级状态定义

| 级 | 状态 | 物理位置 | 命名后缀 | 用途 |
|---|---|---|---|---|
| **L1** | **v0.1 候选** | `16_/<战役名>/` | `_候选.md` | 现场刚抽出，仅"我觉得这是一个可复用决策点" |
| **L2** | **v0.5 已验证候选** | `16_/<战役名>/` | `_候选_v0.5.md` 或正文版本号升 v0.5 | 至少经一次现场/客户/数据验证 |
| **L3** | **v0.8 准入库稿** | `16_/<战役名>/` | `_准入库稿_v0.8.md` 或正文版本号升 v0.8 | 已抽出场景边界 + 标准动作，待 G05 阶段门评审 |
| **L4** | **v1.0 正式入库** | `04_商品诊断与商品力提升/中层决策库/` | 去掉"候选"后缀，统一命名 `M-DEC-XXX_<主题>_v1.0.md` | 通过 G05 评审，正式纳入跨战役复用资产 |

**关键纪律**：
- 状态升级**只能正向**，不能跳级（v0.1 不可直接升 v1.0）。
- 状态升级**仅在该 M-DEC 经过下一级"晋级条件"全部满足后**触发。
- 状态降级（v0.5 → v0.1）**允许且鼓励**：发现验证有问题时主动退级，不要硬扛。
- 物理迁移**只发生在 v0.8 → v1.0**（mv 至 04_/中层决策库/）；v0.1/v0.5/v0.8 全程留在 `16_/<战役名>/`。

## 4. v0.1 → v0.5 晋级条件（"已验证候选"门槛）

**目的**：从"看起来像个决策点"变成"经现场粗验证、可继续打磨"。

**必满足 5 项**：

| 编号 | 条件 | 验证方式 |
|---|---|---|
| 4.1 | **来源明确** | 正文必标 `[[来源战役]]` + `[[来源决策记录]]` + 决策发生日期 |
| 4.2 | **问题真实** | 不是"也许有用"，而是"现场确实遇到了、不解决就卡住" |
| 4.3 | **决策清晰** | 至少给出"做 X 不做 Y"的二元判断，不能停留在"看情况" |
| 4.4 | **24h 验证** | 经过一次 24h 内的现场/客户/数据三选一回路验证（参 [[CLAUDE.md]] §21 24h 硬约束 + `15_/客户验证日志/`） |
| 4.5 | **待确认 + 适用边界标注** | 正文显式列出"已确认事实 / 推断 / 不适用场景"三段 |

**反例**（任一命中 → 退回 v0.1）：
- ❌ 没有任何来源指针 → 4.1 fail
- ❌ "未来 30 家门店也许都会遇到" → 4.2 fail（推测，非现场真实）
- ❌ "需要根据客户具体情况判断" → 4.3 fail（无二元决策）
- ❌ 没有 24h 客户验证日志 → 4.4 fail
- ❌ 通篇没有"适用 / 不适用"边界 → 4.5 fail

## 5. v0.5 → v0.8 晋级条件（"准入库稿"门槛）

**目的**：从"现场可用的决策"打磨为"下一个客户也能照着做的标准件"。

**必满足 5 项**：

| 编号 | 条件 | 验证方式 |
|---|---|---|
| 5.1 | **抽象出适用场景** | 明示"什么类型的客户 / 什么类型的门店 / 什么类型的品类"可套用；剥离花厅坊专名 |
| 5.2 | **提炼判断框架** | 至少给出 3 个可量化判据（如"动销 < X% + 库存周转 < Y 天 + 毛利率 < Z%"），不靠纯经验拍 |
| 5.3 | **给出标准动作** | 至少 1 套"店长可直接执行"的动作清单（What / Who / When / How / 验收点） |
| 5.4 | **保留反例边界** | 明示"什么情况下本 M-DEC 不适用 / 会失效 / 必须降级"，至少 2 条反例 |
| 5.5 | **能挂接到桥接目录** | 明示该 M-DEC 归属于桥接层 02-08 中的哪 1-2 个模块，并在该模块 README 留指针候选 |

**反例**（任一命中 → 退回 v0.5）：
- ❌ 通篇仍带"花厅坊"等专名 → 5.1 fail
- ❌ "需要顾问凭经验判断" → 5.2 fail（没量化判据）
- ❌ 只有"原则上应该…" → 5.3 fail（没动作清单）
- ❌ 通篇没"不适用 / 反例" → 5.4 fail
- ❌ 不知道归到 02-08 哪个模块 → 5.5 fail

## 6. v0.8 → v1.0 晋级条件（"正式入库"门槛）

**目的**：从"准入库稿"变成"vault 跨战役资产"。**触发**：G05 阶段门评审通过。

**必满足 6 项**：

| 编号 | 条件 | 验证方式 |
|---|---|---|
| 6.1 | **可复用** | 至少完成一次"假想跨客户演练"——拿到肇庆/集团其他门店场景下能不能跑通的模拟 |
| 6.2 | **可交付** | 已对应到至少 1 份 `10_咨询交付模板/` 模板（报告 / 看板 / SOP），形成"决策→交付"链路 |
| 6.3 | **可验证** | 已对应到至少 1 条 [[知识库测试用例]] 条目，能跑测试 |
| 6.4 | **可归类** | 已确认归属 `04_商品诊断与商品力提升/中层决策库/` 下的子分类（按桥接模块或主题分） |
| 6.5 | **可回溯** | 双链 `[[来源战役]]` `[[来源决策记录]]` `[[客户验证日志]]` 全部完整，G03 Lint 无断链告警 |
| 6.6 | **可维护** | 显式标注 owner + 下次复审日期（建议 6 个月）+ 失效条件（如"如客户业态变化则降级"） |

**反例**（任一命中 → 退回 v0.8）：
- ❌ 没做跨客户假想演练 → 6.1 fail
- ❌ 不知道对应哪份交付模板 → 6.2 fail
- ❌ 无测试用例对应 → 6.3 fail
- ❌ 子分类不明 → 6.4 fail
- ❌ 双链断链 → 6.5 fail
- ❌ 没 owner / 没复审日期 / 没失效条件 → 6.6 fail

**G05 评审产出**：通过 → 物理 mv 至 `04_商品诊断与商品力提升/中层决策库/`，文件名去 `_候选` 后缀，统一命名 `M-DEC-XXX_<主题>_v1.0.md`；未通过 → 退回 v0.8 并写明退回理由。

## 9. 与花厅坊战役的关系

**定位**：花厅坊样板战役（战役 #1）是 M-DEC 机制的**第一批真实验证场**。

**核心约束**：

1. **M-DEC 的晋级不以"写完"为标准，而以"客户验证 + 可迁移"为标准**
   - 一个 M-DEC 即使在花厅坊跑得很好（客户老板拍手），但若没法迁移到肇庆 / 集团其他门店 → 仍只能停在 v0.8，不入 v1.0
   - 反之，一个 M-DEC 即使写得不漂亮，但满足跨客户演练通过 → 可入 v1.0

2. **不允许为了填库而脱离现场战役节奏**
   - W20-W22 是花厅坊执行期（启明拉数 + 现场调改 + sign off），**M-DEC 升级动作不得占用现场执行精力**
   - 若 M-DEC 评审与现场执行冲突，**优先现场执行**，评审延后
   - 仅在战役 #1 G06 战后复盘 AAR 完成后，才进入 M-DEC 批量升级窗口（预计 W23+）

3. **M-DEC ↔ 战役的双向追溯不可断**
   - 战役章程 [[16_/花厅坊样板/00_战役章程.md]] 中应列出"本战役预计沉淀的 M-DEC 候选清单"
   - M-DEC 入 v1.0 后，应在战役 AAR 中明示"本战役贡献的 M-DEC 资产"

4. **跨战役验证窗口**
   - M-DEC 入 v1.0 后的**首个跨客户验证窗口**预计在战役 #2（肇庆破晓计划 10 店预热）落地阶段
   - 跨战役验证结果回写至 M-DEC 文件 §"实战登记"段（参 [[CLAUDE.md]] §22.5.4 反向回写机制）

## 10. 禁止事项

**硬红线**（违反一条 → M-DEC 立即退回上一级 + 沉淀到 [[工程操作错误案例库]]）：

1. ❌ **未验证即入库**：任何未通过 §4 / §5 / §6 全部条件的 M-DEC，禁止物理迁入 `04_/中层决策库/`
2. ❌ **仅凭观点入库**：M-DEC 必须有"现场来源 + 客户验证 + 数据/事实依据"三足支撑，纯顾问拍脑袋的观点不入库
3. ❌ **把项目日志包装成方法论**：现场会议纪要 / 客户聊天记录 / 调改进度看板 **不是 M-DEC**，不得换个标题就升级

5. ❌ **在 W20 执行期大规模迁移文件**：M-DEC 物理迁移窗口在战役 #1 G06 AAR 之后；W20-W22 严禁批量 mv
6. ❌ **跳级**：v0.1 不得直接升 v1.0（哪怕看起来很成熟）；每一级都必须留下评审痕迹（在 [[16_/花厅坊样板/05_决策记录.md]] 显式记录）

8. ❌ **无主入库**：无 owner / 无复审日期 / 无失效条件的 M-DEC，禁止入 v1.0（vault 不接受"孤儿资产"）
9. ❌ **绕过 G05 评审**：v0.8 → v1.0 必须经 G05 阶段门评审通过；不得"今晚写完明天就 mv"
10. ❌ **违反客户视角防火墙**：M-DEC 正文中涉及客户内部操作的，必须遵守 [[CLAUDE.md]] §13.16 客户视角防火墙（未经客户亲口告知，不写"启明做了 X"等具体内部操作）
