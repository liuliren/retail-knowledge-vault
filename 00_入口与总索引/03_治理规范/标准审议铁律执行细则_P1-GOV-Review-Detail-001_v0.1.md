# 标准审议铁律执行细则 v0.1

> **层级声明**：本文是全局宪法《标准审议铁律》的**项目级执行细则**，只细化不覆盖；冲突按更严格执行。
> **生效状态分层**：§1、§2 两条经六哥 2026-07-02 明确签字生效（Ledger 已登账）；§3、§4、§5 为方法沉淀，随本细则整体走三轮审议后升 stable。
>
> **§1/§2 生效法源（R1 修订·封死自我豁免先例）**：§1/§2 依全局《标准审议铁律》"六哥可在单次会话中连续完成多轮（快速审议）"通道生效——其条文诞生于 2026-07-02 五件 draft 的第 2 轮独立批判审议实战（五个独立 Agent 的判例实证），经六哥当场第 1 轮确认后签字快审通过；第 3 轮专业对照随本细则整体第 3 轮一并补齐。**通则：签字不得替代轮次；任何"六哥签字即刻生效"的跳轮，必须援引明确的快审通道并在 SignoffLedger 落一行说明，否则不成立。** 此条防止 §1/§2 的生效被后续任何标准援引为"签字即豁免三轮"的先例。
> **自举披露（R1 附）**：本细则的第 2 轮批判由起草方主线程派出、prompt 含"优先转正"框架，属程序瑕疵；第 3 轮须换框架（外部审议制度 steelman）或换派遣方/模型，并在签字时按一次性程序豁免落 Ledger。详见 [[2026-07-03_第2轮批判_标准审议铁律执行细则]] §3。

## §1. 自检不算轮次（✅ 六哥 2026-07-02 签字·即刻生效）

**同会话自检不算正式审议轮次。** 正式审议必须同时满足三要素：

1. **独立上下文**——审议者不携带起草会话的上下文（新会话或独立 Agent）；
2. **独立角色**——与起草角色有实质切换（批判/专业/用户视角，非换个说法的作者本人）；
3. **带反例搜索任务**——审议指令中显式包含"找反例/找边界失效/验算数字"的任务，而非"看看有没有问题"。

自检（含双 Pass 自验证）只能作为**作者修订记录**（frontmatter 标 `起草期自检`），不得作为放行依据、不得计入审议轮次。

> 判例来源：2026-07-02 五件 draft 第 2 轮审议——五件全部自称"双 Pass 已修干净"，独立批判角色仍查出数学级硬伤（7D 锚点算不拢 / C6 结构性天花板），实证自检修不出自己的系统性偏差。

**lite 版口径（R2a 修订·与 /critique skill 对齐）**：<50 行且非红线级的小件，可用**单个独立 Agent**完成批判（不必并行多 Agent），但**仍须满足三要素**（独立上下文优先）——即 lite = "独立单 Agent"，**不是**"主线程单 Agent"（主线程携带起草上下文，按要素 1 不构成正式轮次）。/critique SKILL.md 的 lite 版措辞已同步改为"独立单 Agent"。lite 的划界维度以**影响面**（被引用数/是否红线级/是否含公式阈值）为准，行数仅作粗筛。

**与全局快审通道的关系（R2b 修订·消歧）**：§1 的"独立上下文"要求**不取代**全局"六哥可在单次会话连续完成多轮"的快审通道——六哥可同会话连续裁多轮，但**每一轮的批判执行仍须由独立上下文的 Agent 承载**（六哥切换视角亲审、或派独立 Agent），"连续"指六哥的裁决节奏可压缩，不指审议者可携带起草上下文。两者并存无矛盾。

## §2. draft 互引治理（✅ 六哥 2026-07-02 签字·即刻生效）

**未进第 3 轮审议的 draft 之间，不得互相作为执行依据引用。** 执行规则：

- 允许 related 挂链（导航用途）；**正文不得以 draft 结论为执行依据**（R3 修订·向全局"未完成三轮不得被引用为执行依据"收紧）——确需提及时，只能以"待验证假设"身份出现并标 `[draft·待验证假设·不作执行依据]`；标注是披露手段，不是引用许可。
- 引用链形成放行死锁时，修订顺序按**依赖拓扑**排（先修被引者，再修引用者）；
- **互引成环时**（拓扑排序对环无解）：**环内文件视为一个审议单元，整环一次过三轮、签字时同批升级**（R5 修订）。首个适用案例=本细则 §3 与 `/critique` skill 这对互引环（细则称 skill 母版、skill frontmatter 上游指回细则），两者同批走三轮同批升。
- `/lint` 检查项：正文中把 draft/外部参考写成"铁律/依据"而无标注 → 🟡 警告。

## §3. 批判审议五步法（/critique skill 的方法论母版）

| 步 | 动作 | 硬规则 |
|---|---|---|
| 1 **锁定与设角** | 每件待审文件配一个**独立批判 Agent**（魔鬼代言人），多件并行 | 一件一 Agent，不合并（合并=互相稀释火力）；Agent 不携带起草上下文 |
| 2 **四问模板** | ①什么场景产生错误结果 ②与现有规则的逻辑张力 ③遵守成本 vs 轻量替代 ④五年后还成立吗 | 四问必答，答"无"须给论证 |
| 3 **反例与实算** | 找 ≥3 个边界失效案例；**数字必须代入公式实算**；引用出处必须 grep 回源验证 | 每条批判带行级证据；"自称已修"必逐条复核 |
| 4 **判定分级** | 放行 / 有条件放行（列必修项）/ 打回（列阻断项）/ 降格重定位 | 判定否定文本不否定方向；阻断项编号（R1…）供逐条裁 |
| 5 **汇总裁决单** | 跨文件发现（比单件更值钱）+ 依赖拓扑修订排期 + 六哥裁决点 | 原文件零改动；只写审议记录；裁决权永远在六哥（90/10 红线） |

## §3.5 分级审议与存量特赦（R4 修订·必修·防"生效即瘫痪"）

**背景**：治理目录实测 48 个 .md 中 36 个含触发字样（铁律/红线/不得/必须/零容忍/强制），绝大多数未走三轮。若 `/lint` 按全局"未完成三轮不得引用"严格执法，全库 3/4 标黄、全补审 ≈ 千万级 token——规则第一天就注定选择性执法，而选择性执法比没有规则更坏（把守不守规矩变成裁量权）。故设两条泄压阀：

**A · 分级审议（按影响面定轮次，不搞一刀切三轮）**
| 级别 | 判据 | 审议强度 |
|---|---|---|
| 红线级 | 宪法/权限/客户数据/全局生效 | 全三轮（第2轮独立批判 + 第3轮外部 steelman） |
| 普通标准 | 项目级方法/SOP/框架 | 双角色两轮（批判+专业合为一个 Agent 一轮 + 六哥裁） |
| 参数微调/判例追加 | 阈值改数、加一条判例、措辞订正 | 变更单（单 Agent 核"不引入新矛盾"即可，不重跑三轮） |

**B · 存量特赦（把一次性债务摊销成增量成本）**
- 本细则生效日（六哥签字日）前的既有"铁律/标准"文件，统一视为 `遗留未审`；
- `/lint` 对 `遗留未审` 件**不标黄、不阻断**，但也**不得被引用为新执行依据**（可作导航/历史参考）；
- 遗留件仅在**下次实质修订时**按 A 的对应级别补审——不主动清全库存量，避免千万级 token 一次性债务。

## §4. 思想铁律沉淀（2026-07-02 SG-SCI-6.1 全程实证提炼）

1. **先证据后设计**——rg 检索+精读在前，方案在后；证据不足写"已知/缺证/假设/待读/待裁决"五段，不硬写方案（反空壳）。
2. **带结论找反例**——第二轮验证不重复第一轮搜索，而是带着初步结论专门找推翻它的证据（本轮 4 项判断被此法修正）。
3. **外部 Prompt 落地两道闸**——任何外来执行 Prompt（ChatGPT 等）先过：①假设的资产是否存在（rg 一遍，防"Excel 总表 6.0"式幻觉谱系）；②术语是否撞术语口径统一表（防"数据力/增长力"式红线冲突）。
4. **数学实算复核**——凡含公式/权重/阈值的标准件，审议必代数验算，文字自洽 ≠ 数字自洽。
5. **唯一账本**——同类清单（缺口/任务/决策）三源并存即合并，新增只登总账；双账 = 漂移之源。
6. **事件驱动排期**——无真实触发条件的建设（空表/空库/未用框架）不排死日期；"第二客户进场""首个即时零售开通"才是排期单位。
7. **窗口纪律**——交付窗口内只做"随交付实测"的件；纯建设排窗口后（服务交付，不抢注意力）。
8. **快变事实保鲜**——半衰期 <12 个月的事实（佣金率/平台格局）标 `review_by:`，过期未审即降 pending。
9. **版本单点声明**——代码/文档的本体版本只在一处声明（如 ONTOLOGY_VERSION），正文用中性名（"标准本体"），防半迁移漂移。
