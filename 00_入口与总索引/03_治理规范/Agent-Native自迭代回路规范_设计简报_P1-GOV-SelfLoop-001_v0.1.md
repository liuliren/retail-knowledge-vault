## 0. 一句话结论(底层洞察)

> **六哥提的 5 个元能力不是 5 个功能,是同一个 LLM-Wiki × M-DEC 回路,递归套在 3 个尺度上 + 1 个横切。** 整个晟果系统 = 自迭代回路;模型分层调度是它的算力版 90/10。

```
              ┌─────────── 横切:模型分层调度(Opus判断 / Sonnet机械)───────────┐
              │                                                              │
宏观  知识库总回路  物料→编译→双向标签→反复链接→修订认知→输出→外部实践→新输入 →(回)
              │                                                              │
中观  Skill治理回路  动作追踪→3次规律重复→孵化新Skill→复盘迭代→沉淀          │
              │                                                              │
微观  Skill内回路    跑→脱敏→质检→分析→复盘→迭代(M-DEC的D/E相,套在每个skill外)│
              └──────────────────────────────────────────────────────────────┘
```

## 机制 A — 模型分层调度(横切·六哥第1点)〔判定+实现都要写死〕

**判定规则(按"相位"分,不按任务分):**
| 相位 | 特征 | 走哪个模型 | 例 |
|---|---|---|---|
| 机械相 | 确定性 / IO / 格式 / 重复 | **Sonnet(或 Haiku)** | 脱敏·质检·跑脚本·写台账·更进度表·commit message·CSV整形 |
| 判断相 | 判断 / 裁决 / 创造 / 洞察 | **Opus** | 方法论裁决·诊断解读·调改取舍·复盘提炼·对外稿起草 |

**实现(关键澄清):** Claude Code 主会话模型一会话内固定,**"循环切换"不是切主会话,而是主控 Agent 把机械相 phase 委派给 `Agent(model:"sonnet")` / Workflow 分stage model override 的子代理**。主控留 Opus 做裁决,机械活farm给 Sonnet 子代理 → 一次 skill 跑内就实现 Opus↔Sonnet 交替。
**落地动作:** 每个 skill 模板的每个 Step 前标〔Opus〕/〔Sonnet〕标签;skill 正文加"模型分工"小节。先在【单品类诊断】skill 试点(diagnose.sh本就是纯脚本=0模型成本,真正成本在解读 Agent → 解读=Opus,脱敏自检复核=Sonnet)。

## 机制 B — Skill 闭环标准(微观·六哥第2、4点)〔每个skill必须有D/E相〕

**规则:** 任何 skill 流程跑完闭环后,**强制追加 复盘(D)+迭代(E) 两相**,否则视为未完成。即把 M-DEC 的「现象→决策→推理→预期→**actual→lessons**」套在 skill 执行本身上:
- **D 复盘**:本轮 skill 哪步卡了?误判了吗?产物质量?(如单品类诊断:角色闸有没有误杀/漏杀?脱敏自检够不够?)
- **E 迭代**:把 lessons 回写进 skill 模板(改判据/补铁律/调步骤)→ skill 自我进化。
**worked example(新会话从这开始):** 给【单品类诊断】skill 补 Step 6「复盘迭代」;以**方便食品诊断**为首个被复盘对象(它本就是3页active方法论的§11.3首个实证)。

## 机制 C — 动作追踪→Skill孵化治理(中观·六哥第3点)〔写进治理〕

**规则:** 库内追踪所有重复动作;**某类动作规律性重复 ≥3 次(非偶发;≥10 次必触发)→ 生成一条规则,并提示六哥"是否孵化成 Skill 以提效"。** 这是治理要求,要进 CLAUDE.md §7 操作回路。
**实现:** 轻量**动作台账**(append-only,复用已存在的 `98_AI协作中枢/02_Codex/.../7天动作追踪表.csv` 模式) + G03_Lint 扩一条检查:按动作类型计数,≥3 且有规律 → 输出"孵化提示"。**只提示不自动建**(建 skill=改系统结构,须六哥 Z)。
> 台账schema待定:`日期|动作类型|对象|耗时|是否已有skill`。新会话设计。

## 机制 D — 知识库总回路 / LLM-Wiki(宏观·六哥第5点)〔系统级机制,写进去〕

**规则:** 知识库本身是 loop:`物料输入→编译→建双向标签→反复链接→优化原有内容/修订错误认知/改进好认知→输出→外部咨询实践→新输入` → 再循环。**不仅适用大系统,也递归适用各模块/流程管控/效率任务分配/后期状态。**
**与现状焊接:** 这正是 CLAUDE.md §11.1 的「7环生命周期」(Ingest→Compile→Link→Lint→Distill→Publish→Feedback)。本机制 = 给它补上「**外部实践→新输入**」的回流闭合 + 「修订错误认知/改进好认知」的**双向更新**(不只追加,还回头改旧页)。零售老刘321篇逐字工程 = 这个回路跑通的样板。
