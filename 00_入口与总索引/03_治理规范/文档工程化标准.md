# 文档工程化标准

本文件定义知识库文档的**标准结构**，包括 frontmatter 规范和正文结构模板。所有新建重要文件应遵循本标准。

---

## 一、Frontmatter 标准模板

所有重要 Markdown 文件（方法论、案例、模板、PRD）须包含以下 frontmatter：

```yaml
---
title: 文件标题
version: v0.1
status: draft
owner: 六哥
created: YYYY-MM-DD
updated: YYYY-MM-DD
module: 所属一级目录名
tags:
  - 标签1
  - 标签2
source_type: method / case / template / data / prd / content / research / reference
confidence: high / medium / low / pending
quadrant: I / II / III / IV
client_safety: internal_only / with_disclaimer / client_ready
# v1.3 新增 fact_layer 4 元字段（涉及客户事实 / 数据 / 推断的文档强制 / 参 §13.16-19）
fact_layer:
  observed: []      # 顾问亲眼/亲耳观察的事实
  client_told: []   # 客户亲口告知的事实
  inferred: []      # 顾问推断（必须明示 / §13.17）
  pending: []       # 待确认
client_internal_ops: false  # 是否触及客户内部具体操作的细节判断
related:
  - "[[关联文件1]]"
  - "[[关联文件2]]"
---
```

### 字段说明

| 字段 | 说明 | 必填 |
|------|------|------|
| `title` | 文件标题（与 H1 标题一致）| 必填 |
| `version` | 文件版本号 | 必填 |
| `status` | 文件状态（见下方说明）| 必填 |
| `owner` | 文件责任人 | 必填 |
| `created` | 创建日期 | 必填 |
| `updated` | 最近更新日期 | 必填 |
| `module` | 所属一级目录 | 必填 |
| `tags` | 标签（参考 [[标签体系说明]]）| 必填 |
| `source_type` | 内容类型 | 必填 |
| `confidence` | 置信度 | 必填 |
| `quadrant` | 四象限治理归属（详见下方）| 必填（自 v1.1 起，老文件大版本升级时回填）|
| `client_safety` | 客户可见性等级（详见下方）| 必填（自 v1.1 起，老文件大版本升级时回填）|
| `fact_layer` | **事实分层 4 元字段**（observed / client_told / inferred / pending）| **强制（自 v1.3 起 / 涉及客户事实 / 数据 / 推断的文档）** |
| `client_internal_ops` | 是否触及客户内部具体操作的细节判断 | **强制（自 v1.3 起 / 默认 false / 设为 true 时 client_told 必须非空）** |
| `related` | 关联文件双链 | 建议填写 |

### status 取值（2026-07-03 六哥签字定版 · 全库唯一合法集 · 7 值）

| 值 | 含义 |
|----|------|
| `draft` | 草案，不可作执行依据 |
| `review` | 审议中，可讨论不可下游调用 |
| `candidate` | 候选，可小范围试用（自行可升，无需签字·沿 CLAUDE.md §11.3） |
| `active` | **六哥签字生效**（落 SignoffLedger），可执行可引用，仍可迭代（签字门不变） |
| `stable` | 双轨（六哥2026-07-06 C4裁）：**业务/方法论件**=签字+多案例实证封稳；**治理/标准件**=三轮审议齐+签字（无需案例实证）。变更均需重签 |
| `deprecated` | 已废弃，仅存查证 |
| `archived` | 归档追溯（99_归档/·60_archive已并入） |
| `delivered` | 交付件终态（已交客户/已发布定稿·六哥2026-07-06 D1裁增补第8值）：内容冻结，修订须另起新版 |

> **映射与落日**（六哥 2026-07-03 裁决）：`seed` = `draft` 的合法别名（lint 不报错，逢修订改正名）；旧值 `validated` 落日——存量视为 `candidate`，逢触碰改正名；其余乱值（pending 等 21 种）逢触碰归位，**不开专项**。active 签字门沿 CLAUDE.md §11.3 不放松。

### confidence 取值

| 值 | 含义 |
|----|------|
| `high` | 已验证，高度确定 |
| `medium` | 基于经验判断，合理但未充分验证 |
| `low` | 待验证，谨慎参考 |
| `pending` | 待确认，内容存在待核实部分 |

### quadrant 取值（自 v1.1 起强制）

四象限是 vault 顶层最高纲领，定义任何元素的治理归属。完整说明见 `CLAUDE.md`（原§19=旧版编号·历史留痕，现行见 §11）与 [[2026-05-04_知识库重构方案_v0.2_完整作战方案]] §2。

| 值 | 名称 | 内容举例 | 变更门槛 |
|----|------|----------|----------|
| `I` | 内核稳固（不变 + 可复制）| 反幻觉规则 / 逻辑一致性 / 文档工程化 / 三大优先级 / 三回归原则 / 品牌内核 | 最高 |
| `II` | 内核灵活（不变 + 权变）| 客户访谈话术 / 现场危机判断 / 报告对老板的表达色调 | 中 |
| `III` | 外延复制（变化 + 可复制）| SOP 套件 / 销售包 / 报告模板 / 数据清洗规范 / DisplayMap 模板 | 低 |
| `IV` | 外延权变（变化 + 权变）| 创造性方案设计 / 区域文化差异 / 客户家族关系 / 行业趋势判断 | 低（但每次发挥后必须强制 AAR）|

**回填策略**：新文件强制；老文件不强制回填，但每次大版本升级（v0.x → v0.5 / v1.0）必须补齐。

### client_safety 取值（自 v1.1 起强制）

定义文件的客户可见性等级，是对外发布、客户演示、销售包派生的核心控制字段。

| 值 | 含义 | 典型场景 |
|----|------|----------|
| `internal_only` | 仅内部使用，不可对客户展示 | 战役档案 / 内部诊断分析 / 决策记录 / 客户敏感信息 |
| `with_disclaimer` | 可对客户展示，但需带免责说明或去标识化 | 销售包模板 / 战役章程对客户透明部分 / 案例对外讲述 |
| `client_ready` | 可直接对客户交付 | 客户报告终稿 / 客户演示版 / 老板看板 / 报价单正本 |

**派生原则**：`internal_only` 文件如需对外，必须派生 `with_disclaimer` 或 `client_ready` 版本，不允许直接降级使用。

### fact_layer 取值（自 v1.3 起强制 / 反幻觉硬约束 §13.16-19）⭐

vault 历史上把"推断 / 假设 / 类比 → 写成事实"是反复出错的 #1 模式（参 [[feedback_数据治理与反幻觉机制升级]]）。`fact_layer` 4 元字段强制 vault 在文档级别**显式分层**所有事实/推断陈述。

| 子字段 | 内容 | 例 |
|---|---|---|
| `observed` | 顾问**亲眼 / 亲耳**观察到的事实（vault 在场）| 5/6 老板访谈现场 / G4-A 调改实操 |
| `client_told` | 客户亲口告知的事实（视频会议 / 访谈 / 微信文字记录）| 启明 5/8 晚电话告知"业绩 30% 基于 2025 年销售额" |
| `inferred` | 顾问推断（**必须明示** / 文档正文中也要 `[推断]` 前缀 / §13.17）| 推断"调味组 +33% 是同步调改" |
| `pending` | 待确认 / 已识别但未问到答案 | 执行店长姓名 / 文员姓名 / 调味是否同步调改 |

**规则**：
- 文档里**任何客户事实陈述**必须**至少出现在 4 个子字段之一**，不允许空挂
- `observed` 字段**只能**写顾问在场观察到的 / 不能写"客户做了 X"（除非客户告知）
- `client_internal_ops: true` 时，**对应内容必须在 `client_told` 段**（不是 `observed` / `inferred`）
- 升级文档版本 ≥ v0.5 时 / fact_layer 4 元字段必须装填实质内容（不能空数组）

### client_internal_ops 取值（自 v1.3 起强制 / 客户视角防火墙 §13.16）⭐

定义文档**是否触及客户内部具体操作人**（启明 / 高彬 / 张力 / 佳妮 / 文员 / 等）的判断。

| 值 | 含义 | 配套要求 |
|----|------|----------|
| `false` | 文档**不**写客户内部具体操作（默认 / 安全）| 用门店主体表述（"客户方提供 / 门店 POS 导出"）|
| `true` | 文档触及客户内部具体操作的细节 | **必须**：① `client_told` 子字段非空 ② 文档正文标注每条信息来源（"启明 5/8 告知" / "5/6 老板访谈现场观察"）③ 客户视角防火墙 §13.16 自检 |

**反幻觉对照示例**：

| ❌ 推断写成事实 / `client_internal_ops` 错填 | ✅ 中性主体 / 防火墙合规 |
|---|---|
| `observed: ["启明导出 89 .xls"]` | `observed: ["5/8 16:50 客户方提供门店 POS 全店全品类导出 89 个 .xls"]` |
| `observed: ["佳妮负责采购"]`（除非客户告知）| `client_told: ["启明告知佳妮负责财务+休食采购"]` |
| `client_internal_ops: false` 但正文写"启明亲自做" | 应 `client_internal_ops: true` + 信息来源标注 |

### version 字段升版语义（自 v1.2 起强制）

详细约定见 [[文件命名规范]] v1.2 §二 2.1-2.2。本节锚定 frontmatter 一致性原则：

- frontmatter `version` 字段 **必须等于** 正文 §版本记录最后一行（实际 issued 行，不含"v1.0(规划)"等占位行）的版本号
- 文件名版本后缀（如 `_v0.1.md`）按 [[文件命名规范]] §二 2.2 处理（major rename / point in-place）
- frontmatter `version` ↔ 文件名后缀 ↔ §版本记录三者**任何不一致 = §14 逻辑一致性硬冲突 / G03 必扫**

**反幻觉补充**：
- 升 frontmatter version 时 **必须同步**更新 `updated` 字段为当日日期
- §版本记录新行的"变更内容"必须**列出实际改动**，禁止"小修订" / "完善" 等空泛措辞

## 二、正文结构标准模板

### 方法论文件标准结构

```markdown
# 文件标题

## 1. 文件定位

## 2. 核心定义

## 3. 适用场景

## 4. 与其他方法论的关系

## 5. 操作规则

## 6. 判断标准

## 7. 常见误区

## 8. 示例

## 9. 待完善事项

## 关联上游
- [[上游方法论]]

## 关联下游
- [[下游工具或模板]]

## 关联案例
- [[真实案例]]

## 版本记录

| 版本 | 日期 | 变更内容 |
|------|------|----------|
| v0.1 | YYYY-MM-DD | 初始建立 |
```

### 案例文件标准结构

```markdown
# 文件标题

## 1. 案例背景

## 2. 原始素材说明（来源、时间、范围）

## 3. 已确认事实

## 4. 分析判断

## 5. 行动建议

## 6. 执行记录

## 7. 复盘结论

## 8. 可复用经验

## 9. 待确认事项

## 关联上游
## 关联下游
## 版本记录
```

### 模板文件标准结构

```markdown
# 模板名称

## 使用说明（何时用、谁来用）

## 模板正文

## 填写示例

## 常见填写错误

## 版本记录
```

---

## 三、注意事项

- 方法论文件中，**事实、分析、规则、建议、待确认必须可区分**，可用加粗、引用块（`>`）或列表前缀区分。
- 避免在一个文件里混写多个独立主题。
- 模板文件应尽量简洁，让使用者一眼看懂如何填写。
