# SENSITIVE-GOV-001｜客户数据脱敏与 Git 保密治理规范

> **核心原则(先行)**:不是"客户相关内容一律不进 Git",而是——
> **客户原始数据、客户经营判断、客户派生分析结论、客户诊断日志、dry-run / execute 结果表,默认不进普通 Git。**
> 普通 Git 只管理方法论、模板、治理规则、以及**无客户经营判断**的工程记录。

## 1. 定义(对象)
本规范处理 9 类对象:① 客户原始数据;② 客户经营数据;③ 客户派生分析结论;④ 客户诊断日志;⑤ AI 执行日志;⑥ dry-run / execute 结果;⑦ 治理元数据;⑧ 方法论与模板;⑨ agent_rules。

## 2. 为什么必须治理
1. **Git 会保留历史**——删除当前文件 ≠ 删除历史。
2. **`.gitignore` 只挡未来跟踪**,不能清除已提交的历史。
3. 客户敏感风险**不只来自条码/进价/供应商**,也来自**毛利率、PSD、死货占比、品类诊断、门店对比、金矿候选**等派生结论。
4. **AI 协作日志易混入客户诊断内容**,必须单独治理。
5. **客户信任是零售咨询交付的底线**。

## 3. 四区隔离模型
| 区 | 名称 | 内容 | Git 处理 |
|---|---|---|---|
| **A** | 可同步公开区 | 方法论 / 模板 / SOP / 通用工具说明 / agent_rules / 字段字典 / 不含客户数据的案例框架 | ✅ 进 Git |
| **B** | 可同步内部治理区 | 当前任务队列 / 执行前置状态登记 / 脱敏检查记录 / Git 边界声明 / **无客户经营判断**的工程执行日志 / **无客户派生结论**的治理记录 | ✅ 进 Git |
| **C** | 本地私有客户区 | 客户诊断报告 / 门店经营分析 / 毛利率·PSD·动销·库存诊断 / dry-run·execute review / 客户项目执行日志 / 门店对比结论 / 客户派生分析结论 | ⛔ 默认 gitignored 或本地私有 |
| **D** | 原始数据隔离区 | xls/xlsx/csv/db / SKU 明细 / 条码 / 进价 / 供应商 / 销售流水 / 库存金额 / 会员数据 / 员工信息 / 合同与个人信息 | ⛔⛔ 原则上**永不**进普通 Git |

> **规则**:Git 默认只管 **A + B**;**C** 默认 gitignored / 本地私有;**D** 永不进普通 Git。

## 4. 五级敏感分级(L0–L4)
| 级 | 名称 | 定义 | 示例 | Git | AI 可读 | 可同步 | 需脱敏 |
|---|---|---|---|---|---|---|---|
| **L0** | 公开方法论 | 通用、与客户无关 | 7D 诊断框架、52周MD 方法 | ✅ | ✅ | ✅ | 否 |
| **L1** | 治理元数据 | 治理状态/闸门,无经营判断 | 任务队列、execute 前置登记 | ✅(原则上) | ✅ | ✅ | 否 |
| **L2** | 客户派生汇总 | 客户数据派生的汇总/结论 | 死货占比、金矿候选数、品类诊断 | ⛔ 默认不进 | 谨慎 | ⛔ | **是** |
| **L3** | 客户明细数据 | SKU/条码/进价/流水明细 | 商品明细表、POS 流水 | ⛔⛔ 绝不 | 仅授权 | ⛔ | 必须 |
| **L4** | 个人/会员/合同 | 个人信息、会员、合同 | 会员手机号、员工信息、合同 | ⛔⛔ 最高保护 | 仅授权 | ⛔ | 必须 |

## 5. Git 入库判断标准(可执行)
**凡是别人看到后,能判断某客户门店的:经营好坏 / 商品结构问题 / 库存压力 / 毛利水平 / 动销问题 / 供应商关系 / 采购策略 / 门店对比优劣 / 具体调改方向——默认不得进普通 Git。**

## 6. 允许进入普通 Git 的内容
1. 通用方法论;2. 通用模板;3. 通用字段字典;4. 通用 SOP;5. 无客户经营判断的任务队列;6. 无客户经营判断的执行闸门;7. 脱敏状态登记;8. 反幻觉规则;9. agent_rules;10. 仅描述"如何治理"的规范文档。

## 7. 默认不得进入普通 Git 的内容
1. 客户原始 Excel/CSV/DB;2. SKU 明细;3. 条码;4. 进价;5. 供应商裸名;6. 销售/库存/毛利明细;7. 毛利率/PSD/死货占比/金矿候选数;8. 品类诊断结论;9. 门店对比结论;10. dry-run/execute 结果表;11. 客户诊断类执行日志;12. 未脱敏客户案例;13. 客户项目复盘中的经营判断。

## 8. 客户派生分析结论规则(关键)
**"无条码、无进价、无供应商裸值" ≠ 可提交。** 只要含**客户经营判断或客户派生分析结论**,即进入 **L2 或更高**。
示例(默认不进普通 Git):"某店死货率偏高"、"剔除生鲜后候选数以百计"、"金矿候选数以千计"、"某店库龄污染严重"、"某品类毛利率异常"、"方案 C 更适合该店"。

## 9. AI 执行日志治理规则
| 日志类型 | Git |
|---|---|
| 工程执行日志 | ✅ tracked |
| 治理执行日志 | ✅ tracked |
| 客户诊断聚合日志 | ⛔ 不 tracked |
| 客户项目执行日志 | ⛔ 默认不 tracked |

## 10. .gitignore 规则建议(只写文档,不本轮改)
原则:精准忽略**客户诊断聚合日志 / 原始数据 / dry-run·execute 结果表**;**不误伤** `04_执行日志/` 治理日志、方法论、模板、规范文档。
建议规则示例(**仅记录于本文档,未授权前不改 .gitignore**):
```
**/*.xlsx
**/*.xls
**/*.csv
**/*.db
/客户敏感执行日志/
/internal_logs/
/dry-run_results/
/execute_results/
98_AI协作中枢/01_Claude_Code/Claude执行日志.md
```

## 11. 提交前检查清单
1. `git status --short`;2. `git diff --cached --name-only`;3. `git diff --cached --stat`;4. `git diff --check`;5. staged 是否含 xls/xlsx/csv/db;6. staged 是否含**客户名 + 经营数字**;7. staged 是否含**毛利率/PSD/死货/金矿/动销/库存诊断**;8. staged 是否含 dry-run/execute 结果;9. staged 是否含客户诊断日志;10. **禁止 `git add .`**。

## 12. 文件裁决表模板
| 文件 | 类型 | 敏感级别 | 裸值红线 | 含客户经营判断? | Git 裁决 | 处理方式 |
|---|---|---|---|---|---|---|
| … | A/B/C/D | L0–L4 | 0/有 | 是/否 | 见下 | 见下 |

裁决选项:可提交 / 暂不处理 / 不可提交 / 脱敏后再议 / 移入 gitignored 区 / 只读评估。

## 13. 脱敏改写规则
客户案例要沉淀为方法论,必须:① 去客户名;② 去门店名;③ 去具体数字;④ 去供应商/商品/SKU 识别信息;⑤ 改成区间或相对描述;⑥ 保留方法论结构,不留客户经营细节;⑦ 标注"案例已去客户化"。
**示例**:
- 原句(⚠️下列数字为虚构演示，不对应任何真实门店):XX 店死货 47%，剔生鲜 900，金矿候选 1500。
- 改写:某社区超市项目中,老库存与低动销叠加形成明显库存污染,需先通过非快销品筛选与库龄闸门建立候选池。

## 14. 历史残留处理原则
1. `git rm --cached` 只停止未来跟踪;2. `.gitignore` 只阻止未来误入;3. 历史敏感内容如已提交,必须**另立 SENSITIVE-HISTORY-AUDIT 只读评估**;4. **禁止未经用户明确授权**执行 `git filter-repo` / BFG / rebase / 强推;5. 历史改写前必须确认远端、分支、备份、协作状态。

## 15. Claude / Codex 执行提示词固定条款(可复制)
> 执行前必须遵守:
> - 不得提交客户原始数据。
> - 不得提交客户派生分析结论。
> - 不得提交客户诊断日志。
> - 不得提交 dry-run / execute 结果表。
> - 不得提交 xls/xlsx/csv/db。
> - 不得 `git add .`。
> - 每次提交前必须核对 staged 全量文件。
> - 如发现客户毛利率、PSD、死货占比、金矿候选、品类诊断、门店对比,**默认停止并报告**。

## 17. 最小执行流程(7 步)
1. 判断文件类型(A/B/C/D);2. 判断敏感级别(L0–L4);3. 检查裸值红线;4. 检查客户经营判断;5. 给出 Git 裁决;6. 精确 `git add`;7. 提交前 staged 全量复核。

## 18. 反模式
1. 认为"无条码/无进价"就安全;2. 把客户诊断结论写进执行日志;3. 用 `git rm` 代替 `git rm --cached`;4. 依赖记忆而非 `.gitignore`;5. 用 `git add .`;6. 把 dry-run review 当普通治理稿提交;7. 在未审历史前执行 filter-repo;8. 客户案例未脱敏就沉淀为方法论。
