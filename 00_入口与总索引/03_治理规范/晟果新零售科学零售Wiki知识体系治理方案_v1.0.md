## 1. 治理定位

**本方案解决什么**
- 把分散在 `00_入口与总索引/03_治理规范/`、`02_方法论索引/`、`_wiki入口/` 的多份治理文件，**统合为一套与 v0.5 架构对齐的 Wiki 治理总则**；
- 让「总纲 / 三力 / M1-M8 / M-DEC / RetailOS / 产品线 / 案例 / 工具模板 / 外部素材」之间形成**稳定、可导航、可引用、可扩展**的关系；
- 把 30 天商品提效包验证出来的实践（物理拆分客户/内部版、client_safety 分级、产品线闭环结构）**上升为可复用的通用治理规则**。

**本方案不解决什么**
- 不重写任何方法论（不动总纲/三力/M1-M8/M-DEC/RetailOS 正文）；
- 不做批量改名、移动、断链修复、归档清理（这些是后续分步专项，且需逐项确认）；
- 不落盘术语口径表（那是 P1-A）；
- 不补组织力正文（那是 P1-C）。

**与 v0.5 的关系**：v0.5 锁定的是**晟果新零售科学零售知识体系的内容架构**（三力主骨架/数据底座支撑层/顾客营销横切/L1-L4/M1-M8 唯一主定义/P0-1 降级/RetailOS 外壳/M-DEC 决策层）。P1-B 锁定的是**承载这套架构的物理治理规则**（目录/文件/命名/frontmatter/链接/密级/生命周期）。一个是「内容是什么」，一个是「内容怎么放、怎么连、怎么维护」。

**与 P1-A 的关系**：P1-A（术语口径统一表）是 P1-B 的**子模块/依赖**。本方案只给出「术语条目应如何登记、归属哪个主定义」的**规则位**，不落盘术语条目本身；术语表落盘前不得据其重构全库（§17 风险）。

**与 30 天产品线的关系**：产品线是 P1-B 的**首个样板与压力测试**。§10 产品线治理规则直接以它为标准结构固化；client_safety 四级（§9）直接来自它的实战。

> **一句话定位**：Wiki 治理不是重写方法论，而是让晟果新零售科学零售知识体系**可导航、可引用、可维护、可扩展**——服务真实交付，不做知识库洁癖。

## 4. 主定义文件治理规则

**主定义文件标准**：①对某一概念/产品/方法是**唯一权威来源**；②被 `核心概念主定义索引` 登记；③其他文件只能引用不得重定义；④修改它需检查全部引用一致性。

**不能成为主定义**：README（导航）、索引（聚合）、案例（验证）、旧素材（原料）、AI 输出区草稿（未确认）、规划件（未实现）。

| 类型 | 是否可作主定义 | 判断标准 | 示例 |
|---|---|---|---|
| L3 模块表 | ✅ 是（M1-M8 唯一）| 方法武器权威定义 | `00_模块索引与骨架映射.md` |
| 产品主定义 | ✅ 是（产品级）| 产品权威定义 | `社区超市30天商品提效包_主定义_v0.1` |
| 模板定义 | ✅ 是（模板级）| 模板字段/枚举权威 | `SKU动作建议表_模板定义_v0.1` |
| README | ❌ 否 | 只导航不定义 | `30天商品提效包/README` |
| 索引 | ❌ 否 | 只聚合不定义 | `核心概念主定义索引`（它登记主定义，自身非主定义）|
| 旧素材归位地图 | ❌ 否 | 素材索引 | `…旧素材归位地图_v0.1` |
| 案例文件 | ❌ 否 | 样板验证 | 花厅坊 G4-A 记录 |
| P0-1 Clippings 能力地图 | ❌ 否（已降级）| 导航索引 | `P0-1_Clippings能力地图索引_v0.1` |

**主定义 vs README vs 索引 vs 案例 vs 旧素材**：主定义=定义概念；README=目录入口导航+状态；索引=跨文件聚合+状态标记；案例=真实验证+反哺线索；旧素材=未提炼原料。四者**均不得与主定义冲突**，发现冲突以主定义为准。

## 5. README 与索引治理规则

**README 职责**：入口导航 / 状态说明 / 当前主文件链接。**不承载**正文方法论、大段历史说明，不与主定义冲突。

**索引职责**：聚合 / 导航 / 状态标记。**不重定义**概念。

**建议 README 模板（统一结构）**

```
## 本目录定位      —— 这是什么、服务什么
## 当前主文件      —— 指向主定义（wikilink）
## 已完成文件      —— 列表 + client_safety + 状态
## 待建文件        —— 待补清单（明确"待建/待校准"）
## 版本状态        —— 当前基线版本 + 锚点
## 注意事项        —— 边界/纪律（引用不另起炉灶等）
```

现有 `30天商品提效包/README` 与 `模板库/README` 已基本符合，可作样板；后续各一级目录 README 逐步对齐（分步、不批量）。

## 6. 文件命名与版本治理规则

| RetailOS | `RetailOS_vX.Y_用途` | 商业交付版独立 | `RetailOS v2.0 商业交付版` | 当新理论层 |

**关键裁决**
- **文件名 v0.1 / frontmatter v0.2 不一致**：默认**以 frontmatter version 为准**（内容真实版本），文件名**保留旧名避免断链**，在 `核心概念主定义索引` 或文件版本记录登记「文件名 v0.1 ＝ 内容 v0.2」的对应说明；**仅当跨大版本（v0→v1）或对外交付时才改文件名**，且需同步全部反链。
- **何时改文件名**：仅大版本跃迁/对外定稿/纠正错别字且无外链时；改名必须先查反链。
- **何时只改 frontmatter**：小版本迭代（v0.1→v0.2）一律只改 frontmatter + 版本记录。
- **日期**：方法论/主定义**不放**日期入名（求稳定）；案例/复盘/审计/AI 输出**可放**日期（求时序）。
- **中文命名**：保留（Obsidian 双链友好）；行业通用词（SKU/PRD/SOP/DisplayMap）可英文。
- **空格/括号**：正文件名避免空格；`（本件）`类括号仅用于清单表述，不入实际文件名。

## 7. frontmatter 治理规则

> 现有文件 frontmatter 已较规范（title/version/status/owner/created/updated/module/tags/source_type/client_safety/related/quadrant）。本方案做**字段统一 + 枚举收敛**，不强制回填存量（分步）。

| 字段 | 是否必填 | 取值规则 | 说明 |
|---|---|---|---|
| title | 必填 | 与文件主题一致 | 可与文件名略异 |
| version | 必填 | `vX.Y` | 内容真实版本，权威 |
| status | 必填 | draft/active/stable/archived/deprecated | 生命周期 |
| owner | 必填 | 默认"六哥" | 责任人 |
| created | 必填 | YYYY-MM-DD | 建档日 |
| updated | 必填 | YYYY-MM-DD | 末次改 |
| module | 必填 | 所属目录路径 | 定位 |
| source_type | 必填 | 见下枚举 | 文件性质 |
| client_safety | 必填 | internal_only/shareable_draft/client_ready/public_ok | 对外密级 |
| priority | 选填 | P0/P0.5/P1/P2/backlog | 任务/资产优先级 |
| related | 选填 | wikilink 数组 | 关联（§8）|
| tags | 选填 | ≤5，中文，先登记 | 沿用标签体系说明 |

**枚举（建议收敛口径）**
- `status`: draft → active → stable → archived → deprecated

## 8. 链接与反链治理规则

**何时必须加 wikilink**：主链路上下游、主定义引用、产品↔模板↔总清单、案例↔方法。
**何时不加**：普通名词、同一文件内重复词、L8 素材的零散提及（统一走归位地图）。
**related vs 正文链接**：related=结构性关联（同级/参考）；正文 wikilink=主链路引用（上下游/主定义）。
**反链最小同步**：A 引用 B 时，若 B 是主定义/主清单，在 B 的「被引用/下游」处最小登记一条即可，**不做全量反链回填**。
**前瞻链接（forward link）**：允许指向尚未建的文件（如早期 `[[30天商品提效包]]`），但必须在**未解析链接登记表**记一条（待建/预期路径）。
**未解析链接**：不即时修复，**登记**到 P1-B-04 清单，分批处理。
**不要为每个词都加链接**（避免链接噪音）。

**主链路链接规则**
- 主定义 → 必须链接：总纲/上游/下游；
- 产品文件 → 必须链接：产品主定义/交付物总清单/模板库；
- 模板 → 必须链接：所属产品主定义；
- 案例 → 必须链接：对应方法（M 簇/M-DEC）与产品；
- 旧素材 → **只在归位地图登记**，不在主链路乱连。

## 12. M-DEC 治理规则

> 沿用现有 [[M-DEC中层决策资产晋级机制_v0.1]]，对齐生命周期。

| 阶段 | 标准 | 证据要求 | 可用范围 |
|---|---|---|---|
| candidate | 从案例/经验提出的候选规则 | ≥1 真实场景线索 | 候选池内部，不对外 |
| validated | 多案例/复盘验证有效 | ≥2 独立案例或一次完整复盘 | 内部交付可引用 |
| stable | 反复验证、纳入标准流程 | 稳定复现 + 模板字段承接 | 产品/RetailOS 可固化 |
| deprecated | 被取代/证伪 | 反例或上位规则 | 仅历史留档 |

**裁决**：①能成为 M-DEC 的=可复用的**落地决策规则**（非一次性判断）；②M-DEC ← 案例证据（无证据不晋级）；③M-DEC 是 RetailOS 决策内核的规则来源，RetailOS 是其封装外壳；④M-DEC 可转化为模板字段/校验；⑤**严禁把单次经验判断直接升级为 stable 规则**（防经验幻觉）。

## 17. 风险与边界

- **过度治理**：把治理做成洁癖，脱离真实交付——以「服务交付」为准绳，非必要不立规。
- **批量改名致断链**：Obsidian 双链脆弱——改名前必查反链，默认保留旧名。
- **README 承载正文**：README 膨胀成方法论——严守「只导航」。
- **主定义泛滥**：人人都想当主定义——经索引登记，一概念一主定义。
- **案例污染主定义**：把花厅坊个案当通则——案例须抽象脱敏才反哺。
- **归档回流污染**：旧正文复制回主库——只抽取重写，不搬运。
- **AI 输出直接进正文**：预览/草稿当定稿——三步确认才入库。
- **术语表未统一前过早重构**：P1-A 未落盘就按假定术语改全库——B-07 须等 P1-A。
- **真实数据泄露**：客户名/SKU/条码/销售额外泄——client_safety 四级 + 物理拆分 + gitignore 三重兜底。
- **命名口径混用**：把「生活性零售」当总体系名——统一为「晟果新零售科学零售知识体系」（见 §0）。
