# 外部资料编译规则 v0.2（draft·未完成三轮审议）

> ⚠️ 本文件涉及"规则/必须/禁止"字样，按全局宪法"标准审议铁律"，签字前均为draft，不具约束力，不得被其他Agent或文件引用为执行依据。

## 触发背景

2026-07-13批量ingest生成35个draft页面后的价值审计发现：约25/35页在"so-what"段落把外部案例（ALDI/山姆/胖东来/美团快乐猴等）直接转写成"花厅坊应该怎么做"式经营指令，越过了"外部案例只做参照、经营转化留给六哥判断"的边界。根因是编译提示词要求"so-what必须落到花厅坊场景"，这个要求把fact层（案例是什么）和应用假设层（对花厅坊可能意味着什么）混在了一起写成陈述句。

## 三层结构（草案）

### Layer 1 · Fact（事实层）

**允许写**：
- 来源事实（谁说的、哪篇文章、什么时间）
- 企业动作（某企业做了什么、数据是多少）
- 行业背景（市场规模、竞争格局、政策变化）

**禁止写**：
- 对晟果客户（花厅坊/好家源等）下结论
- 任何形式的"六哥应该/花厅坊应该"式陈述句出现在这一层

**frontmatter要求**：本层内容不需要额外字段，是正文的基础叙述部分。

### Layer 2 · Inferred（推断层）

**允许写**：
- 从事实中提炼的机制/规律（如"ALDI的护城河是采购规模+费用纪律"）
- 可迁移的观察（如"硬折扣模式在人力密集市场更容易复制"）

**必须标注**：`fact_layer: inferred`（区分于观察到的事实）

**仍然禁止**：直接对花厅坊下结论——这一层是"这个机制/规律是什么"，不是"花厅坊该怎么用这个机制"。

### Layer 3 · Application Pending（应用待验证层）

**所有涉及客户/花厅坊应用的内容，必须放进这一层，且必须用以下格式，不能写成陈述句：**

```markdown
## 应用待验证（pending_application）

- **假设**：[这个案例的机制，如果迁移到花厅坊，可能带来什么效果——用"如果...可能..."句式]
- **验证条件**：[要验证这个假设成立，需要观察什么现象/满足什么前提]
- **所需数据**：[验证这个假设需要采集哪些花厅坊自己的数据]
```

**硬性禁止用词**（出现即视为越界，需要重写）：
- "应该"
- "必须"
- "建议立即"
- 任何祈使句式的经营指令（如"六哥要做XX""花厅坊需要马上XX"）

**替代写法**：
- ❌ "花厅坊应该在1个货架小范围测试ALDI的分权模式"
- ✅ "假设：ALDI分权模式在花厅坊这类小规模门店，可能通过店长自主权提升响应速度；验证条件：需先观察当前门店决策链条的实际响应时长；所需数据：近3个月现场问题从发现到解决的平均耗时"

## 与既有格式（14_外部案例与行业研究模板）的关系

既有模板结构（引用块 → 结论先行 → MECE小节[本质/可落地要点/so-what] → 与方法论的关系 → 开放问题）里，**"so-what"字段是本次问题的直接来源**，本规则草案建议：
- "本质"归入 Layer 1 Fact
- "可落地要点"归入 Layer 2 Inferred（如果是机制提炼）或 Layer 1（如果只是事实罗列）
- "so-what"整体改造为 Layer 3 Application Pending 格式，不再允许自由陈述句

**本节改动涉及既有模板结构调整，是否正式修改模板需要六哥确认**——本文件只是draft草案，不自动生效。
