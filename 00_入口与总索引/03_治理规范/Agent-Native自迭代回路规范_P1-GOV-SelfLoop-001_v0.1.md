## 0. 一句话结论 + 总图

> **晟果系统 = 一个 LLM-Wiki × M-DEC 自迭代回路,递归套在 3 个尺度上,由"模型分层调度"横切供能。** 5 个元能力不是 5 个功能,是这一个回路的不同切面。

```
         ┌──── 横切·机制A 模型分层调度(脚本 < Sonnet 机械相 < Opus 判断相)────┐
         │                                                                   │
宏观·机制D 知识库总回路   物料→编译→双向标签→反复链接→修订/改进认知→输出→外部实践→新输入 ↺
         │                                                                   │
中观·机制C Skill治理回路  动作追踪→≥3次规律重复→生成规则+提示孵化Skill→沉淀  │
         │                                                                   │
微观·机制B Skill内回路    跑→脱敏→质检→分析→【复盘D→迭代E】→skill自进化      │
         └───────────────────────────────────────────────────────────────────┘
```

**判据共识**:三个尺度跑的是同一套 M-DEC(现象→决策→推理→预期→actual→lessons→蒸馏);差别只在"现象"的粒度(一次 skill 跑 / 一类动作 / 整库物料)。

## 1. 机制 A — 模型分层调度(横切·供能层)

### A.1 定义
按**相位(phase)而非任务**,把工作分给不同算力层。是 90/10 护城河的**算力版**、KB-BUILD-001 token 经济学的**算力延伸**:能用更便宜层完成的,绝不上 Opus;判断相绝不下放。

### A.2 判定规则(写死·可检查)
| 相位 | 特征(可被确定性脚本/模板替代?) | 用 | 例 |
|---|---|---|---|
| **脚本相** | 完全确定性,无需语言模型 | **纯脚本(0模型)** | diagnose.sh 脱敏/算指标·git add·文件移动·CSV 整形 |
| **机械相** | 确定性 + 需轻语言,格式/重复/IO | **Sonnet(或 Haiku)** | 脱敏自检复核·质检·写台账/进度表·commit message·摘要套模板·批量打标签 |
| **判断相** | 判断/裁决/创造/洞察,讲不清就不能签字 | **Opus** | 方法论裁决·诊断解读·调改取舍·复盘提炼·对外稿起草·矛盾 steelman |

> **判定口诀**:能写成 SOP 让脚本/模板跑的 → 脚本相;需轻判断但错了能查出来的 → 机械相;**需现场判断、且"可解释=可负责"红线下讲不清就不签字的 → 判断相**。
> **优先级铁律**:`脚本 > Sonnet > Opus`。先问"能不能脚本化",再问"能不能 Sonnet",最后才 Opus。

### A.3 实现(Claude Code 三条路径)
1. **子代理委派(主力)**:主会话一会话内模型固定,**"循环切换"= 主控把机械相 phase 委派给 `Agent(model:"sonnet")` / Workflow 分 stage `model` override 的子代理**。主控留 Opus 做裁决,机械活 farm 给 Sonnet 子代理 → 一次 skill 跑内 Opus↔Sonnet 交替。
2. **会话级切换**:机械-heavy 整段(纯脱敏日/质检日/台账日)用 `/model` 切 Sonnet 会话整段跑,省 Opus 配额(Sonnet 是独立 quota 桶)。
3. **脚本前置**:能脚本化的优先脚本(连 Sonnet 都不花),如 diagnose.sh。

### A.4 落地 + 红线
- **落地**:skill 模板每个 Step 前标〔脚本〕/〔Sonnet〕/〔Opus〕;skill 正文加「模型分工」小节。**先在【单品类诊断】skill 试点**(脱敏/自检/算指标=脚本+Sonnet;诊断解读/调改取舍=Opus)。
- **可检查(lint)**:G03_Lint 查 skill 模板是否每 Step 有模型标签。
- 🚫 **红线(must-not)**:判断相(方法论裁决 / 对赌式签字背书 / 调改取舍)**不得降级**给 Sonnet 或脚本代做——那是驭马人层,降级即破护城河(CLAUDE.md §1)。

## 2. 机制 B — Skill 闭环标准(微观)

### B.1 定义
任何 skill 跑完**功能闭环**后,**强制追加 D 复盘 + E 迭代两相**,否则视为**未完成**。即把 M-DEC 的「actual→lessons」套在 **skill 执行本身**上,让 skill 自进化。

### B.2 规则(写死)
- skill 模板**必须含 Step「复盘D」+「迭代E」**;缺则 lint 报「skill 未闭环」。
- **D 复盘固定问题清单**(跑完即答,≤5 行):
  1. 哪一步卡了 / 返工了?
  2. 有没有误判?(诊断类:角色闸**误杀/漏杀**?脱敏**残留**?事实层标错?)
  3. 产物达交付标准了吗?(§10 世界级:结论先行 / MECE / fact_layer / 0 断链)
  4. 哪条铁律被触发/差点被破?
- **E 迭代**:把 lessons **回写 skill 模板**(改判据 / 补铁律 / 调步骤 / 加/调模型标签)→ skill 版本 +0.1,在 skill 目录 `_复盘台账.md` 记一行(日期·对象·改了什么·为什么)。
- **相位**:D=判断相(Opus);E 中"识别改什么"=Opus,"落字改模板"=Sonnet。

### B.3 落地(worked example)
- 给【单品类诊断】skill 补 **Step 6「复盘迭代」**;以**方便食品诊断**为**首个被复盘对象**(它本就是 3 页 active 方法论的 §11.3 首个实证)。
- 建 `.claude/skills/单品类诊断/_复盘台账.md`。

### B.4 焊接
- = M-DEC 回路(§7)应用到 skill 自身;= §11.1 七环的 ⑦Feedback 在 skill 尺度的落地。

## 3. 机制 C — 动作追踪 → Skill 孵化治理(中观)

### C.1 定义
库内追踪重复动作;**某类动作规律性重复 → 生成一条规则,并提示六哥"是否孵化成 Skill 提效"**。这是治理要求,写进 CLAUDE.md §7 作一等公民。

### C.2 规则(写死)
- **触发**:同类动作 **≥3 次且有规律** → 进「孵化候选」;**≥10 次 → 强制提示**(必须报六哥)。
- **「规律性」三判据(须同时具备,防把偶发杂活误判)**:
  1. **步骤稳定**——可复述为 SOP(每次步骤同构);
  2. **频率周期**——非一次性救火,有重复节律;
  3. **输入同构**——输入形态一致(如"某品类 raw .xls")。
- **动作(must-not 自动建)**:🚫 不自动建 skill(建 skill = 改系统结构 = 须六哥 Z)。只 ① **生成一条规则草稿** ② 向六哥提「是否孵化 skill」**一句话提示**。

### C.3 实现
- **动作台账(append-only)**:schema `日期 | 动作类型 | 对象 | 相位(脚本/机械/判断) | 是否已有skill`。复用现存 `98_AI协作中枢/02_Codex/.../7天动作追踪表.csv` 模式,落点待定(建议 `98_AI协作中枢/00_总控/_动作台账.csv`)。
- **计数检查**:G03_Lint v2 扩**第 7 查「动作孵化查」**——按动作类型聚合计数,≥3 且过三判据 → 输出「孵化提示清单」。
- 相位:记台账=机械相(Sonnet/脚本);判「够不够格孵化」=判断相(Opus)。

### C.4 焊接 / 落地状态
- 写进 CLAUDE.md §7 作 **G12_SkillForge**(或挂 G03_Lint 第 7 查)。台账 schema=本规范定义(可即用);自动计数=`planned`(挂 G03_Lint v2,现 `in_progress`)。

## 4. 机制 D — 知识库总回路 / LLM-Wiki(宏观)

### D.1 定义
知识库**本身是 loop**:`物料输入 → 编译 → 建双向标签 → 反复链接 → 修订错误认知/改进好认知 → 输出 → 外部咨询实践 → 新输入` ↺。**递归适用**大系统 + 各模块 / 流程管控 / 效率任务分配 / 后期状态。

### D.2 规则(写死·相对 §11.1 的两点增量)
- 本机制 = §11.1 七环(Ingest→Compile→Link→Lint→Distill→Publish→**Feedback**)的**完整闭合**,补两点:
  1. **闭合「外部实践 → 新输入」**:已发布内容(公众号/视频号)+ 客户实战(花厅坊诊断结果)**回流为新 source**,不止于发布(呼应原则 8 公开构建)。
  2. **「双向更新」非只增不改**:新料进来既**回链旧页**,又**按新料修订旧页**(修订错误认知 / 改进好认知)。知识库不是只追加的日志。
- **双向标签**:概念页互链(wikilink);新料进库**必须**回链相关旧页。
- **修订纪律(must)**:发现旧页认知错误 → **不就地悄改**;标「修订」+ 记一行(日期·原认知·新认知·依据)。保真 + 可追溯(呼应 §3 保真六哥声音、§2 原则 6 source 不可变)。
- 相位:编译/蒸馏/认知修订=判断相(Opus);建标签/挂链/更 MOC=机械相(Sonnet)。

### D.3 样板(实证)
**零售老刘 321 篇逐字工程** = 本回路跑通的实证:物料(321篇)→精读卡→回填 B 层方法论→升 active(3 页)→花厅坊「方便食品」实证 → 实证结果回流为新 input。**机制 B/C/D 是同一回路在三个尺度的缩影。**

### D.4 焊接
- = CLAUDE.md §11.1 七环 + KB-BUILD-001(raw 冻结 / wiki 编译 / summary 检索)。本规范只补「外部回流闭合」与「双向修订」两笔。

## 6. CLAUDE.md §7 补丁提案(待六哥 Z)
> 在 §7 操作回路新增一条,作一等公民:

```
- G12_SkillForge(动作孵化治理）— 重复动作进 _动作台账;同类 ≥3 次且过「步骤稳定/频率周期/输入同构」三判据 → 生成规则草稿 + 提示六哥是否孵化 Skill；≥10 次强制提示。不自动建 skill（建=改系统=须签字）。
- Skill 闭环铁律 — 任何 skill 跑完功能闭环须含「复盘D + 迭代E」两相，lessons 回写模板。缺则视为未闭环。
- 模型分层调度 — 脚本 > Sonnet（机械相）> Opus（判断相）；判断相（裁决/签字背书/调改取舍）不得下放。
```

## 7. 反模式(必避)
① 把判断相(裁决/签字)下放给 Sonnet/脚本「省钱」(破护城河);② skill 跑完不复盘就收工(无 D/E 相);③ 动作重复多次却不入台账、不提孵化(治理失效);④ 自动建 skill 不报六哥(越签字门);⑤ 新料只追加不回改旧页(知识库退化成日志);⑥ 发现旧页错认知就地悄改(破可追溯);⑦ 把单次救火误判为「规律动作」去孵化 skill(三判据没过)。
