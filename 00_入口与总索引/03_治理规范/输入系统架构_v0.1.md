# 输入系统架构 v0.2 — LLM Input OS

> **本质正名(v0.2)**:这不是知识管理 / 不是 Obsidian 架构,是 **LLM 输入操作系统(Input OS)** —— **计算成本控制 + 检索边界设计**。
> **一句话目标**:**不是让模型更聪明,是让模型更少读错东西。** 把"理解"前移到编译阶段,让 LLM 只做推理、不做阅读。
> **不是新理论**:落实 `CLAUDE.md` §7 G02「编译层优先铁律」+ Karpathy 三层。**与 [[RetailOS_DataAdapter_v1.0_设计与SaaS评估]] 同构**(一个治数据、一个治内容,同一套降 LLM 依赖纪律)。

## 三层 + 检索隔离(工程形态 · 六哥 2026-06-25 收敛)
| 层 | 名 | 性质 | LLM 可见? |
|----|----|------|:--:|
| **L1** | Raw Layer(Clippings) | 外部冻结:文/视/音/图原件 + source + tags + 1段Why | ❌ `.no-ingest` |
| **L2** | Compile Layer(ingestion) | 去冗去噪标准化 → concept/method/dataset summary(≤150词)+ tags + linkage | 🟡 摘要可见 |
| **L3** | Wiki Layer | MOC/方法论/SOP/决策模型,可组合可推理 | ✅ 唯一默认检索层 |

> **Clippings 正名**:❌ 不是知识库 → ✅ **输入流缓冲区(ingestion buffer)**。
> 🟢 **落点(六哥 2026-06-26 签 · B 案)**:进料缓冲区是**系统级基础设施,独立于所有子库**,落根层 `/Users/davidliu/KnowledgeBase/Clippings/`(非 retail、非主库内)。按"去向"分主题桶,路由表见 `Clippings/_README_进料路由.md`。retail/Clippings 仅保留纯零售料(第 2 批将连零售料一并归并到中枢 `零售/` 桶,实现"子库零 raw")。
> **检索隔离四规则**(L1/T3 强制):`.no-ingest` + `.no-index` + Graph view exclude + MOC 不挂载。**T3 = 存在但不参与计算**(冷存储/黑盒)。

## 结论先行
1. **Clippings 不是知识分类,是 raw 进料层**(与方法论/品类不平级)。
2. **我永远只读"编译层"**(MOC+索引+各页 summary 廉价首遍),不扫原始内容池;raw 授权才碰。
3. **多模态必须进料时编译成文本**(音视频→转录、图→caption),不能查询时现读(又贵又慢)。
4. **Token 成本 ∝ 编译层大小,不是洪流大小** —— 这是省 70-90% token 的唯一来源。
5. **立即整改**:Clippings 加 `.no-ingest`;`性爱生活`(T3)隔离出本库(见 §6)。

## 一、三段式管道（映射三层)
```
渠道(公众号/博客/爬虫/音频/视频/图)
  ↓ 进料【冻结·.no-ingest·A层external_reference·§6】
[raw层]  原件 + 一句话摘要(## Why) + 标签
         音视频 → 转录文本 / 图 → caption(进料时一次性编译)   ← Clippings 就是这层
  ↓ /ingest 编译（消化执行层=台账记录"哪篇→ingest进哪页"）
[wiki层] 概念页/方法论/媒体选题(frontmatter 必带 summary: ≤40字)  ← 我真正检索的层
  ↓ 我的调用【G02 铁律】
[我]    ① 标题/标签/摘要(首遍·近0成本) → ② 命中页正文 → ③ 授权才读 raw
```

## 二、我的调用逻辑（三段式廉价检索）
| 段 | 读什么 | 成本 | 何时升级到下一段 |
|----|--------|------|------------------|
| ① 首遍 | MOC + 索引 + 各页 `summary:` | 近 0 | summary 命中相关 |
| ② 正文 | 命中的概念页/方法论页正文 | 中 | 需要原始证据/原话 |
| ③ raw | 原件(Clippings/xls/媒体) | 高 | **仅六哥显式授权** |

> **铁律**:绝不从 ③ 起步。raw 默认在我上下文之外。

## 三、多模态归一（关键)
| 模态 | 进料时编译成 | 我检索的 | raw 处置 |
|------|------------|---------|---------|
| 文章/网页 | 摘要 + 标签 | 摘要/概念页 | 冻结存档 |
| 音频/播客 | **转录文本** + 摘要 | 文本蒸馏 | 音频冻结 |
| 视频 | **字幕/转录** + 关键帧caption | 文本蒸馏 | 视频冻结 |
| 图片 | **caption/OCR** | caption | 图冻结 |

> **为什么**:让我在查询时转录一个视频 = 又贵又慢又不可复用。**进料时编译一次,查询时读文本千次。**

## 四、Clippings 定位（不是分类·是进料层)
| | Clippings | 方法论/品类(wiki) |
|---|---|---|
| 性质 | 外部进料·原件 | 编译后·大脑本体 |
| 可变性 | 存进即冻结 | 可重编译 |
| 进上下文? | **默认不进** | 是 |
| 怎么成长 | 靠 /ingest 被编译进 wiki | 靠 Clippings 喂养 |

- Clippings 按主题归档**够了,别过度建分类**(目录别工程化,模型读文本不读层级)。
- "## Why"一句话摘要 = 已在做的正确第一步(每件带消化锚点)。
- 台账 = 消化执行层(记录 ingest 状态),非知识分类。

## 五、Token 经济（为什么这套省钱)
- 查询成本 ∝ **编译层 summary 总量**,与 raw 洪流体积**解耦**。
- 1 万篇 clippings:我查询只读其 summary(1万×40字 ≈ 40万字一次扫),**不读正文**。
- 渠道越多、内容越杂 → 这道"编译闸"越值钱。**输入系统的本质 = 把洪流挡在我默认上下文外,只喂蒸馏。**
