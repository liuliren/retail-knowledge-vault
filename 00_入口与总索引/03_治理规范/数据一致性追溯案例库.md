## 案例 #001：花厅坊客流/客单数/客单价口径连环订正（2026-05-09）⭐⭐

### A. 触发场景

5/9 v2.0/v2.1/v2.2 综合诊断分析报告中 4 个数字（滞销 7140 / 库存 3300 / 客流 145,490 / 销售 300 万）/ 用户追溯发现多重错误。

### B. 错误做法（vault v2.2 报告）

```
❌ 错误 1：145,490 当成"客流"（成交客户数）
❌ 错误 2：滞销 7,140 = 文件总行数（实际去 header 后 7,139）
❌ 错误 3：库存挤压 3,300 = 文件总行数（实际 3,293）
❌ 错误 4：用"30 天"代替实际 32 天（4/7-5/8）/ 销售 31 天（4/8-5/8）/ 时段错开 1 天
❌ 错误 5：凭"客流 × 客单 ≠ 销售（差 2 倍）"猜测"双倍统计"作为结论

共同根因：vault 把推断 / 假设 / 估计写成事实 / 缺乏 evidence 追溯
```

### C. 用户订正（5/9 2 次连环订正）

**第 1 次订正**（5/9 下午）：
- 用户提问"7,140 怎么来的？库存 3,300 怎么来的？"
- → vault 重新追溯 → 滞销 7,139 / 库存 3,293 / 时段 32 天 vs 31 天

**第 2 次订正**（5/9 晚）：
- 用户告知 "客流应是 2,300-2,500/天 / 现在 4,547 一定不对 / 双倍统计"
- 然后再订正："145,490 实际是**客单数**（订单数）/ 不是客流"
- "客流是另外一种计算路径 / 保留 / 但先按客单数进行"

### D. 数学验证（订正后完美闭合）

```
✅ 145,490 / 32 天 = 4,547 客单数/天
✅ [推断] 双倍统计 → 真实 ≈ 2,273 客单数/天（≈ 用户 2,300-2,500 范围）
✅ 日均销售 96,723 元 / 日均客单数 2,273 = 42.55 元/单
✅ ≈ 报表客单价 42.14 元（差 0.96%）⭐⭐ 完美对上！

→ 客单价 = 销售 / 客单数（POS 笔均）/ 不是"销售/客流"（那是件单）
```

### E. 教训 + vault 沉淀

| 教训 | vault 沉淀 |
|---|---|
| **客流 ≠ 客单数 ≠ 客单价**（三层不同口径 / 不能混用）| 6 表 v1.0.2 字段加 / [[2026-05-09_花厅坊综合诊断分析报告_v2.2_补充与订正]] §A.0 锁定数据三层口径 v1.0 |
| **数字必带 evidence**（文件名 + sheet + 行号 + 时段 + 字段定义）| frontmatter `data_source` 字段强制（[[文档工程化标准]] v1.3）|
| **时段必须精确**（32 天 vs 30 天差 6.7% / 错开 1 天差 1.3%）| [[反幻觉检查清单]] §M 第 2 条 |
| **行数 vs SKU 数**（去 header / 小计行）| [[反幻觉检查清单]] §M 第 3 条 |
| **不一致时列 ≥ 3 种解释 + 数学验证**| CLAUDE.md §13.20 第 3 条 |
| **必须用户 confirm 才能用作分析输入**| CLAUDE.md §13.20 第 6 条 |
