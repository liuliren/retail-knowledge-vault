# 痛点框架

## 1. 为什么痛点在最上层

方法论不是用来"懂"的,是用来**消除某个痛点**的。每个工具、每条公式,都必须回答:**它治哪个痛?** 治不出痛点的方法论 = 自嗨,不进体系。

痛点框架 = 知识体系的**归因起点**和**调用入口**:遇到一个经营问题 → 先归到痛点 → 痛点指向 L3 哪个武器模块 → 武器算出数 → L4 落地。

---

## 2. 两类痛点

### 2.1 顾客痛点(顾客视角 / 决定要不要来、买不买)

| 痛点 | 表现 | 根因 |
|---|---|---|
| **买不到** | 想买的缺货 / 没有 | 订货失真 / 选品缺位 / 缺货管理弱 |
| **买不对** | 找不到 / 选错 / 不知买啥 | 品类结构乱 / 陈列无逻辑 / 价格带断层 |
| **买贵了** | 同物别处更便宜 | 价格带失控 / 毛利结构不合理 |
| **买不好** | 品质不稳 / 临期 / 体验差 | 品质管理弱 / 周转慢 |
| **买不便** | 动线绕 / 找货累 | 货位规划差 / DisplayMap 缺失 |

### 2.2 经营痛点(老板视角 / 决定赚不赚钱)

| 痛点 | 表现 | 根因 |
|---|---|---|
| **缺货** | 畅销断货 / 商机损失 | 追加式订货 / 安全库存一刀切 |
| **滞销** | 卖不动 / 占资金占排面 | 选品错 / 末位淘汰误判 |
| **毛利低** | 卖得多不赚钱 | 价格带 / 品类角色错配 |
| **周转慢** | 现金压库存 | 库存结构差 / 周转天数 > 账期 |
| **客流降** | 来的人少了 | 需求洞察缺失 / 商品力弱 |

## 4. 用法(客户对话 / 自我诊断)

1. 客户说一个现象("总是缺货""卖不动")→ 归到痛点
2. 痛点 → §3 表查主治模块
3. 模块给武器(公式/判据)→ 算出花厅坊的数
4. 数 → 行动 + 沉淀 M-DEC

> **给老板讲**:不讲"我有什么方法论",讲"你这个痛是什么根因,我用什么把它算清、治掉"。痛点框架让方法论永远站在客户那边。
