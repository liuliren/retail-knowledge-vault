# 门店数据治理频次框架

> **一句话定位**：什么数据、多久看一次、看完做什么决策——门店经营的数据节奏表。

---

## 0. 设计原则

1. **频次匹配决策速度**：能影响当日行动的数据日看；影响本周调整的数据周看；影响结构性决策的数据月/季看
2. **看数据 = 触发动作**：每个频次节点必须对应一个可执行动作，看了不做 = 不用看
3. **从花厅坊现有数据能力出发**：有什么数据看什么，不为了"完整"而造数据负担
4. **PCV 回路嵌入**：每个节点的"发现"都是论断候选，值得沉淀的用 `/promote` 写回KB

---

## 1. 数据治理日历

### 🔴 日频（每日·晨会级）

**看什么**：

| 数据项 | 目的 | 触发动作 |
|---|---|---|
| 昨日销售额 vs 上周同天 | 判断趋势正常/异常 | 偏差>10% → 查原因（天气/活动/缺货） |
| A类缺货清单 | 防止畅销品断货 | 当日立即补货/联系采购 |
| 日客单价 | 监控消费行为变化 | 连续3天下降 → 触发选品/陈列排查 |

**花厅坊现实**：目前POS日报能支撑哪几项？先只看能看的，不造数负担。

---

### 🟡 周频（每周一·回顾上周）

**看什么**：

| 数据项 | 目的 | 触发动作 |
|---|---|---|
| 品类动销率（在销÷总SKU） | 发现滞销结构 | 动销率<60% → 该品类需要选品排查 |
| 周转天数变化 | 库存健康监控 | 周转天数连续增加 → 库存积压预警 |
| 本周新品销售表现 | 新品爬坡判断 | 新品第2周动销<30% → 位置/陈列问题排查 |
| 缺货次数统计 | 补货节奏校准 | 缺货频次高的SKU → 提升订货系数 |
| 畅销TOP10变化 | 动态维护畅销目录 | 新晋畅销 → 扩排面；跌出 → 缩排面 |

**决策级别**：执行层调整（补货系数、排面、陈列位），不需要六哥每次审批。

---

### 🟠 半月频（每月1日和16日）

**看什么**：

| 数据项 | 目的 | 触发动作 |
|---|---|---|
| ABCZ重新分类 | 防止分类漂移 | 有SKU从A降到C → 触发汰换流程 |
| 价格带分布 | 结构健康检查 | 高价带占比下降 → 品类结构问题 |
| 供应商到货准确率 | 供应链评估 | 准确率<90% → 双轨制优胜劣汰启动 |
| 临期库存清单 | 损耗控制 | 距保质期<30天 → 折扣促销启动 |

**决策级别**：中层调整，六哥过目，执行层执行。

---

### 🔵 月频（每月底）

**看什么**：

| 数据项 | 目的 | 触发动作 |
|---|---|---|
| 品类毛利结构 | 毛利贡献诊断 | 毛利贡献低于占位 → 品类重新定位 |
| 客单价×品单数趋势 | 客群行为变化 | 品单数下降 = 连带率弱 → 关联陈列排查 |
| 月度ABCZ全量更新 | 选品汰换依据 | D类清单确认 → 汰换提案给六哥 |
| 坪效/货架效率 | 空间资源分配 | 坪效低于均值30% → 品类/陈列调整 |
| 畅销品目录月度调整（10-20%） | 动态维护 | 见零售老刘标准：月调幅10-20% |

**决策级别**：结构性调整，六哥签字级。对应 `/review month`。

---

### 🟢 季度频（每季末）

**看什么**：

| 数据项 | 目的 | 触发动作 |
|---|---|---|
| 畅销品目录大调整（50-60%） | 季节性结构重置 | 见零售老刘标准：季调幅50-60% |
| 品类角色重新评估 | 战略定位校准 | 引流品/利润品/形象品重新分配 |
| 供应商全量评分 | 优胜劣汰执行 | 低分供应商减量或淘汰 |
| 年度目标对照 | 2026三目标进度 | 偏差 → 客户优先级调整 |

**决策级别**：战略级，六哥主导。对应 `/review quarter`（待建）。

## 2. 数据输入节奏（AI协作层）

> 六哥输入数据 → AI 分析诊断 → 触发 PCV 回路 → 命题验证/回流

| 节奏 | 六哥的动作 | AI的动作 |
|---|---|---|
| 日 | 日报截图/数字口述 | 标记异常·生成次日关注清单 |
| 周 | 周销售表格 | 动销率/周转分析·生成本周选品建议 |
| 半月 | ABCZ更新表 | 汰换候选清单·供应商评分更新 |
| 月 | 月度完整数据包 | 全量诊断报告·KB页命题验证回流 |
| 季 | 季报+现场照片 | 品类重新定位建议·下季畅销预测 |

## 3. 花厅坊当前优先建立的数据能力

**第一阶段（现在-8/31）**：只建必须的

| 优先级 | 数据能力 | 原因 |
|---|---|---|
| P0 | 日销售额+A类缺货 | 最基础的日频指标，不建就无法管店 |
| P0 | 周动销率+周转天数 | 汰换/补货的判据，不建选品决策无依据 |
| P1 | ABCZ月度更新 | 结构性决策依据 |
| P2 | 客单价×品单数 | 客群健康监控，有POS就能算 |
| 暂缓 | 转化率（卖场三指标） | 需要客流摄像头，花厅坊暂无条件 |

**第二阶段（8/31后）**：引入更多诊断维度

---

## 4. 张力：这套框架在什么条件下会失效

1. **数据采集成本过高**：如果日频数据需要人工录入，执行层负担过重，会被抛弃
2. **花厅坊POS数据不准**：进价数据不准影响毛利分析，月频和季频的结论可靠性下降
3. **频次过密导致疲劳**：每天都要看数据会让店长厌倦，要区分"必看"和"选看"
4. **数据有了但没有触发动作**：看数据不做决策等于浪费，每个频次必须绑定最少一个具体动作
