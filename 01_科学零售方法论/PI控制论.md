# PI 控制论 v0.2(六哥 2026-07-07 确认定位)

## 结论先行

**PI(Proportional-Integral，比例-积分)反馈控制**是控制论/自动化的经典方法：**用「当前误差」(比例项 P)+「误差累积历史」(积分项 I)两部分驱动调节量**，使系统输出逼近目标值且消除稳态偏差。

六哥 2026-07-07 明确：**这不是零售经营数学工具家族(GMROI/TF函数一类)的用法，而是要作为咱们后期系统工程化的重要设计指标与思想**——即知识库/协作系统/工作流本身的迭代改进，应该借鉴 PI 控制的思路来设计，而不只是零售业务分析工具。

## 核心思想

| 控制论概念 | 含义 | 对我们系统工程化的映射(инferred，待具体落地) |
|---|---|---|
| **误差(Error)** | 当前状态与目标状态的差距 | 如：四维评分现状(逻辑74/稳固84/实用73)与目标(90+)的差距 |
| **比例项(P)** | 根据*当前*误差大小即时调节 | 发现问题立刻修(如本轮lint消债、审计发现即改) |
| **积分项(I)** | 根据*历史*误差累积调节基准线 | 若同类问题反复出现(如多次"先改后审"翻车)，说明比例调节不够，需要**调整系统基准/加护栏**(如批量修复前必查签字清单的新规则)，而不是每次都临时补丁 |
| **消除稳态偏差** | 单纯比例控制会有"永远差一点"的残留误差；积分项才能真正归零 | 只做"发现问题就修"(纯P)会让同类问题反复冒头；只有把"这是第N次犯这类错"沉淀为规则变更(I)，才能让系统性偏差真正收敛 |

## 与本知识库已有实践的对应关系(不是新发明，是给已有做法起名字)

回看本轮系统迭代的几个真实案例，其实已经在无意识地践行 PI 控制的思路：
- **纯比例式修复**：lint 消债批1/批2逐条改断链——这是"当下误差"的即时修正。
- **积分式系统调整**（真正的 PI 闭环）：
  - lint 批2 撞见"批量修复覆盖了六哥07-03已签字的处置决定"→不是只改回来了事，而是新增了一条**永久规则**(批量修复前必查既有签字清单)，防止同类偏差再次累积。
  - P2波2 发现"v2.0→v3.0批量替换造成自指坏链"→不只是修一处，而是提炼成**版本号替换铁律**记入 P2 计划文档，供未来所有类似操作参照。
  - 这正是 PI 控制的精髓：**光修当下的错(P)不够，得看错误有没有反复出现的模式(I)，模式出现就得改系统本身**，否则误差永远消除不了。

## 待落地(下一步，六哥若有具体工程化要求可继续细化)

- 是否要把"积分项检测"正式做成一个机制？例如：同类问题出现≥N次自动触发"生成/修订规则草案"提示(这其实正是 G12_SkillForge 动作孵化治理的判据逻辑，可以视为 PI 控制思想在本库的既有实现之一)
- 是否需要给"四维评分"设计显式的 P/I 双轨反馈：P=单次审计发现即改，I=累计同类审计发现≥N次触发系统性规则调整(而非逐次打补丁)
- 具体应用到"零售运营闭环"(库存/补货)是否也是六哥想要的方向，或本次仅指知识库系统自身工程化——待六哥后续明确
