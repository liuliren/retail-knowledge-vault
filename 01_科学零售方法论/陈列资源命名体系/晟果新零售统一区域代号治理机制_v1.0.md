# 晟果新零售统一区域代号治理机制 v1.0

## 1. 文件定位

本文件是晟果新零售内部用于**所有门店、所有客户、所有项目**的陈列区域代号治理标准。

**适用对象**：
- 乐易购系列门店（花厅坊店 / 沙埔大道店 / 后续新店）
- 沙埔大道店
- 市场店
- Shop 店
- 折扣店
- 标超
- 后续所有客户样板门店

**花厅坊**仅是首个**样板案例**（参 §11）/ 不代表公司通用结论 / 后续可复制到任何新门店。

**本文件解决**：
1. 区域代号如何固化为公司级标准
2. 代号变更如何走治理流程
3. alias 机制如何处理历史代号
4. 跨门店如何复用代号
5. 项目级如何适配
6. 照片入库如何规范
7. 花厅坊案例如何沉淀

**本文件不解决**：
- 具体门店调改计划
- 具体 SKU 决策
- 具体 DisplayMap
- 客户交付方案

⚠️ **与 [[2026-05-19_晟果新零售陈列资源命名治理规范_v1.0]] v1.1 的关系**：
- 命名规范 v1.1 = **细节规则**（22 段 / 字段 / 公式 / 通用对象）
- 本治理机制 v1.0 = **顶层治理**（11 项区域代号 lock / alias / 变更流程 / 跨门店复用）
- 两者协同 / 不替代

## 2. 核心原则（8 条）

| # | 原则 | 说明 |
|:---:|---|---|
| 1 | **公司级代号优先于项目级临时代号** | 任何项目不得发明与公司级冲突的新代号 / 必须先查公司级表 |
| 2 | **区域代号必须短、稳定、可复用** | 2 字符为主（XS / LY / JS 等）/ 一旦 lock 不轻动 |
| 3 | **区域代号不得依赖单店临时叫法** | 单店"洗护区""修饰区""引流区"等口语 → alias / 不进公司标准 |
| 4 | **项目特例必须通过适配表管理** | 单店特例 → 适配表 §特例登记 / 不污染公司级 / 跨 3 家以上才入候选 |
| 5 | **历史旧代号不得强行洗白** | 旧代号通过 alias 表理解 / 历史文件保留 audit trail / 不批量回改 |
| 6 | **新旧代号必须通过 alias 表解释** | deprecated_alias 标识 / 历史可读 / 新文件禁用 |
| 7 | **照片原始素材不得直接覆盖改名** | 永久保留原始素材 / rename map 先行 / 复制升级 / 不覆盖 |
| 8 | **新文件从最新公司级标准生效日起执行** | 本机制 v1.0 生效日 = 2026-05-19 / 含本日起所有新文件用 XS / LY |

## 3. 公司级区域代号表 v1.0（lock）

⭐ **强制锁定**：以下 11 项为公司级标准 / **所有门店全部启用或部分启用** / 不得自创冲突代号。

| 代号 | 区域名称 | 说明 | 是否公司级锁定 | 适用范围 |
|:---:|---|---|:---:|---|
| **XS** | 休食区 / 休闲食品区 | 休闲食品、糖巧、坚果、方便速食、冲调等项目内可细分 | ✅ 是 | 全部门店 |
| **JS** | 酒水区 | 酒、饮料、水饮等 | ✅ 是 | 全部门店 |
| **TW** | 调味区 | 调味品、酱料、复合调味等 | ✅ 是 | 全部门店 |
| **LY** | 粮油区 / 米面粮油 | 米、面、油、杂粮等 | ✅ 是 | 全部门店 |
| **RH** | 日化家清区 | 日化、洗护、家清、纸品等 | ✅ 是 | 全部门店 |
| **SX** | 生鲜区 | 蔬果、肉禽、水产、冷藏生鲜等 | ✅ 是 | 全部门店 |
| **LD** | 冷冻冷藏区 | 冷柜、冻品、低温食品等 | 🟡 待确认 | 视门店 |
| **SY** | 收银区 | 收银台、收银端架、收银小货架 | ✅ 是 | 全部门店 |
| **RK** | 入口区 | 门口、入口堆头、入口引流位 | ✅ 是 | 全部门店 |
| **TD** | 主通道 | 主动线、横向 / 纵向通道 | ✅ 是 | 全部门店 |
| **TC** | 特陈区 | 堆头、主堆、地堆、促销位 | ✅ 是 | 全部门店 |

### 3.1 代号冲突防护

⚠️ **不得混用 / 不得二义**：
- 如历史规范中**已有 SX 表示生鲜区** → 不得再让 SX 表示休食区
- **XS 固定表示休食区** / 避免 HS / SX / XS 混用
- **LY 固定表示粮油** / 避免 LY / MLY / MMLY 混用
- **JS 固定表示酒水** / 不得用 JS 表示家私 / 家私见 RH 子类

### 3.2 公司级标准变更门槛

公司级 11 项代号变更属于 [[CLAUDE.md]] §19 四象限 **I 内核稳固**层 / 门槛**最高**：
- 必须经核心团队议事
- 必须有跨 3 家以上门店实践支撑
- 必须走 §9 变更治理 7 步流程
- 单店特例不能晋升公司标准

## 4. 项目级适配规则

### 4.1 命名

每个门店必须建立项目适配表：

```
{门店简称}_陈列资源命名适配表_v1.0.md
```

存放位置：`09_门店案例与项目复盘/{门店简称}/05_陈列整改/<日期>_<门店简称>_陈列资源命名适配表_v1.0.md`

### 4.2 适配表必含 9 项

| # | 内容 | 说明 |
|:---:|---|---|
| 1 | 公司级代号 | 从 §3 表内引用 / 不得自创 |
| 2 | 门店实际叫法 | 如花厅坊"休食区" / 沙埔"零食区" |
| 3 | 现场口语叫法 | 如"水果通道""糖玩架" alias |
| 4 | 对应货架编号 | G1-A / G2-B / G3-A 等 |
| 5 | 对应通道编号 | A01 / A02 / A03 / A04 等 |
| 6 | 对应端架编号 | T01-T08 等 |
| 7 | 特例说明 | 单店例外 / 不进公司标准 |
| 8 | 是否启用 | ✅ 已启用 / 🟡 待启用 / ⏸ 待确认 |
| 9 | 生效日期 | YYYY-MM-DD |

### 4.3 适配表与花厅坊样板

花厅坊 [[2026-05-19_花厅坊陈列资源命名适配表_v1.0]] v1.1 = 首家样板 / 后续新门店参照其字段结构 / **不直接 copy 取值**（如不假设新门店也是 G1-G4 / 8 端架 / 4 通道）。

## 5. alias 机制

### 5.1 alias 映射表（公司级 lock）

| 旧代号 | 新代号 | 含义 | 状态 | 规则 |
|:---:|:---:|---|:---:|---|
| **HS** | **XS** | 休食区 | deprecated_alias | 历史可读 / **新文件禁用** |
| **MLY** | **LY** | 米面粮油 | deprecated_alias | 历史可读 / **新文件禁用** |

### 5.2 alias 规则（5 条）

| # | 规则 |
|:---:|---|
| 1 | alias **只用于理解历史文件** / 不用于新文件命名 |
| 2 | alias **不得用于新文件命名** / 5/19 起新文件强制用新代号 |
| 3 | 历史文件**不批量回改** / 保留作 audit trail |
| 4 | 新项目**不得沿用 deprecated_alias** / 沙埔大道店等新店不得用 HS / MLY |
| 5 | 如必须引用旧文件 / 必须**注明"历史旧代号"** + 关联 [[2026-05-19_区域代号HS到XS_MLY到LY变更审计_v0.1]] |

### 5.3 引用历史文件标准格式

```
本文件引用 [[2026-05-23_5月19日86张照片入库audit_v0.1]]（原文使用历史旧代号 HS- / 现已订正为 XS- / 参 [[2026-05-19_区域代号HS到XS_MLY到LY变更审计_v0.1]] + [[2026-05-19_晟果新零售统一区域代号治理机制_v1.0]] §5）
```

## 6. 陈列资源完整编码公式

### 6.1 统一公式

```
{客户/门店}-{区域代号}-{资源类型}{编号}-{面位}-{子组}-{层位}-{排面位}-L{品类}
```

### 6.2 字段说明

| 字段 | 取值 | 必填 | 说明 |
|---|---|:---:|---|
| 客户/门店 | LYG-HTF / LYG-SPDD / ... | ✅ | 客户简称-门店简称 |
| 区域代号 | XS / JS / TW / LY / RH / SX / LD / SY / RK / TD / TC | ✅ | 公司级 §3 |
| 资源类型 | G 货架 / T 端架 / A 通道 / D 主堆 / SD 侧堆 / PD 柱旁堆 / M 机位 | ✅ | 命名规范 §4 通用对象 |
| 编号 | NN（2 位）| ✅ | 排号 / 序号 |
| 面位 | A / B | 货架必填 | A 正 / B 反 |
| 子组 | NN（2 位）| 货架按需 | 01-06 等 |
| 层位 | L01-LNN | 精分析必填 | L01 顶层 |
| 排面位 | P01-PNN | 重点 SKU 必填 | 左到右 |
| 品类 | -L{品类名} | 精分析建议 | 对接大湾区品类表 V4.0 |

### 6.3 示例

```
LYG-HTF-XS-G3-B-04-L03-P02-L方便速食
LYG-HTF-XS-T06-G4首-L方便速食
LYG-HTF-XS-A04-M2
LYG-HTF-LY-G1-A-01-L02-P01-L食用油
```

### 6.4 公式解释

- LYG = 客户 / 门店集团代号（乐易购）
- HTF = 花厅坊门店代号
- XS = 公司级区域代号（休食）
- G = 货架
- T = 端架
- A = 通道
- M = 机位
- L = 逻辑品类
- P = 排面位

## 7. 货架 / 通道 / 端架 / 堆头命名规则

```

```


```
{门店}-{区域}-A{通道编号}-M{机位}
```

| 柱旁堆 | `{门店}-{区域}-PD{编号}-L{品类}` |

## 8. 照片命名规则

### 8.1 现场快采命名（速度优先）

```
{门店}_{区域}_{资源位}_{视角}_{序号}.jpg
```

示例：
```
花厅坊_休食区_G4-A_正面全景_01.jpg
花厅坊_休食区_XS-T06_端架全景_01.jpg
```

特点：
- 速度优先 / 现场佳妮 / 员工 / 顾问均可
- 不要求填层位 / L 品类 / 时段
- ❌ 严禁错误词（如休食区误写"洗护"）

### 8.2 入库标准命名（顾问升级）

```
{标准资源位}_{内容}_{YYYYMMDD}_{时段}.jpg
```

示例：
```
LYG-HTF-XS-G4-A-03-L02-P01-L冲调_商品价签_20260519_常规.jpg
LYG-HTF-XS-T06-G4首-L方便速食_全景_20260519_夜灯后.jpg
LYG-HTF-LY-A01-M2_通道全景_20260615_常规.jpg
```

### 8.3 5 禁止事项

| # | 禁止 |
|:---:|---|
| 1 | ❌ 不直接覆盖原始照片 |
| 2 | ❌ 不未经审核批量 rename |
| 3 | ❌ 不把现场快采名当最终入库名 |
| 4 | ❌ 不混用 HS / XS |
| 5 | ❌ 不混用 MLY / LY |

### 8.4 时段字段（沿用命名规范 v1.1 §14.4）

常规 / 夜灯前 / 夜灯后 / 次晨白光 / 24h复拍 / 7d复拍 / 15d复拍 / 30d复拍

### 8.5 双层机制

| 阶段 | 命名 | 路径 |
|:---:|---|---|
| 现场快采 | §8.1 快采 | `99_原始素材/04_现场照片/<日期>/` |
| 入库升级 | §8.2 入库 | `09_/<门店>/05_陈列整改/01_现场图片说明/<日期>_<区域>/` |
| rename map | audit trail | `09_/<门店>/05_陈列整改/<日期>_<...>_rename_map_v0.1.md` |

## 9. 变更治理流程（7 步）

任何区域代号变更**必须**走以下 7 步：

| 步 | 动作 | 输出 |
|:---:|---|---|
| 1 | **提出变更原因** | 文字说明 / 用户 confront 或团队议事记录 |
| 2 | **扫描影响范围** | grep -rl 全 vault / 统计文档数 + 照片数 + 历史引用 |
| 3 | **判断公司级还是项目级** | 公司级 → 走 §3 表更新 / 项目级 → 走 §4 适配表登记 |
| 4 | **建 alias 映射** | 旧代号 → 新代号 / deprecated_alias 状态 / 写入 §5.1 |
| 5 | **只改主文件与活跃文件** | 4 主 + N 活跃 / 不批量改历史 |
| 6 | **历史文件保留** | 通过 alias 理解 / 不重写 commit history |
| 7 | **建审计文件并 commit** | `<日期>_区域代号<旧>到<新>变更审计_v0.1.md` / commit message: `fix(命名治理): ...` |

### 9.1 反例（不做）

- ❌ 全 vault `sed -i 's/HS-/XS-/g'`
- ❌ 重写 git history
- ❌ 删除历史文件
- ❌ 在历史文件内回改字符串

### 9.2 案例

- 5/19 HS→XS / MLY→LY → 已走完 7 步 / 审计文件 = [[2026-05-19_区域代号HS到XS_MLY到LY变更审计_v0.1]] / commit `3d7a357`

## 10. 跨门店复制规则（10 步）

新门店启动时**必须**先做：

| 步 | 动作 | 输出 |
|:---:|---|---|
| 1 | **建门店简称** | 客户简称-门店简称（如 LYG-SPDD）|
| 2 | **复制公司级区域代号表** | 引用 §3 / 不修改 / 仅启用部分 |
| 3 | **建门店适配表** | `<日期>_<门店简称>_陈列资源命名适配表_v1.0.md` |
| 4 | **建货架编号** | G1-GNN / A 面 / B 面 / 子货架 01-NN（按现场实测）|
| 5 | **建通道编号** | A01-ANN（从入口起编号 / 配合现场布局图）|
| 6 | **建端架编号** | T01-TNN（关联货架首 / 次 / 异型）|
| 7 | **建特陈位编号** | D01-DNN / SD01-SDNN / PD01-PDNN |
| 8 | **建照片入库 SOP** | 双层机制（快采 / 入库）+ rename map 模板 |
| 9 | **建 alias 表** | 单店历史叫法 / 口语 alias / 不污染公司标准 |
| 10 | **建审计记录** | 启动日 + 适配范围 + 命名 lock 内容 |

### 10.1 适用门店类型

| 业态 | 适用度 |
|---|:---:|
| 社区超市 | ✅ |
| Shop 店 | ✅ |
| 市场店 | ✅ |
| 折扣店 | ✅ |
| 标超 | ✅ |
| 客户样板店 | ✅ |
| 便利店 | ⏸ 待验证 |
| 生鲜店 | ⏸ 待验证 |

### 10.2 跨门店复用约束

- ✅ 公司级 §3 代号表 + §6 编码公式 + §9 变更流程 + 本 §10 复制规则 → 跨门店完全复用
- ❌ 花厅坊 G1-G4 / T01-T08 / XS-A01-A04 取值 → 不跨门店复用
- ⚠️ 跨 3 家以上门店反复出现的特例 → §3.2 公司级标准变更候选

## 11. 花厅坊案例沉淀

### 11.1 案例摘要

| 项 | 内容 |
|---|---|
| 案例日期 | 2026-05-19（W21）|
| 变更 | 原代号 HS → 新代号 XS / 原代号 MLY → 新代号 LY |
| 触发 | 用户决策（HS 与 House / 货架 混淆 / MLY 过长简化）|
| 执行方式 | **方案 B**（4 主文件 + 4 当前活跃文件 + 1 新建审计 = 9 文件改）|
