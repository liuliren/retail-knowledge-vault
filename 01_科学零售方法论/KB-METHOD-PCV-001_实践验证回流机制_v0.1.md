# 实践-验证-回流机制（PCV Loop）

## 0. 核心命题

> **方法论不是写出来的，是从实践中跑出来的。** 每一次诊断、每一个客户反馈、每一条失败的判断，都是对方法论命题的活体验证。验证结果必须回流——不回流的知识是漏电的知识。

---

## 1. 机制定义

**PCV Loop**（Practice → Claim → Validate → 回流）是本知识库方法论自我生长的核心回路，由四个阶段构成：

```
实践（花厅坊/客户）
    ↓ 产生
论断（candidate命题·尚未验证）
    ↓ 经实践检验
验证（成立/部分成立/不成立）
    ↓ 结果
回流（更新KB页状态 candidate→active→stable·或标注张力·或降级删除）
    ↓ 重新指导
实践（下一轮·带着更精准的方法论）
```

---

## 2. 四个阶段

### 2A 实践（Practice）
- **主战场**：花厅坊、客户诊断、每次品类调整
- **产出**：原始数据、现象观察、问题判断
- **关键**：实践是最高级别的输入，高于理论推导。凡与实践矛盾的命题，命题先降级，而不是解释实践
- **前置条件**：数据质量必须达到基准才能作为验证依据。花厅坊 POS 进价数据不准是已知问题——**数据失真的实践不能用来验证命题，只能用来发现问题**。数据质量未达标时，实践结果标注"数据待核实·暂不用于验证"

### 2B 论断（Claim）
- **定义**：从实践现象归纳出的可检验命题，标注为 `candidate`
- **格式要求**：必须是可证伪的——能说出"在什么条件下这条命题不成立"
- **工具**：`/ingest` 把论断写入对应KB页，`/deep-read` 从外部知识提炼候选论断
- **禁止**：模糊的感想不是论断。"顾客需要好商品"不是论断，"花厅坊饼干T03品类中，A类前3SKU动销率低于60%意味着陈列位分配过散而非商品力弱"才是

### 2C 验证（Validate）
- **方式**：
  1. **同一场景复测**：下次做同类诊断时，主动用该命题预测结果，看预测是否成立
  2. **跨客户迁移**：同一命题在第2家客户是否仍然成立（增加泛化置信度）
  3. **反例搜索**：主动寻找让命题失效的边界条件
- **验证达标门槛**（升 active 的最低标准）：
  - 同一场景下，命题预测结果与实际结果一致 ≥2次，且未发现反例
  - 数据质量已达基准（见 §2A 前置条件）
- **结果处理**：
  - 完全成立且达门槛 → 升 `active`，补充场景范围
  - 部分成立 → 标注限制条件，维持 `candidate`
  - 不成立 → 标注"已证伪·保留反例"，不删除（反例本身是知识）

### 2D 回流（Backflow）
- **动作**：用 `/promote` 把验证结果写回KB页
- **状态机**：`candidate` → `active`（有实践验证）→ `stable`（六哥签字·跨场景验证）
- **回流粒度**：不是每次现场即时判断都要回流。触发回流的条件：①命题预测被实践结果明确验证或证伪；②该结论对下次同类诊断有参考价值。即时决策（现场拍板、临时调整）不需要回流，有沉淀价值的验证结论才回流
- **铁律**：验证结果不回流 = 知识漏电。每次诊断结束，用一句话说出"哪条命题被验证了"，再决定是否值得写入KB

## 3. 与操作四档的关系

PCV Loop 是内容层的迭代机制，不受时间硬锁约束——内容迭代随时可做。

唯一需要慎重的是**结构层操作**（目录大改·批量路径变更），按C/D档执行：

| 操作 | 所属层 | 处理方式 |
|---|---|---|
| KB页新增命题、升级状态 | 内容层 | A/B档·随时做 |
| 新建KB页、新增wikilink | 内容层 | A/B档·随时做 |
| 目录重组、文件批量移动 | 结构层 | C档·列清单确认 |
| 修改CLAUDE.md原则 | 宪法层 | D档·签字 |

## 4. 张力：机制失效的条件

以下条件下，PCV Loop 会失效或产生错误结果：

1. **实践数据质量差**：花厅坊 POS 数据不准（已知问题）→ 产生的论断基于噪声，验证结论不可信。对策：数据到位前，论断标注"数据依赖·候选"
2. **样本量不足**：只有花厅坊一个客户时，"验证通过"不等于"普遍成立"。对策：跨客户迁移前，命题最高升 `active`，不得升 `stable`
3. **验证不回流**：做了诊断但没有把结果更新到KB页。对策：每次诊断结束后，`/promote` 是必须步骤，不是可选项
4. **论断粒度太粗**：命题太泛则无法被实践证伪（如"要做好选品"）。对策：KB页质量铁律三项检验·每条命题必须有具体场景和失效条件

## 5. 花厅坊落地：下一个具体动作

**当前机制运转状态**（截至 2026-07-01）：

- ✅ 输入端：零售老刘321篇精读已编译入方法论（充足弹药）
- ✅ 命题状态：数量管理/单品管理/数据底座均已升 `active`
- ⚠️ 验证端薄弱：大量 `active` 命题尚未经花厅坊实践验证，仍停留在"理论成立"阶段
- ⚠️ 回流动作稀少：诊断结果（如T03饼干）没有系统性地更新命题状态

**最高优先级行动**：
1. 每次花厅坊诊断结束后，明确说出"哪条命题被验证了/被证伪了"
2. 用 `/promote` 把结果写回对应KB页
3. 凡经花厅坊验证且成立的命题 → 从 `active` 升 `stable`（六哥签字）
