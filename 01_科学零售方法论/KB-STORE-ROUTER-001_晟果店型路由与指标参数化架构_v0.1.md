## 0. 结论先行

1. **"分几类店型"是错问题——它正是杀死 STORE-001 的坑。** 正确对象不是一张扁平的店型清单，而是一个 **4 轴描述符坐标空间**：`region_tier(消费力带) × strategy_type(战略店型 S1–S4) × chain_stage(连锁阶段) × size_band(面积带)`。一家店 = 空间里一个坐标点；"店型"= 该点上被实例化的一份 **profile**。类别数不预先枚举(枚举=盖大教堂=老刘 1800 笛卡尔积过度工程化)，**只在真实有店的坐标上惰性实例化 profile**。

2. **系统保持一套，变的只是"叶子值"。** 把体系拆成三层：**架构不变量**(双投影 C 码锚点、RM-01 主方程、7D/COMP-7 维度定义、标准化工序、隐私护栏)——永远单套、跨店型共享；**参数化基准**(客单/毛利/生鲜占比/权重/价格带配比等阈值与目标值)——随坐标变；**适用性开关**(某指标/方法在某型开或关，如"品类完备性"只对 S2 成立)。**参数化 vs 分叉的终裁 = 参数化(见 §4)**；分叉只在"跨业态"(前置仓/大卖场/社区团购)才允许，因为那时变的是结构不是值。

3. **路由 = 描述符判别 → profile 级联覆盖 → 实例化标准 → 数据可行性门 → 校准回路。** 判别只对 strategy_type 一个轴做(用 3 条 POS 判据)，其余三轴直接测量/查表；profile 用**级联覆盖**(CSS cascade 式)合成，冲突按覆盖优先级裁决；实例化后过 **Pre-flight 数据可行性门**(复用 MULTISTORE §0.5，判 ACTIVE/DEGRADED/LOCKED)；诊断后 observed 值回填 profile 抬升 fact_layer。

4. **我实质推翻/修正了主线程 3 条假设中的 2 条**(诚实标注，详见 §7)：① 描述符里的"生鲜占比"被逐出——它是被基准的**输出/结果变量**，拿它当路由输入是循环论证；② "客群×商圈"不作独立轴——社区粒度不可独测(STORE-001 死因)，已并入 strategy_type；③ 新增 `region_tier(消费力/城市线级)` 轴——现系统全部按大湾区校准，好家源(三线)一进来就暴露此轴缺失；④ 数据降级**不挂在 profile 上**、而是独立的、按"本店实际数据清单"触发的可行性门——降级是 per-store 运行时事实，不是 per-type 属性。

## 1. 店型 Taxonomy：4 轴描述符坐标空间

### 1.1 为什么是坐标空间而非扁平清单

STORE-001 与 STORE-TYPE-SG-001 的审议卡头号反例是同一个：**只要把"面积段"当第一判别维，混合商圈就把商圈性质不同的店归成同型**。根因不是"选错了第一维"，而是**"店型"被当成一维标量**。真实差异来自多个正交因子的组合，任何单维硬分类都会在其它维度上失真。解法是把"店型"升维为多轴坐标，并明确**哪一轴是判别主维、哪些轴只是描述字段**。

MULTISTORE §3.4 的 R5 三治理字段(`store_size_band / store_type_strategy / store_stage_band`)已经无意中发现了"三轴正交、禁止用单一 store_type 字段混装三义"。本页做两件事：**(a) 把这三轴正式确立为描述符空间的三轴；(b) 补上被它漏掉的第四轴 `region_tier`**——因为三治理字段全部在大湾区单一消费力带下定义，跨线级时会系统性污染基准(好家源三线客单混入大湾区客单库 = 基准中毒)。

### 1.2 四轴定义与判别方式

| 轴 | 值域(inferred) | 角色 | 取值方式 | 主要影响的指标 |
|---|---|---|---|---|
| **A1 · region_tier(消费力带)** | 一线/新一线(GBA核心) ／ 二线 ／ 三线及以下县域 | **绝对值标尺轴**——先定价格与客单的量纲 | 查表(城市线级 + 商圈房价/人均可支配收入代理) | 客单价基准、价格带绝对区间、损耗承受度、SKU 量级上限 |
| **A2 · strategy_type(战略店型 S1–S4)** | S1刚需菜场平替 ／ S2家庭厨房主力 ／ S3便利鲜食复合 ／ S4品质精选 | **第一判别主维**(客群×商圈的可测代理) | **判别**：3 条判据(见 §1.3)= **1 条 POS(时段结构) + 1 条促销日历(促销日客流比) + 1 条现场调研(500m竞对锚)** → 主型+副型。〔归型演练破绽②订正：非纯 POS，需促销日历+现场表方可闭合〕 | 生鲜占比目标、价格带配比(P/M域)、CHI权重、五角色SKU比例、完备性开关 |
| **A3 · chain_stage(连锁阶段)** | 单店 ／ 2–4店 ／ 5–10店 ／ 10–30店 ／ 30+店 | **结构成本轴**——连锁共享供应链→毛利/周转系统性异于单店 | 查(客户门店数) | 毛利率基准、周转天数、可否跑场景A内部横比、数据可得性 |
| **A4 · size_band(面积带)** | 500–800 ／ 800–1200 ／ 1200–2000㎡ | **描述/次级调制字段**(非判别主维) | 测量(卖场净陈列㎡) | SKU 量级带、完备性上限、坪效基准**值**(分母口径本身=L-INV 常量·见 §2.1) |

> **判别主维的降级铁律(修正 STORE-TYPE-SG-001 的隐患)**：`size_band` 从"第一判别维"降为"描述+调制字段"。归型只由 A2 的 3 条 POS 判据驱动；A1/A3/A4 是**直接取值**(查表/测量)，不参与"判别"、只参与"实例化"。这一步直接采纳店型审议卡反例①与专业对标①(业界通行做法=商圈客群第一维、面积次维)。

### 1.3 strategy_type 的判别(沿用并锁定 STORE-TYPE-SG-001 §2–§3)

3 条判据不重造，直接引用 STORE-TYPE-SG-001：① 促销日客流比(≥3倍→S1 / <1.5倍→S4)；② 时段客流结构；③ 500m 竞对锚。归型规则：**≥2 判据同型→主型；余判据他型→副型(以时段/品类模块表达)；三判据三分裂→判"混合商圈"，强制主型+副型双标签，禁止取中间值**。

> **边界店/混合商圈的鲁棒解**：不在 A2 上取加权平均(两头挨打)，而是**输出双坐标 + 副型以模块表达**。即路由输出允许一店挂"主 profile + 副 profile 模块补丁"，而非一个折中 profile。这是比"面积单维"鲁棒的根本原因——描述符是坐标不是标量，混合态可用"主+副"表达而无须坍缩。

### 1.4 惰性实例化(反过度工程化)

4 轴笛卡尔积理论上有数十个坐标；**禁止预先为全部坐标写 profile**(违反宪法"少而深/别盖大教堂"，重蹈老刘 M3-T5 的 1800 笛卡尔积)。规则：**只为"当前真实有店"的坐标实例化 profile**；未populated 坐标在被访问时走 §3.4 fallback 级联到最近已定义祖先。花厅坊 populate 一个点，好家源 populate 另一批点，profile 库随客户到来生长。

## 2. 指标参数化 Schema

### 2.1 三层切分：不变量 / 参数 / 适用性开关(这是全部设计的核心决策)

| 层 | 判据(什么进这层) | 例 | 是否随坐标变 |
|---|---|---|---|
| **L-INV 架构不变量** | 定义"怎么想/怎么算"的结构，与店型无关 | 双投影 C 码锚点、per-SKU 角色属性 schema(`is_traffic/is_margin/is_image`)、RM-01 主方程、7D(D1–D7)与 COMP-7(C1–C7)维度定义、标准化工序(÷面积/客流/时段)、**坪效分母口径=实用销售区㎡(排除仓储/后场·所有 profile 坪效字段须附面积口径标注)**、场景A/B/C 隐私护栏、ID/Johnny.Decimal 约定 | **否**——永远单套 |
| **L-PAR 参数化基准** | 插进不变结构的**阈值/目标值/权重**，有证据随轴变且变化会改变决策 | 客单价基准、毛利率目标、生鲜销售占比基准、生鲜区面积占比目标、SKU活跃率/空销率基准、周转天数基准、价格带配比(P/M域)、CHI 七维权重、五角色 SKU 比例、损耗率基准 | **是**——按 §2.3 profile 覆盖 |
| **L-APP 适用性开关** | 某指标/方法在某坐标"开/关/降级"的布尔位 | 品类完备性指标=S2 ON、S3/S4 OFF/反转；GBA-001"完备性>专精度"断言=成立域仅 S2；权重表"生鲜驱动"配置=生鲜占比<50% 时须换表 | **是**——按坐标取布尔 |

> **参数化的边界铁律(防伪精确/防膨胀)**：一个变量**只有在"有真实门店证据证明它随某轴变、且这个变化会改变一个具体动作"时**，才从 L-INV 提升为 L-PAR。默认留在 L-INV 当全局常量。**禁止投机性参数化**(给每个指标都配 4 套值 = 伪精确、维护地狱、老刘 M3-T5)。不变量→参数的晋升是**经验驱动**(被一家真实店证伪才晋升)，不是设计时一次性铺满。

> **坪效分母 L-INV 常量(v0.2 补·归型演练破绽③)**：坪效分母口径 = **实用销售区㎡(排除仓储/后场)**，是 L-INV 强制工序常量、**不是** L-PAR 参数——分母口径若随店漂移则跨店坪效根本不可比(花厅坊同月销用合同1800㎡→¥1,378、用实用1500㎡→¥1,653，差20%)。坪效的**基准值**仍是 L-PAR(随坐标变)，但**分母定义**冻结为全局常量。所有 profile 的坪效字段须附面积口径标注。〔六哥 2026-07-04 定花厅坊实用1500㎡已隐含拍此·档A即改〕

### 2.2 与 KB-CAT-ARCH-001 双投影的正交性(不冲突证明)

- CAT-ARCH 作用在**品类/商品轴**(L1–L5 分类 + C 码锚点 + per-SKU 角色属性)。本 profile 作用在**门店轴**。两者天然正交。
- profile **绝不触碰** C 码锚点、分类层级或角色属性 schema——那是 L-INV，是 SSOT，本页只读不改。
- profile 里所有品类级参数**以 L3 C 码为 key**(与 MULTISTORE 用 C 码做 JOIN 键同源)。例：`价格带配比` 参数不是改 L4 分类、也不是改 `is_traffic` 字段定义，而是给出"本坐标下 A01叶菜 的目标角色分布 = 引流盘为主"。**改的是目标分布(profile 数据)，不是 schema(架构)**——恰如 v1.1 裁决"价格带是 per-SKU 角色属性、与 L4 分类轴正交"。store profile 正是这套角色属性的**按店型目标分布层**，骑在锚点上、不重定义锚点。

### 2.3 Profile 数据结构

```yaml
profile:
  # —— 坐标(描述符) ——
  coord:
    region_tier: 三线            # A1
    strategy_type: {primary: S1, secondary: S2}   # A2 主型+副型
    chain_stage: 10-30店         # A3
    size_band: 800-1200          # A4（可为 per-store 覆盖）
  provenance: {source: 好家源, n: 0, calibrated: false}

  # —— L-PAR 参数覆盖(每个值三元组：value + fact_layer + source) ——
  params:
    客单价基准:     {value: [25, 38], fact_layer: inferred, source: 三线消费力代理}
    毛利率目标:     {value: [19, 24], fact_layer: inferred, source: 连锁采购+V5.0}
    生鲜销售占比基准: {value: [62, 75], fact_layer: inferred}
    CHI权重:        {value: {C1:.18,C2:.10,C3:.10,C4:.18,C5:.22,C6:.12,C7:.10}, fact_layer: inferred}
    价格带配比:     {value: {P:.55, M:.40, Pref:.05}, fact_layer: inferred}  # by C码可再细化
    # ... 其余 L-PAR 指标同构

  # —— L-APP 适用性开关 ——
  applicability:
    品类完备性指标: off        # S1 主型：完备性非本型考核轴
    完备性优于专精: false      # GBA-001 断言在 S1 不成立
    场景A内部横比:  on         # chain_stage≥2店 → 解锁 10 店内部横比

  # —— 覆盖来源链(级联，见 §3.3) ——
  cascade: [base(region_tier), strategy(S1), chain(10-30店), size(per-store), override(六哥)]
```

> **fact_layer 三态**沿用 MULTISTORE：`inferred`(结构推断，零实证) → `observed`(N=1 单例锚点) → `calibrated`(N≥3 区间，过三轮方可对外)。每个参数独立带自己的 fact_layer，**禁止整份 profile 一刀切标 observed**。

## 3. 路由功能实现机制

### 3.1 数据流(端到端)

```
门店属性输入
   │
   ├─(A1)region_tier ← 查表(城市线级/房价代理)          ─┐
   ├─(A3)chain_stage ← 查(客户门店数)                    ├─ 直接取值(不判别)
   ├─(A4)size_band   ← 测量(卖场净陈列㎡)                ─┘
   └─(A2)strategy_type ← 判别器(3条POS判据)→ 主型+副型   ── 唯一需"判别"的轴
   ▼
[归型] → 坐标 coord = (region_tier, {primary,secondary}, chain_stage, size_band)
   ▼
[级联合成] profile = resolve(coord)   # §3.3 CSS-cascade 覆盖
   ▼
[实例化标准] for metric in L-PAR∪L-APP: 取值 + 附 fact_layer + provenance
   ▼
[数据可行性门] Pre-flight Gate(复用 MULTISTORE §0.5)
      每条标准 × 本店实际数据清单 → ACTIVE / DEGRADED / LOCKED
   ▼
[输出] 实例化标准集 + 置信标注 + 降级说明 + 缺数动作清单
   ▼
[校准回路] 诊断产出 observed 值 → 回填 profile.params(inferred→observed→calibrated)
```

### 3.2 判别器伪代码(strategy_type)

```python
def classify_strategy(pos_data, field_survey):
    votes = {}
    votes['促销日客流比'] = bucket_promo_ratio(pos_data.promo_day_traffic /
                                              pos_data.normal_day_traffic)  # ≥3→S1; <1.5→{S3,S4}
    votes['时段结构']     = bucket_time_profile(pos_data.hourly_traffic)     # 早市→S1; 晚峰→S3; 均衡→S2/S4
    votes['500m竞对锚']   = bucket_anchor(field_survey.competitors_500m)     # 菜场/折扣→S1; 高端盘→S4
    tally = Counter(votes.values())
    top, n = tally.most_common(1)[0]
    if n >= 2:
        primary = top
        secondary = next((t for t in tally if t != top), None)   # 剩余判据他型→副型
    else:                       # 三判据三分裂
        primary, secondary = mixed_zone_resolve(votes)           # 强制主+副，禁取中间值
    return {'primary': primary, 'secondary': secondary,
            'fact_layer': 'inferred', 'evidence': votes}
```

### 3.3 级联覆盖(profile 合成的核心机制)

**覆盖顺序(低→高优先级)**，后者覆盖前者的同名参数：

```
1. base(region_tier)      # 先定绝对值标尺：客单/价格带量纲
2. strategy(primary)      # 定结构：生鲜占比/角色配比/权重/完备性开关
   + strategy(secondary)  # 副型只打"模块补丁"(时段/品类局部)，不整份覆盖
3. chain(chain_stage)     # 调毛利/周转；开/关场景A内部横比与数据可得性
4. size(size_band)        # 调 SKU 量级带/完备性上限/坪效分母
5. override(六哥)          # 人工覆盖，最高优先级，逐条记 log
```

**冲突裁决**：高层覆盖低层(cascade)。语义分工保证冲突可控——各轴主管不同参数子集(A1 管量纲、A2 管结构、A3 管成本、A4 管规模)，真正硬撞的参数极少；撞到时按优先级取高层值并标注 `overridden_by`。

### 3.4 Fallback / Override / 校准 三回路

- **Fallback(坐标未 populate)**：目标坐标无 profile → 沿 cascade **向上找最近已定义祖先**，缺失轴用"轴默认值 × 跨轴缩放因子"补(如 S3×三线 未定义 → 取 S3×大湾区 结构 × region_tier 客单缩放)，结果强制降 `fact_layer` 一级并标 `fallback_from`。**禁止**用 fallback 值对客户称"标准"。
- **Override(人工)**：六哥可逐条覆盖任一实例化值；**档位判定**——覆盖单店单参数=档A(即执行告知)；改某 profile 影响多店=档C(列清单确认)；改轴定义/参数化 schema=档D(签字+记 log)。
- **校准回路(inferred→calibrated)**：诊断后 observed 回填，沿 MULTISTORE §3.4 阶梯：N=1 单例(仅参照)→N=2(内部趋势)→N≥3(区间，过三轮可对外)→N≥5(高可信)。**回填必须带全 4 轴坐标字段**(见 §6 对 MULTISTORE 的变更)，否则跨坐标污染基准。

### 3.5 数据可行性门 = 独立层(不与店型耦合)

**关键正交结论**：profile 回答"这坐标下标准**应该**是多少"(规范)；Pre-flight Gate 回答"这家店的实际数据**能不能**算"(可行)。二者独立——小店"倾向"缺 POS/会员是**倾向不是定义**，好家源某店可能有完整 POS。故降级**只按本店实际数据清单触发**，直接复用 MULTISTORE §0.5 的 LOCKED/DEGRADED 判定，**不在 profile 里为每个店型预写降级解**(这是对主线程假设③的修正)。RFM 是否可用、坪效是否 LOCKED，取决于会员渗透率/面积是否测得，与 S1–S4 无关。

## 4. 通用性 vs 专用性：终裁

**终裁：参数化(单系统 × profile 级联)，不分叉 N 套系统。** 边界：**在"实体社区生鲜超市 500–2000㎡"业态内一律参数化；跨业态(前置仓/大卖场/社区团购/便利店)不假装一套覆盖、需另立系统。**

裁决理由(MECE)：

1. **成本结构**：分叉 = N 套 7D/双投影/标准化工序各维护一份 → 违反"少而深"，每套独立漂移，改一处主方程要改 N 次。参数化 = 不变量单套维护，只维护叶子值表。
2. **学习复利**：分叉后每套的校准样本各自为政，永远凑不够 N≥3(每个客户 1–2 店)。参数化让所有店的 observed 值按坐标回填**同一个** profile 库，同坐标样本可跨客户池化——这是唯一能攒够统计基准的路径(宪法"一切复利为资产")。
3. **正确性**：店间真实差异是**值差**(客单/毛利/占比)与**开关差**(某指标适用与否)，**不是结构差**——7D 的七个维度、RM-01 主方程、双投影锚点在 S1 和 S4 完全一样，只是阈值/权重不同。值差用参数、结构差才需分叉；社区生鲜超市内部无结构差,故参数化足矣。
4. **反方 steelman(诚实给出)**：若某店型需要一个在别型**无意义**的维度(不是取值不同、是根本不存在)，参数化会被撑成"一堆全是 off 的适用性开关"，此时分叉更诚实。**判据**：当某坐标的适用性开关 off 比例 > ~40%(多数指标不适用)，说明它已越出业态、应触发"是否另立系统"审查。这正是业态边界(前置仓/大卖场)的信号线。**参数化的失效点 = 业态的边界**，与 MULTISTORE §8.6 失效边界一致。

## 8. 开放问题(留给第 2/3 轮)

1. `region_tier` 的取值代理是什么最省？城市线级太粗(同城有高端盘与县域)，建议用"商圈房价/人均可支配收入代理"，但采集成本待定——可能沦为第二个"消费阶段不可测"陷阱，须第 3 轮压测。
2. strategy_type 副型的"模块补丁"如何形式化？副型只打局部补丁 vs 整份覆盖，接口未定义。
3. `chain_stage` 分档(单/2-4/5-10/10-30/30+)是否过细？好家源=10店恰在 5-10/10-30 边界，边界店归档规则待定。
4. profile 参数的"晋升门槛"(inferred→L-PAR)判据是"一家真实店证伪"，但 N=1 证伪够不够触发全局参数化？还是需 N≥2？
5. 4 轴是否已穷尽？是否遗漏"线上/即时零售渗透"轴(承接 STORE-TYPE 第2轮问④)——若前置仓/到家一体化成常态，可能是第 5 轴或直接是业态边界。
6. **(v0.2 补·归型演练破绽④)生鲜主导业态下 strategy_type 的 S1–S4 四分是否过细？** 花厅坊(S2·大湾区)生鲜占比 67.6% 恰落在架构给三线 S1 的 [62,75] 区间内——生鲜占比在生鲜主导业态里 S1/S2 高度重叠、几乎无区分力(此点反向加固了 §7 假设②"逐出生鲜占比"的正确性)。含义：A2 判别**必须**靠客单/促销弹性/竞对(恰是数据最缺的三项)，一旦缺失，S1/S2 在生鲜社区超市里可能本就难分。S1–S4 是否应对生鲜主导业态并档为 2–3 型？留第 2/3 轮压测。
7. **(v0.2 补·归型演练破绽⑤)region_tier 缩放因子未被压测**：花厅坊正处基准位(大湾区)，只验证了 A1 的"取值"顺畅(广州→一线)，`§3.4 fallback` 用的"region_tier 客单缩放因子"**完全未压测**，须等好家源三线进场验证。本 N=1 对 A1 的验证不完整(取值✓、缩放✗)。
