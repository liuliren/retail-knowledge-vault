# 数据与系统能力底座

## 1. 文件定位

本文件定义**数据与系统能力底座**的核心概念、构成和对三大支柱的支撑关系。它是 `08_经营分析与数据看板`、`11_系统产品与PRD`、`13_数据分析与工具脚本` 等模块的方法论上游依据。

## 2. 核心定义

**数据与系统能力底座**不是独立于商品力、组织力、运营力之外的第四力，而是支撑三大业务能力运行的基础设施。它让科学零售从经验判断升级为可观察、可分析、可执行、可复盘、可复制的管理体系。

底座本身不创造经营目标，也不替代业务判断，它的价值在于让商品、组织、运营三大能力更高效、更准确、更可持续地运行。

### 2B. 数据化管理 = 底座之上的「分析→运用」管理方法（逐字批43回填·零售老刘《数据分析架构系列》08·10·11·六哥2026-06-26「先回填」签）

> 来源:零售老刘《数据分析架构系列》08/10/11(转述)。**把"数据底座"(基础设施)与"数据化管理"(用底座做事的方法)分开**:底座是地基,数据化管理是地基之上的经营动作。

**结论先行:数据化管理 = 运用分析工具,对客观、真实的数据做科学分析,并把分析结果落到销售/业务流程各环节的管理方法。**

1. **三阶段闭环**:数据收集整理 → 分析 → 运用。缺任一环都不成立——只收集不分析 = 数据坟墓;只分析不运用 = PPT 好看不落地。
2. **关键差异(vs 常规数据管理软件)**:数据化管理会**区分数据真假、剔除错误数据**。销售数据天然有水分(退货、错单、促销虚高、刷单),不剔除就分析,等于在沙子上盖楼——这正是 §3「数据清洗·异常数据识别」、§4B「数据混乱 = 经营混乱」铁律的方法论母题。
3. **金句**:"数据不会骗人,关键是如何运用和解释它。" 顾客嘴里说的 ≠ 真打算做的——问卷里说"愿意买有机菜"的人,购物小票未必有一笔有机菜;**看小票(行为数据)验证嘴(口头数据)**,才是数据化管理的真功夫。
4. **工程化落点**:剔除错误数据 = Data Adapter 的**数据质量治理**(§3 数据清洗/异常数据识别 + §4B 三准则之"业务可解释性"/"实时性可分级")的方法论上游——adapter 在工程层做的"脏数据隔离",就是数据化管理在方法层要求的"剔除错误数据"。

## 3. 为什么不是第四力

| 判断维度 | 第四力（错误认知） | 底座（正确定位） |
|----------|------------------|----------------|
| 是否独立产生经营价值 | 是 | 否，依附于三大支柱 |
| 是否有独立业务目标 | 是 | 否，服务于商品/组织/运营目标 |
| 是否可以独立考核 | 是 | 否，以三大支柱的落地质量为衡量标准 |
| 核心作用 | 独立驱动 | 支撑放大 |

**结论：** 把数据与系统能力单独列为第四力，会导致为了数字化而数字化，脱离真实业务问题；正确做法是把它嵌入三大支柱的执行链路中，让每项业务能力因底座而更强。

## 4. 数据与系统能力底座的核心构成

| 构成模块 | 核心内容 |
|----------|----------|
| 数据采集 | 销售数据、库存数据、客流数据、价格数据、陈列现场数据的采集规范 |
| 数据清洗 | 异常数据识别、条码匹配、字段标准化、历史数据修复规则 |
| 指标体系 | 销售额、毛利额、动销率、库存周转、客单价、坪效、人效等核心指标定义 |
| 分析模型 | SKU ABCD分类、ABC-Z模型、品类贡献分析、价格带分析、活动效果分析 |
| 管理系统 | 门店经营管理软件、任务系统、巡店工具、报表平台 |
| AI工具 | Claude Code辅助分析、智能报告生成、异常预警、辅助决策 |
| 报告输出 | 周报、月报、老板看板、品类报告、商品诊断报告的标准化模板 |
| 执行闭环 | 数据分析结果转化为任务清单、跟踪执行、验证结果的完整链路 |

### 4B. 两大基础表 = "哼哈两将"（零售老刘逐字批23回填·六哥 2026-06-25 签）

> 来源:零售老刘《品类建模》149-154 商品MD系列(转述)。**点明数据底座最底层的物理地基**:不是先有分析模型,而是先有两张干净的基础表。

- **商品信息表(Item/属性) + 商品结构表(分类骨架) = 零售经营的"哼哈两将"**,二者通过结构表最细类别(小类/细类)与信息表建关系 = 一切数据建模的核心关系。**商品信息表=数字化地基**(印证 Data Adapter 13 字段 schema);**商品结构表=分类骨架/"战略地图"**,最细一级=需求单元("一个小类=一个需求")。
- **铁律:数据混乱 = 经营混乱**。凡后台数据严谨、清晰、与业务流程关联的超市,经营至少良性;反之数据一团糟的,几乎无良性经营——**两表的整洁度可作诊断一家店经营好坏的辅助判据**(花厅坊"建错层级/进价不准"即此类坑)。
- **典型错配**:500㎡ 小店套 3000 小类大卖场结构表 = 结构与需求严重不符(对照 [[多品类压力测试报告_v2.0]] 的 SKU 冗余/结构过密失真)。
- **商品表=业务神经中枢**:一个数据触发全链条反应(周转率低→采购减单+销售促销+配送降频+财务计提);三条黄金准则=**①维度一致性(同一 product_sk 指向同一维度源·避免数据孤岛)②业务可解释性(字段名要一线看得懂)③实时性可分级(销售 T+1·绩效按周)**——这三条正是 adapter/底座的设计准则。

### 4C. 库存系统的交易逻辑（零售老刘逐字批39-40回填·六哥 2026-06-26 签）

> 来源:零售老刘《库存管理系列》06-12(转述)。**库存数据怎么在系统里被记录/流转——是 adapter 之上"库存维"的工程地基**。fact_layer=inferred(外来系统知识)。

**库存管理「六位」框架 = §4C 货位管理统领框架(逐字批46回填·零售老刘《商品库存管理分析建模》12·六哥2026-06-26「先回填」签)**

> 来源:零售老刘《商品库存管理分析建模》12(转述)。**这是把此前散落在 §4C 的货位三层 / 三位一体 / 满陈列量 / 货位5类单品数收成的总框架**——下文各条都是某一"位"的细则,本块是统领。fact_layer=inferred(外来系统知识)。

**结论先行:商品在卖场的「位置 + 数量」,统领为「六位」= 定位 / 定数 / 定品 / 定架 / 定价 / 定管。**

1. **定位(商品的"家")**:每个商品都要有货位编码。狭义=商品在卖场的位置,广义=商品定位——新店做商圈调研定类别结构、出落位图,老店按需求先调小类。**小类货位 BAY 分配量 = 该小类综合权重占比 × 总 BAY 数**(例:综合占比 0.2% × 300 组 = 6 组 BAY)。〔收编:本节"货位三层 仓库→库区→库位"与"三位一体"的库位/货位,都落在「定位」下〕
2. **定数(合理数量)**:确定各货位/类别的合理数量,是选品优化的前提(各类别数量分配)。〔收编:"满陈列量=纵×横陈列面"(批41)、"货位编码内单品数=5类"(批45)都是「定数」维度的数量口径〕
3. **定品(选品)**:末位淘汰太粗糙,要依品类管理——先搭商品完整性(集群·"面粉没酵母=过不去")+场景营造,再谈汰换。**本条只点"定品"在六位中的位置,生鲜选品详见 [[品类管理]] §3.1D 生鲜专题,不在此重复承载(wikilink 互指)。**
4. **定架(货架位)**:确定货架位。卖场调整多是"减货架"——根因常是原定位就错 + 没按商圈调 + 没维护 BAY 销售状态。
5. **定价**:价格是最好的经营工具。有生鲜的超市要"表演便宜·烟火味·离顾客近",不能让顾客觉得贵。
6. **定管**:以货位为业务主线,定流程规范 / 绩效考核。〔收编:"三位一体"的人位(课组归属、谁管谁考核)即「定管」的落点〕

**收编说明(不重复承载)**:§4C 既有各条按"位"归位——货位三层 / 三位一体(库位·货位)→定位;满陈列量 / 5类单品数→定数;三位一体(人位)→定管。六位是统领,各条是某一"位"的细则,彼此互指、互不重复。下文各条保持原样,只在概念上挂到对应"位"下。

- **货位三层结构**:仓库(库房·最上层)→库区(分区)→库位(货位·具体编号)。**陈列的准确定义=解决商品在卖场里的"数量和位置"**(与 [[数量管理]] §6.6 "库存两点=数量+位置"互为表里)。
- **复式库存(核心)**:每笔变动同时记减(−)和增(+),**所有库位库存加总恒为零**(类似会计借贷);出入库明细只记"变动"不记绝对数,某库位某品数量=目标库位数−源库位数。**库存价值"两张皮"**(财务 vs 库管口径差)靠"每笔移动自动生成会计记录"统一(出库−/贷方、入库+/借方)。
- **实际库存 vs 虚拟库存**:实际库存=现有实有;**虚拟库存=现有实际+未来一段入库−未来一段出库(与时间相关)**——**下订单要参考"到货时点的虚拟库存",不是现有库存**(避免重复下单/缺货)。
- **库存六状态**:可销售(库存≥0)/残次/调配/在途/锁定(促销买赠占用)/虚拟/报损——库存状态是动态非恒值,**是库存管理中最难维护的部分**,可用参数表+度量(ABC/可销天数)动态计算。
- **库存流水五要素**:单据号及来源类型/变动方向(增·扣·锁·释)/变动数量/流水时间/变动信息(商品ID·SKU·货主·仓库·库位);多模块(采购/WMS/财务/PDA/配送)须实时同步,否则超卖缺货难发现。
- **复用 §4B 铁律**:库存系统的一切分析都以"商品组织结构表"为地基——500㎡ 小店套 3000 小类结构,连盘点都做不下去,遑论库位分析。
- **满陈列量 = 再订货点的咬合(逐字批41回填·零售老刘《零售超市库存管理系列》17/19·六哥2026-06-26「先回填」签)**:**满陈列量 = 一个商品在卖场(不含内仓)可允许的最大陈列量 = 纵向陈列数 × 横向陈列面数**。系统的自动补货建议表有一项「**按库存量(满陈列量)指标自动补货**」——这正是货位「位置」与「数量」两要点的**咬合点**:**满陈列既是陈列上限,又是系统的再订货点依据**。承接上文货位三层(仓库→库区→库位)只定了商品的"位置",满陈列量则把"数量"锁死在位置上:位置定"放哪",满陈列定"放满到几个、补到几个就停"——两者合起来才是 [[数量管理]] §6.6"库存两点=数量+位置"在系统补货侧的落地。
- **库位 + 货位 + 人位「三位一体」(逐字批43回填·零售老刘《数据分析架构系列》08·10·11·六哥2026-06-26「先回填」签)**:承上文货位三层(仓库→库区→库位)只解了"商品放哪",商品数据库管理还要把**三类"位"设成相互关联的一张网**——**①库位(系统数据库位:配送中心/门店/专柜,即商品在哪个实体/法人里)②货位(卖场 BAY,即上文货位三层的物理排面)③人位(员工类别/课组,即这块货归谁管、对谁考核)**。三位关联同时解三件事:**商品定位优化、人员区域商品管理、绩效考核分配**(哪个课组管哪个 BAY 的哪些品、卖好卖坏算谁的,一张表说清)。落地可用 Power BI / Excel 建表关系做"库位分析"——把三位当三张维度表与销售事实表关联(呼应下文 §4D「维度 × 度量」)。
- **货位编码内单品数 = 5 类(货位健康度诊断判据)(逐字批45回填·零售老刘《商品库存管理分析建模》4·6·7·六哥2026-06-26「先回填」签)**:承上文货位三层(仓库→库区→库位)与三位一体只定了"货位是什么、归谁管",本条再补**同一个货位编码下"单品数"有 5 种口径**,口径之间的落差正是诊断货位健康度的判据。**5 类单品数**:① **系统信息数**(系统里最初配置的单品数·一般最大)② **在陈单品数**(实际已陈列的)③ **在销单品数**(有销售记录的)④ **能配单品数**(这个货位合适陈列的单品数)⑤ **参考 BAY / 单品数**(同类店 / 竞争店同位置的单品数)。**诊断用法(看落差找病)**:① **系统信息数大,但能陈列的少 = 货位久未维护**(系统里挂了一堆、货架上摆不下,典型"该淘汰的没淘汰"积压在系统里);② **能配数远大于在陈数 = 拉排面拉得厉害、缺单品**(货位本能放很多品,实际只摆了几个畅销品,把排面拉宽撑场——即**"良品驱赶劣品"现象**:畅销品越拉越宽,把本该进场的其他品挤没了)。这 5 类单品数是 §4C 货位三层 / 三位一体在"数量口径"维度的延伸,可作 report-export 货位健康度量的判据来源。
- **货位 BAY 分配精化 = 销售占比 × 毛利占比(交叉比例)的比值分配(逐字批49回填·零售老刘《分析建模》04·六哥2026-06-26「先回填」签)**:承六位框架「定位」既有「小类货位 BAY 分配量 = 综合权重占比 × 总 BAY 数」,本条把"分配权重"再算细一层——**用销售占比 × 毛利占比的交叉比例算出比值,据此分配区位的货位(BAY)数 + 货位的单品数**。比单纯"综合权重占比"更精:既看走量(销售占比)又看赚钱(毛利占比),两者交叉,避免"只走量不赚钱"或"赚钱却走不动"的品占错排面。**作为既有 BAY 分配公式的精化补充,不替换、二者择优**(简版用综合权重占比×总BAY,精版用销售×毛利交叉比)——挂在「定位/定数」下:交叉比定 BAY 数(定位)、再定货位单品数(定数)。
- **货位商品状态判定 = ABC × 数量属性 → 6 类诊断标签(货位健康度可落地诊断)(逐字批49回填·零售老刘《分析建模》04·六哥2026-06-26「先回填」签)**:承上文「5 类单品数」是货位健康度的"数量口径",本条把口径落成**可落地的诊断标签**——结合 **ABC 分类 + 数量属性(在销 / 在陈 / 已选 / 库存)**,逐货位判出商品状态:**① 有销无库(有销售记录但库存为 0)② A 类无货(A 类畅销品却无货)③ 已选无库(已选品但无库存)④ 有库未陈(有库存却没上架陈列)⑤ 畅销缺货(畅销品断货)⑥ 此货暂缺(暂时性缺货)**。这是 5 类单品数(批45)的**诊断化运用**,可进 report-export 做"货位健康卡"。**数量参数表补充(喂这套判定的字段)**:商品总信息表 / 在销单品数 / 在陈单品数(盘点机扫货位条码得) / 已选单品数 / **货位条码差异表(预估配置数 vs 已配置数的差异)**——差异表正是上文"系统信息数 vs 在陈数"落差在工程层的载体,是状态标签的数据来源。
- **各类单品数状态 = 8 类口径(批45「货位5类单品数」的扩展·5类→8类)(逐字批55回填·零售老刘《分析建模》21·六哥2026-06-26「先回填」签)**:**结论先行——承上文批45「货位编码内单品数=5类」,本条把"单品数"口径从 5 类扩到 8 类,把"系统侧 vs 卖场侧"两条线分得更细,落差越多越准地暴露货位病。** 8 类单品数状态:① **总单品数**(店里所有商品的总口径)② **库位单品数**(系统库位口径·**一个 SKU 一个家**·即批45"系统信息数"的细化)③ **货位单品数**(卖场货位实际陈列口径·即批45"在陈单品数")④ **满陈列单品数**(货位摆满时的单品数·咬合上文「满陈列量=纵×横陈列面」)⑤ **在销单品数**(有销售记录的·同批45③)⑥ **选品配置数**(选品配置应有的单品数·对应批45"能配单品数"/批49"已选单品数")⑦ **当前库存数**(此刻实有库存的单品数)⑧ **此货暂缺数**(**可订、可销、需陈列,但没陈列出来的·≠无库存**——它是"该摆没摆"的缺口,不是断货)。**承接说明(不重复承载)**:批45 是 5 类(系统信息/在陈/在销/能配/参考BAY),本条在其上扩出"总单品数、当前库存数、此货暂缺数"三个新口径,并把"系统信息数"拆成"库位单品数(一SKU一家)"、把"在陈"对齐"货位单品数",其余沿用批45 不重复定义。**核心判据:库位数(系统口径) vs 货位数(卖场口径)不一致 = 单品数异常;两者落差越大、异常项越多,经营管理水平越差**——这一条把批45"看落差找病"从 5 类推到 8 类的系统化版本,可进 report-export 做"库存异常诊断卡"的数量底座。
- **⭐7 大异常商品 = 卖场货位形态的库存异常清单(货位健康诊断标签集·可落地)(逐字批55回填·零售老刘《分析建模》21·六哥2026-06-26「先回填」签)**:**结论先行——承上文批49「ABC×数量属性→6类诊断标签」,本条把"单品数异常"落成一张可直接扫出来、可发店长的库存异常清单,分三个形态层。** **① 7 大异常商品(卖场货位形态)**:**㈠负库存**(账面库存为负·盘点/收货错单)**㈡高库存**(库存远超满陈列+合理周转·压资金)**㈢负毛利**(售价低于进价·卖一个亏一个)**㈣畅缺货**(畅销品却断货·最该补没补)**㈤零现象**(销售/库存/毛利/进货任一为零·四零里有一零就该查)**㈥有销未订**(有销售却没下订货·迟早断)**㈦终止商品**(已终止/淘汰却还挂在货位上)。**② 单品组合形态(单品自身的状态组合)**:有陈列无库存 / 有库存无销售 / 有销无订货 / 此货暂缺 / 临过期——这五种是上文 8 类单品数两两交叉出的"病态组合"。**③ 系统库位形态(商品生命周期状态机)**:新品 → 试销 → 评估 → 正常品 → 禁止下单 → 淘汰 → 删除——商品在系统库位里的状态流转,异常往往是"卖场形态"与"系统状态"对不上(如系统已"禁止下单"卖场还在销、系统"正常品"卖场却"此货暂缺")。**核心:库位数(系统) vs 货位数(卖场)不一致 = 单品数异常,异常程度直接反映经营管理水平**;这 7 大异常 + 两类形态是批45 5类/批49 6标签的最终落地清单,可进 report-export 做"库存异常诊断卡"。

### 4D. 数据分析三部曲 = 维度 × 度量 × 输出（逐字批43回填·零售老刘《数据分析架构系列》08·10·11·六哥2026-06-26「先回填」签）

> 来源:零售老刘《数据分析架构系列》08/10/11(转述)。**给 §4「分析模型」补一个可操作的最小动作框架**:任何一次零售分析都能拆成"从哪个维度、看哪个度量、怎么输出"。

**结论先行:数据分析的本质 = 从哪个维度(角度)去了解哪个度量(指标),然后展示(输出)。一次分析 = 维度 × 度量 的组合。**

1. **Power BI 三步落地**:加载数据 → 数据建模 → 拖透视/度量输出。**这三步正好咬合 §4B 两表与本页统一 13 字段 schema**:加载 = 喂事实表 + 维度表;建模 = 按 §4B「结构表最细类别建关系」搭维度;输出 = 拖度量。
2. **零售常用 7 维度**:时期 / 门店 / 仓库 / 类别 / 商品 / 供应商 / 客户。
3. **零售常用 9 度量**:销售额 / 销量 / 毛利 / 毛利率 / 单品数 / 客单 / 客流 / 交叉率 / 动销率。
4. **组合即分析**:例「6 月份 × 001 店 × 休闲食品 → 动销率 52%」= 3 个维度锁定切片 + 1 个度量取值。
5. **与底座 schema 的对应(关键)**:**维度表 ↔ 维度,事实表 ↔ 度量**——§4B 商品结构表/信息表是维度的来源,销售/库存流水(§4C 库存流水五要素)是度量的来源;统一 13 字段 schema 让"维度可对齐、度量可累加",否则维度对不齐、度量加错口径,三部曲第一步就崩。

**收束:数据分析的本质 = 建模(再往上收一层·逐字批46回填)**——把上面三部曲再收一层,落到"为什么是维度×度量":**① 数据分析的首要任务 = 建模**(哪怕只是拉一张 Excel 表,也是在建模)。数据模型 = **维度 + 度量**,二者组合成业务逻辑;核心一问 = **从什么角度(维度)看什么指标值(度量)来服务决策**——三部曲的"维度×度量"正是建模的最小单元,不是堆指标。**② 零售数据分析的主维度 = 商品·条码(货号)= 最细粒度**:条码是颗粒度最细的那根轴,其余维度(类别[**最细小类·先与商品信息表关联**]/时期 Date/供应商/门店)都是挂在它周围的"关系主维度列"。**③ 业务单据(数量度量)+ 财务单据(金额度量)可统一进一个数据模型**(业财联合)——同一套维度、两类度量、一张模型。**对 Data Adapter schema 设计的原则(关键)**:**条码 = 主键 = 最细粒度**(印证本页统一 13 字段 schema 的 product_sk 主键设计);类别维度**"先与商品信息表关联"再上挂**(呼应 §4B"结构表最细类别与信息表建关系")——建模动作在工程层就落成"以条码为主键、小类先关联"的 schema 骨架,业财两类度量共用这套维度即"业财联合"模型。(逐字批46回填·零售老刘《商品库存管理分析建模》12·15·六哥2026-06-26「先回填」签)

### 4E. 卖场三指标 = 转化率 / 驻停时间 / 价值指数（卖场诊断核心指标集）（逐字批51回填·零售老刘《分析建模》09卖场经营指数·六哥2026-06-26「先回填」签）

> 来源:零售老刘《分析建模》09 卖场经营指数(转述)。**承 §4D「维度×度量」是分析的最小动作,本节给"度量到底要落到哪三个卖场指标上"——把一堆经营指标收口成卖场诊断的三个核心切口**。fact_layer=inferred(外来系统知识)。

**结论先行:所有经营指标,最终都回到卖场指标;卖场指标(维度化、量化之后)= 转化率 / 驻停时间 / 价值指数 三个。这是卖场诊断的核心指标集,转化率作首要切口。**

1. **⭐转化率(最关键)= 来店客流 ÷ 有效购物客流**(例:来店 1600 ÷ 有效购物 2000·实务一般控制在 90% 左右)。**这是超市数据分析最关键的指标——所有经营活动(商品/货位/陈列/绩效/服务)最终都反映在这一个数上,可由它初判商品好坏、货位好坏、陈列好坏、绩效好坏、服务好坏。国内卖场很少关注它,正是诊断切口。**
   - **商圈三率**:入店率 / 市场占有率 / 购买率(进店数 ÷ 购买数·**小票数即客流口径**)。
   - **货位转化率**:坪效 / 库效率 / 陈列转化率 / 区位转化率——往下打到货位级,落到"**一品一功能,一品一市场,一品一顾客**"(每个单品都要担一个功能、占一块市场、对一类顾客)。这与 §4C 六位框架「定位/定数」的货位健康度判据互为表里:货位转化率低 = 该货位的位置/数量/选品没担起它的"功能"。
2. **驻停时间**:顾客在卖场停留的时长。**首要影响 = 环境、布局、动线**;次要 = 商品氛围、陈列、音乐、气味;**等待环节(收银、找货)则越短越好**——前者要拉长(多停多看多买),后者要压短(别赶客)。
3. **价值指数**:见 [[品类管理]] 客群相关章节(本节只把它列为卖场三指标之一并 wikilink 互指,**不重复承载**)。
4. **建模三方向(这三指标怎么算)**:① **历史数据**(同比·和过去比)② **即时结果**(实时·注意是"实时"非"及时",看当下切片)③ **洞悉未来**(由前两者推趋势)。三方向正好咬合 §4D「时期」维度的三种取法。

**定位**:卖场三指标是卖场诊断的核心指标集,可进 report-export 做"卖场健康卡";转化率作首要诊断切口(国内少人看 = 差异化诊断价值),驻停时间看动线环境,价值指数看客群质量。

**⭐购买率(成交率)= 六大顾客购物指标的最终绩效指标(逐字批61回填·零售老刘《品类度量07》《货位调整建议》·六哥2026-06-26「先回填」签)**:**结论先行——承上文 §4E 转化率「商圈三率」里的「购买率」(批51·小票数即客流口径),本条把它扩成「六大顾客购物指标」完整体系,并点明购买率=复合综合指标+最终绩效指标。** **六大顾客购物指标 = 客流量 / 进店率 / 成交率 / 客单价 / 连带率 / 复购率**。其中 **购买率(成交率)= 有购物小票的顾客数 ÷ 进入卖场总顾客数**(进店的人里有几个真买了东西)——它既是**复合综合指标**(综合反映商品/陈列/服务/价格的合力),又是**最终绩效指标**(所有经营动作的结果最终都收口在这一个数上)。**最大盲点 = 进店量不清楚**:多数门店只有 POS 小票数(=购买客数),却不知道进了多少人,分母缺失 → 无法对比真实成交率;解法 = **客流统计系统(摄像头对接 POS)**,把"进店总人数"这根分母补上,购买率才算得准。**承接说明(只补增量)**:批51 已点明"购买率=转化率的客流口径"(购买率=进店数÷购买数·小票数即客流口径),本条不重复定义,只补三点增量——① 把购买率放进"六大购物指标"完整框架(给转化率一个上位指标家族);② 补"进店量不清楚"这一盲点与"摄像头对接 POS"客流统计系统的解;③ 补**进店后约 87% 的顾客是否真正关注到商品、取决于卖场环境/陈列科学性/显眼度**——即购买率高低有一大半在"进店之后、买单之前"的陈列环节被决定,呼应 §4E 货位转化率"一品一功能一市场一顾客"与 §4C 六位框架「定架/定位」。可进 report-export 卖场健康卡补"购买率口径(含分母客流统计)"一栏。

### 4F. 小票客群五度量 = 没有会员系统也能做客群分析的度量底座（逐字批52回填·零售老刘《分析建模》11顾客需求变化指数·六哥2026-06-26「先回填」签）

> 来源:零售老刘《分析建模》11 顾客需求变化指数(转述)。**承 §4E「价值指数看客群质量」却把客群分析甩给了 [[品类管理]],本节补上"社区超市连会员系统都没有,客群分析的度量到底怎么落地"——用一张小票当客群分析的最小颗粒**。fact_layer=inferred(外来系统知识)。

**结论先行:一张购物小票 = 一个顾客的一次购物行为;社区超市会员管理普遍缺失,小票比会员卡更可靠——所以以小票替代客群,搭出客群分析的五个核心度量。**

1. **核心前提**:超市会员系统普遍缺失、就算有也只覆盖一部分客、数据还常年不维护;而**每一笔成交都必有一张小票**——所以**小票客群比会员客群更可靠、更全量**。一张小票 = 一个顾客的一次购物行为,这是没有会员系统时客群分析的最小颗粒度。
2. **五核心度量(DAX/口径)**:
   - **客流数 = 不重复小票数**(`DISTINCTCOUNT(小票号)`)——这是整套度量的**基础度量**,其余四个都从它派生。
   - **客单价 = 销额 ÷ 客流数**——反映整体定价水平 + 商圈购买力。
   - **客单量 = 销量 ÷ 客流数**(含称重,**≠品单数**)——一次购物买走多少量。
   - **品单数 = 销售表行数 ÷ 客流数 = 连带率**——反映商品关联性,一张小票上几个不同单品(呼应购物篮·批04)。
   - **品单价 = 客单价 ÷ 品单数**——反映商品结构的平均价格带 + 商圈消费水平。
3. **要点**:**客流是基础度量,先有它才有后面四个**;Power BI 里用 `DISTINCTCOUNT(小票号)` 一行即得,**Excel 难做即时客流**(去重计数即时算很别扭)——这正是**很多门店分析很少涉及客流的根本原因**,也是用 Power BI 替代 Excel 的一个硬理由。分析方法 = **两期对比**(本期 vs 上期),才能洞悉某个小类的客流、客单是怎么变化的——孤立看一期的绝对值没意义,变化才有诊断价值。
4. **定位**:这是 **M5 小票客群法 / Data Adapter 的客群度量底座**——没有会员表也能跑客群分析,只要有小票号这根轴。可进 report-export 做"**客群健康卡**"(客流/客单价/客单量/品单数/品单价五度量 + 两期对比)。**品单数 = 连带率**,与购物篮分析(批04)同源,呼应 [[品类管理]] §3.1E;价值指数(§4E)看客群质量,本节的五度量则是价值指数往下落到"可算的客群口径"的工程载体——两者一个看质、一个给量,互为表里。

### 4G. 采购/供应商数据策略 = 双轨制优胜劣汰供货 + 降提前期（逐字批59回填·零售老刘《分析建模》31·六哥2026-06-26「先回填」签）

> 来源:零售老刘《分析建模》31(转述)。**承 §4「分析模型」「报告输出」一直停在商品/货位/客群侧,本节补上"采购与供应商"这条最上游的数据策略——把降成本、分单、降库存都从'谈判技巧'升级成'可算的数据动作'**。fact_layer=inferred(外来系统知识)。

**结论先行:采购的核心竞争力是"采购技术"而非"采购技巧"——靠分析控制(成本模型 + 双轨制分单 + 降提前期)降成本,不靠谈判嘴皮子。评价一个采购,看他的"技术性"而非"技巧性"。**

1. **⭐采购技术 > 采购技巧**:降成本的正道是**通过分析来控制**——建成本模型、把成本意识下沉到小类/单品(每个单品的成本结构都清楚),而不是靠和供应商砍价的技巧。技巧是一次性的、靠人;技术是可复制的、靠数据。**这是 §4B"数据混乱=经营混乱"铁律在采购侧的延伸:采购成本一团糟,谈判砍得再狠也补不回结构性的贵。**
2. **⭐双轨制优胜劣汰供货(核心机制)**:**同一个单品由两家/多家供货**,好的多订、差的少订 → 形成"越好越多订、越差越少订"的良性循环,自然完成供应商优胜劣汰。
   - **订货量分配公式 = (质量 ÷ 价格) × 关系**:**质量** ∝ 该供应商供货品的销量(卖得好=质量被市场认);**价格**按批次成本算(同品不同批进价不同);**关系** = 供应商综合表现系数(0–1),由**商品淘汰率 < 10% / 送货准确率 > 90%**等供应商生命周期指标喂出来(达标给更高系数)。
   - **公开公平公正**:把这套公式和打分摆到台面上 → 供应商心服口服、自己也知道差在哪、关系反而更稳。**这正是 §4C「定管」绩效考核分配逻辑在"对外供应商"侧的镜像**——对内按三位一体考核课组,对外按双轨制公式考核供应商,同一套"用数据说话、按贡献分配"的治理观。
3. **⭐降提前期 = 库存优化的总抓手**:库存优化别只盯安全库存的数值,要盯它背后的**提前期**——**安全库存 ∝ √提前期、管道库存(在途) ∝ 提前期**,提前期一降,两类库存一起降。三个抓手:① **缩短采购周期**(月配 → 半月配 → 周配,逼近高频);② **系统按"再订货点 + 满陈列量"实时触发订单**(承 §4C「满陈列量=再订货点的咬合」,把人工周期性下单改成系统事件触发,等效缩短决策提前期);③ **供应商协同降送货提前期**(把送货天数也纳入双轨制"关系系数",送得快的多分单)。
4. **定位 = 采购与供应商数据策略**,呼应本库已有 `supplier_agg.py` / `supplier_dryrun.py`(花厅坊 POS 供应商汇总脚本)——**双轨制的数据底座现成**:`supplier_dryrun.py` 已在做"一个归一条码主键 → 供应商数 > 1"的冲突检测,这批"一品多供"记录正是双轨制要分单的对象;`supplier_agg.py` 的供应商能力库已产出**价格带(低/中/高)+ 采购周期带(高频/周配/半月配/低频)+ 停购状态 + 各供应商×类别 SKU 数**,其中「采购周期带」就是本节「降提前期」的现成度量轴,「价格带 + SKU 数(近似销量/质量)」可直接喂(质量÷价格)分单公式。可进 report-export 做"**供应商评分卡**"(双轨制分单建议 + 提前期缩短清单)。

### 4H. 百货课诊断范本 = 一条可直接套花厅坊的单品类诊断链（逐字批61回填·零售老刘《品类度量07》《货位调整建议》·六哥2026-06-26「先回填」签）

> 来源:零售老刘《品类度量07》《货位调整建议》(转述)。**承 §4C 货位健康度判据、§4E 卖场三指标都是单点度量,本节把它们串成一条完整的"诊断链"——一个百货课(品类)从坪效偏低查到即时补货的端到端范本。这是 [[单品类诊断]] skill 的方法骨架,可进 report-export 诊断卡。** fact_layer=inferred(外来系统知识)。

**结论先行:一个完整的单品类诊断 = 六步链——坪效 → 动销率 → A类识别 → 货架数测算 → 三占比对比 → 畅销丢失 + 即时补货。每步都给可算判据,最后落到"砍货架 + 补A类"两个动作。**

1. **第一步·坪效切口(发现病)**:先看坪效(如某品类 **22 元/㎡** 明显偏低)——坪效低 = 这块地没产出,是启动诊断的信号灯。(呼应 §4E 货位转化率"坪效"切口)
2. **第二步·动销率定性(病多深)**:两个口径分开看——**在销动销率 = 有销 SKU ÷ 总 SKU**(选品层面有多少在动);**在库动销率 = 有销 SKU ÷ 有库存 SKU**(有货的里有多少在动)。**60% 为正常线**,低于线 = 大量 SKU 占着位不出货。
3. **第三步·A类识别(谁在扛)**:用 ABC 分类识别 A 类畅销品——这是后面"该保谁、该补谁"的依据(调 §4C 批49 ABC×数量属性判定)。
4. **第四步·货架数测算(该给多少地)**:**应配货架组数 = 有销 SKU ÷ 每组 25 单品**(例:**737 个有销 SKU ÷ 25 = 29 组**)。若实际占了 **40+ 组** → 应**减约 20 组**——多出来的货架是被无销/低动 SKU 占着的虚胖。依据 = **"货架公平分配原则"**:每个品类按它真正能动销的 SKU 量分地,不按历史惯性占地。(咬合 §4C 六位框架「定位/定架」BAY 分配 + 批49"良品驱赶劣品"现象)
5. **第五步·三占比对比(错配在哪)**:把该品类的 **销售占比 / 单品(SKU)占比 / 陈列占比** 三个数摆一起对——**陈列占比 > 销售占比 = 占地过多(虚胖,要缩面);销售占比 > 陈列占比 = 被压制(要扩面)**。三占比错位处就是货架调改的下刀点。
6. **第六步·畅销丢失 + 即时补货(救A类)**:① **去年 A 类"畅销丢失"排查**——把去年是 A 类、今年掉出去的品拉出来,查是主动汰换还是断货误杀;② **结合实时库存对"有销无库"的 A 类即时补货**——A 类且有销售记录却库存为 0 的最该补、最先补(直接调 §4C 批49"有销无库 / A类无货"诊断标签)。

**定位**:这条六步链是 [[单品类诊断]] skill 的方法骨架,把 §4C(货位/动销/A类标签)、§4E(坪效/货位转化率)、§4B(数据地基)串成一条可落地的诊断流;可直接进 report-export 做"**单品类诊断卡**"(坪效 → 动销 → 该减几组货架 → 补哪些 A 类,四句话发店长)。花厅坊可直接套:任取一个百货小类,跑完六步即得一张诊断卡。

## 8B. 零售数学的分层计算边界（简单类 L3 算 / 复杂类 L4 算）

> 来源：v3.2 `05_零售数学模型`（打捞 SALVAGE-BACKLOG #3）。修正现役"全部在 L3 汇总算"导致的指标失真。承接 [[品类管理]] §3.1B 的 L4 启用层。
> 📖 **可执行引擎(SSOT)**：[[复杂类分层计算引擎_v1.0]]（3级复杂度模型 + L3/L4/SKU决策树 + IP-Chart防失真 + 三品类真实验证）；本节是原则,引擎是落地规则。

**核心修正**：不是新增一套数学，而是**修正数学的应用边界**——
> **简单类在 L3 算，复杂类在 L4 算。**

| 类型 | 价格带/效率/补货的第一计算层 | L3 角色 |
|------|------------------------------|---------|
| 普通类 | **L3**（直接作分析主层） | 分析主层 |
| 复杂类（已启用 L4） | **L4 决策组** | 汇总 + 经营判断 |

**为什么**：同一 L3 下若含多个决策逻辑，直接在 L3 算会失真——
- 主流价位被拉歪 · 双波峰判断失真 · GMROI/周转对比失真 · 补货安全库存失真。

**复杂类正确顺序**：`L4 计算 → L3 汇总 → L2 判断`。

**四个模型的应用层**：
| 模型 | 计算层 |
|------|--------|
| IP-Chart（价格带图） | 复杂类优先在 **L4 决策组**做（否则形状失真） |
| CTM | 仍以 L2/L1 为主；L4 可做复杂类内部结构诊断 |
| GMROI / 周转 | 复杂类优先在 **L4 比较**，再决定扩面/缩面/替换 |
| 补货模型（安全库存/订货点/最小陈列量） | 按 **L4 真实消费与补货节奏**设置 |

> **接 G4-A 冲调**：冲调是多品牌多价格带的复杂类——价格带图/GMROI 应在 L4（麦片/咖啡/奶粉等决策组）分别算，再汇总到 L3，避免"全冲调一锅算"把主流价位拉歪。

## 10. 常见误区

- **误区1：** 把数据与系统能力当成独立目标——"数字化转型"本身不是目的，让三大支柱跑得更好才是目的。
- **误区2：** 以为有了系统就有了能力——系统是工具，不替代判断；数据出来了，还需要人解读和执行。
- **误区3：** 在基础数据不准确的情况下追求高级分析——数据清洗和字段标准化先于分析模型，基础不牢则分析结论不可信。
- **误区4：** 只建报表，不建执行闭环——数据到看板就停了，没有转化为行动清单，等于没用。
- **误区5：** 忽视社区超市的实际数字化能力——照搬大型连锁的系统方案，会超出门店承载能力，反而形成负担。
