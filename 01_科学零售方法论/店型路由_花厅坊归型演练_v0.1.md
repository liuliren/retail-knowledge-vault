# 店型路由架构·花厅坊归型演练 v0.1（N=1）

> [!important] 状态声明
> 本页是 KB-STORE-ROUTER-001（draft·轮次1/3）落地第一步（该文 §6.3 第1步）。**目的是压测架构、暴露破绽**，不是对外交付。全部数值 `fact_layer` 逐字段标注，混合三态。花厅坊真名仅内部用，对外须脱敏。本演练成本≈0（复用已有 POS/首诊/进货单），不依赖签字。

---

## 0. 结论先行

1. **花厅坊 4 轴坐标（inferred 主型）**：`region_tier=一线/GBA核心(广州黄埔) × strategy_type={primary: S2, secondary: 待现场判据} × chain_stage=连锁·在手样本N=1 × size_band=1200–2000㎡`。
2. **架构基本可用**：4 轴能把花厅坊落到一个坐标点，profile 级联结构装得下花厅坊实证值，fact_layer 三态并存（客单/生鲜占比/坪效=observed，综合毛利/损耗/价格带=inferred或LOCKED）——这正验证了架构"每参数独立带 fact_layer、禁止整份一刀切"的决策。
3. **但暴露 5 处破绽**（详见 §4，含 1 处架构内部自相矛盾）：① §5.3 把 `chain_stage=单店` 与 `场景A LOCKED` 画等号，违反 §3.5 自己的解耦原则；② A2 判别"成本≈0/POS已有"过度乐观，实测仅 1/3 判据数据现成；③ 坪效分母未定义，花厅坊三个面积口径给三个坪效值；④ 生鲜占比对 S1/S2 几乎无判别力（花厅坊 S2 大湾区 67.6% 落在架构给三线 S1 的 [62,75] 区间内）；⑤ region_tier 在花厅坊=基准位，缩放机制未被压测（要等好家源）。
4. **花厅坊 profile 作为参照基准就位**：客单 42.14、生鲜占比 67.6%、月坪效 ¥1,653/㎡ 三个 observed 锚点，供好家源三线连锁 profile 对比。

## 1. 4 轴坐标定位（逐轴给值 + 判据依据）

| 轴 | 取值 | fact_layer | 取值方式 | 判据依据（源） |
|---|---|---|---|---|
| **A1 · region_tier** | 一线 / GBA核心 | observed | 查表 | 广州市黄埔区云埔街道（门店基础档案）→ 一线城市、大湾区核心消费力带。**恰好落在现系统的基准校准位**。 |
| **A2 · strategy_type** | primary **S2**（家庭厨房主力）；secondary 待现场判据 | **inferred** | 判别器（3判据，仅1条数据现成） | 见 §1.1 逐判据。综合信号：客单 42.14 偏高（非刚需菜场平价）、客数驱动增长 +14.97%（非促销驱动）、生鲜综合区 f≈70.74% 家庭厨房主力盘、电子价签/现代明亮/客群多元 → 偏 S2；非 S1（客单太高、非促销敏感）、非 S4（客单 42 未达精品、含大量刚需米面油日化）。 |

| **A4 · size_band** | 1200–2000㎡ | observed | 测量 | 实用销售区 1500㎡（合同 1800㎡ / 仓储 500㎡，六哥二访订正）→ 落 1200–2000 带。**描述/调制字段，不参与判别。** |

### 1.1 A2 三判据实跑（这是暴露破绽②的地方）

| 判据 | 数据现成？ | 花厅坊取值 | 归型指向 |
|---|---|---|---|
| ① 促销日客流比（≥3→S1；<1.5→S3/S4） | **否**——需促销日历 + 促销日/平日客流拆分，花厅坊 POS 未见此拆分导出 | 缺 | — |
| ② 时段客流结构（早市→S1；晚峰→S3；均衡→S2/S4） | **部分**——客流时段分析_*.xlsx 有 18/时段行，但本演练未细拆早/晚峰 | 高峰存在收银排队（首诊卡 C 级），早/晚峰未定 | 待细拆 |
| ③ 500m 竞对锚（菜场/折扣→S1；高端盘→S4） | **否**——门店基础档案竞品栏全"待确认"，需现场调研 | 缺 | — |

> **归型结论**：3 判据仅 ①/③ 缺、② 未细拆 → **无法用架构规定的"≥2判据同型"闭合**。当前 S2 主型是靠客单/生鲜结构/客群/环境等**辅助信号 inferred**，而非架构主判据。**这正是破绽②**：架构声称 strategy_type 判别"成本≈0、POS已有"，实测 2/3 判据非纯 POS（一条现场调研 + 一条促销日历）。归型闭合待补：促销日历 + 500m 竞对现场表。

### 2.1 数据可行性门（Pre-flight Gate）实跑——花厅坊缺小票/损耗→哪些降级

| 标准/指标 | 门判定 | 触发原因（本店实际数据清单） |
|---|---|---|
| 全店/品类坪效 | 🟢 ACTIVE | 面积 1500㎡ 已校准 + POS 品类销售齐 |
| 客单价 / 生鲜占比 | 🟢 ACTIVE | POS 笔均 + L1 分类齐 |
| ABC/尾部结构 | 🟢 ACTIVE | SKU 级 POS 齐（滞销7139/挤压3623 可算） |
| 综合毛利 / GMROI | 🟡 DEGRADED | 生鲜真进价可得，但标品周转/综合毛利未取数 |
| 损耗率 | 🔴 LOCKED | 无日清/损耗数据 |
| 篮分析/连带率/RFM | 🔴 LOCKED | **无小票行级数据**（客流是另一计算路径，暂用客单数代替；无会员渗透） |
| 场景A 跨店横比(PSD) | 🔴 LOCKED | 仅花厅坊一店数据在手 |

> **验证结论**：数据门与店型解耦（架构 §3.5）在花厅坊成立——LOCKED 的三项（损耗/小票/跨店）都是 per-store 运行时事实，与"S2"这个店型属性无关。**架构 §3.5 的正交设计在真店上兑现。**

## 4. 架构在真店上暴露的破绽（本演练核心产出）

**破绽①（架构内部自相矛盾·最严重）**：§5.3 表格把 `chain_stage=单店(N=1)` 与 `场景A LOCKED` 直接画等号，**违反 §3.5 自己确立的"数据门与 chain_stage 解耦"原则**。花厅坊实为乐易购连锁的一店（集团战略层存在），chain_stage 按"客户门店数"取值应 >单店；场景A LOCKED 的真实原因是**姐妹店数据未在手**（运行时数据事实），不是"chain_stage=单店"（结构定义）。二者被 §5 错误绑定。
→ **修订建议**：§5.3 表格"chain_stage: 单店(N=1，场景A LOCKED)"改为"chain_stage: 连锁但在手样本 N=1 → 场景A LOCKED（数据门触发，与 chain_stage 定义无关）"。这恰是架构 §3.5 正交设计的最佳注脚，却被 §5 自己踩了。

**破绽②（判别成本被低估）**：§1.2/§6.3 称 strategy_type 判别"3条POS判据、成本≈0、POS已有"。实测仅"时段结构"是纯 POS；"促销日客流比"需促销日历+客流拆分（花厅坊 POS 未导出）；"500m竞对锚"是现场调研（门店档案竞品全"待确认"）。**1/3 判据数据现成**。花厅坊在缺 2/3 判据下无法单义闭合归型，只能靠辅助信号 inferred S2。
→ **修订建议**：§1.2 A2 取值方式改为"1条POS(时段) + 1条促销日历 + 1条现场调研"；§6.3 第1步删去"成本≈0"，改为"需补促销日历+500m竞对现场表方可闭合"。

**破绽③（坪效分母未定义）**：架构 §1.2 说坪效分母="卖场净陈列㎡"，但 §2.3 profile 没锁死口径。花厅坊有三个面积口径：合同 1800㎡ / 实用销售区 1500㎡ / 仓储 500㎡。同样月销 2,479,172 元，用 1800㎡→¥1,378、用 1500㎡→¥1,653，差 20%。跨店坪效不可比的根因就在这。
→ **修订建议**：把"坪效分母=实用销售区㎡（排除仓储/后场）"提升为 **L-INV 标准化工序的强制定义**（不是 L-PAR 参数），并要求所有 profile 的坪效字段附面积口径标注。这是架构漏掉的一个 L-INV 常量。

**破绽④（生鲜占比对 S1/S2 无判别力·佐证架构逐出它的决策）**：架构 §7 假设②把生鲜占比逐出路由输入（因循环论证）——**方向对**。但花厅坊进一步暴露：花厅坊（S2·大湾区）生鲜占比 67.6% 恰好落在架构给三线 S1 的 [62,75] 区间内。说明生鲜占比在生鲜主导业态里，S1 和 S2 高度重叠、几乎无区分力。
→ **含义**：这加固了"逐出生鲜占比"的正确性，但也提示一个风险——strategy_type 的判别**必须**靠客单/促销弹性/竞对（恰是数据最缺的三项，见破绽②），而这些一旦缺失，S1/S2 在生鲜社区超市里可能本就难分。开放问题：架构的 S1–S4 四分，对生鲜主导业态是否过细？留给第2/3轮。

**破绽⑤（region_tier 缩放未被压测）**：花厅坊正好在基准位（大湾区），本演练只验证了 A1 的"取值"顺畅（广州→一线），**完全没压测 §3.4 fallback 用的"region_tier 客单缩放因子"**。缩放机制要等好家源三线才能验证。
→ 诚实标注：本 N=1 演练对 A1 的验证是不完整的（取值✓、缩放✗）。

**验证为正的部分（架构站得住的地方）**：
- 4 轴能装下花厅坊，坐标点唯一（除 A2 副型待定）；
- fact_layer 三态在一份 profile 内并存（§2.3 决策正确，禁止一刀切被兑现）；
- 数据门与店型解耦（§3.5）在花厅坊成立（LOCKED 三项都与"S2"无关）；
- 惰性实例化（§1.4）落地——花厅坊只 populate 一个点，未强行铺满笛卡尔积。
