# 晟果新零售陈列资源命名治理规范 v1.1

## 1. 文件定位

本文件是晟果新零售内部通用命名标准，用于统一管理不同门店中的陈列资源、照片、货架、端架、通道、堆头、商品与陈列位置映射。

适用对象：
- 社区超市
- Shop 店
- 市场店
- 标超店
- 折扣店
- 客户样板门店
- 后续所有咨询项目门店

本文件解决：
1. 陈列位如何命名
2. 货架如何命名
3. 通道如何命名
4. 端架如何命名
5. 堆头如何命名
6. 照片如何命名
7. 商品如何与陈列资源绑定
8. 每家门店如何做适配
9. 命名变更如何审计

本文件不解决：
1. 某家门店具体有哪些货架
2. 某个 SKU 是否淘汰
3. 某个品类如何调改
4. 某个门店的具体 DisplayMap
5. 具体业务复盘结论

⚠️ **本文件来源**：基于花厅坊样板门店 [[2026-05-23_花厅坊休食区陈列资源命名与照片入库规范_v1.0]] 实践，抽象为公司级标准。花厅坊命名规范不再代表公司通用结论（参 §20）。

## 2. 命名治理总原则

1. **通用标准先行，门店适配落地**：公司层定方法，门店层定取值
2. **物理位置优先于品类名称**：货架编号锁定物理 / 品类是当下取值
3. **编号稳定优先于口语名称**：口语名作 alias 不作主键
4. **一物一号，不重复、不混用**：跨门店同号必须靠门店编码区分
5. **照片命名必须服务后续检索、复盘和品效分析**：不可为了简短牺牲可解析
6. **原始素材不覆盖，升级命名走 rename map**：永远保留原始 audit trail
7. **门店特例必须登记，不得污染公司级标准**：特例 → 登记表 / 跨 3 家以上反复出现才晋升标准候选
8. **命名规则必须可被员工理解、顾问执行、系统解析**：三视角同时通过才算成立
9. **所有推断必须标注，不把推断写成事实**：fact_layer 强制（参 [[CLAUDE.md]] §13.17）
10. **命名最终目标是建立"商品 × 陈列资源 × 时间 × 经营结果"的数据底座**：不只为整理照片

## 3. 四层命名体系

### 3.1 公司通用标准层

定义字段、编码、公式、治理规则。**本文件即此层**。

### 3.2 门店适配层

每家门店定义：
- 门店编码
- 区域编码启用表
- 货架编码
- 通道编码
- 端架编码
- 堆头编码
- 特殊陈列资源编码

每家新门店必须建立独立适配表（参 §16）。

### 3.3 执行采集层

现场照片、视频、手工记录、商品陈列表、员工反馈。
- 现场快采（速度优先 / 员工 / 店长 / 顾问均可）
- 入库升级（顾问按公司标准 + 门店适配表升级）

### 3.4 数据应用层

用于：
- 商品 × 陈列资源映射
- SKU 动销分析
- 排面效率分析
- 端架效果分析
- 调改前后对比
- 复盘报告
- 后续系统化工具开发（小程序 / PC 后台 / 数据库）

## 4. 通用编码对象

公司级通用代号（不等于每家门店都必须使用 / 门店没有该资源则不启用）：

| 对象 | 通用代号 | 说明 |
|---|---|---|
| 门店 | STORE | 门店唯一编码 |
| 区域 | Z | Zone / 物理区域 |
| 货架排 | G | Gondola / Grid / 主货架 |
| 货架面 | A/B/C | 货架正反面或多面 |
| 子货架 | NN | 横向第几组（强制 2 位）|
| 层位 | LAYER | 纵向层数 |
| 排面位 | POS | 从左到右第几个排面位 |
| 通道 | Aisle / A | 货架之间或区域之间通道 |
| 端架 | T | Endcap / 端架 |
| 主堆 | D | Display / 主堆 |
| 侧堆 | SD | Side Display |
| 柱旁堆 | PD | Pillar Display |
| 促销台 | PT | Promo Table |
| 冷柜 | COLD | 冷藏 / 冷冻设备 |
| 收银端 | CASH | 收银区陈列 |
| 异型陈列 | X | 非标准陈列 |

## 5. 门店编码规则

### 5.1 公式

```
<客户简称>-<门店简称>
```

### 5.2 示例

| 编码 | 含义 |
|---|---|
| `LYG-HTF` | 乐易购花厅坊店 |
| `LYG-SPDD` | 乐易购沙埔大道店 |

### 5.3 要求

1. 客户简称统一大写拼音或英文缩写
2. 门店简称必须唯一
3. 不用口语别名作为标准编码
4. 门店编码一旦 lock，不得随意变更
5. 如门店有历史别名 / 登记在门店适配表 / 不进入主编码

## 6. 区域编码规则

### 6.1 公司级通用区域代码建议（v1.1 / 2026-05-19 升级）

| 区域 | 通用代码 | 说明 |
|---|---|---|
| 休食区 | **XS** | 休闲食品 / 方便速食 / 冲调等 / 可按门店定义（v1.1 / 原 HS）|
| 生鲜区 | SX | 蔬果 / 肉禽 / 水产等 |
| 酒水区 | JS | 酒 / 饮料 / 水 |
| 调味区 | TW | 调味品 |
| 米面粮油 | **LY** | 米面粮油（v1.1 / 原 MLY 简化）|
| 日化家清 | RH | 日化 / 家清 / 洗护 |
| 收银区 | SY | 收银台 / 收银端架 |
| 主通道 | TD | 主通道 |
| 入口区 | RK | 入口 |
| 社区服务区 | SQ | 社区服务 |
| 冷柜区 | LG | 冷藏 / 冷冻柜 |
| 特陈区 | TC | 特殊陈列 / 临时陈列 |

### 6.2 要求

1. 区域编码为公司**建议标准**
2. 每家门店可启用部分区域
3. 不得让某家门店的区域特殊叫法污染公司标准
4. 门店适配表中可以写"本店实际叫法"作为 alias

### 6.3 ⭐ 历史代号 alias 映射表（v1.1 / 2026-05-19 加 / deprecated_alias）

| 旧代号 | 新代号 | 含义 | 状态 | 使用规则 |
|:---:|:---:|---|:---:|---|
| **HS** | **XS** | 休食区 | deprecated_alias | 历史文件可读 / **新文件禁用** |
| **MLY** | **LY** | 米面粮油 | deprecated_alias | 历史文件可读 / **新文件禁用** |

**变更原因**：
- HS → XS：HS 容易与"货架 / House / 其他缩写"混淆 / XS 更贴近"休食"拼音首字母
- MLY → LY：简化区域代号 / 避免过长编码 / 米面粮油可在名称说明中保留完整含义

**强制规则**：
- ✅ 历史文件中出现 HS / MLY → 按本表 alias 理解 / 不批量回改
- ❌ 5/19（含）起所有新文件 / 新照片 / 必须使用 XS / LY / 不得继续用 HS / MLY
- 完整变更治理见 [[2026-05-19_区域代号HS到XS_MLY到LY变更审计_v0.1]]

## 7. 货架命名公式

### 7.1 通用公式（含门店编码）

```
<STORE>-<Z>-G<排号>-<面>-<子货架>
```

### 7.2 简化公式（门店内 / 上下文已确定 STORE + Z）

```
<Z>-G<排号>-<面>-<子货架>
```

### 7.3 示例

```
LYG-HTF-XS-G1-A-01
XS-G1-A-01
```

### 7.4 字段解释

- `STORE` = 门店编码
- `Z` = 区域编码
- `G1` = 第 1 排货架
- `A` = A 面
- `01` = 从左到右第 1 个子货架

### 7.5 规则

1. 货架排号从顾客主入口或门店适配表定义的起点开始
2. A/B 面必须在门店适配表中定义朝向
3. 子货架编号强制两位数（前导零）
4. 不允许写成 `G1_A` / `G1A` / `G-1-A` 等混用
5. 门店内部可快采简写 / 入库必须升级为标准格式

---

## 8. 层位命名规则

### 8.1 通用公式

```
<货架位>-L<层号>
```

### 8.2 示例

```
XS-G1-A-01-L01
XS-G4-A-06-L03
```

### 8.3 层号规则

1. 默认从上到下编号
2. `L01` = 顶层
3. 最后一层 = 底层
4. 如果门店习惯从下往上 / 必须在门店适配表中明确 / 不得混用
5. 挂钩层使用 `H01`/`H02` 或 `HOOK01`/`HOOK02` / 在门店适配表中定义

### 8.4 商业层位语义

| 层位语义 | 说明 |
|---|---|
| 顶层 | 形象展示 / 轻货 |
| 黄金视线 | 顾客视线层 |
| 黄金手拿 | 最容易拿取层 |
| 中层 | 常规动销层 |
| 底层 | 大包装 / 重货 |
| 围货 | 库存补充 |
| 挂钩 | 挂袋 / 小件 |

⚠️ **层号是物理事实**（不可改）/ **层位语义是商业解释**（可随调改变化）。两者都可记录 / 但不能混淆。

## 9. 排面位命名规则

### 9.1 公式

```
<货架位>-L<层号>-P<排面位>
```

### 9.2 示例

```
XS-G1-A-01-L03-P01
XS-G1-A-01-L03-P02
XS-G1-A-01-L03-P01-03   # 同 SKU 占 3 个排面位（01 起 03 止）
```

### 9.3 规则

1. 排面位从左到右
2. 两位数编号（前导零）
3. 重点 SKU 必须记录排面位
4. 普通照片可不记录到排面位
5. SKU × 陈列位映射必须记录到排面位或至少记录到层位

## 10. 通道命名规则

### 10.1 公式

```
<STORE>-<Z>-A<NN>
<Z>-A<NN>
```

### 10.2 示例

```
LYG-HTF-XS-A01
XS-A01
```

### 10.3 通道定义

通道是**两个陈列资源之间的顾客移动空间** / 不等于货架本身。

### 10.4 命名规则

1. `A01` / `A02` / `A03` 从门店适配表定义的起点开始
2. 通道两侧必须记录
3. 通道不得用口语名称作为唯一标准
4. 口语名称可作为 alias
5. 通道拍照必须记录机位

### 10.5 通道适配表字段

| 通道编号 | 左侧资源 | 右侧资源 | 口语名称 | 主动线方向 | 是否重点通道 |
|---|---|---|---|---|---|

---

## 11. 通道机位命名规则

### 11.1 公式

```
<通道编号>-M<NN>
```

### 11.2 示例

```
XS-A03-M01
XS-A03-M02
XS-A03-M03
```

### 11.3 建议

- `M01` = 通道入口端
- `M02` = 通道中段
- `M03` = 通道出口端
- 如通道较长 / 可扩展 `M04`/`M05`
- 同机位复拍必须使用同一 M 编号（用于调改前后真实可比）

## 12. 端架命名规则

### 12.1 通用公式

```
<STORE>-<Z>-T<NN>-<关联货架><首/次/异>
<Z>-T<NN>-<关联货架><首/次/异>
```

### 12.2 示例

```
LYG-HTF-XS-T06-G4首
XS-T06-G4首
```

### 12.3 规则

1. `T` = 端架
2. `NN` 为两位数
3. 每家门店端架编号在门店适配表中 lock
4. 端架必须绑定关联货架或标注为异型
5. 不同区域可以有相同端架编号 / 但完整编码不冲突（如 `XS-T01` ≠ `RH-T01`）
6. 端架商品变动必须拍照留证

### 12.4 端架照片命名（C 方案 / 沿用花厅坊样板）

```
<P>-T<NN>-<关联G>[首/次/异][-L品类]_<内容>_<YYYYMMDD>_<时段>.jpg
```

## 13. 主堆 / 侧堆 / 柱旁堆命名规则

### 13.1 公式

| 类型 | 公式 |
|---|---|
| 主堆 | `<Z>-D<NN>` |
| 侧堆 | `<Z>-SD<NN>` |
| 柱旁堆 | `<Z>-PD<NN>` |

### 13.2 示例

```
XS-D01
XS-SD01
XS-PD01
```

### 13.3 要求

1. 临时堆头也必须编号
2. 起止日期必须记录
3. 促销商品必须绑定 SKU 或商品组
4. 异型堆不得混入端架编号
5. 主堆 / 侧堆 / 柱旁堆数量需现场盘点后由门店适配表锁定 / 不预编公司级编号

## 14. 照片命名规则

### 14.1 现场快采命名（速度优先 / 员工 / 顾问均可）

```
<门店>_<区域>_<陈列资源编号>_<视角>_<序号>.jpg
```

示例：
```
花厅坊_休食区_G4-A_正面全景_01.jpg
花厅坊_休食区_XS-T06_端架全景_01.jpg
```

特点：
- 速度优先 / 不要求现场填层位 / L 品类 / 时段
- 不得使用错误词（如休食区误写"洗护"）
- 必须包含资源编号

### 14.2 入库标准命名（顾问升级）

```
<STORE>-<Z>-<陈列资源位>[-L品类]_<内容>_<YYYYMMDD>_<时段>[_<序号>].jpg
```

示例：
```
LYG-HTF-XS-G4-A-01-L03-L冲调_商品价签_20260519_常规.jpg
LYG-HTF-XS-T06-G4首-L方便速食_全景_20260519_常规.jpg
LYG-HTF-XS-A03-M01_通道全景_20260520_常规.jpg
```

### 14.3 字段表

1. STORE
2. Z
3. 陈列资源位
4. L 品类（可选 / 建议）
5. 内容
6. 日期
7. 时段

### 14.4 时段字段

| 时段 | 说明 |
|---|---|
| 常规 | 任意 / 非调改场景的随手拍 |
| 夜灯前 | 22:00 调改前 / 暖光 |
| 夜灯后 | 02:00 调改完成 / 暖光 |
| 次晨白光 | 09:00 次晨 / 自然光 / 24h 同机位真对比 |
| 24h复拍 | 24h 后复拍 |
| 7d复拍 | 7 天后复拍 |
| 15d复拍 | 15 天后复拍 |
| 30d复拍 | 30 天后复拍 |

### 14.5 内容字段

| 类别 | 字段 |
|---|---|
| 全景 | 全景 / 正面全景 / 左45度 / 右45度 |
| 局部 | 上半区 / 下半区 / 局部 |
| 价签 | 商品价签 / 价签错 |
| 异常 | 异常点 / 缺货 / 围货异常 / 坏品 |
| 调改 | 调改前 / 调改后 |
| 通道 | 通道全景 / 动线观察 |
| 端架 / 堆头 | 端架全景 / 堆头全景 |

## 16. 门店适配表规则

### 16.1 命名公约

```
<日期>_<门店名>_陈列资源命名适配表_v1.0.md
```

## 17. 特例治理规则

### 17.1 特例登记表字段

| 特例编号 | 门店 | 问题 | 标准规则 | 本店特例 | 是否临时 | 是否进入公司标准 |
|---|---|---|---|---|---|---|

### 17.2 特例分三类

1. 门店布局特例（如 G2 仅 1 个端架 / 无次端架）
2. 历史命名遗留（如 v0.3 "G1-G6" 实际是 G1-G4）
3. 业务临时活动（如临时堆头 / 节令陈列）

### 17.3 晋升公司标准的硬约束

只有跨 **3 家以上门店**反复出现的特例 / 才允许进入公司级标准候选。
单店特例 / 双店特例 → 仅登记 / 不污染公司标准。

## 18. rename map 与审计规则

### 18.1 触发场景

凡照片或文件**批量改名** / 必须先生成 rename map。

### 18.2 字段

| 原文件名 | 新文件名 | 修改原因 | 依据 | 是否确认 | 执行时间 | 操作人 |
|---|---|---|---|---|---|---|

### 18.3 硬约束

1. 不覆盖原始素材
2. 不直接批量改名
3. 先 audit / 再 rename map / 再用户确认 / 再执行
4. 执行后生成 rename log
5. 原始文件保留或备份（永久 / 不删）

## 19. 命名成熟度分级

| 等级 | 状态 | 说明 |
|:---:|---|---|
| L0 | 未命名 | 只有照片 / 口语描述 |
| L1 | 快采命名 | 现场临时命名 |
| L2 | 标准命名 | 符合公司命名规则 |
| L3 | 映射完成 | 商品 × 陈列位绑定 |
| L4 | 数据可分析 | 可做销售 / 库存 / 品效分析 |
| L5 | 系统化 | 可进入工具 / 小程序 / 数据库 |

⚠️ 同一家门店不同区域可处于不同等级 / 整店等级 = 各区最低值。

## 20. 从花厅坊抽象出的公司级结论

### 20.1 花厅坊提供的样板经验（公司级可复用）

- 货架 G + 数字 + A/B 面 + 子货架 2 位编号
- 子货架从左到右 / 强制 2 位前导零
- 层位 1=顶 / 末=底
- 端架 T + 数字 + 关联货架 + 首/次/异
- 通道 A + 数字 / 通道机位 M + 数字
- 主堆 D / 侧堆 SD / 柱旁堆 PD
- 照片快采 → 入库升级双层机制
- rename map + audit trail
- 5 步入库 SOP

### 20.2 花厅坊特例（**不进入公司通用固定值**）

- ❌ XS 一定等于 G1-G4（公司标准 = G + 数字 / 数量按门店定）
- ❌ 每家店都有 8 个端架（公司标准 = T + 数字 / 数量按门店定）
- ❌ 每组货架一定 6 个子货架（公司标准 = 子货架 2 位 / 数量按门店定）
- ❌ 每家店都有 G4-A 冲调（品类 = 当下取值 / 不进编码）
- ❌ 每家店都用同样通道数量（公司标准 = 通道 A + 数字 / 数量按门店定）
- ❌ 花厅坊的 T06/T07 端架关系（公司标准 = 关联货架字段 / 取值按门店定）

### 20.3 公司级标准只保留方法

- 怎么编号
- 怎么适配
- 怎么审计
- 怎么映射
- 怎么扩展
