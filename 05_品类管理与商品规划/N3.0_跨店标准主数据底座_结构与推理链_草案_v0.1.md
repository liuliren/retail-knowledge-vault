# N3.0 跨店标准主数据底座 · 结构与推理链（草案 v0.1）

> **结论先行**：把现在散在 N1/N2/N2.5/2.6 的脚本，收口成**一个产品**——"任意门店 POS 原始导出 → 一张归一到 V6.0 标准（SSOT·DEC-20260702-01）的商品主数据表 + 一份可发客户的诊断"。可收费单元 = **每店一次标准化主数据交付**。机器做 90%（清洗/归一/识别/诊断派生），六哥签 10%（角色裁定/复杂类 L4/对赌背书）。本草案给**数据结构 + 推理链 + 分阶段路线**，需六哥在 §9 逐点签字后才动工。**不在花厅坊交付窗口内抢工，按"作交付副产品长出来"推进。**

---

## 1. 这是什么 / 卖什么（产品定位）

- **当前状态**：N2.6 已把"类目归一到单一本体（2026-07-02 起为 V6.0·DEC-20260702-01）+ store-agnostic 路由"做成（`category_mapper.infer_l3` 四层路由 + `STORE_OVERRIDES` 钩子）。这是底座的**第一块能力（Normalizer）**。
- **产品化目标**：门店制 = 多店重复跑同一条流水线。可收费单元有三档（由浅到深）：
  1. **标准主数据交付**（最薄）：客户给 POS 导出 → 产出归一到 V6.0 的商品主数据表（含 ABC/动销/呆滞/价格带）。机器活为主。
  2. **单店品类诊断**（中）：在 1 之上叠 7D/品类力诊断卡（已有 `单品类诊断` skill 雏形）。
  3. **跨店对标 / 连锁主数据治理**（最厚，护城河最深）：N 家店同一本体 → 横向对标、选品复用、汰换基准库。**只有单本体成立才可能。**
- **90/10 落点**：机器 = 清洗/归一/识别/诊断派生/草表；六哥 = 角色裁定、复杂类 L4 分层、对赌式签字、跨店长期跟踪。**判断相不下放（治理 §7 模型分层调度红线）。**

## 2. 数据结构（分层 + 跨店维度表）

### 2.1 五层数据流（与 §6 权限层对齐）
| 层 | 内容 | 权限 | 状态 |
|---|---|---|---|
| **L0 raw** | 各店 POS 原始导出（xls/csv） | D `raw_sensitive`，只读冻结、不入 git | 已有（99_原始素材）|
| **L1 normalized** | adapter 归一字段（条码/品名/类目/价/量/库存…标准 schema） | C `client_confidential` | 已有（N1 `retail_clean`）|
| **L2 routed** | L3/L4 类目 + 品牌 + 规格，**每字段带 value+confidence+method** | C | 部分（N2.6 类目单本体 ✓；品牌/规格待加 confidence 列）|
| **L3 enriched** | ABC / DOS / 呆滞 / 毛利(仅参考) / 角色弱建议 | C | 已有（N2 `build_master`）|
| **L4 deliverable** | 脱敏一页诊断（0 条码 0 进价） | E `client_shareable_summary` | 部分（report-export skill）|

### 2.2 三张跨店主数据维度表（SSOT · 跨店共用，复利资产）
1. **类目标准表 `dim_category`** = V6.0 68 个 L3 / 385 个 L4（L1/L2/L3/L4决策组 + 编码；勘误：原文「V4.0 273 个 L3」为笔误——V4.0 实为 63 L3/308 L4，本体已依 DEC-20260702-01 切至 V6.0）。**唯一本体**，所有店映射到它。已有源：《完整品类表 L4拆分版》。
2. **品牌主数据 `dim_brand`** = canonical + aliases + 国别 + 母公司（现 `brand_alias.csv`，已扩到 ~104 条）。
3. **映射知识库 `map_raw_to_std`** = 跨店"报表类别名→标准 L3"通用基座（`RAW_CAT_TO_L3`）+ 每店覆盖（`STORE_OVERRIDES`）+ 二级规则（`SECONDARY_RULES`，如 调味→基础/复合/佐餐）。**这是越跑越全的核心资产（M-DEC 回路）。**

> 铁律：跨店维度表是**复利资产**，每新增一店的人工校准都回写进它，下一店命中率更高（边际人工递减）。客户裸值永不进维度表（只进各店 C 层 CSV，gitignore）。

## 3. 推理链（pipeline 七段 · 每段可观测）

```
① Ingest      L0 原始导出 → 冻结登记（store_id, period, 来源文件清单）
② Clean       N1 retail_clean：读表/清洗/跨文件 join → L1 标准 schema（带 drop_log）
③ Normalize   L1 → L2：每字段独立路由，输出 value + confidence + method + conflict_flags
                ├─ 类目  : infer_l3(name, src_cat, store_id)  四层路由（已实现）
                ├─ 品牌  : SKUExtractor + dim_brand 前缀匹配（confidence 已在算法内，待落列）
                └─ 规格  : SpecExtractor（confidence 已在算法内）
④ Enrich      L2 → L3：ABC / DOS / 呆滞 / 毛利(仅参考) / role_weak
⑤ Calibrate   生成"待人工校准队列"，按 ABC × (1-confidence) 排序（高价值 × 低置信 优先）
⑥ Signoff     六哥只看队列头部：角色裁定 / 复杂类 L4 / 低置信品牌类目 → 签字（10%）
⑦ Feedback    签字结果回写 dim_brand / map_raw_to_std（M-DEC：越跑越全）→ 回 ①下一店
```

**关键升级点（相对现在）**：
- **③ 每字段三件套 value+confidence+method**：现在类目有 `method` 列、品牌有 `brand_confidence`，但未统一成"每字段都带置信度"。统一后 ⑤ 才能按置信度排校准队列。
- **⑤ 校准队列 = coverage→confidence 的落地**：不再"全集都标待校准"，而是**只把高价值×低置信的推到六哥面前**，其余自动放行。这是把人工 10% 真正压到 10% 的机制。

## 4. store-agnostic 机制（新店接入流程）

新店接入 = **零改基座**，只做三件：
1. 写一个 adapter（或复用）把该店 POS 列名映射到 L1 标准 schema。
2. 跑流水线 → 看 `map_raw_to_std` 命中率；未命中的类别名进 `STORE_OVERRIDES[store_id]`（仅"同名异类"）或补进通用基座（更常见的新类别名）。
3. 校准队列头部由六哥签字 → 回流维度表。

> 命中率随店数上升（复利）：花厅坊把通用基座从 0 喂到了 ~92 条精确映射 + 调味二级规则，下一家社区生鲜超市大概率直接复用 70%+。

## 5. 三能力分阶段（拒绝顾问"冲突消解器优先"的错配）

| 阶段 | 能力 | 为什么这个顺序 | 触发条件 |
|---|---|---|---|
| **N2.6（已做）** | **Category Normalizer** | 门店制命脉：不归一无法跨店对标 | 已完成 |
| **N3.0 next** | **品牌×类目一致性校验** | 成本低、立刻提质：蒙牛→乳品 OK，卫龙→乳品则 flag。给品牌打"类目先验"，纠偏错挂 | 跑第 2 家店、出现跨类品牌噪声时 |
| **N3.1 later** | **Conflict Resolver（置信度仲裁）** | 仅当"源类目 vs 品名 vs 关键词"高频冲突时才值得；现在顺序决策已够用 | 冲突率实测 >某阈值才做，不预建 |

> 顶你那位顾问一句：冲突消解器是"系统精致度"，对**单店**有用、对**跨店复制**无杠杆；先做归一与品牌校验，把跨店底座铺开，冲突仲裁等数据说话再上。**不在没数据时预建抽象（少而深）。**

---

## 6. M-DEC 回流（越跑越全的飞轮）

每次六哥签字 = 一条 decision：现象（机器低置信/冲突）→ 决策（归哪类/哪品牌）→ 回写维度表。积累后：
- `map_raw_to_std` / `dim_brand` 持续增厚 → 新店人工量递减；
- 沉淀出"社区生鲜超市标准品类×角色基线库"，成为**比 SOP 更难被 AI 追平的资产**（真实多店裁决数据）。

---

## 7. 红线（不可违）

- **脱敏**：维度表与对外件 0 条码 / 0 进价 / 0 供应商；裸值只在各店 C 层 CSV（gitignore）。
- **D 层只读**：L0 原始数据冻结、不清洗回写、不入 git（治理 §5.3 / §6 D）。
- **判断相不下放**：角色裁定 / 复杂类 L4 / 对赌签字只有六哥（§7 模型分层调度红线）。
- **可解释=可负责**：每个机器结论都带 method + confidence，讲不清的不放行（§10.1.5）。

---

## 8. 不做什么（防膨胀）

- 不现在建全功能"决策引擎 UI / SaaS 前端"——先把**命令行流水线 + 维度表**跑顺，N≥3 店验证复用率后再谈产品壳。
- 不预建冲突仲裁、不预建归一化压缩算法——等数据暴露真问题（少而深，§8）。
