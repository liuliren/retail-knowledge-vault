## 一、批判性审视：吸收 ChatGPT 方案前的诚实对话

> **原则**：吸收有价值的内核，批判执行层的根本缺陷。不做改良版的 ChatGPT——做从实战出发的晟果版。

### 1.1 值得真正吸收的三点

**① 三系统解耦**（结构层 / 经营层 / 行为层分离）  
这是正确的第一性原理。品类树（L1–L4）负责"是什么"；经营标签（角色/毛利带/价格层级）负责"怎么卖"；陈列行为（场景/动线/促销）负责"在哪卖"。三系统不得混入同一维度。这一点晟果已在 v4.x 反复验证。

**② 冲突显式化**  
当 SKU 存在多归属路径时，必须显式标注、裁决，不允许静默归类。这不是学术洁癖——是防止系统腐烂的纪律。晟果历史上"豆浆到底算乳品还是冲调""蛋黄派算饼干还是糕点"，如果没有裁决记录，每次导入都会重新混乱。

**③ 稳定性约束**  
"结构变更须满足 SKU 映射不重构率 > 90%"——这是品类树冻结策略的量化表达。体系频繁变动等于历史数据断档，等于分析失效。晟果的"冻结底稿"哲学与此一致。

---

### 1.2 四条本质缺陷（不是修正——是重写）

**缺陷 ①：L6（SKU）不应进品类树**

ChatGPT 把 SKU 定义为 L6 并放入品类树。这是概念混淆：品类树是**商品的坐标系**，不是容器。正确关系是 SKU 表有外键指向品类树节点，而不是品类树包含 SKU。把 SKU 放进树，后果是：新增/下架任何商品都要"修改品类树"，系统永远不稳定。

> 晟果实践：SKU 表独立，`category_code` 字段指向品类树节点编码。品类树冻结，SKU 随业务增减。

**缺陷 ②：L1 定义为"损益科组"在中国社区超市不可执行**

大卖场（沃尔玛/大润发）有分品类核算 P&L 的能力，社区超市通常没有。把无法落地的概念放在 L1，整套体系在门店层会直接失效。L1 的正确定义是**经营管理边界**（谁负责这个区域的采购、陈列、损耗、补货），而不是财务核算单位。

**缺陷 ③：L5（消费决策子类）是前端逻辑，混入后端树会失稳**

口味/规格/形态是消费者选择维度，应该属于**前端陈列分类**，而不是采购管理分类。一旦放入后端树，每次加新规格（如"低糖款""家庭装"）就要改树，带来巨大的主数据维护成本。

> **这正是本文设计两张表的核心原因**：前端（消费者决策逻辑）和后端（经营管理逻辑）本来就不应该是同一棵树。

**缺陷 ④：过度工程化，可执行性为零**

`score = rule_match + context_match + frequency_weight` 在论文里很漂亮，在社区超市门店执行层没有任何意义。社区超市的采购员需要一张表、5 条判断规则、20 分钟能上手。一套 95 分但没人用的系统，比不上一套 80 分但全店执行的系统。

---

### 1.3 我的核心判断：两张表 > 一张表

**品类管理从来不是一张表能解决的问题。**

所有行业最佳实践（7-Eleven 商品体系、永旺 CDT 分类、胖东来品类组织）都是双层结构：
- **后端经营表**：服务采购员、品类经理、财务、老板——按商品经营逻辑组织（采购频率、毛利带、品类角色、启用阈值）
- **前端消费表**：服务店长、陈列员、顾客——按购物任务与动线逻辑组织（区域名称、温区、购物任务）
- **连接层**：确保两张表在 L2 层面可以互通对照

三张表的设计原则完全不同，不可混为一谈。国内大多数品类系统失效的根本原因是：用一张表同时服务两种人，结果两边都服务不好。

---

### 1.4 晟果独有的设计约束（国际方案没有的）

1. **大湾区特化**：叶菜高频、烧腊/腌腊季节性强、煲汤滋补刚需、活鲜水产是核心竞争力——这些都影响 L1 权重和品类角色定义
2. **3+1 层架构**：L1/L2/L3 为稳定主干，L4（决策组）仅在高复杂度类启用——避免过度分层
3. **温区优先**：生鲜/冷链/冷冻/常温/烟草是门店管理的物理约束，必须在前端表中明确
4. **启用阈值量化**：条件类目必须有可量化的启用条件（SKU 数/销售占比/陈列空间），不能靠感觉决策

> **用途**：采购决策、品类角色分配、KPI 设定、SKU 数目标、主数据挂载  
> **更新频率**：半年一次（结构稳定性原则）

> **说明**：品类角色缩写：D=目标品(Destination) / P=偏好品(Preferred) / R=日常品(Routine) / S=季节品(Seasonal) / C=便利品(Convenience)；毛利带：低<15% / 中15-30% / 高>30%

## 三、表二：前端品类消费展示表（门店陈列/店长/顾客导向）

> **用途**：门店陈列规划、货架标签、购物动线设计、区域名称命名  
> **更新频率**：季节节点前调整（节庆模板切换）  
> **主责**：门店店长 / 陈列员  
> **核心原则**：区域名称必须是顾客能理解的日常语言，不是后端管理术语

---

### 前端区域组织原则

1. **温区优先**：顾客路径设计以温区（常温→冷藏→冷冻→活鲜）为物理约束，不可跨温区拼陈列
2. **购物任务驱动**：每个区域的命名逻辑是"顾客来这里是为了解决什么问题"
3. **动线逻辑**：生鲜区在门店主动线入口（引流）→日用消耗品在中间通道（保障）→冲动购买品在收银台前（提客单）
4. **关联陈列**：煲汤料旁边配汤料干货；火锅丸滑旁边配蘸料调味

### 工具 A：品类宽度诊断 · 5态判定法

**来源**：`06_历史归档内品类资产/02_品类宽度诊断方法论`  
**用途**：挂码完成后，判断各 L3/L4 的 SKU 广度是否合理，输出"调改清单"，将"凭眼看"变为"凭数据看"

**5态判定逻辑**

| 状态 | 判定条件 | 业务含义 | 行动 |
|---|---|---|---|
| 🔴 **空缺-紧急** | 真实SKU=0 且 建议下限≥8 | 应是品类支柱，完全没卖 | P0：本周内引进 |
| 🟡 **空缺-建议引进** | 真实SKU=0 且 建议下限<8 | 应有但当前空缺 | P1：两周内引进 |
| 🟠 **过窄-建议补充** | 0 < 真实 < 建议下限 | 有卖但SKU不足 | P1：补充至下限 |
| 🟢 **健康** | 建议下限 ≤ 真实 ≤ 建议上限 | 广度合理 | 维持 |
| 🟠 **过剩-建议精简** | 真实 > 建议上限 | SKU过多挤占货架 | P2：精简至上限 |

**大湾区社区超市参考基准（1000–2000㎡）**

| 商品角色 | 建议SKU区间 | 占比 |
|---|---|---|
| 引流款 | 8–14个 | 40–50% |
| 利润款 | 6–10个 | 25–35% |
| 形象款 | 3–5个 | 10–15% |
| 补充款 | 2–4个 | 5–10% |

**大湾区特化基准上调（必须用）**

| 品类 | 上调幅度 | 原因 |
|---|---|---|
| 凉果类（嘉应子/陈皮/话梅/甘草梅/无花果丝） | +30% | 大湾区高频刚需 |
| 即食海苔（岩烧/寿司海苔） | +30% | 港式生活方式 |
| 早餐配粥类（豆豉鲮鱼罐头/咸菜） | +30% | 煲仔粥饭文化 |
| 凉茶/广式特色饮料 | +50% | 夏季高频引流 |
| 煲汤食材包（干货/淮山/红枣/枸杞） | +50% | 煲汤季刚需 |
| 打边炉调味（沙茶酱/腐乳） | +30% | 冬季火锅场景 |
| 腌腊制品（腊肠/腊肉） | +40%（秋冬） | 季节性核心品 |

**大湾区慎用（基准下调或不引进）**

- 中式糕点（月饼除外）
- 北方挂面（饺子皮除外）
- 东北腌渍肉类（东北红肠等）

**双维度联动诊断**：只看 SKU 数广度不够，还要看角色覆盖度

```
维度1：真实SKU数 vs 建议宽度（基础诊断）
维度2：真实SKU的角色覆盖（深度诊断）

典型错位：
- 薯片架上12个SKU（数量健康），但全是中高价进口品 → 数量健康但"引流角色缺失"
- 牛肉干8个SKU（健康），5个中价+3个低价 → 利润款应主导，反而低价货占多 → "结构倒挂"
```

### 工具 B：品类经营 8 步闭环（管理用）

**来源**：`06_历史归档内品类资产/03_品类经营模型`  
**用途**：将三张表从"静态分类"变为"动态经营系统"——品类经营不是一次性分类，而是持续闭环

```
1. 品类定义（从顾客购买逻辑出发）
   ↓
2. 品类角色（D/P/R/S/C，不同角色承担不同任务）
   ↓
3. 品类评估（9个维度：顾客价值/销售/毛利/流量/客单/搭配/周转/场景/战略）
   ↓
4. 品类评估表（量化输出）
   ↓
5. 品类战略（做强/做深/做宽/做稳/精简/替换/季节强化/价格锚点——必须体现取舍，不允许全部写"做强"）
   ↓
6. 品类战术（商品结构战术/价格带战术/陈列战术/促销战术）
   ↓
7. 品类实施（负责人/周期/目标/资源/协同/复盘时间）
   ↓
8. 品类回顾（预设目标是否达成/是否有结构副作用/下轮方向）
```

**类别增长率分群·资源再分配法**（来自零售老刘·M6品类结构簇）

| 分群 | 判定标准 | 资源动作 |
|---|---|---|
| 增长类 | 同期增长 > +20% | 加排面/加订货/引进新SKU |
| 稳定类 | ±10%以内 | 维持现状/精细化运营 |
| 下降类 | 同期下降 < −20% | 缩排面/汰换SKU/重新定角色 |

> **一个商品小类 = 一个顾客需求**（零售老刘根基命题）。对小类不重视 = 对顾客需求不重视 = 战略级问题。

### 工具 C：L4 启用判断 · 5大复杂类模板

**来源**：`06_历史归档内品类资产/02B_重点复杂品类决策稿_v3.2`  
**用途**：决定某个 L3 是否需要启用 L4（决策组），避免过度分层或分层不足

**核心原则**：L4 的启动不是因为"SKU多"，而是因为"顾客已经不把这组商品当成同一种购买路径"

| 模板编号 | 模板名称 | 触发条件 | 典型适用类 |
|---|---|---|---|
| CC-PATH-01 | 路径分叉型 | 顾客按不同购买路径筛选后，再在路径内比较 | 饮料（无糖/含糖）、零食（儿童/成人）、个护（功效路径） |
| CC-BAND-01 | 价格带/规格带分叉型 | 价格带或规格带已经表达完全不同的购买任务 | 饮料、调味品、乳品 |
| CC-SCENE-01 | 场景表达型 | 顾客先按使用场景做第一层判断 | 调味品（家常/火锅/烧烤）、家清（厨房/卫浴/地面） |
| CC-FUNC-01 | 人群/功能分叉型 | 功能或人群差异已主导购买判断 | 洗护（去屑/控油/儿童温和）、乳品（儿童/功能/基础） |
| CC-GIFT-01 | 礼赠/时段强化型 | 礼赠路径与日常路径明显不同 | 坚果炒货礼盒、水果礼盒、酒类礼赠 |

**L4 启用决策流**：

```
1. 这个L3服务什么顾客任务？
   ↓
2. 顾客先拿什么做第一层比较？（路径/价格带/场景/功能/礼赠）
   ↓
3. 这些比较逻辑在同一L3内是否会冲突？
   ↓
4. 如果不拆，分析/规则/陈列建议是否会一起失真？
   ↓
5. 是 → 启用L4，按上述5个模板选择分法
   否 → 不启用L4，经营维度外挂标签即可
```
