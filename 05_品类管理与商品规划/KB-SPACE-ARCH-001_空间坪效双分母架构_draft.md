## 结论先行（读这一段就能决策）

1. **系统命门**：SKU→货位桥表（Planogram）。CAD 给空间，POS 给销售，中间缺「谁在哪个货位摆几个排面」——必须现场手工盘两天，AI 无法代劳。没有它，坪效只能停在品类级，做不到货位级。

2. **双分母是核心公式决策**：不能只用占地投影面积——冷链引流品（水产缸/低温奶柜）占地大但引流强，单分母会建议砍掉水产区。必须用「占地投影 + 陈列展开线性米」两个分母交叉看，引流品类叠加连带贡献修正。
   > **哲学根基（为什么必须系数化，源:零售老刘06·卡批50）**：卖场为每件商品付出的其他成本（占位/服务/损耗）被按整体销售**平摊**了——占黄金位的商品和塞角落的商品，成本被算成一样，不公平也不精细。所以坪效不能只看销售额，必须按「每件商品实际占用的空间/服务成本」系数化修正。这段 why 是六哥「精细化」IP 的外部印证：没有它，双分母只是 Excel 技巧；有了它，它是方法论主张。〔来源:零售老刘/零售数据化企划·external_reference〕
   > **失效边界**：此哲学在标品货架成立；散装/称重/现制区成本平摊本就难拆到单品，系数化修正退化为区域级。

3. **150天MVP三闭环**（严格按顺序）：
   - Week 1–2：品类级坪效红黑榜（不需桥表，现在就能出）
   - Week 3–6：Top20% SKU 局部 planogram + 货位坪效
   - Week 7–10：3 个堆头试点 ROI 追踪 → 供应商谈判弹药

4. **可复用三层分离**：engine/（客户无关公式）+ config/（客户YAML）+ data/（客户数据）。换客户只改 config。

## 核心计算公式

### 双分母坪效

```
全店月坪效 = 月销售额 ÷ 净卖场面积(3000㎡)

品类坪效-占地 = 品类月销售额 ÷ Σ footprint_m2(该品类货位)
品类坪效-展开面 = 品类月销售额 ÷ Σ linear_facing_m(该品类货位)

货位坪效 = 分摊到该货位的月销售额 ÷ footprint_m2
```

### 引流修正（社区生鲜必须做）

```python
修正货位价值 = 直接坪效 + 连带贡献系数 × 关联篮销售额
# 若无小票数据 → 引流保护名单兜底（水产/蔬菜/低温奶不进淘汰逻辑）
```

**「假A类」识别（源:零售老刘06·卡批50）**：坪效红黑榜按销售额判 A/B 类之前，必须先排除**断货掩盖的真需求**——A 商品卖 20 个后断货、B 商品 5 天卖 50 个，数据看 B 好，但真需求是 A（断货逼顾客改买 B）。「**畅销的往往是缺货的**」，分析里遍地是**假A类**。
```python
# 判假A类的两个触发条件（任一命中即复核，勿直接砍/降位）:
#   ① 该SKU在统计期内 有库→无库 状态切换 ≥N 次（配 O-04b 货位状态矩阵的"畅销缺货"态）
#   ② 高价 + 低销量 + 占黄金位 → 伪A类，须按"净品效(剔除断货天数后的日均)"复核
```
> **失效边界**：判「假A类」需库存/断货时点数据；花厅坊 POS 若不留每日库存快照 → 退化为店长经验标注「这个经常断货」，不进自动逻辑。

### 区位自动分类

```python
traffic_score = 0.4 × 入口可达性 + 0.4 × 通道等级 + 0.2 × 视觉焦点
# 黄金区 ≥0.66 | 普通区 0.33–0.66 | 死角区 <0.33

红牌 = 黄金区(traffic≥0.66) AND 低坪效(<品类中位数×50%)
# 死角区低坪效不报警（本就该放慢销品）
```

### 堆头 ROI

```
增量毛利 = (促销期日均销售 - 基线日均销售) × 促销天数 × 毛利率
机会成本 = 该位置品类月坪效/30 × 占地㎡ × 促销天数 × 毛利率
ROI = (增量毛利 + 供应商费用) / 机会成本

⚠️ 需求前移检测：
若 T+8–T+30 销量显著低于基线 → 真实增量打折（防促销是「借未来」）
```

## 桥表实操：货位管理五表体系（零售老刘实证）

> **来源：零售老刘 04-货位商品配置与维护（2022-10-01）·L1 Action Layer**

**这解决了什么**：`bridge_sku_location` 在架构图里是一张桥表，但实际建表时需要 5 种原始数据的支撑。这五表就是"手工盘点两天"的标准产出物：

| 表名 | 内容 | 建立方式 | 对应我们的字段 |
|---|---|---|---|
| **①商品结构表** | 每个区位该摆哪些品类（L1-L4）| 设计图纸＋品类系统V4.0 | `dim_sku.category_l1~l4` |
| **②货位类别关联表** | 哪个货位编号归属哪个区位/品类 | 现场一对一对照 | `dim_fixture.zone`+`category` |
| **③区位货位数量统计表** | 各区的货位数、层位数、线性米 | 现场量尺＋数格子 | `dim_fixture.shelf_layers`+`linear_facing_m` |
| **④在陈单品表**（★桥表核心） | 哪个货位摆了哪条SKU、几个排面 | **扫码枪逐一扫**（或拍照+AI识别） | `bridge_sku_location.*` |
| **⑤货位条码差异表** | 实际在陈条码与系统已选商品的差异 | ④与POS商品档案做diff | 用于发现"陈列了但POS没在管"的幽灵SKU |

**花厅坊下次现场执行（L1指令）**：
```
优先级 P0（两天可完成）:
  1. 扫码建在陈单品表（④）：带蓝牙扫码枪，从入口开始，逐货位扫一行SKU中任意条码，
     记录：货位编号、条码、排面数、层位（golden/upper/lower/floor）
  2. 与dim_fixture初稿合并 → 得到完整 bridge_sku_location 初始版本

优先级 P1（补充）:
  3. 统计每个区位货位数+层位数（③） → 更新dim_fixture.shelf_layers
  4. 货位条码差异表（⑤）：Python自动做POS商品档案 diff，不需手工
```

**区位货位面积分配公式**（L1可执行）：
```
某区货位数 ∝ 该区销售占比 × 毛利占比（两者交叉比例）
= √(销售占比 × 毛利占比)  ← 几何平均，防止纯销售导向压缩高毛利区
```
> 花厅坊验证：日化区实际货位数 ÷ 全店货位总数（约14%）vs. 交叉比例（估算4-5%）→ 超配3倍，理论上有大量腾挪空间。

## 时空系数法：货位层位打分与调整追踪

> **来源：零售老刘 06-货位陈列系数案例（2022-10-14）·L1 Action Layer**

**两个系数的意义**：

```
空间系数（静态）= 各层位的结构价值分
  黄金层（85-165cm）  → 空间系数 = 1.0
  中层（45-85cm）     → 空间系数 = 0.7
  下层（<45cm）       → 空间系数 = 0.4
  头顶层（>165cm）    → 空间系数 = 0.3

时间系数（动态）= 记录货位变更的时间窗口
  目的：当某SKU从A货位移到B货位时，扩展为"SKU在A货位的有效销售天数"
  用途：防止"移位前销售被算到移位后货位"的数据污染
```

**核心用法（AI可执行）**：
```python
# 理想位置评分公式
货位层位期望销售 = SKU类别销售占比 × 空间系数
实际层位空间系数 = 由layer_position字段映射（golden=1.0/upper=0.7/lower=0.4/floor=0.3）

陈列质量检验 = 实际层位空间系数 / 理想应有空间系数
# <0.7 → SKU位置偏差，应调整到更高层位
# >1.3 → 高价值位置放了低贡献SKU，浪费黄金资源
```

**与 bridge_sku_location 的映射**：
- `layer_position` 字段（golden/upper/lower/floor）即空间系数来源
- `valid_from / valid_to` 字段即时间系数的实现——每次货位调整记为新行，旧行关闭 valid_to

**花厅坊下次现场执行**：建在陈单品表（④）时，同步记录层位信息（是上层/中层/下层）→ 空间系数就有了；`valid_from` 填写盘点日期作为初始值。

### ⚠️ 失效边界：静态空间系数在社区生鲜的两处塌方（源:零售老刘06·M4-T2）

老刘06 明确说空间系数「**是不变的**」（黄金位固定四档 1.0/0.7/0.4/0.3）。这是**大卖场高货架标准，不是社区红线**，两处失效：

1. **矮货架塌档**：社区生鲜货架多 1.6–1.8m，几乎无「>165cm 头顶层」→ 四档系数**塌缩为 2–3 档**。上表直接搬「85–165cm=golden=1.0」会把社区店本就只有的中层误判为普通位。→ **应对**：按各店实际货架高度重标系数区间，`config/*.yaml` 里做成客户参数，不硬编码四档。
2. **黄金位随时段漂移**：「静态不变」假设在生鲜不成立——早市买菜动线 vs 晚市即食/熟食动线不同，同一货位 traffic 值随时段变。→ **应对**：`traffic_score` 对生鲜区拆早/晚市两组，或货位价值取时段加权；标品货架可维持静态。
> 不把大卖场四档系数当社区红线：矮货架 + 时段动线是社区生鲜特有约束。

## 货位商品状态判断矩阵（坪效外第二诊断层·源:零售老刘04·M4-O4）

> **这解决了什么**：坪效红黑榜只看「已在售且已陈列」的商品，看不见**没上架的隐性断货**。老刘04 末尾一句带过、但极有用——把「在陈 × 在销 × 实时库存 × ABC属性」四表叠加，可自动判出每个货位的商品状态，是坪效之外的**第二诊断层**。

**6 态判定（在陈单品表建成后加一列 `sku_status`）**：

| 状态 | 布尔组合（在陈/在销/有库/ABC） | 含义 | 花厅坊下一个动作 |
|---|---|---|---|
| **有库未陈** | 未陈 + 有库 | 仓里有货但没上架＝**隐性断货** | ★店长晨会看「有库未陈」清单→立即补架（零成本捡回销售）|
| **畅销缺货** | 在陈 + 在销 + 无库 + A档 | 真需求被压制（配「假A类」识别）| 提前补货/加安全库存；勿因缺货误判该SKU弱 |
| **A类无货** | A档 + 无库 | 主力商品断供 | 优先级最高的补货警报 |
| **有销无库** | 在销 + 无库 | 系统在卖但实物没了 | 查串货/盘亏；关掉线上可售 |
| **已选无库** | 已选品 + 从未有库 | 选了品但从没进过货 | 采购跟进或从商品档案剔除（幽灵SKU反向）|
| **此货暂缺** | 在陈位空 + 无库 | 货位有排面但空着 | 补货或收缩排面给相邻畅销品 |

- **与架构的衔接**：SPACE-ARCH 边界失效表原来只有单向的「幽灵SKU（陈列了但POS没管）」；本矩阵是**双向状态机**，「有库未陈」「畅销缺货」两态是它抓不到的。
- **落地**：桥表 `bridge_sku_location` 建成后，用 4 个布尔字段 IF 判 6 态，日更给店长。
> **失效边界**：矩阵依赖「实时库存」字段；花厅坊 POS 若无每日库存快照，「有库未陈/畅销缺货」两态判不出 → 退化为盘点日单点快照，时效性打折。散装称重区因下条 PLU 建表失效，本矩阵在生鲜区同样降级为品类级。

## 边界失效场景（批判性评估）

| 序号 | 失效场景 | 应对 |
|---|---|---|
| 1 | 桥表腐烂（陈列变了但没更新表） | 货位级结论标有效期90天；过期降级为品类级 |
| 2 | 称重生鲜qty单位塌方（kg vs 件混算） | unit字段强制分叉，周转率只在同unit内比 |
| 3 | 引流品被冤杀（水产坪效低但引流强） | 保护名单硬编码；连带贡献修正 |
| 4 | Facing弹性因果幻觉（排面多≠销量因） | 系数带红标；只用于A/B实测设计 |
| 5 | 需求前移伪装堆头成功 | T+8–T+30 回落检测；净真实增量打折上报 |
| 6 | 花厅坊 POS 导不出单据号 | 引流修正退化为名单兜底；第一周验证 |
