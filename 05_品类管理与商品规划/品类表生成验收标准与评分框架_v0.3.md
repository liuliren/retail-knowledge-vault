## 〇、结论先行

**一张 AI 生成的品类表，验收分两关：先过 6 道硬门禁（任一不过 = 一票否决打回，不进评分），再打 6 维 100 分（≥80 可交付，<65 重做）。AI 生成时必须附自评分卡，六哥抽查 ≥2 维后签字，签字前一律 status: draft。**

设计原理：门禁管"错不错"（对齐 SSOT、数据真实、客户安全——错了就是事故），评分管"好不好"（结构、判定、可执行性——差了可以修）。两者不混在一个量表里，是为了让"事故级问题"不可能被"其他维度的高分"平均掉。

## 一、适用范围与验收对象

**验收对象**：AI（Claude Code 及子代理）生成的门店级/品类级品类表交付件，含三种形态：

| 形态 | 样例 | 说明 |
|---|---|---|
| ① 单品类专项表 | [[2026-05-07_花厅坊休食商品品类表_v1.0]] | 某 L2 下的 L3 结构基础表（词表+角色+价格带+占比） |
| ② 全店品类表 | 花厅坊全店版（本周焦点·待生成） | 覆盖 L1-A~G2 全域的后端经营表实例化 |
| ③ 前后端对照表实例 | 三张表方法论·表三的门店落地版 | L2→前端区域映射 + 资源优先级 |

**不适用**：方法论模板本身（走标准审议流程）、诊断卡（走 /diagnose 与 mdcard 体系）、原始数据清洗件（走 /posclean 七坑防护）。

**上游权威（生成必须对齐的 SSOT）**：

## 二、第一关：6 道硬门禁（Gate·一票否决）

> 任一 Gate 不过 → 直接打回重做，附具体违规位置清单。不进入评分环节。
>
> **T1 修订（第3轮·planogram 合规校验 + 同行评审独立性）**：门禁分两类，**执行主体不同**——
> - **机械门禁 G2/G4/G5/G6**（T码模式/条码正则/Schema与断链/词表集合差）= **确定性可校验项**，应由**独立于生成器的 checker**（脚本或独立 Agent 实例）跑，**零 token、不进 LLM 自评**（工业界 planogram 合规从不让生成货架图的系统自评这些项）。checker 脚本上线前，过渡期由 ② AI 自检覆盖，但**须由非生成本表的独立 Agent 实例执行**（作者自评不算评审）。
> - **判断门禁 G1/G3**（SSOT 语义对齐/数据锚定与失真判断）= 需领域判断，保留 LLM + 六哥复核。

### G1 · SSOT 对齐（结构不许自造）
- [ ] 食品侧所有 L3/L4 节点名称与编码 = **V6.0 编译层镜像**（[[品类系统V6.0_精读卡_v0.1]] 的 mapper 对照表——DEC-20260702-03 指定的唯一 SSOT 映射件），逐一可对上；xlsx 本体仅在六哥显式授权时抽验（G02 raw 只读铁律，R3 修订）
- [ ] 非食品节点与 C 码 = KB-CAT-ARCH-001 v1.2（注意母婴已迁 H20/21/30/31/40）
- [ ] **跨界节点 tiebreak**（R3 修订·堵反例5）：食品但挂在 ARCH-001 专区的节点（婴幼奶粉/辅食 H20/H21、保健食品、宠物食品等），以 **ARCH-001 专区归属优先**，不按"食品侧走 V6.0"误判为节点对不上；两权威都查无 → 才进待裁决清单
- [ ] **0 个自造节点**。发现真实商品无处可挂 → 显式列入"待裁决清单"提交六哥，不得静默加节点（冲突显式化原则）

### G2 · T码作用域正确
- [ ] 跨系统引用（对外报告/脚本/跨店对比）一律 8 分法
- [ ] 门店内部细分用 12 分法时，表头必须带消歧注记 + 8分法对照表（照休食表 §0 格式）
- [ ] 无 T 码混用（同一列内两套编码混排 = 不过）

### G3 · 数据锚定（不许编数）
- [ ] 所有实测数字（SKU数/占比/动销率）可溯源至 **/posclean 产出的标准 master 表**，标注数据窗口期（起止日期）；**禁止回读 99_原始素材**（R3 修订·G02 raw 只读铁律，A 维抽 SKU 试挂同样只用标准 master）
- [ ] `fact_layer` 三分严格执行：observed（POS 实测）/ inferred（推断·须写推理依据）/ pending（无数据·不得填估计值冒充）
- [ ] **pending 计分归属与占比上限**（R6 修订·堵空心表博弈）：判定=pending 的行**不算** B-1"缺判定"（诚实优于编数），但 pending 行占比 **>30% → 总评自动封顶 C 级**（列修订清单复评）；管理规范客户上限收紧至 15%（§7-2 的"收紧机关"落成参数）
- [ ] 已知失真必须标注：生鲜品类毛利字段按双轨改判标注"结构性无进价·毛利失真"，不得直接引用伪高毛利率

### G4 · 客户安全（红线）
- [ ] 正文 0 条码（EAN-13）、0 进价裸值、0 供应商明细——只允许汇总口径
- [ ] frontmatter 有 `client_safety:` 字段；含客户可识别信息的标 `confidential`
- [ ] 对外版本（client_shareable）另行脱敏，不与内部版混用同一文件

### G5 · Schema 与链接完整
- [ ] frontmatter 必填齐：`title / version / status / summary(≤40字) / client_safety / fact_layer / related`
- [ ] 0 断链（所有 `[[wikilink]]` 目标存在）
- [ ] **生成器只能标 `status: draft`**——AI 及子代理禁止自填 active/signoff（代签红线，有判例）

### G6 · 词表闭合与角色合法
- [ ] `category_l3` 列的值域 = 表内定义的标准词表，无表外值、无近义漂移（"果干蜜饯"≠"蜜饯果干"）
- [ ] 品类角色列取值 ∈ {D, P, R, S, C} 及其组合（如 D/P），无自造角色
- [ ] 每个条件启用类（L4/酒类区/小家电等）标注了启用条件是否满足及依据

## 三、第二关：6 维评分框架（100 分）

> 门禁全过后打分。每维给分必须附证据指针（行号/字段名），不许裸给分。

### A · 结构质量（25 分）
| 检验项 | 分值 | 判分要点 |
|---|---|---|
| 划分 MECE | 10 | 无双占（一个商品集合只有一个 L3 归属）、无遗漏（门店在售商品全部可挂）；抽 10 个边界 SKU 试挂，双占/无处挂每个 −2 |
| L4 启用有据 | 8 | 启用 L4 的必须对应 5 大复杂类模板之一（CC-PATH/BAND/SCENE/FUNC/GIFT）；"因为 SKU 多"不是理由，每个无据启用 −3 |
| 粒度适配 | 7 | 一个小类 = 一个顾客需求（零售老刘根基命题）；粒度过粗（多任务混一类）或过细（同任务拆多类）每处 −2 |

### B · 数据判定质量（20 分）
| 检验项 | 分值 | 判分要点 |
|---|---|---|
| 5 态判定完整 | 10 | 每个 L3 行有宽度状态（🔴空缺紧急/🟡空缺建议/🟠过窄/🟢健康/🟠过剩）+ 对应行动；缺判定每行 −1 |
| 双维度联动 | 6 | 不只看 SKU 数，还看角色覆盖（引流/利润/形象/补充款结构）；只做单维度 −4 |
| 增长分群 | 4 | 有同期数据时按 ±20% 分增长/稳定/下降三群并挂资源动作；无同期数据标 pending 不扣分 |

### C · 角色与策略判定（20 分）
| 检验项 | 分值 | 判分要点 |
|---|---|---|
| 角色判定三源有据 | 8 | 每个 D/P/R/S/C 判定至少援引一源：POS 销售结构 / 购物任务图谱 / 大湾区特化基准；裸断言每处 −2 |
| 策略体现取舍 | 6 | 必须有做深/做稳/精简/替换的分布。**全表零取舍（100% 做强）→ 本项 0 分且总评强制降一等级**（R5 修订：与设计原理一致——无战略取舍属结构性缺陷，不许被其他维度高分平均掉） |
| 角色-资源自洽 | 6 | D 目标品不得配 P3 优先级、C 便利品不得配最大面陈等矛盾；每处矛盾 −2 |

### D · 可执行性（15 分）
| 检验项 | 分值 | 判分要点 |
|---|---|---|
| 20 分钟上手测试 | 6 | 采购员/店长视角：一张表 + ≤5 条判断规则能直接用；需要读第二份文件才能操作 = −3 起 |
| 管控指标带阈值 | 5 | 每 L2/L3 行有关键管控指标且量化（动销率≥X%/损耗≤Y%/周转≤Z天）；有指标无阈值每行 −1 |
| 直接可调用 | 4 | KB 页质量铁律：读完任一行能立刻生成"花厅坊→下一个具体动作"；抽 3 行测试 |

### E · 前后端联动（10 分）
| 检验项 | 分值 | 判分要点 |
|---|---|---|
| 区域映射齐全 | 5 | 每个后端 L2 有对应前端区域（A–K）+ 陈列分区；缺映射每行 −1 |
| 温区标注 | 3 | 常温/0–4°C/-18°C/活水标注完整，无跨温区拼陈列建议 |
| 顾客任务语言 | 2 | 前端侧命名是顾客日常语言，非管理术语（"今日蔬菜"✓ /"叶菜类SKU组"✗） |

### F · 大湾区特化（10 分）
| 检验项 | 分值 | 判分要点 |
|---|---|---|
| 特化基准应用 | 5 | 凉果+30%/煲汤料+50%/腌腊+40%（秋冬）等上调基准落进 SKU 目标数；漏用每处 −1 |
| 季节日历挂接 | 3 | S 季节品挂大湾区节奏日历节点（腊味季/煲汤季/年货季）及备货提前量 |
| 慎用清单核查（**scope 条件化**·T4/D2 修订） | 2 | **中式糕点冲突已裁决（六哥 2026-07-03「照专业建议办」·D2）=条件化，非二选一**：ECR Category Role 本按店群 cluster 赋予，同品类不同 cluster 角色可相反；两份 stable 都对、只漏写作用域。**判分规则**：先判本表 cluster（客群×商圈×店型）——① 广府 cluster（花厅坊=荔湾广府核心）→ 中式糕点=**Destination 级差异化品**，建议扩充算加分，慎用清单**不适用**；② 非广府 cluster → 慎用清单生效，扩中式糕点扣分。北方挂面/东北腌渍类同理按 cluster 判（花厅坊仍慎用）。裸判（不声明 cluster）→ 本项判 0 |

## 四、裁定等级

| 总分 | 等级 | 裁定 |
|---|---|---|
| ≥90 | **A · 优秀** | 可交付 + 列为跨店模板候选（第二客户复用池） |
| 80–89 | **B · 通过** | 可交付花厅坊使用 |
| 65–79 | **C · 有条件通过** | 列修订清单，3 日内修订复评，只复评失分维度 |
| <65 | **D · 打回** | 重做；重做前先诊断是生成方法问题还是上游数据问题 |

**门禁失败不评分**——G1–G6 任一不过，无论其他部分多好，直接打回。

## 五、验收流程（谁做什么）

```
① AI 生成品类表（status: draft 强制）
   ↓
② AI 自检：跑 G1/G3 判断门禁 + 6 维自评分卡（每项附证据指针）
   ↓  自评不过 → 自修后再进 ③，不许带病提交
③ 独立 checker 校验（T1·同行评审独立性）：机械门禁 G2/G4/G5/G6
   　由**独立于生成器的 checker**（checker 脚本上线前=非生成本表的独立 Agent 实例）跑
   　+ lint-kb 阶段A（断链/孤儿·现有）；G03_Lint v2 上线后扩 Schema/敏感值/T码冲突（R7·据实表述）
   ↓  机械项不由生成器自证——作者自评不算评审
④ 六哥定向抽查（R2 修订·打击虚高激励，不再随机）：
   　a. 必抽 = 自评最高分维 + 权重最大维（A 维）——虚报者必被查
   　b. 复评触发线按维度满分归一化：单维相对差 >20%（替代绝对 5 分线）
   　c. A 维复核用 AI 自评同一批样本 SKU，不许换样本比分（隔离抽样噪声与判定虚高）
   　d. 首 2 张表强制全维人工复评，建立自评-复核校准基线后方可启用抽查制
   ↓  相对差超线 → 全表人工复评 + 记入生成方法缺陷台账
⑤ 六哥验收签字 = 交付许可 → **按件分两轨（T3/D1 修订·六哥 2026-07-03 裁=豁免）**：
   　【交付件轨】客户交付表（花厅坊全店/单品类表·1店0案例）走 fitness+签字，
   　　　终态 status: **delivered**（交付即终态，**豁免 §11.3 的 ≥3案例/≥2门店 实证门槛**——
   　　　那是方法论推广门槛，套到单店交付表是工序错配）；落 SignoffLedger。
   　【方法论轨】从多张交付表蒸馏出的可复用方法论/模板件，才受 §11.3 约束：
   　　　draft→candidate→active（≥3案例 OR ≥2门店 observed + 签字）。
   ↓
⑥ post-ship POS 检验环（T2·7-Eleven tanpin kanri 検証）：交付表=**假设非终值**——
   　上架后用真实 POS 回填复核（宽度判定对不对/角色判得准不准/管控阈值合不合理），
   　偏差回写"生成方法缺陷台账"+ 校准 §7-4 权重。花厅坊数据基建撑不起快循环 →
   　检验节奏随交付节点（非每周），但**必须有这一环**，否则表永远停在"未证伪的假设"。
```

**分工红线**：①②③ 是 AI 的活（②判断相自评/③机械相独立校验），④⑤⑥ 是六哥的活（判断相·签字背书·实测裁决），不得下放（90/10 护城河）。

## 七、张力与失效条件（本框架什么时候不成立）

1. **店型漂移即失效**：SKU 目标数、特化基准全部锚定"1500–2000㎡ 近全品类社区店（花厅坊/乐易购沙浦大道型）"。接到便利店型、纯生鲜型、或非大湾区客户 → F 维整体停用，A/B 维基准须重校准后才可评分。
2. **数据不规整的容忍度**：花厅坊管理水平不高、采购周期无规律——B 维评分容忍过程字段缺失（标 pending 不扣分），但**不容忍编数**（G3 是门禁不是评分）。换了管理规范的客户，pending 容忍度应收紧。
3. **框架随 SSOT 迭代**：V6.0 → V6.x 升级、或 KB-CAT-ARCH-001 再改版时，G1 的对照目标同步切换，旧表按当时 SSOT 验收的结果不追溯重判。
4. **分值权重是假设不是实证**：25/20/20/15/10/10 的权重分布是第 1 轮拍的（结构>判定>执行>联动>特化的直觉排序），须经花厅坊全店品类表实跑 1–2 次后回填校准——如果实跑发现"可执行性差"是最大返工原因，D 维权重应上调。
