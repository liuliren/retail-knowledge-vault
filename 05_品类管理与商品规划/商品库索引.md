## §1. 一句话定位 ⭐⭐⭐

> **商品库不是商品资料表 / 而是连接客户数据、品类知识、经营判断、匹配算法和咨询交付的数据资产层。**

---

## §2. 5 个角色

商品库未来承担 5 个角色（缺一不可）：

1. **客户 SKU 标准化的目标库** — 客户原始 SKU 文本 → 匹配到标准品名 / 标准品类
2. **V4.0 品类树的实例化数据层** — V4.0 是结构骨架 / 商品库是血肉实例
3. **顾问经验沉淀的规则库** — keyword_rule + brand_alias + package_unit 沉淀顾问判断
4. **经营属性判断的知识库** — 品类角色 / 形象 / 流量 / 毛利 / 民生 / 端架适合性
5. **四层匹配算法的持续学习底座** — 边建库 / 边匹配 / 边反哺 / 持续迭代

---

## §3. 与 V4.0 品类树的关系

```
V4.0 品类树（308 L4 节点）→ 结构 / 骨架 / 唯一最终版
商品库（A+B+C 三层）        → 实例 / 血肉 / 多 SKU 挂在 L4 节点下

约束：商品库每个 SKU 必须有 category_l4_code FK 指向 V4.0
关系：1 对多（L4 → 多 SKU）

例：
  V4.0 L4 [10103] = 调味品/酱油/生抽
  商品库 SKU：
    A-00001 / 海天金标生抽 500ml 玻璃瓶 / category_l4=10103
    A-00004 / 海天生抽王 1.9L 家庭装 / category_l4=10103
    ...
```

---

## §4. 与品类管理八环节的关系

商品库是**品类管理八环节**（[[品类管理]] 主定义）的**实例化数据底座**：

| 环节 | 商品库支撑 |
|---|---|
| 1. 品类定义 | B 层骨架（category_l1-l4 + business_role）|
| 2. 品类角色 | B 层 business_role / is_essential_category |
| 3. 品类评估 | B + C 联合（品类骨架 + 客户战役表现）|
| 4. 品类记分卡 | C mapped 层（销售 / 毛利 / ABCD）|
| 5. 品类策略 | A + C（标准 SKU 选择 + 客户特殊判断）|
| 6. 品类战术 | C mapped 层（端架 / 黄金层 / business_action）|
| 7. 品类执行 | DisplayMap 关联 + 客户战役 |
| 8. 品类复盘 | C mapped 层 feedback_to_library 反哺 |

---

## §5. 与四层匹配算法的关系

```
商品库（基础设施）+ 4 层匹配算法（应用层）= 持续学习闭环

闭环 8 步：
  客户原始 SKU → 清洗标准化 → 初步匹配 → 人工审核 → 
  结果回写 → 规则沉淀 → 商品库升级 → 下一客户复用

→ 商品库不是 4 层算法的"前置一次性工程" / 是"持续学习底座"
```

详见 [[13_数据分析与工具脚本/商品库_v0.1/README]] §2 闭环 8 步。

## §6. 在客户咨询项目中的使用方式

| 场景 | 用途 |
|---|---|
| 客户首访前 | 提前用商品库标准化客户提供的 SKU 列表 |
| 客户诊断中 | 用商品库提供品类骨架 + 经营属性 |
| 调改方案 | 用 mapped 层 business_action 给出处置建议 |
| 客户交付 | 标准品名 + 标准品类 / 客户能听懂 |
| 战役复盘 | 反哺商品库 / 沉淀新规则 |

## §8. 5 大原则（v1.1 / 来自 ChatGPT 综合）

1. **先品类，后单品** — 早期不执着每个 SKU 精准识别
2. **先白盒，后智能** — 先做规则 + 关键词 / 后做 embedding + LLM
3. **先反哺，后规模** — 不靠一次性导入做大 / 靠客户项目做准
4. **必须绑定 V4.0 品类树** — category_l4_code 强 FK 约束
5. **业务数据资产 / 不是治理支柱** — 不是 vault 第 7 大支柱
