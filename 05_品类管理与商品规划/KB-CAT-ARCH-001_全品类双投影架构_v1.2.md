# 全品类商品品类表 · 双投影架构 v1.0

> **v1.2 变更（2026-07-02·六哥 D 档签字）**：修复 C-码双占——母婴专区 B01/B02/B10/B11/B20 迁至 **H20/H21/H30/H31/H40**（表A + 表B M6 + 锚点表同步）；表A 粮油 B 码不动。执行时发现原 diff 拟用的 H01/H02/H10/H11 已被日化占用，依 diff 顺延条款按"L2→十位段"模式改配（H1婴儿食品→H2x / H2母婴用品→H3x / H3孕产妇→H4x）。凭据：C码双占修复diff_D档待签_v0.1 + SignoffLedger。

> **结论**：本体系由两张投影表 + 一个共用锚点组成。表 A（前端/顾客视角）和表 B（后端/运营视角）在 L1–L2 各自独立分群，在 **L3 C-码**汇聚为唯一锚点——同一个 C-码，在两张表里意义不同但指向同一商品集合。
>
> **来源定位**：本文件是顶层架构定义。V4.0 大湾区版（xlsx / 308 L4 / 食品）是食品侧 L3–L5 的主数据源，本文件扩展覆盖范围至非食品类，并定义前后端双投影逻辑。
>
> **裁决规则（优先级）**：管理链路强度 > 物理形态（例：婴幼米粉→B5 特配婴儿辅食，不落入 B1 粮油）

## §1. 架构原则：双投影 + 单锚点

```
顾客视角 (表A)            锚点              运营视角 (表B)
L1 购物区域           →   L3 C-码    →     L1 管理域
L2 购物目的群              (唯一)           L2 管理模块
       ↘                   ↑                 ↗
        L4 商品小类    L3 品类(锚)    L4 管理指标
              ↓                              ↓
             L5 典型SKU                  L5 主要动作
```

**锚点规则**：
- L3 C-码全库唯一，格式 `X##`（大类字母 + 两位序号），由 V4.0 食品册沿用，非食品补充 `G`/`H` 系列
- 同一 C-码在表 A 落在「顾客去哪拿」，在表 B 落在「我们怎么管」
- L1–L2 冲突正常——冲突不是问题，锚点才是真相

## §5. 使用说明

1. **查品类决策(M-DEC)时用表B**：找到 M-DEC 对应的 C-码，在表B定位管理逻辑域，用 Domain C3/D4 等判断。
2. **规划门店陈列动线时用表A**：顾客从 A→B→C... 的行走路线就是你的主通道规划依据。
3. **数据交叉分析时用C-码做JOIN**：两表共用 L3 C-码，直接作为数据库联表键。
4. **扩展新品类**：先在 V6.0 确认 L3 C-码，新码沿用系列；若无对应码，在此表扩编后同步 V6.0。
5. **V6.0关系**：**V6.0 大湾区版（68L3/385L4·2026-05-21）是当前 L3–L5 食品侧主数据源**（取代V4.0·六哥 2026-06-28 三裁决确认）；本文件扩展了非食品类（G/H/F系）并叠加了双逻辑层。非食品归属裁决：冷冻→F系、酒精→M4、火腿肠→F02（六哥 2026-06-28 签字）。vault路径：`Documents/30/.../晟果新零售_品类系统V6_0_迭代升级审核版.xlsx`。
6. **L4 定义裁决（六哥 2026-06-29 签字）**：L4 = 细分类（commodity sub-type，与 V6.0 385 个 L4 口径一致）。**价格带不是 L4 维度**——它是每个 SKU 独立的角色属性标签（`is_traffic`/`is_margin`/`is_image` 布尔字段），与 L4 分类轴正交，不得混入品类层级。任何试图将"高价带/低价带"等写入 L4 节点的设计，视为违反本条裁决。
