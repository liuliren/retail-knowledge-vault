# B1条码分组统计读取修复记录 v0.1

> **结论先行**：B1不存在真正的"假xls"问题。118个文件全部是标准OLE2二进制xls格式(`file`命令确认CDFV2 Microsoft Excel，magic bytes D0CF11E0)，用pandas默认的`xlrd`引擎读取会报错(该引擎新版本已不支持老式.xls二进制格式)，**改用`engine='calamine'`后118/118文件100%读取成功**，零失败。

---

## 1. 报错现象

上一批评估中，抽样打开B1文件时用pandas默认方式(`pd.read_excel(f)`，隐式走xlrd引擎)复现"国产POS假xls"式报错(`Expected BOF record; found...`)。

## 2. 文件真实格式判断

对3个抽样文件用`file`命令+十六进制头部检查：

| 文件 | file命令输出 | 头部magic bytes |
|---|---|---|
| 商品销售条码分组统计_90天_3个月百货数据 | CDFV2 Microsoft Excel | D0CF11E0(真OLE2二进制) |
| 商品销售条码分组统计_2025年_猪肉档数据 | CDFV2 Microsoft Excel | D0CF11E0 |
| 商品销售条码分组统计_120天_4个月蔬菜数据 | CDFV2 Microsoft Excel | D0CF11E0 |

**判断：全部是真xls(国产POS系统"棚割/进销存"打印导出的老式Excel二进制格式)，不是HTML伪装、不是CSV伪装、不是编码错乱文本。** 与本项目此前B8/B10/B5等文件同源同构，均属"国产POS老式xls只能被calamine正确解析"这一类已知模式。

## 3. 尝试方案

| 方案 | 结果 |
|---|---|
| `pd.read_excel(f)`(默认引擎，通常隐式选xlrd) | ❌ 报错(xlrd 2.x不支持老式.xls二进制格式) |
| `pd.read_excel(f, engine='calamine')` | ✅ 成功 |

## 4. 成功读取方案

```python
import pandas as pd
df = pd.read_excel(文件路径, engine='calamine', header=None)
```

无需其他前置处理(不需要转码、不需要先转xlsx、不需要HTML解析)。
