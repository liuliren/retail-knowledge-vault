# POS 数据清洗与商品标准化规范

## 一、文件定位

### 1.1 本文件是什么

本文件规范客户 POS 原始销售流水 / 商品档案 / 库存表从**"客户系统导出状态" → "诊断可用标准数据资产"**的全过程。

**核心任务：** 把"客户给的 .xls"按规范变成"可分析、可复盘、可追溯的标准数据"。

### 1.2 本文件不是什么

- ❌ 不是代码实现（Excel 模板 / Python 脚本 / AI 工具的具体实现另立项）
- ❌ 不替代 [[数据与系统能力底座]] 方法论
- ❌ 不处理客户财务凭证 / 会员隐私 / 供应商合同等非授权数据
- ❌ 不是商品力诊断报告（[[诊断输出质量控制规范_v0.1]] 的事）

### 1.3 在 4 文件销售包中的位置

```
[[社区超市商品力诊断首访执行手册_v0.1]]（首访产生原始 POS）
       ↓
[[客户项目资料入库标准_v0.1]]（原始 POS 入库 99_原始素材/01_POS原始数据/）
       ↓
本文件（清洗 → 标准化 → 输出诊断可用数据）
       ↓
[[诊断输出质量控制规范_v0.1]]（清洗后数据进入分析层 + 证据等级标注）
       ↓
客户诊断报告 / 调改方案
```

### 1.4 服务目标

服务三个月生存战 P2 新客户开拓 + P1 花厅坊样板的"**数据基础设施**"——这是后续所有诊断结论的可信度根基。

## 二、适用场景与不适用场景

### 2.1 适用场景

- ✅ 客户 POS 导出的销售流水 .xls / .xlsx / .csv
- ✅ 客户 ERP 导出的商品档案
- ✅ 客户当前库存表（金额 + SKU 级）
- ✅ 客户品类销售汇总表
- ✅ 客户提供的"商品 - 供应商关系表"（如有）

### 2.2 不适用场景

- ❌ 会员隐私数据（手机号 / 身份证 / 消费记录与个人关联）
- ❌ 财务凭证（账目 / 银行流水 / 报税资料）
- ❌ 供应商合同（采购价 / 账期 / 返点等敏感信息）
- ❌ 未授权敏感数据（客户未明示授权使用的数据）

**接收原则：** 客户给什么处理什么；不主动索取超出诊断需要的数据。

## 三、数据清洗总原则（8 条不可妥协）

| # | 原则 | 解释 |
|---|------|------|
| 1 | **原始表不覆盖** | 任何清洗动作必须存为派生文件；原始 .xls 永不修改 |
| 2 | **清洗表可追溯** | 每行派生数据可追溯到原始文件 + 行号 |
| 3 | **字段统一优先于分析** | 字段标准化未完成 → 不进入分析；不允许"半统一"分析 |
| 4 | **条码优先** | 主键以 barcode 为准；商品名仅作辅助 |
| 5 | **原始商品名必须保留**（`sku_name_raw`）| 用于后续复核 / 客户沟通 |
| 6 | **标准商品名用于分析**（`sku_name_std`）| 按 §八 命名规范生成 |
| 7 | **自动匹配结果必须可复核** | 每条匹配结果含 `match_confidence` + `match_method` 字段 |
| 8 | **低置信度结果不得直接进入正式结论** | 参 §十 置信度规则 + [[诊断输出质量控制规范_v0.1]] §六 证据等级 |

## 四、数据处理流程 SOP（S1-S11）

> 11 阶段标准流程。每阶段含输入 / 动作 / 输出 / 验收标准 / 风险提醒。

### S1 接收客户原始表

| 项 | 内容 |
|---|---|
| 输入 | 客户邮件 / 微信发送的原始 .xls / .xlsx / .csv |
| 动作 | 1. 落入 vault `99_原始素材/01_POS原始数据/`（参 [[客户项目资料入库标准_v0.1]] §四.3）；2. 按命名规范重命名 `客户简称_门店简称_POS_品类_日期区间_原始.xls`；3. 不打开文件 |
| 输出 | 命名标准化的原始文件 |
| 验收标准 | 文件未被打开 / 修改 / 重命名前的内容；命名规范完整 |
| 风险提醒 | 用 mv 而非"另存为"；Excel 打开后保存会改格式 |

### S2 原始表归档

| 项 | 内容 |
|---|---|
| 输入 | S1 命名后的原始文件 |
| 动作 | 1. 在《资料索引表》新增 1 行（`file_type=原始` / `category=POS`）；2. 标记 `status=已入库` |
| 输出 | 索引表更新 |
| 验收标准 | 索引表 `original_path` + `standard_path` 双字段完整 |
| 风险提醒 | 不允许"先归档后补索引" |

### S3 建立数据文件索引

| 项 | 内容 |
|---|---|
| 输入 | 已归档的原始文件 + 当次入库的所有相关数据表 |
| 动作 | 在《数据文件索引清单》（嵌入资料索引表的子集筛选）登记本次清洗任务的所有原始数据文件 |
| 输出 | 数据文件清单（含文件清单 + 总行数 + 时间区间）|
| 验收标准 | 含 ≥ 3 类表（销售流水 + 商品档案 + 库存）|
| 风险提醒 | 缺任一类表则进入降级方案（参 [[社区超市商品力诊断首访执行手册_v0.1]] §四.5）|

### S4 判断表类型

| 项 | 内容 |
|---|---|
| 输入 | 已索引的原始数据文件 |
| 动作 | 打开（仅查看，不修改）原始文件，识别属于：① 销售流水；② 商品档案；③ 库存表；④ 品类汇总；⑤ 供应商关系；⑥ 其他 |
| 输出 | 表类型标签 |
| 验收标准 | 索引表 `category` 字段填写正确 |
| 风险提醒 | 表头不规范的客户（如多张 sheet / 含合并单元格 / 含图表）需特殊处理 |

### S5 字段映射

| 项 | 内容 |
|---|---|
| 输入 | 表类型 + 客户原始字段 |
| 动作 | 按 §六 字段映射规则，把客户字段映射到本规范 §五 标准字段 |
| 输出 | 字段映射清单（客户字段 → 标准字段）|
| 验收标准 | 必填字段 100% 映射；缺失字段标记 `待确认` |
| 风险提醒 | 客户字段命名混乱（如"金额"可能是销售金额或毛利额）需对照客户访谈确认 |

### S6 数据类型清洗

| 项 | 内容 |
|---|---|
| 输入 | S5 映射完成的字段 |
| 动作 | 按 §七 数据类型与格式标准清洗每个字段（日期 / 条码 / 数字 / 空值 / 负值 / 重复 等）|
| 输出 | 类型清洗后的中间表 |
| 验收标准 | 每行数据通过类型校验；异常行进入 §十一 异常清单 |
| 风险提醒 | 异常数据**不删除**，进异常清单备查 |

### S7 商品名标准化

| 项 | 内容 |
|---|---|
| 输入 | S6 清洗后的中间表 |
| 动作 | 按 §八 商品命名统一规范，从 `sku_name_raw` 生成 `sku_name_std` |
| 输出 | 含双字段（raw + std）的中间表 |
| 验收标准 | `sku_name_raw` 100% 保留；`sku_name_std` 覆盖率 ≥ 90%；无法生成的标 `待确认` |
| 风险提醒 | 不允许覆盖 raw；不允许凭直觉编 std |

### S8 商品表 × 标准品类表匹配

| 项 | 内容 |
|---|---|
| 输入 | S7 标准化后的商品档案 + 标准品类表（v0.1 由方法论文件 + Excel 标准品类表共同定义）|
| 动作 | 按 §九 4 层匹配规范执行：一级条码精确 → 二级关键词 → 三级品牌+规格+口味 → 四级人工复核 |
| 输出 | 含品类标签（L1-L4）+ `match_method` + `match_confidence` 的商品档案 |
| 验收标准 | L1 覆盖率 100%；L4 覆盖率 ≥ 70%；置信度低的标记进入复核 |
| 风险提醒 | 标准品类表 v0.1 待校准；匹配结果以 [[品类管理]] / [[购物任务图谱_v1.1]] 方法论为最终依据 |

### S9 异常数据清单输出

| 项 | 内容 |
|---|---|
| 输入 | S6-S8 各阶段识别的异常数据 |
| 动作 | 按 §十一 异常类型分类，输出《异常数据清单》 |
| 输出 | 异常数据清单（13 类异常 + 字段：issue_id / source_file / row_no / barcode / sku_name_raw / issue_type / issue_desc / impact_level / suggested_action / status / owner）|
| 验收标准 | 每条异常含建议动作 + 影响等级（高/中/低）|
| 风险提醒 | 异常**不直接删除**；标记后人工决策处理 |

### S10 生成诊断可用标准表

| 项 | 内容 |
|---|---|
| 输入 | S6-S9 全部产出 |
| 动作 | 输出 §十二 6 张标准表（销售流水 + 商品档案 + 库存 + 商品-品类匹配 + 异常清单 + 清洗日志）|
| 输出 | 6 张标准表（按命名规范，含 `_清洗后` 后缀）|
| 验收标准 | 6 张表齐全；每张表通过 §十四 验收 10 项 |
| 风险提醒 | 标准表落入 `99_原始素材/` 内独立 `_清洗后` 子目录 或 `03_商品诊断/` 等业务模块（参 [[客户项目资料入库标准_v0.1]] §五）|

### S11 复核与版本记录

| 项 | 内容 |
|---|---|
| 输入 | 已生成的标准表 |
| 动作 | 顾问独立复核 + 版本记录入清洗日志（§十三）|
| 输出 | 清洗日志 + 资料索引表 status 升级到 `已清洗` |
| 验收标准 | 索引表完整 + 清洗日志记录全部异常处理决策 |
| 风险提醒 | 清洗日志是后续诊断结论可追溯的根基；不允许跳过 |

## 五、标准字段字典

> ⚠️ 字段命名采用英文 snake_case，**支持后续自动化处理**（Excel 公式 / Python pandas / AI 工具）。

### 5.1 销售流水标准表（`sales_log_standard`）

| 字段 | 中文名 | 类型 | 必填 | 客户原字段别名 | 清洗规则 | 异常处理 |
|---|---|---|---|---|---|---|
| `sale_date` | 销售日期 | DATE (YYYY-MM-DD) | ✅ | 销售日期 / 日期 / 交易日期 / 销售时间 | 统一为 YYYY-MM-DD；时分秒去除 | 缺失行进异常清单；不可推断 |
| `store_name` | 门店名称 | STRING | ✅ | 门店 / 门店名 / 店名 / 分店 | 标准化为"客户简称_门店简称"（如 客户A_主店）| 单店项目可填默认值 |
| `barcode` | 商品条码 | STRING（≥ 8 位）| ✅ | 商品编码 / 条形码 / 国际条码 / EAN / UPC | 去除空格；前导零保留；标准 13 位 EAN 保留；非数字字符进异常 | 无条码进异常清单 |
| `sku_name_raw` | 原始商品名 | STRING | ✅ | 商品名称 / 品名 / 商品全称 / 商品 | **保留客户 ERP 原始字符（不修改）** | 缺失进异常清单 |
| `sku_name_std` | 标准商品名 | STRING | ✅ | 派生字段（清洗生成）| 按 §八 商品命名规范生成 | 无法生成则标"待确认" |
| `qty` | 销售数量 | DECIMAL(10,2) | ✅ | 销售数量 / 数量 / 销量 / 件数 | 转为数字；负数保留（退货）| 异常正负值进异常清单 |
| `sales_amount` | 销售金额 | DECIMAL(12,2) | ✅ | 销售金额 / 实收金额 / 金额 / 销售额 | 转为数字（元）；保留 2 位小数 | 负数 = 退货保留；0 异常 |
| `gross_profit` | 毛利额 | DECIMAL(12,2) | 否（待确认）| 毛利 / 毛利额 | 转为数字；可为负 | 缺失字段记录 source_file 标"毛利字段缺失" |
| `category_raw` | 原始品类名 | STRING | 否 | 品类 / 分类 / 类别 | 保留原文 | 缺失则待清洗 |
| `source_file` | 来源文件 | STRING | ✅ | 派生字段 | 记录原始文件名（含路径）| 必填以便追溯 |

**完整性 KPI：**
- `barcode` 覆盖率 ≥ 95%
- `sku_name_raw` 覆盖率 100%
- `sale_date` / `qty` / `sales_amount` 覆盖率 100%
- `gross_profit` 覆盖率（待确认）

## 六、字段映射规则

> 客户字段千差万别，本节定义常见映射对照表。**任何客户具体字段映射前必须经过 S5 字段映射阶段确认。**

### 6.1 高频映射表（销售流水）

| 客户原字段（任一别名）| 标准字段 |
|---|---|
| 商品编码 / 条形码 / 国际条码 / EAN / UPC / SKU编码 | `barcode` |
| 商品名称 / 品名 / 商品全称 / 商品 / 名称 | `sku_name_raw` |
| 销售数量 / 数量 / 销量 / 件数 / 数 | `qty` |
| 实收金额 / 销售金额 / 金额 / 销售额 / 营业额（按行）| `sales_amount` |
| 毛利 / 毛利额 / 利润 | `gross_profit` |
| 销售日期 / 日期 / 交易日期 / 销售时间（取日期部分）| `sale_date` |
| 门店 / 门店名 / 店名 / 分店 / 店面 | `store_name` |

### 6.2 高频映射表（商品档案）

| 客户原字段 | 标准字段 |
|---|---|
| 品牌 / 制造商 / 厂家 / 厂商 | `brand` |
| 规格 / 单品规格 / 包装 / 净含量 | `spec` |
| 单位 / 计量 / 计量单位 | `unit` |
| 零售价 / 售价 / 标价 / 吊牌价 | `retail_price` |
| 进价 / 成本价 / 采购价 | `cost_price` |

### 6.3 字段映射的 5 条纪律

1. **客户原字段歧义时**（如"金额"可能是销售金额或毛利额）→ 必须**与客户访谈确认**，不能臆断
2. **客户字段含中英文混排**（如"sales_数量"）→ 按英文部分映射；忽略中文修饰
3. **客户字段含合并单元格**（多字段合一）→ 拆分为多个标准字段
4. **客户字段含计算公式而非数值**（Excel 公式列）→ 转换为数值后映射
5. **客户原字段无法映射到任一标准字段**（如"客户备注"）→ 进入"扩展字段"区，不强制清洗

## 七、数据类型与格式标准

| 数据类型 | 标准格式 | 异常处理 |
|---|---|---|
| **日期** | `YYYY-MM-DD`（仅日期，不含时分秒）| 多种格式（如 2026/5/3）统一转换；模糊日期（如"5月"）进异常 |
| **条码** | STRING（≥ 8 位数字）；保留前导零；标准 EAN-13 / UPC-A | 含非数字字符（如汉字 / 字母）进异常；< 8 位进异常 |
| **数量** | DECIMAL(10, 2)（最多 10 位整数 + 2 位小数）| 文本"3 袋"等进异常清洗；保留负数（退货）|
| **金额** | DECIMAL(12, 2)（元，2 位小数）| 含 ¥ / RMB 等符号去除后转换；负数保留（退货）|
| **毛利额** | DECIMAL(12, 2) | 可为负（亏损 SKU）；缺失字段不强制 |
| **空值处理** | 必填字段空值 → 异常清单；非必填字段空值 → 标 `NULL` | 不允许用 0 / "未知" 替代真实空值 |
| **负数处理** | 销售/库存负数保留；价格负数 → 异常 | 销售退货负数标记 `is_refund=true` 派生字段 |
| **重复行处理** | 完全相同的 2 行 → 保留 1 行 + 异常清单标记 | 同 barcode 同日期不同金额 → 异常进清单不合并 |
| **多门店字段** | `store_name` 标准化；多店项目必须每行有 store_name | 单店项目省略可，但要在清洗日志记录 |
| **时间区间统一** | 不同表的时间区间必须对齐（如 POS 90 天 vs 库存盘点日 = 同一节点）| 不对齐时记录差异 + 进异常 |

## 八、商品命名统一规范

### 8.1 标准商品名建议格式

```
品牌 + 品名 + 口味/属性 + 规格 + 单位
```

**示例：**
- `卫龙_大面筋_麻辣_26g_袋`
- `三只松鼠_每日坚果_独立装_25g×30_盒`
- `怡宝_纯净水_555ml_瓶`
- `化州橘红_凉果_原味_200g_袋`

### 8.2 双字段并存

| 字段 | 来源 | 用途 | 是否可修改 |
|---|---|---|---|
| `sku_name_raw` | 客户 ERP 原始字符 | 复核 / 客户沟通 | ❌ 永远不修改 |
| `sku_name_std` | 顾问按规范派生 | 内部分析 / 去重 / 匹配 | ✅ 可迭代升级（v0.1 → v0.2 等）|

**核心纪律：** `sku_name_std` 出错可重新派生；`sku_name_raw` 一旦覆盖即不可恢复。

### 8.3 商品名清洗规则表

| 清洗类型 | 客户原始可能形态 | 清洗动作 | 异常处理 |
|---|---|---|---|
| 促销词剔除 | "卫龙大面筋促销价 26g" | 剔除"促销价"等促销词 | — |
| 品牌缺失 | "大面筋 26g" | 从客户访谈或档案补"卫龙" | 无法判断 → "待确认_品牌缺失" |
| 规格缺失 | "卫龙大面筋" | 从档案补"26g" | 无法判断 → "待确认_规格缺失" |
| 单位混乱 | "卫龙大面筋 26g 1袋" / "卫龙大面筋 26g/包" | 标准化为"袋"/"包"统一选一 | 标客户原口径 |
| 同品不同名 | "卫龙大面筋"="卫龙麻辣大面筋"（实际同 SKU）| 通过 barcode 合并 | 异常清单备查 |
| 一品多码 | 同一商品有 2 个条码 | 异常清单标"一品多码" | 待确认是否合并 |
| 多品一码 | 不同商品共用 1 个条码（错码）| 异常清单标"多品一码" | 必须人工分拆 |
| 无法判断 | 商品名严重残缺（如"零食"）| 标 `待确认` | 进入异常清单 |

### 8.4 命名规范的禁止事项

- ❌ 不允许覆盖 `sku_name_raw`（原始字符必须保留）
- ❌ 不允许凭直觉编 `sku_name_std`（必须可追溯到 raw 或档案）
- ❌ 不允许在 `sku_name_std` 中含特殊符号（: * ? " < > | / \）
- ❌ 不允许使用 emoji 或全角符号

### 9.2 4 层匹配规范

| 层级 | 匹配方法 | 输出 | 适用场景 |
|---|---|---|---|
| **一级：条码精确匹配** | `barcode` 与标准品类表一致 | `match_method=barcode_exact` | 标准品（百分百直接命中）|
| **二级：商品名关键词匹配** | `sku_name_std` 含关键词（如"薯片"→ L3 膨化薯片）| `match_method=name_keyword` | 条码缺失或新品 |
| **三级：品牌+规格+口味辅助匹配** | `brand + spec + 口味` 组合命中 | `match_method=brand_spec_combo` | 同品多码 / 商品名不规范 |
| **四级：人工复核** | 顾问根据档案 + 商品名 + 现场照片综合判断 | `match_method=manual_review` | 自动化无法处理 |

### 9.3 输出匹配结果字段

| 字段 | 含义 |
|---|---|
| `barcode` | 商品条码 |
| `sku_name_raw` | 原始商品名 |
| `sku_name_std` | 标准商品名 |
| `brand` | 品牌 |
| `spec` | 规格 |
| `category_raw` | 客户原始品类名 |
| `category_l1` | 标准品类 L1 |
| `category_l2` | 标准品类 L2 |
| `category_l3` | 标准品类 L3 |
| `category_l4` | 标准品类 L4 |
| `match_method` | 匹配方法（参 §9.2 4 层）|
| `match_confidence` | 匹配置信度（高/中/低/冲突，参 §十）|
| `review_status` | 复核状态（未复核/已复核/有争议/已修正）|
| `reviewer` | 复核人代号 |
| `notes` | 备注（异常说明 / 待确认事项）|

### 9.4 匹配的执行纪律

1. 先一级（条码精确）→ 后二级（商品名）→ 后三级（组合）→ 最后四级（人工）
2. 每条匹配结果**必须**含 `match_method` + `match_confidence` 字段
3. 一级匹配命中率（条码精确）应 ≥ 70%；不达标说明客户商品档案与标准品类表偏差大
4. 四级人工复核**必须记录复核日期 + 复核人**，不允许"匿名修正"

## 十、匹配置信度规则

> ⚠️ 置信度直接决定数据是否可进入正式诊断（参 [[诊断输出质量控制规范_v0.1]] §六 证据等级）。

| 置信度 | 含义 | 诊断使用边界 |
|---|---|---|
| **高** | 一级条码精确匹配 + 商品名一致 + 品牌一致 | ✅ 可直接进入正式分析；可写入客户报告（A 级证据）|
| **中** | 二级关键词匹配 + 品牌一致 / 一级匹配但商品名小差异 | 🟡 可用于初步分析，但需标注"待复核"（B 级证据）|
| **低** | 三级组合匹配 + 信息不全 / 二级匹配但跨品类风险 | ❌ 不得进入正式结论；必须人工复核后升级（C 级证据）|
| **冲突** | 多种匹配方法给出不同 L4 / 客户原品类与标准品类表冲突 | ⛔ **必须暂停使用**；人工裁决前不得进入任何分析 |

### 10.1 置信度判断示例

| 场景 | 证据 | 置信度 |
|---|---|---|
| 商品 barcode = {{EAN13_已脱敏}}；标准品类表精确命中"卫龙大面筋 26g 袋"；L4="风味豆制品_辣条"| 一级精确 + 商品名一致 + 品牌一致 | **高** |
| 客户商品名 "卫龙麻辣面筋"；标准品类表无精确条码；按关键词命中 L3="风味豆制品" | 二级关键词 + 品牌一致 | **中** |
| 客户商品名 "辣条"（无品牌）；通过规格"30g"猜测命中"卫龙大面筋"| 三级组合 + 信息不全 | **低** |
| 客户原品类标"糖巧"；标准品类表 L1 标"风味豆制品"| 客户与标准冲突 | **冲突** |

## 十四、验收标准

### 14.1 验收检查清单（10 项）

## 十五、风险红线（8 项）

| # | 红线 | 违反后果 |
|---|------|------|
| 1 | ❌ 不覆盖原始表 | 数据完整性破坏 |
| 2 | ❌ 不删除客户原始数据（任何理由）| 客户授权风险 + 不可追溯 |
| 3 | ❌ 不擅自修正客户真实数据（清洗必须存为派生 + 记录依据）| 与原始版混淆 |
| 4 | ❌ 不把低置信度匹配当成已确认 | 诊断结论不可信 |
| 5 | ❌ 不把负毛利、负库存等异常直接删除 | 真实问题被掩盖 |
