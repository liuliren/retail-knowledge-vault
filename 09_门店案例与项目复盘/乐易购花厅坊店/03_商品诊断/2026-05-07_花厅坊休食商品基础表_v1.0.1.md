# 花厅坊休食商品基础表 v1.0（静态档案 / 全量历史）

> **定位**：休食单品**全量历史档案**（含在售 + 停售 + 历史售卖 / 12 个月以上回溯），是表 B 决策 / 表 C 结构 的上游数据源。
>
> **设计依据**：[[POS数据清洗与商品标准化规范_v0.1]] §5.2 sku_master_standard 14 字段 + 花厅坊定制扩展（货架 / 通道 / 包装实测）+ [[09_门店案例与项目复盘/乐易购花厅坊店/03_商品诊断/_品类匹配算法包_v1.0/02_辅助算法_SKU属性提取_v0.1/README|SKU属性提取算法_v0.1]] 自动化兼容。
>
> **使用方式**：vault `.md` 锁字段定义；本地 Excel 装实测数据（约 900-1200 SKU 含历史）。
>
> **更新频率**：月更（财务字段）/ 季更（供应 / 规格 / 包装尺寸）/ 实时（状态 / 货架字段）。
>
> **反幻觉**：参 [[CLAUDE.md]] §13 + [[feedback_display_layout_confirmation]]（实测优先）+ [[feedback_mvp_first_validate_in_practice]]（v1.0 起步 / 实战迭代）。

## §1. 字段定义（35 字段 / 8 维度）

> 字段命名采用英文 `snake_case`，与 [[POS数据清洗与商品标准化规范_v0.1]] / [[09_门店案例与项目复盘/乐易购花厅坊店/03_商品诊断/_品类匹配算法包_v1.0/02_辅助算法_SKU属性提取_v0.1/README|SKU属性提取算法_v0.1]] 统一，**支持自动化处理**。

### A. 商品 ID（8 字段）

| # | 字段 | 类型 | 必填 | 说明 | 来源 |
|---|---|---|---|---|---|
| 1 | `barcode` | STRING (≥8 位) | ✅ P0 | 13 位 EAN 主键 / 前导零保留 | POS 系统 |
| 2 | `sku_id` | STRING | 否 | 门店内部编码 | POS 系统 |
| 3 | `sku_name_raw` | STRING | ✅ P0 | **原始品名 / 不可修改** | POS 系统 |
| 4 | `sku_name_std` | STRING | ✅ P0 | 标准品名 / 按 POS §八 格式生成 | [[09_门店案例与项目复盘/乐易购花厅坊店/03_商品诊断/_品类匹配算法包_v1.0/02_辅助算法_SKU属性提取_v0.1/README|SKU属性提取算法_v0.1]] / 人工复核 |
| 5 | `brand` | STRING | ✅ P1 | 品牌 | SKU 属性提取 v0.1 / 99.3% 命中 |
| 6 | `spec` | STRING | ✅ P1 | 规格如 "75g" / "500ml" | SKU 属性提取 v0.1 / 93.2% 命中 |
| 7 | `unit` | ENUM | 否 P2 | 袋 / 盒 / 瓶 / 罐 / 包 / kg / g / ml / L | 人工 / spec 解析 |
| 8 | `box_pack` | STRING | 否 P2 | 箱规 如 "24 袋/箱" | 采购合同 / 经销商资料 |

### B. 品类层级（5 字段 / 与 vault 4 层标准对齐）

| # | 字段 | 类型 | 必填 | 说明 | 来源 |
|---|---|---|---|---|---|
| 9 | `category_l1` | STRING | ✅ P0 | 大类如 "休食饮料" | 标准品类表 v0.1 / 4 层匹配 |
| 10 | `category_l2` | STRING | ✅ P1 | 中类如 "休闲零食" | 同上 |
| 11 | `category_l3` | STRING | ✅ P0 | 小类（FK→ 表 C `category_l3`）/ 12 类如 "薯片" "膨化" "饼干" "坚果炒货" "果干蜜饯凉果" "糖果" "巧克力" "果冻布丁" "肉脯" "辣条豆制品" "饮料伴侣" "其他" | 同上 |
| 12 | `category_l4` | STRING | 否 P2 | 细类如 "原味薯片" | 同上 |
| 13 | `purchase_task_main` | STRING | 否 P1 | 购物任务主标签 / 权重 ≥0.5 / 来自 [[购物任务图谱_v1.1]] 休食 9 任务（解馋独享 / 办公室零食 / 聚会 / 朋友分享 / 家庭时光 / 郊游 / 年货送礼 / 儿童零食 / 中老年常备）| 人工打标 |

### C. 商品状态（4 字段）

| # | 字段 | 类型 | 必填 | 说明 | 来源 |
|---|---|---|---|---|---|
| 14 | `sku_status` | ENUM | ✅ P0 | 在售 / 停售 / 待汰换 / 待引进 / 历史归档 | 人工 + POS |
| 15 | `first_listed_date` | DATE | ✅ P1 | 进店日 (YYYY-MM-DD) | POS / 老板访谈 |
| 16 | `delisted_date` | DATE | 否 | 停售日（status≠在售时填）| POS / 决策记录 |
| 17 | `category_role` | ENUM | ✅ P1 | **vault 标准 6 类**：目标性 / 常规性 / 季节性便利性 / 形象 / 流量 / 利润；门店执行可用简化 4 类（主角 / 流量 / 利润 / 形象）| [[品类管理]] §3.3 / [[05_README]] |

### D. 财务（4 字段）

| # | 字段 | 类型 | 必填 | 说明 | 来源 |

## §3. 反幻觉红线（参 [[CLAUDE.md]] §13）

| # | 红线 | 违反例 | 正确做法 |
|---|---|---|---|
| 1 | 财务字段必须 POS 导出 | "凭记忆填进价" | POS 导出 / 供应商对账双源 |
| 2 | 货架号 / 通道 / 层 / 排面 **必须现场实测** | "估值 75g 薯片放第 3 层" | 现场量货架（[[feedback_display_layout_confirmation]]）|
| 3 | 包装尺寸必须实测 | "典型包装假设" | 卷尺实测 / 包装标注 |
| 4 | 供应商账期 / 配送频次 必须 sign off | "顾问推断" | 老板 / 财务 sign off |
| 5 | `sku_name_raw` **永远不修改** | "改成标准名好看" | raw 留 / std 派生 |
| 6 | `sku_status = 历史归档` 的 SKU **保留不删** | "清理表格删除停售" | 是淘汰复盘的依据（[[单品管理]] §10）|
| 7 | `match_confidence = 冲突`时**禁止用于决策** | "凑合用" | 人工裁决前不进表 B / 表 C |

## §4. 数据装载工作流

```
S1 POS 一次导出（全量含历史归档）
   → 字段 1/3/9/14/15/18/19 P0 全覆盖
   ↓
S2 SKU 算法 v0.1 自动提取
   → 字段 4/5/6 自动生成 + match_confidence
   ↓
S3 现场实测（首日）
   → 字段 25/26/27/28 货架 4 字段全填
   ↓
S4 标准品类匹配（POS §九 4 层）
   → 字段 9/10/11/12 + 33/34 治理标记
   ↓
S5 老板访谈（首周）
   → 字段 17/22 + 23/24 供应字段
   ↓
S6 SKU 打标（首周）
   → 字段 13 购物任务主标签
   ↓
S7 包装实测（按需）
   → 字段 29-32 P2 字段
```
