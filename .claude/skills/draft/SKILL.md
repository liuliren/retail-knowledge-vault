# /draft · 媒体稿起草引擎 Skill v0.1

> **一句话**：从知识库取料，用六哥的声音起草，草稿本身回流知识库——让输出成为输入的新起点。

---

## 输入

```
/draft <主题或选题>
/draft <主题> --format <公众号|视频号|小红书>
/draft <主题> --angle <经营者视角|行业观察|方法论>
```

默认格式：公众号长文（1500-3000字）

---

## 六哥写作铁律（必须贯穿全稿）

**风格约定**：
- **结论先行**：第一段必须是最核心的观点，不做铺垫悬念
- **MECE 结构**：分点必须互斥穷尽，不做"一二三四五……"流水账
- **数字说话**：有数据用数据，没数据给具体场景，不用"很多""大量"
- **可执行**：每个观点后面必须有"所以你该怎么做"的动作指引
- **真实感**：引用知识库中的真实案例（花厅坊等，脱敏处理），不造假

**禁止清单**（违者重写）：
- ❌ 空泛励志："在这个竞争激烈的时代……"
- ❌ 正确废话："要做好品类管理，首先要了解品类……"
- ❌ 列举式填充："第一点……第二点……第三点……"（无内在逻辑的罗列）
- ❌ 无根据断言："据研究表明……"（没有来源的引用）
- ❌ 被动受害者视角："老板们都不重视……"

## 执行步骤

### Step 1 · 理解选题与确定角度

- 如果六哥给了明确主题：直接进 Step 2
- 如果主题模糊：用一句话确认"你是想写【X视角的X主题文章，核心读者是X，最想让读者做到X】吗？"等确认后进 Step 2

### Step 2 · 从知识库取料

**搜索相关 KB 页**：
```bash
grep -rl "<主题关键词>" \
  "/Users/davidliu/KnowledgeBase/retail-knowledge-vault/01_科学零售方法论" \
  "/Users/davidliu/KnowledgeBase/retail-knowledge-vault/05_品类管理与商品规划" \
  "/Users/davidliu/KnowledgeBase/retail-knowledge-vault/14_外部案例与行业研究" \
  2>/dev/null | head -10
```

**读相关 M-DEC**（有实战数据）：
```bash
grep -rl "<主题关键词>" \
  "/Users/davidliu/KnowledgeBase/retail-knowledge-vault/16_客户与战役档案" \
  2>/dev/null | head -5
```

**读对应研究文章**（外部数据来源）：
```bash
find "/Users/davidliu/KnowledgeBase/retail-knowledge-vault/14_外部案例与行业研究" \
  -name "*.md" | xargs grep -l "<关键词>" 2>/dev/null | head -5
```

读取 3-5 份最相关的文件，提取：
- 核心论点（直接用，不改写）
- 关键数据（原始数字）
- 案例（脱敏后引用）

**若知识库无覆盖需要外部调研**（2026-07-07 补入，源自"写作系统"CLAUDE.md v2.0 merge）：
- 优先信源：权威零售媒体（Retail Dive、NRF、联商网、亿邦动力网、第一财经商业数据中心、零售老板内参）；官方财报/研究院报告（商务部研究院、招商证券等券商研报）
- 完全忽略：知乎/百度个人经验帖、超过半年的供应链/即时零售趋势旧文、营销软文
- 同一数据至少两个来源交叉验证，标注引用时间

### Step 3 · 构建文章骨架

在起草正文前，先输出骨架让六哥确认（3-5行即可）：

```
骨架：
- 核心论点：[一句话]
- 结构：① [第一板块] ② [第二板块] ③ [第三板块]
- 关键案例：[脱敏后案例名]
- 行动指引：[读者看完应做的1件事]
确认后我起草正文。
```

若六哥说"直接写"，跳过骨架确认。

### Step 4 · 起草正文

**公众号格式**（默认）：
```
[标题：疑问式或数字式，20字以内]

[导语：1-2句，直接给结论或反直觉的观点]

[主体1-3个板块，每板块：论点→证据→动作]

[收尾：1句话，让读者知道该做什么]
```

**视频号/小红书格式**（短）：
```
[钩子：第一句必须能让人停下来]
[核心：1-3个要点，每点1-2句]
[行动：做一件具体的事]
[互动：一个问题引导评论]
```

### Step 4.5 · 三遍审校（降AI味·2026-07-07 补入，源自"我的写作工作区"CLAUDE.md v1.0 merge）

正文起草后、进隐私检查前，必须过三遍：

**第一遍·内容审校**：数据准确？逻辑无矛盾？无编造案例？论据充分？

**第二遍·降AI味**（核心）：
- 删AI套话：「在当今XXX的时代」「综上所述」「值得注意的是」「众所周知」「不可否认」「首先…其次…最后」连用
- 拆AI句式：「不仅…而且…」连用、「一方面…另一方面…」、「通过…实现…」、把动词名词化（「进行分析」→「分析」）
- 换书面词：「显著提升」→ 给具体数字；「充分利用」→「用好」；「深入了解」→「搞清楚」
- 改口语化：「对此进行了优化」→「改好了」
- 加真实细节：抽象表达换成具体数字/场景（「效率显著提升」→「原本一天的活，现在3小时搞定」）
- 加个人态度：不要「客观中立各打五十大板」，明确站边（「这方法确实好用，但有个问题很烦人」）

**第三遍·细节打磨**：句子15-25字为主不超30字，超长必拆；段落手机屏3-5行为佳；标点多用句号少用逗号连长句。

### Step 5 · 隐私检查（硬拦截门·2026-07-07 P7-3红队测试P0/P1修复后升级）

起草完成后，自动检查：
- 有无客户真名未脱敏（名单动态取自注册表，非硬编码——见下）
- 有无进价/条码等 raw 数据
- 有无未经验证的数据引用

**名单来源不再硬编码**：真名词表从 `.claude/客户代号注册表_LOCAL.txt` 动态取（格式 `真名=代号`），新增客户自动生效，无需改本 skill。

```bash
# 检查真实草稿文件（用 Step 6 写入的实际路径替换 DRAFT_FILE）
DRAFT_FILE="retail-knowledge-vault/98_AI协作中枢/01_Claude_Code/Claude输出区/YYYY-MM-DD_draft_<slug>.md"
REG="retail-knowledge-vault/.claude/客户代号注册表_LOCAL.txt"
test -f "$DRAFT_FILE" || { echo "⛔ 草稿文件未找到——不得进 Step6，先确认落盘路径"; exit 1; }

NAMES=$(grep -v '^\s*#' "$REG" 2>/dev/null | grep '=' | cut -d= -f1 | grep -v '^\s*$')
HIT=0
while IFS= read -r name; do
  [ -z "$name" ] && continue
  grep -q "$name" "$DRAFT_FILE" && { echo "🔴 命中客户真名: $name"; HIT=1; }
done <<< "$NAMES"
grep -nE "进价|条码" "$DRAFT_FILE" && HIT=1

[ "$HIT" = "1" ] && echo "⛔ 隐私检查未过——见下方硬拦截规则，禁止进 Step6" || echo "✅ 隐私检查通过"
```

> **硬拦截铁律**：草稿必须先落盘（Step 6 写入 Claude输出区）才能检查，不接受 echo 占位符替代真实文件扫描。**若命中客户真名，必须先查 `David-Liu-Vault/80_个人资料_敏感/客户名称对外使用授权链清单.md`：该代号 + 本次内容场景（正面/批评·渠道）标 ✅ 且覆盖 → 可保留真名，记一行依据；否则一律脱敏泛化后才能进 Step 6。命中进价/条码 → 一律先删除/汇总化。这是硬拦截，不是提示——命中且未过授权链核验 = 不得写入 Step 6。**

### Step 6 · 写入输出区并登记为 source

**写入草稿**：
`retail-knowledge-vault/98_AI协作中枢/01_Claude_Code/Claude输出区/YYYY-MM-DD_draft_<slug>.md`

**回流登记**（产物本身成为新 source）：
追加一行到内容消化台账：
`KnowledgeBase/_内容消化台账.md`

格式：
```
| YYYY-MM-DD | draft | <标题> | <主题标签> | output_source | 待发布 |
```

### Step 7 · 停在签字门

输出草稿后说：
> 草稿完成，等六哥：① 审阅内容 ② 确认脱敏 ③ 签字后发布（D档）。
> 发布路径：草稿 → doocs/md + 纽约客CSS → 公众号后台 → 六哥签字发出。
