# /深度精读 skill（逐字精读引擎 · 持续可续跑）

> **这是什么**:把"读 RESUME→定位原文→逐篇全文精读→出精读卡→更进度→更 RESUME→精确 commit"固化成一条可重复、可自动续跑的回路。区别于 [[ingest]](命题级编译·一次性消化)——本 skill 是**逐字全文精读**,密度更高,服务"/goal 真精读门控"。
> **护城河定位**:精读卡=Harness(会随模型贬值);**回填进正文 + 升 active = 驭马人层(六哥签字,不贬值)**。本 skill 只产 candidate/draft,**回填与升级一律停在签字门**(CLAUDE.md §3②)。

## 何时用
- 六哥说"续读/接着读/精读下一批/深度精读 X/读语料"。
- 当前 canonical 工程:**零售老刘 321 篇逐字精读**(进度见 `14_外部案例与行业研究/零售老刘体系/逐字精读卡片集/_逐字精读进度_321.md`)。
- 未来扩展:任何长语料系列(万维钢、其他公众号系列、课程转录)套同一回路,只换"工程三件套"路径。

## 项目章程（零售老刘逐字精读 · canonical · 项目模式）
> 把整件事当**一个项目**跑,而非一批批审批。六哥 2026-06-26 定调。
- **🎯 目标**:零售老刘 321 篇**全量逐字精读 + 强候选回填进 B 层方法论**(/goal 门控:完成前不进 G4-A 交付)。
- **📏 标准(质量门)**:每篇读全文(非臆测)→ 精读卡含核心论点(公式/数据/判据)+关键词+星级;回填带溯源标、保真六哥声音、结论先行。
- **🗺 计划**:按有序索引序号、6篇/批(定量技术系列降速3-4篇)逐批推进;每批=读→自动回填上批强候选→出卡→更进度表+RESUME→精确commit。
- **⚙️ 实施**:`/loop /深度精读` 自配速、**全程不打断**、commits 即审计轨迹。
- **🖊 会签点(只两个·末尾集中)**:① 升 active(攒进 RESUME 待签清单,工程完+客户实证后一次过签);② 对外发布/删除(G4-A 等·工程完后)。回填=已授权自动,不算会签点。

## 铁律(每次必守)
1. **断点续传,不重读全库**:开机只读"工程三件套"(进度表 + 有序索引 + RESUME),定位下一批,不回头重扫已读。
2. **逐字全文真读**:每篇读全文(原文常 >2KB,用 Bash cat 落 persisted-output 再 Read),核心论点须是**读后提炼**,不得据标题臆测。
3. **序号对齐以"系列内编号"为准**:有序索引 txt 的序号列可能漂移(如零售老刘"品类建模217缺号"致 ~1 偏移)。**按文章自身系列编号(库存管理系列05/06…)对齐,不认 txt 行号**;遇主题切换在卡片标注。
4. **节奏自适应**:连贯叙事系列可 3批/轮提速;**全新/定量/技术主题(如库存管理、数据分析)放慢到单篇细读或 1 批(6篇)/轮**,质量优先。
5. **回填自动 + 签字门不中途停,累积「待签清单」一次性审核**(2026-06-26 六哥指令改造):
   - **回填自动**:六哥已给逐字工程**标准「先回填」站定授权**——loop 每轮开头**自动**把上批强候选回填进 B 层正文(带溯源标),**无须每批停下等签**。
   - **签字门不中途跳出**:升 active / 对外发布 / 删除 这三类 §3② 签字门,**loop 一律不中途停**;而是把它们**累积进 RESUME 的「📋待签清单」**(标好目标页+理由),继续往下读。
   - **末尾一次性审核**:loop 自然收尾(整个工程读完)、或六哥主动回来时,**一次性呈交「待签清单」让六哥批量审核签字**(逐条 ✅/✗)。**绝不代签**——签字仍是驭马人层,只是把"打断式逐次确认"改成"攒一批末尾一次审"。
   - 精读卡里仍列"回填候选",但状态从"待签"变为"已自动回填/待升active"。
6. **精确 git add,不 push**:只 add 本轮动的精读文件(卡片 + 进度表 + RESUME),不裹挟工作区其他工程的未提交改动;commit 不 push。
7. **收口必更两表**:每轮结束**覆盖更新 RESUME** + 更新进度表(已读数/批次行/下一批占位)。

## 步骤

### Step 0 — 定位(只读三件套)
1. Read RESUME → 看"主线工程"区:上次到哪、下一批起点、上批回填候选(是否已签"先回填")。
2. Read 进度表 tail + 有序索引对应段 → 确认下一批起读的**系列编号**。
3. `find /Users/davidliu/KnowledgeBase/Clippings/零售/零售老刘 -name "*<系列>*"` 定位原文文件(2026-06-26 已迁中枢)。

### Step 1 — (若上批已签"先回填")先回填
六哥若已签字,把上批卡片"回填候选"逐条写进对应 B 层方法论页正文(保真六哥声音、标 source_attribution),并在 RESUME 标记已回填。**未签则跳过。**

### Step 2 — 逐篇全文精读
按节奏读本批 N 篇(`cat` 多篇落 persisted-output → Read)。每篇提炼:**核心论点**(读后整理·含关键公式/数据/判据)+ **关键词**(检索锚点)+ 星级(⭐⭐⭐ 撬动性强/⭐⭐ 有用/⭐ 一般)。

### Step 3 — 出精读卡(固定模板)
新建 `批NN_<主题>_v0.1.md`,frontmatter:`id: KB-LLL-CARD-BNN` / `status: draft` / `source_attribution` / `summary:`(≤40字·密度高·覆盖全批要点) / tags。正文结构:
```
# 逐字精读卡片集 · 批NN <主题>
> 📌 本批主线/主题切换提示
## <序号>. <系列编号> <标题> ⭐⭐⭐
- **核心论点**:…(读后提炼·公式/数据/判据/反误区)
- **关键词**:A / B / C
…(逐篇)
---
## 回填候选(待签·下轮「先回填」=§3②六哥签字后入正文)
- **⭐⭐⭐ <命题>** → [[目标方法论页]](理由/呼应)
## 关联
[[_逐字精读进度_321]] · [[上一批卡]] · [[相关方法论页]]
```

### Step 4 — 更进度表
- 顶部"已精读:N/总数"+百分比。
- 表里把"NN-"占位行替换为本批完成行(范围/篇数/卡片 wikilink/✅)+ 新占位行。

### Step 5 — 覆盖更新 RESUME（含「待签清单」）
- "上次到哪"加一行日期+本批摘要;主线工程区更新进度数、下一批起点、本批回填候选。保持 ≤1 屏。
- **维护「📋待签清单」**:本轮若产生新的升 active 候选(某方法论页回填已成体系、够格升级)/对外发布/删除需求,**追加进 RESUME 的待签清单**(标目标页+理由),**不中途跳出找六哥**。

### Step 6 — 精确 commit(不 push)
```bash
git add "<批NN卡片>" "<进度表>" "<RESUME>"
git commit -q -m "逐字精读 批NN(<系列范围·要点>)+进度N/总数+RESUME收口"
```
> 只 add 这三类文件,核对 `git status` 不误带其他工程。

### Step 7 — 项目模式:不每批回报（六哥 2026-06-26 指令）
**默认项目模式**:loop 续跑期间**不每批向六哥汇报**、不问"要不要接着跑"——直接进下一批。**commits + 进度表 + RESUME 就是审计轨迹**,六哥要看进度自己翻这三处即可。**只在两种情况浮出**:① 整个工程读完(Step 8 出全量会签包);② 真阻塞(原文缺失/索引错乱/judgment 拿不准)。原则:明确目标→制定标准→确定计划→逐步实施→保证质量→**减少干预和审批环节**。

### Step 8 — 末尾一次性审核（仅 loop 收尾 / 六哥主动回来时）
- 触发:整个工程读完,**或**六哥主动回来喊停/查看。
- 动作:**一次性呈交 RESUME「📋待签清单」**全量,让六哥逐条 ✅/✗ 批量签字(升 active / 对外发布 / 删除)。**绝不代签**。
- 这是把"中途逐次打断确认"改成"末尾攒一批一次审"(2026-06-26 六哥指令)。

## 自动续跑(六哥要的"自动运行")
本 skill 单次跑一批。要连续多批无人值守:
- **自配速 loop(推荐·当前在用)**:`/loop /深度精读`(无间隔=动态自配速)—— 每轮自己读下一批、出卡、**自动回填**、commit,用 ScheduleWakeup 自动唤醒下一批,**全程不打断六哥**。升 active/发布/删除**不中途停**,累积进 RESUME「📋待签清单」,**末尾一次性审核**(见 Step 8 + 铁律5)。
- **定时**:`/schedule` 起一个 cron 云agent 每日跑 1-2 批(同样不中途停·累积待签),适合"睡觉时也在替我读"。
- **节奏护栏**:定量/技术系列自动降速单篇细读;连贯系列可提速。**门控**:零售老刘逐字工程完成前,G4-A 客户交付保持锁定(见 RESUME)。
