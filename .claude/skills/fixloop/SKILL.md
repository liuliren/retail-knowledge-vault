# /fixloop — 系统优化/测试/迭代/修复/审计验收 闭环

> **一句话**：拿到一份"审计发现的问题清单"，别急着改——先核实证据、按可逆性分档，A/B 档直接修完再报告，C/D 档才停下来问；改完必须用同样的检测方式复验，不清零就缩小范围再来一轮。

## 何时用 / 何时不用

- **用**：`/lint` 或任何审计/体检产出了一份问题清单（断链、孤儿页、schema 缺项、代码 bug、脚本回归失败等），需要批量修复并验收。
- **不用**：单文件的一次性小改（直接改，不必走全流程）；涉及"标准/铁律"文件升级的判断性审议（走 `/critique` 三轮审议，不是本 skill 的范围）；对外发布决策（走 D 档签字，本 skill 只负责把"待发布"这个动作本身列进验收清单，不代替发布签字）。

## 核心原则（六哥 2026-07-07 指令）

> **中途非必要不用签**——判断门槛是**可逆性**，不是"看起来重不重要"。可逆 + 增量修复 = 直接做（A/B档，做后报）；只有真正不可逆或跨边界的动作（C/D档）才停下来找六哥。别把"我不确定"也当成"必须问"——先自己按下面的判据分类，分不清楚才问。

## 执行五步

### Step 0 · 锁定范围与核实证据（禁止盲信报告数字）
- 输入必须是一份**已存在的扫描/审计产出**（lint 报告、测试失败日志、bug 清单等），不是本 skill 现场瞎猜。
- **报告数字只是线索,不是事实**：动手前用 grep / 实际运行逐条核实真实的引用位置、出现次数、上下文。上一轮实跑中，报告称"133 次"M-DEC 断链，实核 127 处可修 + 6 处历史证据例外 + 1 处归档区豁免——数字对不上是常态，以现场核实为准。
- 分辨"真问题"与"证据性文本"：审计/设计稿里用反引号 `` `[[X]]` `` 或表格单元格引用问题本身作为例子（描述 bug 的文档），不是需要修的活链接——修了反而抹掉审计记录。判断标准：这行文字的作用是"导航到 X"还是"讨论 X 这个问题"？后者不动。

### Step 1 · 按操作四档分类（判据 = 可逆性，不是重要性）
沿用全局宪法《AI 操作四档执行规则》：

| 档位 | 触发条件 | 本 skill 里的典型例子 | 执行方式 |
|---|---|---|---|
| A 即执行 | 可逆 + 单文件/单字段增量 | 修一条断链、补一个 frontmatter 字段 | 直接做，事后一句话带过 |
| B 做后报 | 可逆 + 多文件批量/跨目录 | 批量改同一模式的断链（如本次 127 处 M-DEC 短码） | 执行后列结构化报告（文件清单+处数+复验结果） |
| C 列清单确认 | 不可逆（移动/覆盖/重命名）OR 跨 vault OR 架构调整 | 目录级重命名、跨 vault 迁移文件 | 先列清单 → 六哥确认 → 执行 |
| D 强制签字 | 删除 OR 改宪法/schema OR 对外发布 OR 动 T2/T3 | 物理删除文件、status 升 active/stable、发布对外内容 | 提案 → 六哥签字 → 执行 → 记 log |

**分档产出**：一张"问题→档位"映射表，A/B 档直接进 Step 2，C/D 档单独列出等六哥回应（不阻塞 A/B 档先跑）。

### Step 2 · 修复执行（A/B 档）
- **禁止无脑批量替换**——每一类修复动手前，先用 grep 看几条命中样本的上下文，确认"改的方式"对（如本次用 Obsidian 别名语法 `[[完整文件名|显示文本]]` 保留原显示文本，而不是粗暴替换整行）。
- 用脚本/perl/sed 批量执行时，**排除已识别的例外文件**（历史证据、归档区、只读客户数据区）。
- 每个类别修完立刻用同样的检测方式局部复验一次，别攒到最后才发现方法本身有误一路错到底。

### Step 3 · 复验（用原始检测方式重跑）
- 用 Step 0 发现问题时**同一套命令**（同样的 grep 正则 / 同一个 lint 脚本 / 同一组测试）重新跑一遍。
- 期望：目标问题数归零或只剩 Step 0 已判定保留的例外；**若和例外清单对不上，说明分类判断有误，回 Step 1 重新审视，不能强行合理化**。

### Step 4 · 迭代（未清零 / 有新发现则收窄范围重来）
- 复验中发现遗漏或新暴露的问题（比如改断链时顺带发现另一类命名不一致），不在本轮范围内强行处理——单独记一条，缩小范围开下一轮，不无限扩大本次改动面（呼应宪法 §7"聚焦纪律：少而深"）。

### Step 5 · 审计验收报告
产出结构化报告（默认落 `98_AI协作中枢/01_Claude_Code/Claude输出区/`），必含四段：
1. **修复量化**：按问题类型列"发现数 / 实修数 / 例外数(附理由) / 涉及文件清单"。
2. **例外清单**：每条例外必须有**可复核的理由**（不是"我觉得不用改"），例如"该行为审计文档引用问题本身作为证据，非活链接"。
3. **复验结果原文**：贴上 Step 3 重跑命令的**原始输出**，不是转述"已确认无误"（呼应 verification-before-completion：证据先于断言）。
4. **待决事项**：C/D 档清单（如有），逐条附"为什么这条到不了 A/B 档"的一句话理由，等六哥批。

## 中途签字纪律（本次六哥明确澄清）

- A/B 档：**不中途打断**，跑完整个五步再一次性报告结果。
- C/D 档：即使被夹在同一批问题里，也要单独摘出来，不能因为"顺手"就跟着 A/B 档一起做掉。
- 报告本身不是签字请求——除非报告里明确列了 C/D 档待决项，否则六哥读完即视为知悉，不需要再确认"可以吗"。

## 红线

- 不碰 `99_归档/`、门店原始数据、xls/csv、`Clippings/`（除非任务显式指定）。
- 不用 `sed -i` 之类无差别全库替换；必须先圈定文件清单再逐类执行。
- 不因为"看起来像同一个问题"就跨过 Step 0 的证据核实直接开修——审计文档引用 bug 本身作为例子 vs 真断链，肉眼一致，语义不同，见上次踩过的坑。
- 不在 Step 5 报告里省略复验原始输出——只写"已验证"不算完成（verification-before-completion 铁律）。
