# /critique — 批判审议

> **一句话**：给每件待审文件派一个不认识作者的"魔鬼代言人"，带着四问和反例任务去拆它；拆不动的才配升级。

## 何时用 / 何时不用

- **用**：draft 标准/框架/SOP 进第 3 轮前的第 2 轮审议；六哥对任何方案说"批判一下"；外部 Prompt/方案采纳前的对抗检验。**第 3 轮行业/专业视角审议、或对象涉及多方利益时 → 用 `--steelman` 模式（见下）**。
- **不用**：纯事实性文件（台账/日志/数据表）；已 stable 件的日常小修（走正常修订）；<50 行的小 note（用 lite 版：**独立单 Agent** 四问，不开多 Agent——仍须独立上下文，不得用携带起草上下文的主线程；对齐审议细则 §1 lite 口径）。

## 执行五步（细则 §3 的操作化）

### Step 1 · 锁定与设角
- 确认对象文件清单（1-N 件）；每件派**一个独立 general-purpose Agent**，多件并行发出。
- Agent prompt 必含：角色=批判/魔鬼代言人；**不携带起草上下文**；原文件绝对路径。

### Step 2 · 四问模板（每个 Agent 必答）
1. 这套规则/公式/流程在**什么具体场景下产生错误结果**？（给反例，不给"可能"）
2. 与库内**现有规则的逻辑张力**？（指名文件，grep 验证引用真实性）
3. **遵守成本**是什么？有没有更轻量的替代方案？
4. **五年后还成立吗**？（哪些是耐久内核、哪些是速朽参数——参考 90/10：马具会贬值）

### Step 3 · 反例与实算（硬纪律）
- ≥3 个边界失效案例；
- 凡公式/权重/阈值：**代入数字实算**（文字自洽≠数字自洽）；
- 凡引用：grep 回源验证出处真实、原义未被扭曲；
- 凡"自称已修"（双 Pass/自检）：逐条复核是否真修干净。

### Step 4 · 判定分级
`放行 / 有条件放行（必修项 R1…）/ 打回（阻断项 R1…）/ 降格重定位` —— 判定针对文本不针对方向；每条判定带行级证据。

### Step 5 · 汇总裁决单
全部 Agent 回传后，主线程出一份汇总：判定总表 + **跨文件发现**（同词异义/构念重叠/互引死锁等，往往比单件修订更值钱）+ 依赖拓扑修订排期（先修被引者）+ 六哥裁决点编号。

## Agent Prompt 模板（复制改路径即用）

```
你是《标准审议铁律》第2轮的批判角色（逆向/魔鬼代言人）。对象：<绝对路径>
任务：通读全文，写第2轮审议记录。必答四问：①什么场景产生错误结果（给具体反例）②与库内现有规则的逻辑张力（指名文件并grep验证）③遵守成本与轻量替代 ④五年后还成立吗。
另外：找≥3个边界失效案例；公式/阈值代入实算；引用回源验证；"自称已修"逐条复核。
输出：写入 <输出区审议目录>/<日期>_第2轮批判审议_<文件ID>.md
frontmatter：status: draft、审议轮次: 2、角色: 批判-魔鬼代言人、summary。
正文：结论先行（放行/有条件放行/打回/降格）→ 四问逐答 → 反例清单 → 编号修订建议。
纪律：绝不修改原文件；不读 xlsx/csv/99_原始素材/_client_private；每条批判给行级证据。
```

## `--steelman` 模式 · 利益相关者对抗式审议（2026-07-08 六哥授权新增）

> **目标**：不是模拟固定角色，而是根据审议对象**动态选择真实利益相关者**，通过利益冲突发现隐藏问题。默认模式（上文五步·魔鬼代言人）拆逻辑；`--steelman` 拆利益——两者互补，不互替。Steelman 立场：先把对象最强的一面说透，再指必须补的洞。

### S1 · Stakeholder Selection Engine（选角引擎）

- **必须覆盖四类**：① 买单者（客户/老板——谁为此付钱）② 使用者（店长/学员/采购/一线——谁拿它干活）③ 专业挑战者（同业最强对手视角，steelman 本体）④ 系统维护者（知识管理/可复用性——谁替它长期还债）。
- **可按项目增加**：⑤ 风险控制者 ⑥ 终端用户（顾客）。
- 角色数量 **4–6 个**；**禁止固定角色模板**——每次按对象重新选角，角色必须是该文件的真实利益相关者，且角色处境要具体到能产出通用角色给不出的意见（实例：行话准则审议选 头部咨询顾问/区域商品总监/粤语背景个体老板/零售培训专家/知识管理负责人，"老板只有流水账没有损益表"这类洞见即来自处境具体性）。

### S2 · Evidence Anchoring Rule（证据锚定铁律）

每个角色每条意见必含**五要素**，缺一即视为空转：

`代表谁 | 核心利益 | 反对点 | 证据来源（文件/行号/实例，grep 可验） | 建议动作`

- 禁止无证据的泛泛评价（"可能不够落地"类语句一律打回重写）；
- 角色意见**先独立成段、后合议**；分歧显式列出，禁止和稀泥式折中。

### S3 · 固定输出模板（六节）

1. **角色矩阵**：角色 | 代表谁 | 核心利益 | 一句话总评
2. **关键冲突**：利益相互顶牛处（往往比单条修订更值钱，对应默认模式 Step 5 跨文件发现的地位）
3. **分歧裁决**：合议——采谁弃谁 + 理由
4. **修改建议**：编号、可执行
5. **stable/draft 判断**：明确判定；若否，列差什么
6. **待决事项**：≤6 条，每条带建议

### Agent Prompt 模板（--steelman，复制改路径即用）

```
你是《标准审议铁律》第3轮的利益相关者对抗审议执行者（steelman 模式）。对象：<绝对路径>
选角：按 S1 规则为本对象动态选 4-6 个真实利益相关者（必含 买单者/使用者/专业挑战者/系统维护者，可加 风险控制者/终端用户），禁用通用角色模板，每个角色写明具体处境。
每角色独立给意见，每条意见五要素齐全：代表谁|核心利益|反对点|证据来源(文件/行号，grep可验)|建议动作。禁止无证据泛泛评价。
Steelman 纪律：先把对象最强的一面说透，再指必须补的洞。
输出按 S3 六节模板（角色矩阵/关键冲突/分歧裁决/修改建议/stable-draft判断/待决事项≤6条带建议），写入 <输出区审议目录>/<日期>_第3轮steelman审议_<文件ID>.md
frontmatter：status: draft、审议轮次: 3、角色: steelman-利益相关者合议、summary。
纪律：绝不修改原文件；不读 xlsx/csv/99_原始素材/_client_private；判定建议不等于裁决，裁决权在六哥。
```

## 出口纪律（红线）

- **原文件零改动**——审议只产记录，修订是裁决后的另一动作；
- **裁决权在六哥**——skill 产出判定建议，放行/打回由六哥签；判断相不下放（模型分层调度红线）；
- **自检不算轮次**——本 skill 的运行本身才构成一次正式轮次（细则 §1）；
- 审议记录落输出区审议目录，汇总单登缺口总账对应行。

## 复盘 D + 迭代 E（Skill 闭环铁律）

每次运行后追问两句并回写本文件：①哪个 Agent 的批判空转了（四问答得泛）→ 收紧该问的模板措辞；②有没有跨文件发现漏网 → 补 Step 5 检查角度。lessons 记于下方：

### Lessons
- 2026-07-08 `--steelman` 模式原型跑（行话准则第3轮·五角色）：动态选真实利益相关者产出通用角色给不出的洞（粤背景老板→"损益表个体户没有"、商品总监→"客户总部≠同行，报告会向下漂移"、知识管理→抓出三色标签 SSOT 指针失真）；据此六哥授权将模式固化入本 skill（不新建独立 skill，边界=只改本文件）。待下次实战验证 S1 选角规则可复制性。
- 2026-07-02 首跑（5件）：并行5 Agent 全部产出行级硬伤，零空转；最大收益点=Step 3 的"实算+回源"两条硬纪律（抓出锚点算不拢与引用出处标错）；Step 5 跨文件发现（store_type 同词异义窗口期）价值高于任何单件修订——保留为必做步。
- 2026-07-03 单件跑（品类表验收框架 v0.1）：单件也值得开独立 Agent——三条最硬发现全是"带起草上下文者不会去查"的角度：抽查概率实算（抽2/6维仅1/3命中虚报维）、两个 stable 上游互斥（三张表 vs GBA-001 对中式糕点一贬一褒且转引丢"月饼除外"）、流程条款撞上位宪法（§5 跳 candidate 违 §11.3）。沉淀两条可复用批判角度进 Step 3：凡含"抽查/采样"设计→必做命中概率实算；凡引用 ≥2 个 stable 上游→必做上游互斥检查。
