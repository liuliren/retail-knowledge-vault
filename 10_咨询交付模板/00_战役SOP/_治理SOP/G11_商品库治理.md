# G11 商品库治理审计 SOP v0.2

> **本 SOP**：vault 第 5 个月度审计 / 商品库层（与 G07/G08/G09/G10 协同）。
>
> **v0.2 升级**：加错误处理机制 7 项（采纳 ChatGPT 控制点 4）。
>

## §1. SOP 总流程（5 步 / 共 30 分钟）

```
Step 1（5 min）：扫商品库 SKU 总数 + 三层分布 + 双层 raw/mapped 一致性
Step 2（10 min）：去重检测（同 sku_id / 同 barcode / 同 standard_product_name）
Step 3（5 min）：FK 完整性（category_l4 → V4.0 / brand_alias_id / package_unit / 规则 target_id）
Step 4（5 min）：跨客户匿名化 + 错误处理 7 项检测
Step 5（5 min）：沉淀月度报告 + [[Vault健康仪表]]
```

## §3. 错误处理机制 7 项（v0.2 ⭐⭐ ChatGPT 控制点 4）

详细处理逻辑见 [[商品库_v0.1/03_商品库维护SOP]] §6。本 SOP 月度审计扫描的关键点：

### 3.1 重复 SKU 合并

```
扫描：A 层 sku_id 唯一 + barcode 重复检测 + standard_product_name 重复检测
处理：6 哥 sign off 合并方案（保留高 source_confidence）
记录：update_log.md + status = archived（不物理删除）
```

### 3.2 错误品类回滚

```
扫描：mapped 层 category_l4_code 与 raw 客户原始品类差异 > 阈值
处理：人工修正 mapped 层 / 不动 raw / 留 audit log
特殊：V4.0 升级时商品库批量重新分类 / 升级前 backup
```

### 3.3 人工修正记录

```
扫描：manual_review_status 变化的 mapped 行 / manual_reviewer 必填
处理：update_log.md 文本记录（修改前 → 修改后 / 原因）
原则：不可物理删除 / 保留 audit log
```

### 3.4 规则冲突处理

```
扫描：keyword_rule.csv 同 pattern 的多条规则
排序：priority 高 → confidence 高 → 否则 status = pending_review
裁决：6 哥 / 顾问 sign off
```

### 3.5 客户独有商品入库判定

```
扫描：mapped 层 feedback_to_library = true 的候选
判定 4 项（必须全过）：
  ✅ 商品有标准条码（非客户内部码）
  ✅ 商品在多个客户战役出现（≥ 2）
  ✅ 商品在中国零售普遍可见
  ✅ 6 哥 + 启明 sign off
```

### 3.6 跨客户复用边界

```
✅ 允许跨客户：标准 SKU（A）/ 品类骨架（B）/ keyword_rule / brand_alias / package_unit
❌ 不允许跨客户：销售数据 / 毛利数据 / 库存数据（脱敏后才聚合）/ 客户内部码 / 门店级判断 / 客户特殊业务
```

### 3.7 脱敏规则

```
必须脱敏：
  - client_internal_code → 不上 git
  - 销售金额 / 毛利 / 库存 → 转相对值或区间
  - 客户名 / 门店名 → 用 client_id（hcp/lyg/zq）替代
  - 客户内部联系人 → 仅在 16_/ 战役档案 / 不入商品库
```

### 6.5.1 git 跟踪策略合规检查（4 项）

| # | 检查项 | 目标 | 异常处理 |
|---|---|---|---|
| 1 | **结构层 CSV 是否已被 git 跟踪** | A_core_sku_seed / B_category_skeleton / _客户SKU模板 / _maintenance/* 共 6 文件全部 staged | 缺失 → 检查 .gitignore 白名单 |
| 2 | **客户 raw/mapped 是否仍被 .gitignore 忽略** | git check-ignore 命中 **/*.csv 规则 | 失效 → 立即修复 .gitignore |
| 3 | **git status 是否存在误加入客户敏感数据** | grep _raw.csv\|_mapped.csv 应空 | 命中 → git reset + 沉淀 [[工程操作错误案例库]] |
| 4 | **.gitignore 白名单是否被异常扩大** | 6 条精确白名单 / 不应有通配符 | 异常 → 6 哥 sign off 才允许扩大 |

### 6.5.2 客户经营数据 git 同步风险检查（2 项）

| # | 检查项 | 目标 |
|---|---|---|
| 5 | **是否存在用 git 同步客户经营数据风险** | 走加密备份 / 私有云 / 不入 git |
| 6 | **是否需要建立 _anonymized/ 目录的评估记录** | Year 2+ 触发条件未到 → 不建 / 触发 → 单独立项 |

## §11. 分类争议规则在调改现场的强制触发点（v0.4 加 / 5/19）

### §11.1 5 类强制触发点

| # | 触发点 | 责任人 | 落地动作 |
|:---:|---|---|---|
| 1 | **每个货架调改前** | 6 哥 + 佳妮 | 跑 [[2026-05-19_休食区货架分类争议预检查模板_v0.1]] 1 份 / 未完成 = 该货架 NO-GO |
| 2 | **每次从 ERP / POS 按类目筛 SKU** | 顾问 / 启明 | 筛 SKU 时同步加"商品名常识扫"步骤 / 异常 SKU 标 `[分类争议候选]` |
| 3 | **每次发现商品名与类目不一致** | 全员（现场即时）| 立即填预检表 1 行 / 不留口头未追踪 |
| 4 | **每次现场商品不在预期货架** | 顾问 + 佳妮 | 立即填预检表 1 行 / 拍照 / 问采购历史归属 |
| 5 | **每次采购说法与 ERP 不一致** | 6 哥 | 填预检表 1 行 / 后续按 §6 现场核验 SOP |

### §11.2 与 §10 月度审计的关系

| 维度 | §10 月度审计 | §11 调改现场实时 |
|---|---|---|
| 周期 | 每月 1 次（30 分钟）| 每个货架调改前 + 每次触发即时 |
| 形态 | 月度报告 §6.8 段 | 单货架预检表 1 份 |
| 责任人 | 6 哥（月度）| 6 哥 + 佳妮（现场即时）|
| 沉淀 | 月度报告 | 累积成 [[2026-XX-XX_花厅坊商品分类争议池_v0.5]] |

### §11.3 现场触发不替代月度审计

- ✅ §11 现场触发 = 实时捕获争议 / 不让问题溜走
- ✅ §10 月度审计 = 系统回顾 / 跨货架统计 / 跨月趋势
- ❌ 两者**不互替代** / 必须同时执行
