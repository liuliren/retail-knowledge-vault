# G08 vault 数据治理 SOP v0.1

> **本 SOP**：vault 数据进库时的**强制治理操作** / Oracle ACID 纪律 / 4 层治理底盘（数据契约 + 规则 + lint + Constitutional）。
>
> **触发**：每次新数据进 vault（POS / ERP / 客户提供 / 现场实测 / 行业研究）。
>
> **核心假设**：客户数据 100% 准确（POS+ERP）/ 我们的分析问题在我们这一侧（字段定义 / 时段 / 行数 / 计算公式）。

---

## §1. SOP 总流程（6 步 / 强制顺序）

```
Step 1：数据来源 evidence 标注（必填）
Step 2：frontmatter `data_source` + `fact_layer` 字段强制
Step 3：4 层治理过 §13.16-20（推断 + 客户视角 + 类比 + 订正 + 一致性）
Step 4：数学验证（重推 + 倒推 + 交叉验证）
Step 5：不一致时列 ≥ 3 种解释 + 用户 confirm
Step 6：沉淀至 [[数据一致性追溯案例库]]
```

---

## §2. 数据来源 evidence 标注（Step 1）

### 2.1 必填项

| 字段 | 内容 |
|---|---|
| **文件名** | 绝对路径 |
| **Sheet** | 工作表名 |
| **行号** | 起止行（如 2-7140 / 不含 header）|
| **时段** | 起止日期（如 2026-04-08 至 2026-05-08 / 31 天）|
| **字段定义** | POS / ERP 字段语义（如"客流"字段实际是"客单数"）|

### 2.2 反例（5/9 v2.2 错误做法）

```
❌ "30 天客流 145,490"
   - 缺文件名 / Sheet / 行号
   - 时段错（实际 32 天 4/7-5/8）
   - 字段定义错（实际是"客单数"）

✅ "32 天客单数 145,490 / 来源：01_门店数据材料/客流时段表.xlsx / Sheet1 / 行 2-578 / 时段 2026-04-07 至 2026-05-08 / 字段定义：POS '客流' = 实际客单数（订单数）/ [推断] 双倍统计 / 真实 ≈ 72,745"
```


### 3.2 `fact_layer` 4 元字段（vault 文档级）

```yaml
fact_layer:
  observed:        # vault 实测 / POS 真实数据
    - "30T 销售总额 2,998,401 元（来源：销售报表 / 行 2-32）"
```


## §5. 数学验证（Step 4 / 重推 + 倒推 + 交叉验证）

### 5.1 重推（forward）

```
原始数据
  ↓
[每一步检查：字段 / 时段 / 行数 / 公式 / 单位]
  ↓
我们的分析结论
```

### 5.2 倒推（reverse）

```
我们的分析结论
  ↓
[每个数字检查：来自哪个文件 / sheet / 行 / 时段]
  ↓
原始数据
```

### 5.3 交叉验证

```
独立数据 A：销售报表
独立数据 B：客流报表
独立数据 C：客单价报表

A × B = C ?
不通过 → 必有错（错在 A / B / C 之一）→ 列 ≥ 3 种假设 + 数学验证
```

## §6. 不一致时处理（Step 5）

### 6.1 列 ≥ 3 种解释假设

```
假设 1：双倍统计（145,490 / 2 = 72,745）
  - 数学验证：销售 96,723 / 客单数 2,273 = 42.55 ≈ 42.14 ✅
  - 内部一致性：高
  
假设 2：时段错开 1 天（32 天 vs 31 天 vs 30 天）
  - 数学验证：差 3.2% / 不解释 2 倍差 ❌
  - 内部一致性：中
  
假设 3：字段定义错认（客流 vs 客单数）
  - 数学验证：客单数 = POS 订单数 / 一个客户多张单 ✅
  - 内部一致性：高
  
组合假设 1+3：双倍统计 + 客流字段实际是客单数
  - 数学验证：完美闭合 ✅✅
  - 内部一致性：最高
```

### 6.2 用户 confirm

```
不允许 vault 自行决定 / 必须用户 confirm
标 [推断 / 推理链] / 不是结论
等待 confirm 后才能用作分析输入
```

### 6.3 案例沉淀

```
沉淀至 [[数据一致性追溯案例库]]
5 段固定结构：触发场景 / 错误做法 / 用户订正 / 数学验证 / 教训
```

## §7. 反例 vs 正例（5/9 实战）

### 7.1 反例（vault v2.2 错误做法）

```
❌ "30 天客流 145,490 / 客单 42.14 元 → 销售 = 145,490 × 42.14 / 32 = 191,615 元/天"
   - 字段定义错（"客流"实际是客单数）
   - 时段错（30 天 实际 32 天）
   - 数学验证不通过（191,615 vs 实际 96,723 / 差 1.98 倍）
   - 没列 ≥ 3 种解释
   - 没让用户 confirm
   - 写成事实
```

### 7.2 正例（5/9 用户驱动 / 完美闭合）

```
✅ "32 天客单数 145,490（来源：01_门店数据材料/客流时段表.xlsx / Sheet1 / 行 2-578 / 时段 2026-04-07 至 2026-05-08 / 字段语义：POS '客流' = 客单数）"
✅ "[推断] 双倍统计 → 真实 ≈ 72,745"
✅ "数学验证：日均销售 96,723 / 日均客单数 2,273 = 42.55 元/单 ≈ 报表 42.14（差 0.96%）⭐⭐ 完美闭合"
✅ "客单价定义锁定：客单价 = 销售 / 客单数（POS 笔均）"
✅ "数据三层口径 v1.0：客流 / 客单数 / 客单价（不可混用）"
✅ "启明 confirm 双倍统计 + 客流计算路径（pending）"
✅ "沉淀至 [[数据一致性追溯案例库]] 案例 #001"
```
