# G09 工程操作纪律审计 SOP v0.1

> **本 SOP**：vault 第 5 大治理支柱（工程操作纪律）的**月度审计**操作 / 与 G07 业务层月度审计互补 / Tesla First Principles + Bias to Action。
>
> **触发**：每月 1 日（人工触发 / 月度纪律）/ 6 哥每月固定 1 小时执行。
>
> **目标**：vault 工程操作纪律 6 大类规则集执行率 ≥ 95% / 错误案例库累积 / 同类错误不重复。

---

## §1. SOP 总流程（6 步 / 共 60 分钟）

```
Step 1（10 min）：扫工程操作错误案例库新增数（vs 上月）
Step 2（15 min）：6 大类规则执行率审计
Step 3（10 min）：pre-flight check 实战覆盖率审计
Step 4（10 min）：同类错误重复检测（Tesla 不容忍）
Step 5（10 min）：跨业态可复用度评估（M-DEC v1.0 候选）
Step 6（5 min）：沉淀 [[Vault健康仪表]] + 月度报告
```

---

## §2. 6 大类规则执行率审计

| 类 | 名称 | 审计指标 | 阈值 |
|---|---|---|---|
| 类 1 | 目录与路径纪律 | cd /tmp 等切走 cwd 次数 | ≤ 0（绝不允许）|
| 类 2 | Git 操作纪律 | 不破坏性操作绕过用户次数 | ≤ 0 |
| 类 3 | 文件操作纪律 | Read 先于 Edit 违规次数 | ≤ 0 |
| 类 4 | 数据操作纪律 | 数字无 evidence 次数（与 §13.20 协同）| ≤ 0 |
| 类 5 | 进程与执行纪律 | sleep 轮询次数 | ≤ 0 |
| 类 6 | 工具调用纪律 | Bash 用 cat / sed 等次数 | ≤ 5（warning 级别）|

## 追加纪律(2026-07-03 六哥签字)

### 改名三步铁律(Rename Discipline)
任何正式文件的**改名/移动**,必须三步连做,缺一即违规:
1. 改名/移动(本身须过 D 档签字,原规则不变)
2. `grep -r "旧名" --include='*.md'` 全库扫旧名引用
3. 逐处更新引用(指向新名)或注记(目标已删→纯文本+台账指引);零引用也须在 commit message 声明"已扫,零引用"

> 来源:2026-07-03 链接审计——当日三批改名(S4脱敏/V4降级/短ID)造成5处断链回归,证明"改名不联动反链"是系统性漏洞。
> 配套:批量改名优先用 alias 方案(目标文件 frontmatter 加 aliases)而非全库改链接。

### 敏感扫描正例自检铁律(同批签字·复盘迭代项#4落地)
任何"扫描无发现"的阴性结论(敏感数据/断链/越界),扫描命令必须先跑**正例自检**——确认该命令能找到一个已知存在的目标——否则阴性结论无效,不得写入报告。
> 来源:2026-07-02 S1缺陷——quotepath转义致grep假阴性,"git追踪敏感表=0"错误结论进入已签档案。
