# 经营底盘诊断报告 模板定义 v0.1

> **本文是模板定义（结构标准 + input schema + HTML占位符模板），不是某客户的真实报告。** 首个实例：好家源三店经营底盘诊断报告（2026-07-07 测试版，输出区）。
> **状态：draft。** 本文含「铁律」字样，按全局宪法《标准审议铁律》未完成≥3轮多角色论证+六哥签字前，不得被其他 Agent 引用为已生效红线；当前仅作模板参照使用。

---

## §0 定位与边界（必读）

**一句话**：本模板是四模块对外口径（[[KB-DX-FULLSTORE-001_全店7D诊断框架2.0_v0.1]] §0.5，2026-07-07 六哥裁定）中「**经营底盘**」模块（内部代号 D5/全店经营总诊断）的**对客户 HTML 报告载体**——回答「这家店/这组店的财务、坪效人效、现金流承不承得住其余三力」。

### 0.1 与既有交付件的关系（分工，不重叠）

| 既有件 | 对象/口径 | 与本模板关系 |
|---|---|---|
| 全店经营诊断卡 T-WHOLE-DX-001 | 单店五模块100分**评分**全景（内部工具+实例卡） | 本模板是其 M5/D5 财务健康维度的**单模块深挖对客版**；全店卡回答「先打哪个山头」，本模板只深挖「底盘」这一个山头。评分逻辑不在本模板重定义，取全店卡/7D框架结论 |
| KB-DX-FULLSTORE-001（7D框架） | 方法论层 | 本模板的口径 SSOT（§0.5 四模块映射表）。对客户只说「经营底盘」，**D5/M5/7D 字样永不出现在客户面前** |
| 多店综合诊断报告 T-MULTISTORE-DX-001 | N店全维度横向聚合 | 本模板可单店可多店，但只覆盖底盘一个模块；全维度多店对标仍走 T-MULTISTORE |
| 效果基线确认单 T-BASELINE-001 | 调改前效果基线锁定 | 不重叠。本模板是诊断件，非基线确认件；两者可共用同一批 POS 口径 |
| 报告导出系统 report-export v1.0 | 渲染底座（HTML→PDF+PNG） | **本模板是 report-export 体系下的「长报告变体」**：复用其 CSS class 体系与 render.sh，不新建渲染脚本、不构成第二套输出体系。单品类5段诊断卡仍走 template.html 原结构 |

### 0.2 张力（什么时候本模板会出错）

- 客户没给财务数据时，本模板**只能出「前置观测层」版本**（如好家源测试版）——若把 POS 衍生信号写成经营底盘结论，即为误用（见 §4 数据边界）。
- 面积/业态超出方法论锚定区间（500–2000㎡社区生鲜）时，坪效/人效判读基准失效，须在报告内注明。

## §1 输入结构（input schema）

> 体例对齐 [[报告导出系统_report-export_v1.0]] §4。`fact_layer` 三层强制：`observed`=POS直接观测 / `inferred`=推断需现场验证 / `pending`=无数据待客户提供。**pending 字段值一律填 `null`，禁止估算填充。**

## §3 使用说明

- **何时用**：客户签了含「经营底盘」模块的诊断服务（四模块口径）；或全店卡诊断后需就财务/坪效/现金流单独出对客深挖报告。财务数据未到位时只可出「前置观测层」版（version_label 标"预览版"，boundary_statement 必填）。
- **何时不用**：单品类货架问题 → `/diagnose`；五模块全景评分 → 全店卡 T-WHOLE-DX-001；多店全维度对标 → T-MULTISTORE-DX-001；商品/柜组结构问题 → [[商品力诊断报告_模板定义_v0.1]]。
- **谁来填**：Agent 填 input schema 的 observed/inferred 层与 HTML 实例化；**pending 字段的补数责任在客户**；每店判读段的 So-What 与对外定论由六哥现场核实后签字（10%不可代签）。
- **脱敏**：客户可识别信息进云层前一律代号化；真名进对外层前必查授权链清单（P1-GOV-ClientNameAuth-001），查不到=脱敏。

## §4 数据边界铁律（draft·待标准审议）

> **费用率、净利率、真实坪效/人效（成本口径）、现金周转天数——必须由客户提供财务数据，不得用 POS 衍生数据冒充。** 具体：
> 1. POS 只能给销售/毛利率/动销等**收入侧信号**；"净销×毛利率"只是销售端毛利额估算（inferred），不是利润。
> 2. 销售额÷面积=收入坪效（inferred），不是扣成本后真坪效；销售额÷在编人数=人效线索（pending，未核兼岗排班），不是人效结论。
> 3. 无数据字段在报告中**不填、不猜、不估**——显式标 pending 并进「待补全清单」。诚实的局限声明优于制造可信度假象。
> 4. 违反本条=把「前置观测层」冒充「经营底盘诊断」，属带病交付，正式版禁止出门。
