# 客户项目资料入库标准

## 一、文件定位

### 1.1 本文件是什么

本文件规范客户项目资料从**现场采集 / 客户提供 / 顾问整理 → Obsidian vault 入库 → 命名 → 索引 → 复查**的全过程标准。

**核心任务：** 让任何客户项目的资料都能"放对位置、命名规范、可追溯、不丢失"。

### 1.2 本文件不是什么

- ❌ 不负责 POS 数据清洗细则（[[POS数据清洗与商品标准化规范_v0.1]] 的事）
- ❌ 不负责诊断结论写法（[[诊断输出质量控制规范_v0.1]] 的事）
- ❌ 不负责首访现场动作（[[社区超市商品力诊断首访执行手册_v0.1]] 的事）
- ❌ 不负责客户交付报告模板（PPT / PDF 模板另立项）

### 1.3 服务目标

服务三个月生存战 P2 新客户开拓 + P1 花厅坊样板的"**资料治理基线**"。

---

## 二、适用场景与不适用场景

### 2.1 适用场景

- ✅ 新客户首访后的资料入库
- ✅ 二访补采后的资料补充入库
- ✅ 客户后续陆续提供数据的入库
- ✅ 顾问内部产出（看店有感、诊断报告等）的存放
- ✅ 项目交付报告归档

### 2.2 不适用场景

- ❌ 内部方法论文件（属于 `01_科学零售方法论` 等模块）
- ❌ 销售包模板（属于 `10_咨询交付模板` 顶层）
- ❌ 个人成长日记（属于 `15_刻意练习与成长`）
- ❌ 卷宗 / 素材暂存（属于 `90_素材暂存与待整理`）

## 三、入库总原则（6 条不可妥协）

| # | 原则 | 解释 |
|---|------|------|
| 1 | **原始资料不覆盖** | 客户给什么就存什么；任何修改都必须存为派生文件 |
| 2 | **原始资料不删除** | 即使被新版本取代，旧版本也归档而非删除 |
| 3 | **原始资料不直接改名破坏来源** | 命名标准化时保留原文件名作为元数据 |
| 4 | **派生文件必须可追溯** | 每个清洗后/分析后的文件，必须能回溯到原始文件 |
| 5 | **客户敏感信息必须保护** | 文件名不暴露老板姓名、门店地址、报价数字等 |
| 6 | **所有资料必须进入资料索引表** | 不入索引 = 不存在；3 个月后没人找得到 |

| 99_原始素材 | 客户提供 + 现场采集的原始资料；含 7 个二级子目录（参 §4.3）| 不存任何派生文件；不擅自修改 | `99_原始素材/01_POS原始数据/客户A_主店_POS_休食_20260501-20260731_原始.xls` |

### 4.3 99_原始素材 二级子目录详解

| 二级子目录 | 存放内容 | 禁止事项 | 示例文件 |
|---|---|---|---|
| **01_POS原始数据** | 客户 ERP 导出的 POS .xls / 销售明细 / 销售汇总；保留导出当时的字段顺序 | 不存清洗后的 SKU 主表（→ 03_商品诊断 或独立清洗层）；不存毛利分析（→ 02_生意诊断）| `客户A_主店_POS_休食_20260501-20260731_原始.xls` |
| **02_客户原始商品档案** | 客户 ERP 导出的商品主数据（条码 / 品名 / 规格 / 价格 / 供应商）| 不存与标准品类表匹配后的派生文件（→ 03_商品诊断）| `客户A_主店_商品档案_全店_20260503_原始.xlsx` |
| **03_客户原始库存数据** | 客户 ERP 导出的库存表（金额 + SKU 数 + 盘点日期）| 不存周转天数分析派生文件（→ 02_生意诊断）| `客户A_主店_库存_全店_20260503_原始.xlsx` |
| **04_现场原始照片** | 首访 + 二访现场照片（按 §6.2 命名规范）| 不存裁剪 / 编辑后的照片（→ 派生文件存对应业务模块）| `客户A_主店_休食区_第3组_正面平拍_001_20260503.jpg` |
| **05_原始访谈记录** | 访谈录音 .mp3 + 文字纪要原始版 | 不存编辑 / 整理后的访谈纪要（→ 00_项目总览 或对应业务模块）| `客户A_主店_访谈_老板_20260503_原始.mp3` |
| **06_竞品原始素材** | 周边竞品现场照片 / 调研素材 | 不存竞品分析报告（→ 01_商圈与客群）| `客户A_竞品_赵一鸣_入口_正面_001_20260503.jpg` |
| **99_其他待整理** | 未分类原始资料 / 客户主动给的 PDF / Word / PPT | 不在此目录长期沉淀；定期分流到 01-06 子目录 | `客户A_客户提供_品牌方案.pdf` |

**核心纪律：**
- 原则上每份原始资料必须落入 01-06 二级子目录之一
- 99_其他待整理 仅作为临时缓冲区；每周清理一次
- 二级子目录之间不允许重复（同一文件只属于 1 个子目录）

### 4.4 业务模块（00-08）与原始素材（99）的轴关系

```
99_原始素材（原始层，只读）
       ↓ 清洗后产生派生文件（不动原始）
       ↓
01-08（业务模块层，按业务流程组织）
       ↓ 形成正式产出
       ↓
08_交付报告（客户版输出）
```

**核心纪律：** 99_ 是**只读**层，01-08 是**工作**层。从 99_ 到 01-08 必须经过清洗 / 分析 / 整理。

## 五、三层资料管理法

### 5.1 三层定义

| 层级 | 物理位置 | 存什么 | 不存什么 | 责任人 | 验收标准 |
|---|---|---|---|---|---|
| **原始层** | `99_原始素材/01-06_*/` | 客户原始 POS / 商品档案 / 库存 / 照片 / 访谈录音 + 文字纪要原始版 / 客户提供文档 | 任何派生文件 / 任何修改后的文件 | 顾问 | 文件命名规范 + 进入资料索引表 + 不被覆盖 |
| **清洗层** | `99_原始素材/` 内独立 `_清洗后` 文件 或 `03_商品诊断/`、`02_生意诊断/` 等业务模块下的派生工作表 | 清洗后的 SKU 主表 / 商品 × 品类匹配表 / 异常清单 | 原始文件 / 分析结论 | 顾问（依据 `/posclean` skill·`13_数据分析与工具脚本/POS清洗库_v0.1/`；原 [[POS数据清洗与商品标准化规范_v0.1]] 已由其取代，2026-07-07）| 字段标准化 + 异常处理 + 可追溯到原始 |
| **分析层** | `02-08/` 业务模块子目录 | 诊断报告 / SKU ABCD 清单 / DisplayMap / 调改方案 / 复盘报告 | 原始数据 / 清洗中间表 | 顾问 | 每条结论可追溯到清洗层 + 证据等级标注（参 [[诊断输出质量控制规范_v0.1]] §六）|

### 5.2 三层之间的边界

- **原始层 → 清洗层**：所有清洗动作记录在清洗日志（清洗执行走 `/posclean` skill·`13_数据分析与工具脚本/POS清洗库_v0.1/`；原 [[POS数据清洗与商品标准化规范_v0.1]] 已由其取代，2026-07-07）
- **清洗层 → 分析层**：所有分析结论标注证据等级（参 [[诊断输出质量控制规范_v0.1]]）
- **分析层 → 客户**：所有客户交付物来自 `08_交付报告/`，不直接拿分析层中间产出给客户

## 六、文件命名规范

### 6.1 通用命名格式

```
客户简称_门店简称_资料类型_内容说明_日期_版本.扩展名
```

**字段说明：**

| 字段 | 标准 | 示例 |
|---|---|---|
| 客户简称 | 2-4 字去标识化代号 | "客户A" / "Z 老板店" / "肇A" |
| 门店简称 | 主店 / 分店1 / 分店2 | "主店" |
| 资料类型 | POS / 商品档案 / 库存 / 照片 / 访谈 / 诊断 / 报告 | "POS" |
| 内容说明 | 品类 / 主题 / 编号 | "休食" / "水饮" / "全店" |
| 日期 | YYYYMMDD（单日）或 YYYYMMDD-YYYYMMDD（区间）| "20260501-20260731" |
| 版本 | _原始 / _v0.1 / _v1.0_FINAL / _审计 / _清洗后 | "_原始" |
| 扩展名 | .md / .xlsx / .pdf / .jpg / .docx / .mp3 | ".xls" |

**完整示例（含二级子目录路径）：**
- `99_原始素材/01_POS原始数据/客户A_主店_POS_休食_20260501-20260731_原始.xls`
- `99_原始素材/02_客户原始商品档案/客户A_主店_商品档案_全店_20260503_原始.xlsx`
- `99_原始素材/03_客户原始库存数据/客户A_主店_库存_全店_20260503_原始.xlsx`
- `03_商品诊断/客户A_主店_诊断_商品力快速诊断_20260510_v0.1.md`
- `08_交付报告/客户A_主店_报告_首访诊断报告_20260510_v1.0_FINAL.pdf`

### 6.2 照片命名规则

```
客户简称_门店简称_区域_位置_视角_序号_日期.jpg
```

参 [[社区超市商品力诊断首访执行手册_v0.1]] §八.3。

**完整示例（含路径）：**
- `99_原始素材/04_现场原始照片/客户A_主店_休食区_第3组_正面平拍_001_20260503.jpg`
- `99_原始素材/04_现场原始照片/客户A_主店_收银区_前置_全景_001_20260503.jpg`
- `99_原始素材/06_竞品原始素材/客户A_竞品_赵一鸣_入口_正面_001_20260503.jpg`

### 6.3 各类资料命名补充规则

| 资料类型 | 命名特殊要求 | 落入二级子目录 |
|---|---|---|
| **原始 POS** | 必须含日期区间 + "_原始" 后缀；保留客户原 ERP 导出时的字段顺序 | `99_原始素材/01_POS原始数据/` |
| **原始商品档案** | 必须含 "_原始" 后缀；保留 ERP 导出字段顺序 | `99_原始素材/02_客户原始商品档案/` |
| **原始库存数据** | 必须含盘点日期 + "_原始" 后缀 | `99_原始素材/03_客户原始库存数据/` |
| **现场照片（自店）** | 严格按 §6.2 | `99_原始素材/04_现场原始照片/` |
| **现场照片（竞品）** | 严格按 §6.2，区域字段写"竞品_品牌名" | `99_原始素材/06_竞品原始素材/` |
| **访谈录音** | 含被访者代号（"老板" / "店长"）+ 访谈日期 + "_原始" | `99_原始素材/05_原始访谈记录/` |
| **派生 SKU 主表** | 必须含 "_清洗后" 后缀；记录清洗日期 | `03_商品诊断/` 或独立清洗工作表 |
| **数据表派生文件** | 含数据周期；多版本时必须含版本号 | 对应业务模块 |
| **访谈纪要派生文件** | 含被访者代号 + 整理日期 | `00_项目总览/` 或对应业务模块 |
| **交付报告** | 必须含 v 版本号；正式交付版本必须含 "_FINAL" | `08_交付报告/` |
| **看店有感** | 命名格式：`客户A_主店_看店有感_20260503.md` | `00_项目总览/` |

### 6.4 客户敏感信息保护规则

- ❌ 文件名不出现老板真实姓名（用"客户A" / "Z 老板"代替）
- ❌ 文件名不出现门店真实地址（用"主店" / "分店1"代替）
- ❌ 文件名不出现报价具体数字（"X 万" / "X 千"代替）
- ✅ 客户敏感信息只存于内部备注 + 资料索引表的"备注"字段

### 6.5 命名禁止字符

- ❌ 不使用全角符号（全角空格 / 全角括号）
- ❌ 不使用特殊字符（: * ? " < > | / \）
- ❌ 不使用 emoji
- ✅ 使用半角下划线 `_` 作为字段分隔符
- ✅ 使用半角破折号 `-` 作为日期区间分隔符

## 七、资料索引表标准

### 7.1 索引表文件

每个客户项目必须建立 1 份资料索引表。

**建议文件名：**
```
客户简称_门店简称_项目资料索引表_v0.1.xlsx
```

**存放位置：** `09_门店案例/客户简称_门店简称/00_项目总览/`

### 7.2 索引表字段定义

| 字段 | 类型 | 含义 | 填写标准 | 示例 |
|---|---|---|---|---|
| `file_id` | 整数 | 资料 ID（自增）| 0001-9999 | 0001 |
| `file_name` | 字符串 | 文件名（标准化后）| 严格按 §六 命名规范 | `客户A_主店_POS_休食_20260501-20260731_原始.xls` |
| `file_type` | 枚举 | 资料类型 | 原始/清洗/分析/交付 中选一 | 原始 |
| `category` | 枚举 | 资料大类 | POS/商品/库存/照片/访谈/诊断/报告/竞品 | POS |
| `source` | 字符串 | 资料来源 | 客户提供/现场采集/顾问产出 | 客户提供 |
| `original_path` | 路径 | 原始来源路径 | 客户提供时原文件名（可含中文）| `导出_2026-05-03.xls` |
| `standard_path` | 路径 | vault 内标准化路径（相对 vault 根）| 严格按 §四 目录结构 | `09_门店案例与项目复盘/客户A_主店/99_原始素材/01_POS原始数据/客户A_主店_POS_休食_20260501-20260731_原始.xls` |
| `version` | 字符串 | 版本号 | 原始/v0.1/v1.0_FINAL 等 | 原始 |
| `status` | 枚举 | 资料状态（参 §九）| 9 种状态之一 | 已入库 |
| `owner` | 字符串 | 责任人 | 六哥 / 助理咨询师代号 | 六哥 |
| `created_date` | 日期 | 入库日期 | YYYY-MM-DD | 2026-05-03 |
| `notes` | 字符串 | 备注 | 客户敏感信息 / 异常说明 / 待办 | "客户原文件名为 XXX.xls，与商品档案不全对应" |

### 7.3 索引表使用纪律

- ✅ 每入库 1 个文件 → 立即更新索引表
- ✅ 每修改 1 个文件状态 → 立即更新索引表
- ✅ 每月底回看索引表，识别 30 天未动的资料
- ❌ 不允许"先入库再补索引"——索引必须实时更新

### 7.4 自动化预留

索引表设计支持后续 Excel 公式 / Python 脚本 / AI 工具自动处理：
- `file_id` 数字化主键
- `file_name` / `standard_path` 严格命名规范，便于 regex 匹配
- `status` 枚举值固定，便于状态机校验
- `original_path` 与 `standard_path` 双字段，支持自动校验"派生 → 原始"追溯链

## 八、入库流程 SOP（S1-S8）

### S1 接收资料

| 项 | 内容 |
|---|---|
| 输入 | 客户邮件 / 微信 / 现场拷贝 / 顾问拍摄的原始资料 |
| 动作 | 1. 在桌面建临时收件夹 `INBOX_客户A_20260503/`；2. 拷贝原始文件进入；3. 拍照备份关键截图（如客户微信发来的图片）|
| 输出 | 临时收件夹完整 |
| 验收标准 | 原始文件未被打开 / 修改 / 重命名 |
| 风险提醒 | 不要直接打开 .xls 后保存（Excel 会改格式）|

### S2 原始资料归档

| 项 | 内容 |
|---|---|
| 输入 | 临时收件夹的原始资料 |
| 动作 | 1. 按 §6 命名规范重命名（保留原名为备注）；2. 按 §4.1 一级目录 + §4.3 二级子目录放入对应位置 |
| 输出 | 99_原始素材 / 对应二级子目录 内有完整原始文件 |
| 验收标准 | 原始文件命名标准化 + 二级子目录位置正确 + 原文件名记录在索引表 notes |
| 风险提醒 | 重命名时不要打开文件；用 mv 而非"另存为" |

### S3 建立资料索引

| 项 | 内容 |
|---|---|
| 输入 | 已归档的原始文件 |
| 动作 | 在《客户A_主店_项目资料索引表_v0.1.xlsx》新增 1 行，按 §7 字段填写 |
| 输出 | 索引表完整 |
| 验收标准 | 字段全填 + status="已入库" + standard_path 字段含完整二级子目录路径 |
| 风险提醒 | 不要批量入库后再补索引——容易遗漏 |

### S4 判断资料类型

| 项 | 内容 |
|---|---|
| 输入 | 已索引的原始资料 |
| 动作 | 判断这份资料属于：① 直接可用（不需要清洗）；② 需要清洗（POS / 商品档案 / 库存）；③ 需要标准化（照片重命名）；④ 需要纪要补录（录音）|
| 输出 | 资料类型标签 |
| 验收标准 | 索引表 file_type 字段填写正确 |
| 风险提醒 | 不要把"看似复杂的 .xls"误判为"直接可用"——通常都需要清洗 |

### S5 生成标准化派生文件

| 项 | 内容 |
|---|---|
| 输入 | 原始资料 + 类型判断 |
| 动作 | 用 `/posclean` skill（`13_数据分析与工具脚本/POS清洗库_v0.1/`；原 [[POS数据清洗与商品标准化规范_v0.1]] 已由其取代，2026-07-07）执行清洗（POS / 商品档案）；按 §6 标准重命名（照片）；按访谈纪要模板补录（录音）|
| 输出 | 派生文件（清洗层）|
| 验收标准 | 派生文件能追溯到原始文件（索引表中 original_path 字段 + standard_path 字段双向对应）|
| 风险提醒 | 派生文件必须独立命名（不要覆盖原始文件）|

### S6 放入对应业务模块

| 项 | 内容 |
|---|---|
| 输入 | 派生文件（清洗层）|
| 动作 | 按 §4.1 一级目录放入对应业务模块（如清洗后的 SKU 主表 → `03_商品诊断/`；DisplayMap → `05_陈列整改/`）|
| 输出 | 业务模块层完整 |
| 验收标准 | 文件位置正确 + 索引表更新 status |
| 风险提醒 | 不要把派生文件留在 99_ 层 / 不要把原始文件挪到业务模块层 |

### S7 更新资料索引状态

| 项 | 内容 |
|---|---|
| 输入 | 已入业务模块层的派生文件 |
| 动作 | 索引表状态从"已入库" → "已清洗"；新增派生文件行（含 file_id 引用原始文件）|
| 输出 | 索引表反映最新状态 |
| 验收标准 | 索引表中能完整追溯：原始 → 清洗 → 分析 |
| 风险提醒 | 不要漏更新——3 个月后会找不到 |

### S8 复查是否可进入诊断

| 项 | 内容 |
|---|---|
| 输入 | 索引表 + 业务模块层 |
| 动作 | 对照《首访验收表》（参 [[社区超市商品力诊断首访执行手册_v0.1]] §十）100 分制评分 |
| 输出 | 验收分数 + 缺失清单 |
| 验收标准 | ≥ 70 分可进入诊断；< 70 分需补采 |
| 风险提醒 | 验收不通过强行启动诊断 = 诊断报告必有盲区 |

## 九、资料状态标准（9 种）

| 状态 | 含义 | 触发条件 |
|---|---|---|
| **原始** | 刚接收，未入库 | S1 完成 |
| **已入库** | 已按规范命名 + 归档 99_ + 索引表登记 | S2-S3 完成 |
| **已清洗** | 已生成派生文件 + 入业务模块层 | S5-S6 完成 |
| **已复核** | 已二次核对清洗准确性 | 由顾问或助理完成复核 |
| **已用于分析** | 已进入诊断报告 / 调改方案等分析层 | 分析层文件引用 |
| **已归档** | 项目结束 + 移到归档文件夹（如 `99_归档/客户A_20260801_done/`）| 项目交付完成 |
| **待补充** | 资料缺失 / 不完整 / 待客户补提供 | 验收时识别 |
| **待确认** | 资料字段含义模糊 / 需向客户确认 | 清洗时识别 |
| **不可用** | 文件损坏 / 格式无法解析 / 客户撤回 | 异常处理时标记 |

## 十、入库验收标准

### 10.1 验收清单（10 项）

- ☐ 原始资料是否完整保留（无覆盖、无删除）
- ☐ 是否有资料索引表（含全部已入库文件）
- ☐ 文件名是否规范（符合 §6 命名规则）
- ☐ 文件夹是否正确（一级 + 二级子目录均符合 §4 目录结构）
- ☐ 派生文件是否可追溯到原始文件（索引表 original_path 字段）
- ☐ 客户敏感信息是否受控（无泄露在文件名中）
- ☐ 是否标注缺失项（待补充资料 + 待确认字段）
- ☐ 是否形成二访补采任务（若验收 < 90 分）
- ☐ 三层资料管理是否分开（99_原始 vs 业务模块层）
- ☐ 索引表 status 字段是否更新到最新状态

### 10.2 验收分级

| 验收通过项 | 分级 | 后续动作 |
|---|---|---|
| 10/10 项 | ✅ **可进入诊断** | 跑 `/posclean` skill（POS清洗库_v0.1·原 [[POS数据清洗与商品标准化规范_v0.1]] 已被取代，2026-07-07）后启动诊断 |
| 8-9 项 | 🟡 **可进入诊断，但需补充** | 同步补充缺失项 |
| 6-7 项 | 🟠 **暂不可诊断，必须补采** | 二访补采为主 |
| ≤ 5 项 | 🔴 **入库失败，必须返工** | 全面整理后重新验收 |

## 十一、风险红线（8 项）

| # | 红线 | 违反后果 |
|---|------|------|
| 1 | ❌ 不覆盖原始 POS 数据 | 数据完整性破坏，无法回溯 |
| 2 | ❌ 不删除客户原始文件（任何理由）| 客户授权风险 + 不可追溯 |
| 3 | ❌ 不擅自修改客户原始文件（清洗必须存为派生文件）| 与原始版混淆 |
| 4 | ❌ 不把未清洗数据用于正式结论 | 诊断报告可信度归零 |
| 5 | ❌ 不把客户敏感信息暴露在公开文件名中 | 商业伦理 + 法律风险 |
| 6 | ❌ 不让分析文件失去来源追溯（索引表 original_path 必填）| 3 个月后无法复盘 |
| 7 | ❌ 不在客户离开诊断关系后保留原始资料超过 30 天 | 数据安全责任 |
| 8 | ❌ 不擅自跨客户共享原始文件（即使是同业态客户）| 客户信任崩塌 |
