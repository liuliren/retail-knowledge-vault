# 社区超市30天商品提效包 主定义 v0.1

> 本文件是「社区超市30天商品提效包」的**产品主定义（产品脊梁 / 收口索引）**，对内使用，**非对外销售页、不直接给客户看**。
> 它**只引用、不重定义**任何方法论：方法骨架见科学零售三力，方法武器见 M1-M8，方法引擎见 RetailOS，决策规则见 M-DEC，实战验证见花厅坊样板。
> 历史归档的 PDF/DOCX 套件是本产品的**素材库**，不是主定义（见 §7 新旧校准）。

## 1. 产品定位

- 「社区超市30天商品提效包」是晟果新零售面向社区超市的**标准化阶段项目产品**；
- 它**不是全店重构**，也**不是长期陪跑**；
- 它是「诊断费 → 阶段项目 → 长期陪跑」三阶商业化路径中的**第二阶（阶段项目）**（路径见 [[商品力顾问_报价单_v0.1]]）；
- 它是**科学零售体系进入客户门店的首个商品力产品锚点**——把 L2 三力 + L3 武器 + RetailOS，封装成 30 天可交付、可见结果的单/双品类深调改。

## 2. 适用客户

- 社区超市；
- 生鲜加强型社区店；
- 中小连锁单店样板（先打样板店，再谈多店复制）；
- 营业面积 500–5000㎡可适用，**标准版优先 500–2500㎡**；
- 有基础 POS 数据更佳（无数据则先做轻诊断 + 现场判断，数据质量影响精度，见 §20）。

## 3. 解决的问题（对接 L1 痛点）

- 卖不动（动销差 / 滞销厚）；
- 毛利低（价格带结构错配）；
- 库存重（积压 / 周转慢）；
- 商品结构乱（超配欠配 / 角色不清）；
- 缺货与滞销并存；
- 陈列资源错配（黄金位喂低效 SKU）；
- 老板知道有问题，但不知道**先改哪里**；
- 店长只会补货，不会做**商品经营**（对接组织力，见 §16/§19）。

> 痛点归因详见 [[痛点框架]]；本产品按"先止血（汰滞销/调价）、再造血（选品/陈列）"的顺序处置。

## 5. 与科学零售体系关系

- 本产品属 **L4 落地 / 商业产品封装**；
- **引用** L2 三力（商品力为主，组织力/运营力为辅）；
- **调用** L3 武器 M1-M8（选品 M2 / 价格带 M3 / 陈列 M4 / 库存 M1 为主）；
- **使用** L4 决策规则 M-DEC（如 M-DEC-010 价格带预警）；
- 通过 **RetailOS** 把上述压成"店员可照做"的门店动作。

## 6. 与 RetailOS 关系

- **RetailOS = 方法引擎 / 门店决策操作系统**（A-G 减法层 + v2.0 选品加法层，操作系统外壳定位不变）；
- **30天商品提效包 = 商业产品外壳**（定义边界/流程/交付物/商业模式/定价）；
- **两者不混同**：RetailOS 回答"怎么决策"，本产品回答"卖给谁、交付什么、多少钱、多少天"；
- 30天包**调用** RetailOS：选品层 4 闸（真需求/有位置/算得过来/供得上）、减法层汰换、SKU 动作、门店执行动作。

## 7. 新旧架构校准声明

> 历史归档中的《30天商品提效包》相关 PDF / DOCX / 模板，是本产品的**历史交付资产与模板素材库**。当前主定义文件**不直接复刻旧套件**，而是在 v0.5 科学零售知识体系下，以 RetailOS、M1-M8、M-DEC 和花厅坊样板为方法支撑，对该产品进行**重新定义、归位和版本治理**。

定位关系：
- 旧 PDF / DOCX 套件 = 历史交付物素材，**可引用、可拆解、可复用**；
- 30天商品提效包主定义 = **当前产品主定义**，负责统一边界、流程、交付物、商业模式；
- RetailOS = 方法引擎 / 门店决策操作系统；
- M1-M8 = 方法武器簇；

## 8. 项目边界：做什么 / 不做什么

**做**：
- 1 个重点品类，或 1–2 个重点品类；
- 3–5 个关键货架 / 关键位置；
- 数据诊断（动销 / ABC / 价格带 / 库存）；
- SKU 动作建议（见 §13）；
- 价格带 / 子品类 / 陈列面判断；
- DisplayMap 动作（见 §14）；
- 店长执行清单；
- 周复盘（见 §15）；
- 结项报告；
- 下一阶段建议（90天路线图）。

**不做**：
- 全店全品类重构；
- 长期代运营；
- 全套组织变革；
- ERP 系统建设；
- 完整会员体系；
- PB（自有品牌）开发；
- 多店总部系统搭建；
- **保证业绩增长**（见 §20）。

## 11. 数据输入要求

- 商品档案（含品类/子品类/条码/货号）；
- 销售明细（近 30/60/90 天）；
- 库存表（即时库存 / 库龄）；
- 毛利 / 进价 / 售价；
- 供应商（主供应商 / 可得性）；
- 异常数据说明；
- **数据质量分级**（高/中/低，影响诊断精度与结论强度）。

> 字段口径与清洗规范见 [[POS数据清洗与商品标准化规范_v0.1]]；客户管理成熟度不高时按"结果性代理"务实降级（参科学零售知识体系总纲 §6.6）。

## 12. 现场输入要求

- 门店照片 / 货架照片（6 机位拍摄法）；
- 重点货架编号（统一区域代号）；
- 动线 / 主通道；
- 竞品；
- 端架 / 堆头；
- 店长访谈；
- 老板目标；
- 现场问题记录。

> 拍摄与现场采集规范见 [[SOP_6机位拍摄法_v0.5]] / [[社区超市商品力诊断首访执行手册_v0.1]]。

## 13. SKU 动作体系

标准动作集（10 类）：**保留 / 放大 / 缩面 / 调价 / 换位 / 补货 / 观察 / 淘汰 / 引进 / 待确认**。

> **SKU 动作必须由"数据 + 现场 + 人工确认"共同决定**——不可仅凭单一数据末位淘汰（防误判，参 M-DEC-009 / 选品层 4 闸 / 需求点可得性）。引进动作走 RetailOS 选品层置换式（上新必腾位，一对一挂清退）。

## 14. DisplayMap 与陈列动作

- DisplayMap **不是单纯陈列美化**；
- 它是把**商品结构 + SKU 动作 + 价格带 + 顾客购买路径**转化为**货架动作**；
- 对接 [[M4_陈列货位簇]]（货位陈列系数 / 黄金位 / 排面）；
- 与 RetailOS 门店执行动作相连（陈列动作 → 店长执行清单）。

## 15. 周复盘与结项

- **7 天观察**：SKU 动作是否落地、有无异常；
- **15 天修正**：据动销变化调整 SKU 动作 / 陈列；
- **30 天结项**：SKU 动作复盘 + 陈列执行复盘 + 数据变化复盘 + 老板复盘 + 下一阶段建议。

> 复盘机制对接组织力 P1（[[组织力最小可用模块补建任务卡_v0.1]] 周度商品复盘会议机制）。
> 周复盘见 [[周复盘表_模板定义_v0.1]]；结项见 [[结项报告_模板定义_v0.1]]。

## 16. 客户责任与晟果责任

| 方 | 责任 |
|---|---|
| **晟果** | 诊断、方案、工具、复盘、报告 |
| **客户** | 数据提供、现场执行、补货、调价、人员组织、老板决策 |

> 责任边界对接组织力 RACI 雏形（[[组织力最小可用模块补建任务卡_v0.1]]）。落地效果取决于客户执行与老板参与（见 §20）。

## 17. 价格与商业模式

| 版本 | 范围 | 建议价（内部参考）|
|---|---|---|
| 基础版 | 单店 / 1 个品类 / 轻诊断轻执行 | 3 万左右 |
| 标准版 | 单店 / 1–2 个品类 / 数据诊断 + 陈列动作 + 复盘 | 5 万左右 |
| 加强版 | 多品类 / 多次现场 / 深度复盘 / 初步多店复制 | 8 万左右 |

- 标准周期 **30 天**，可根据门店规模、品类数量、数据复杂度、现场次数和客户执行难度升级为 **4–8 周**；
- 与三阶路径衔接：诊断费可抵扣本阶段、本阶段可抵扣升级陪跑（机制见 [[商品力顾问_报价单_v0.1]]）。

> **价格为内部报价参考，非对外固定价格。对外报价以客户规模、项目边界、数据复杂度、现场次数和销售策略为准。**

## 20. 风险边界

- 不保证业绩增长；
- 不替代客户经营决策；
- 不承诺全店问题一次解决；
- 数据差会影响诊断精度；
- 执行差会影响结果；
- 老板不参与会影响落地；
- 旧模板需经本主定义重新校准后才能对外使用。
