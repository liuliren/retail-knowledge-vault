# 商品力诊断报告 模板定义 v0.1

> **本文是模板定义（结构标准 + input schema + HTML占位符模板），不是某门店的真实报告。** 首个实例：好家源三店商品力诊断报告（顺景/富城/金沙咀，2026-07-07 测试版，输出区，三店体例一致）。
> **状态：draft。** 本文含「铁律」字样，按全局宪法《标准审议铁律》未完成≥3轮多角色论证+六哥签字前，不得被其他 Agent 引用为已生效红线；当前仅作模板参照使用。

---

## §0 定位与边界（必读）

**一句话**：本模板是四模块对外口径（[[KB-DX-FULLSTORE-001_全店7D诊断框架2.0_v0.1]] §0.5）中「**商品力**」模块（内部来源 D1 品类结构 + D2 生鲜运营）的**单店对客户 HTML 报告载体**——回答「这家店的品类结构与外在陈列结构正不正」，颗粒度在**柜组层**（店级俯瞰各柜组），向下派单单品类诊断。

### 0.0 口径边界注记（⚠️ 给未来 Agent，回应 2026-07-07 语音输入教训）

> **商品力 ≠ 组织力。** 六哥语音输入专名易音近出错（全局记忆已载）——若任务描述里出现「组织力诊断」而素材全是品类/柜组/SKU 数据，先停下确认：组织力=D6 人员运营（人员配置/权责利/排班），商品力=D1+D2（品类结构/生鲜运营/陈列结构）。两者是四模块中**不同的两块**，不得混出一份报告。本模板 HTML 顶部保留 `.confirm` 口径确认条，实例化时视沟通情况保留或删除。

### 0.1 与既有交付件的关系（分工，不重叠）

| 既有件 | 对象/口径 | 与本模板关系 |
|---|---|---|
| 单品类诊断卡 `/diagnose` | 一个品类货架（SKU级红黄绿） | 本模板的**下钻件**。本模板只在柜组层给红黄绿与方向，SKU 级裁决一律派单 `/diagnose`——在柜组层直接砍 SKU 必因颗粒度不足而误杀 |
| 全店经营诊断卡 T-WHOLE-DX-001 | 单店五模块100分评分全景 | 本模板是其 M2/M3（D1+D2）维度的**单模块深挖对客版**；派单逻辑（空销率>30%/GMROI<2.0）沿用全店卡 §4.4，不重定义 |
| 30天提效包·商品诊断报告（模板库） | 项目制 S04 环节 Word 长文档（18部分/10模块） | 不重叠：那是30天提效包**项目内**诊断产物（长文档，接 SKU动作表）；本模板是**独立诊断服务**的1页 HTML 快诊报告，不承接 SKU动作表接口 |
| 经营底盘诊断报告 T-BIZBASE-DX-001 | 财务/坪效/现金流模块 | 姊妹件（四模块各一）。毛利率倒挂等信号两报告共享数据，但归因判读各归其模块 |
| 报告导出系统 report-export v1.0 | 渲染底座（HTML→PDF+PNG） | **本模板是 report-export 体系下的变体结构**（5段强结构的店级商品力版），复用 CSS class 与 render.sh，不新建渲染脚本、不构成第二套输出体系 |

### 0.2 张力（什么时候本模板会出错）

- 条码匹配率跌破 95% 防呆线时，任何品类结构判断可能失真——**先修数据可信度，后下结构结论**（顺景 94.8% 实例）。
- 面积超出 500–2000㎡ 锚定区间的门店（如顺景 3200㎡），SKU 目标数/特化基准套用前须先裁是否拆诊断基准。
- 生鲜柜 C 类为品种轮换常态，低销≠该砍；防误杀提示为强制段，不可删。

## §1 输入结构（input schema）

> 体例对齐 [[报告导出系统_report-export_v1.0]] §4。`fact_layer`：observed=POS单源汇总直接观测 / inferred=推断（C级，需现场验证）/ pending=待现场核实或待六哥裁。

```json
{
  "store": "顺景（脱敏代号）",
  "report_type": "prodpower",
  "period": "90天 · 2026-04~06",
  "caliber_confirm": "本报告对应商品力模块（品类结构与外在陈列结构）；如原意确指组织力，请指出另出",
  "conclusion": "≤2句，含关键数字与第一优先动作",          "_fact": "inferred",
  "kpi_cards": [
    { "value": "66.5%", "label": "动销率（三店最高）", "bad": false }
  ],
  "cabinet_rows": [
    { "cabinet": "猪肉柜", "scale": "24.1%净销", "light": "🟢", "action": "毛利20.4%，生鲜头柜健康，保留扩展", "_fact": "observed" }
  ],
  "display_findings": [
    { "item": "A类是否占全店黄金位", "tag": "pending·现场" }
  ],
  "directions": { "keep": "…", "compress": "…", "clear": "…" },
  "field_steps": { "step1_role": "…", "step2_queue": "…", "step3_act": "…" },
  "fact_layer_note": "observed=…；inferred=…；pending=…（防误杀提示同段）"
}
```

## §3 使用说明

- **何时用**：客户签了含「商品力」模块的诊断服务（四模块口径）；POS 三件套（ABCZ/动销/价格带）已跑完、需给老板一张店级柜组俯瞰图时。每店一张（多店项目按店出系列，体例一致）。
- **何时不用**：单个货架 SKU 级裁决 → `/diagnose`；五模块全景 → 全店卡；30天提效包项目内诊断 → 提效包模板库的商品诊断报告；财务/坪效问题 → [[经营底盘诊断报告_模板定义_v0.1]]；人员/权责/排班问题 → 组织力（另出，见 §0.0）。
- **谁来填**：Agent 从三件套结果表填 cabinet_rows/kpi_cards（observed 层）与 HTML 实例化；inferred 归因与 pending 项由六哥现场验证；对外定论（正式版）须六哥签字。防误杀段与 fact_layer 边界段为强制段，Agent 不得删除。
- **脱敏**：同 T-BIZBASE-DX-001 §3——代号化进云层，真名对外必查授权链，查不到=脱敏。
