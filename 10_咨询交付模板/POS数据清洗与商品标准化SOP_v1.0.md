# POS 数据清洗与商品标准化 SOP v1.0

> **定位**:客户原始 POS 数据 → 诊断可用标准表的**跨客户工程规范**。服务新客户快速接入(2026 目标:30 诊断)。打捞 [[2026-06-25_归档打捞backlog_v0.1]] #7。
> **核心铁律**:`sku_name_raw`(客户原始,永不改)+ `sku_name_std`(派生,可迭代)双字段并存;原始表永不覆盖。

## 1. 八条不可妥协原则
| # | 原则 |
|---|------|
| 1 | **原始表不覆盖** —— 清洗动作存为派生文件,原始 .xls 永不修改 |
| 2 | **清洗表可追溯** —— 每行派生数据可追溯到原始文件 + 行号 |
| 3 | **字段统一优先于分析** —— 字段未标准化不进分析,不允许"半统一"分析 |
| 4 | **条码优先** —— 主键以 barcode 为准,商品名仅辅助 |
| 5 | **原始名保留** —— sku_name_raw 永不动 |
| 6 | **标准名用于分析** —— sku_name_std 派生可迭代 |
| 7 | **自动匹配必可复核** —— 每条匹配含 `match_confidence` + `match_method` |
| 8 | **低置信不进正式结论** —— 见 §3 置信度规则 |

## 2. 十一阶段流程
| 阶段 | 动作 | 验收 |
|------|------|------|
| S1 接收 | 客户给什么处理什么,不主动索超需数据 | 文件清单完整 |
| S2 归档 | 原始表归档冻结 | original_path 登记 |
| S3 索引 | 建数据文件索引 | 双路径 original/standard 完整 |
| S4 判类型 | 判断表类型(流水/档案/库存) | category 正确 |
| S5 字段映射 | 客户字段 → 标准字段 | 必填 100% 映射,缺失标"待确认" |
| S6 类型清洗 | 日期/条码/数字/空值/负值/重复 | 通过类型校验,异常进清单 |
| S7 商品名标准化 | sku_name_raw → sku_name_std | —— |
| S8 品类匹配 | 商品 × 标准品类表 4 层匹配 | L1 覆盖100%,L4 ≥70% |
| S9 异常清单 | 输出异常 | 每条含建议动作+影响等级 |
| S10 标准表 | 生成诊断可用标准表 | 6 张表齐全 |
| S11 复核 | 复核 + 版本记录 | —— |

## 3. 四级匹配置信度（决定数据能否进诊断）
| 置信度 | 含义 | 诊断使用边界 |
|--------|------|------------|
| **高** | 一级条码精确 + 名一致 + 品牌一致 | ✅ 可进正式分析/写客户报告(A级证据) |
| **中** | 二级关键词 + 品牌一致 / 一级但名小差异 | 🟡 初步分析,标"待复核"(B级证据) |
| **低** | 三级组合 + 信息不全 / 二级跨品类风险 | ❌ 不进正式结论,人工复核后升级(C级证据) |
| **冲突** | 多法给出不同 L4 / 客户原品类与标准冲突 | ⛔ **暂停使用**,人工裁决前不得进任何分析 |

## 4. 十三类异常标准
| # | 异常 | 影响 | 动作 |
|---|------|:--:|------|
| 1 | 无条码 | 高 | 进异常,从档案补 |
| 2 | 重复条码(同表) | 中 | 合并+备注 |
| 3 | 一品多码 | 中 | 人工复核 |
| 4 | 多品一码 | 高 | 必须人工分拆 |
| 5 | 商品名缺失 | 高 | 从档案补,补不了则不可用 |
| 6 | 规格缺失 | 中 | 从档案/名解析 |
| 7 | 价格异常(0.01/9999/负) | 高 | 进异常,不进结论 |
| 8 | 毛利为负 | 中 | 保留(可能促销)+备注 |
| 9 | 库存为负 | 高 | 进异常,可能错账 |
| 10 | 销量异常(极大极小) | 中 | 校验 |
| 11 | 时间区间异常 | 高 | 进异常 |
| 12 | 品类无法匹配 | 中 | 人工复核 |
| 13 | 客户原品类与标准冲突 | 高 | 人工裁决 |

## 5. 三类标准表（S10 输出）
- **销售流水表**(10字段)· **商品档案表**(14字段)· **库存表**(9字段)+ 商品-品类匹配表 + 异常清单 + 清洗日志 = 6 张。
