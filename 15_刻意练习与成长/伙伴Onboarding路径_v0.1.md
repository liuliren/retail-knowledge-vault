>

## 一、定位与边界

### 1.1 为什么需要伙伴层

vault 的三个时间维度命题（[[CLAUDE.md]] §1）：

- **三个月生存战**：六哥 1 人能撑（花厅坊 + 现金流）
- **三年事业站起来**：必须 2-5 人团队 → **伙伴层从这里开始**
- **十年招牌传下去**：必须有人能独立带战役 + 内核不变

伙伴层 = "把六哥个人能力外化为可复制的团队作业系统"。它和 vault / 战役 SOP / 客户验证日志 共同构成事业平台。

### 1.2 边界（本期不做）

- ❌ 不招实际伙伴（等花厅坊样板对外可讲后才有招募依据）
- ❌ 不写薪酬 / 股权 / 利润分配（三年事业期议题）
- ❌ 不开外部分享课（先把内部 SOP 跑通）

### 1.3 与 [[刻意练习总路径_2年合并版_DP-00]] 的关系

| 维度 | 刻意练习总路径 | 伙伴 Onboarding 路径 |
|---|---|---|
| 对象 | 六哥本人 | 加入团队的伙伴 |
| 时长 | 2 年 | 90 天上手期 + 持续成长 |
| 终点 | 个人能力护城河 | 能独立带战役（合伙顾问层） |

## 二、角色矩阵（5 类）

来源：v0.2 §4.6.1。

| 角色 | 定位 | 能做 | 不能做 | 出现时机 |
|---|---|---|---|---|
| **首席（六哥）** | 战略 / 关系 / 品牌内核 | 全部 | — | 现在 |
| **合伙顾问** | 独立带战役 | 接客户 / 独立交付 / 培训学徒 | 改 I 象限 / 改品牌口径 | Phase 3 起（花厅坊样板后） |
| **项目经理（PM）** | 战役 PM 执行 | 推进 SOP / 维护战役档案 / 主持 G04/G05/G06 | 战略判断 / 关系判断 | Phase 3 / 第二个客户战役起 |
| **数据分析师** | 数据 / 诊断 / 看板 | 跑数据 / 出报告 / 维护 13_ | 客户面对面 | Phase 2 起（兼职可） |
| **学徒 / 助理** | 做执行 + 学经验 | 跑 SOP / 整理素材 / 写初稿 | 独立面对老板 / 独立决策 | Phase 1 起（找助理协助 90_/00_客户原始材料/ 整理） |

### 2.1 角色之间的转化路径

```
学徒 (90 天) → 数据分析师 / PM (1-2 年) → 合伙顾问 (2-3 年)
```

**红线**：学徒在过完整 90 天 Onboarding + 至少跟过 1 个完整战役 + 反幻觉巡检通过率 ≥80% 之前，**不允许**独立面对客户老板。

## 四、RACI 框架

来源：v0.2 §4.6.2。每个战役 SOP 在 §G 段都把"人 / Claude / 客户"扩展为：

```
| 任务 | 首席 | 合伙顾问 | PM | 数据分析师 | 学徒 | Claude | 客户 |
| ... | A | R | C | — | — | C | I |
```

**符号约定**：

- **R** (Responsible) — 实际执行
- **A** (Accountable) — 最终负责（每任务有且仅有 1 个 A）
- **C** (Consulted) — 提供输入 / 被咨询
- **I** (Informed) — 仅被告知结果
- **—** — 不参与

### 4.1 默认 RACI 模板（战役级）

| 任务类别 | 首席 | 合伙顾问 | PM | 数据分析师 | 学徒 | Claude | 客户 |
|---|---|---|---|---|---|---|---|
| 战略判断（接 / 不接客户） | **A/R** | C | I | — | — | C | — |
| 客户老板访谈 | **A/R** | C | I | — | — | — | I |
| 战役章程起草 | A | R | C | — | — | C | I |
| 阶段门评审主持 | A | C | **R** | C | I | C | C |
| 数据清洗与诊断 | I | C | C | **A/R** | C | C | — |
| 商品诊断报告（草稿）| I | A | C | R | C | C | — |
| 商品诊断报告（终版）| **A** | R | C | C | I | C | C |
| 客户报告对老板汇报 | **A/R** | C | C | — | — | — | I |
| 战后 AAR 主持 | A | C | **R** | C | I | C | I |
| 客户验证日志写作 | A | R | C | C | C | C | — |
| 90_ 素材整理 | I | I | C | C | **A/R** | C | — |

### 4.2 RACI 红线

- **A 永不让渡** — 战略判断 / 终版报告 / 客户老板汇报的 A 永远是首席（六哥）。
- **客户始终 I 或 C** — 客户不进 R / A 列。
- **Claude 永不进 R / A** — Claude 是 C（咨询）或 I（被告知），不能背最终责任。

## 五、质控双层

来源：v0.2 §4.6.4。

### 5.1 自动化层（Phase 2 起）

由 `.claude/agents/` 镜像（参 [[_镜像图]] §2.4）：

| Agent | 作用 |
|---|---|
| 反幻觉巡检员 | 扫真实客户结论的依据链 |
| 一致性巡检员 | 扫核心概念是否冲突 / 多定义 |
| 客户视角审稿员 | 站客户角度看报告（看不懂 / 不可执行 / 没价值的标红） |
| 店长视角执行员 | 站店长角度看 SOP（不可操作 / 缺资源的标红） |
| 陈列规则巡检员 | 扫陈列规则一致性 |

配套机制：[[G03_Lint]] 周仪式 + [[Vault健康仪表]]。

### 5.2 人工层（现在就要建的纪律）

- 新人产出**必须**经合伙顾问 review 才能交付客户
- 关键报告（≥客户老板能看到的）**必须**经六哥 review
- [[G04_战役章程]] / [[G05_阶段门评审]] / [[G06_战后复盘AAR]] 三个 PM 仪式**必须**人主持，不允许 Claude 主持
- 对外内容（视频号 / 长文 / 客户报告 / 销售包）发布前**必须**过[[品牌资产手册_v0.1]] §4 三道关

### 5.3 质控的 I 象限不可让渡

无论自动化层做得多好，**I 象限内核（反幻觉 / 逻辑一致性 / 客户始终 I 或 C / 终版必须人 review）永不让渡给 Claude**。这是事业平台 vs 工具的本质区别。

## 六、招募门槛（待 Phase 3 充实）

⬜ **待 Phase 3 第一次招募时定型**。骨架方向：

- **学徒 / 助理**：有零售门店实操经验 ≥1 年；能写中文专业文档；接受 Obsidian + Claude Code 工作流
- **数据分析师**：能独立用 Excel + Python 做 SKU 级数据清洗；能读 POS 数据；零售业务理解优先
- **PM**：有项目交付经验；会用 Notion / Obsidian / 类似工具；零售背景优先
- **合伙顾问**：零售商品 / 品类 / 陈列实战 ≥5 年；带过门店 / 项目；认同科学零售内核
