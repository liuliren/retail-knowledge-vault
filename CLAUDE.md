# 晟果新零售咨询 / 六哥零售知识库项目说明

## 1. 项目定位

本项目是"晟果新零售咨询 / 六哥零售"的长期知识库与咨询作业后台。

它不是普通资料库，而是一套服务真实零售咨询项目的"知识后台""咨询作业系统大脑"和"AI协作知识工程体系"。

项目服务对象包括：
- 晟果新零售咨询
- 六哥零售
- 乐易购超市
- 乐易购花厅坊店
- 未来更多社区超市、零售企业、便利店、生鲜店、折扣店及相关零售项目

项目核心目标是：通过 Obsidian + Claude Code，把零售方法论、真实门店案例、数据分析、咨询交付、系统产品设计、自媒体内容转化，统一沉淀成可检索、可复用、可迭代、可落地的知识资产。

---

## 2. 项目目标

1. 建立"科学零售知识库"，沉淀商品力、组织力、运营力方法论。
2. 支持零售企业生意诊断、商圈分析及客群调研、商品诊断、品类管理、陈列优化、营销活动策划与实施、经营分析、咨询交付。
3. 支持把门店案例、现场图片说明、Excel 分析、PRD、SOP、DisplayMap、视频文案统一整理为结构化知识资产。
4. 默认使用中文输出，风格要求专业、克制、结构化、可落地。
5. 所有输出优先服务于"晟果新零售咨询""六哥零售""乐易购超市""乐易购花厅坊店"等真实零售咨询场景。
6. 知识库要像软件产品一样维护，具备版本管理、引用关系、反幻觉机制、逻辑一致性检查和持续迭代能力。

---

## 3. 总体工作原则

1. 所有内容必须服务真实零售业务，不写空泛理论。
2. 优先结构化表达，便于 Obsidian 检索、双链、复用。
3. 所有方法论内容要区分：理论、规则、案例、模板、行动清单。
4. 不确定的信息统一标注"待确认"，不要编造事实。
5. 修改重要文件前，先说明修改范围；涉及重要版本时，先生成备份或给出 diff。
6. 不要随意删除文件。
7. Markdown 文件要适配 Obsidian。
8. 文件命名使用中文为主，必要时加版本号，例如 v1.0、v1.1、v2.0。
9. 内容要尽量沉淀成可复用资产，而不是一次性回答。
10. 对真实项目资料，要区分原始素材、过程分析、正式交付、复盘结论。
11. 输出要兼顾老板能看懂、店长能执行、顾问能复用、系统能产品化。
12. 不要只给观点，要尽量给判断依据、执行路径、检查清单和复盘方式。

## 7. 金额管理、数量管理、品质管理定义

### 7.1 金额管理

金额管理是**经营目标层**。

关注：销售额、毛利额、毛利率、库存金额、周转金额、费用金额、损耗金额、促销让利金额、坪效、人效、品类贡献额、单品贡献额。

回答：这家店、这个品类、这个项目最终要拿到什么经营结果？

### 7.2 数量管理

数量管理是**运营执行层**。

关注：SKU 数量、品类 SKU 数、陈列面数量、排面数、层板数、堆头数、端架数、库存数量、进货数量、补货数量、销售数量、客流数量、客单件数、动销 SKU 数、缺货 SKU 数、淘汰 SKU 数、新品引进数、活动商品数。

回答：为了实现金额目标，现场到底要管哪些数量动作？

### 7.3 品质管理

品质管理是**价值控制层**。

关注：商品品质、价格品质、结构品质、陈列品质、库存品质、毛利品质、销售品质、数据品质、执行品质、顾客体验品质。

回答：金额和数量是否健康、真实、可持续，而不是短期虚胖？

### 7.4 三者关系

金额管理确定目标 → 数量管理拆解动作 → 品质管理控制质量 → 经营复盘反馈修正。

## 8. 52周MD 定义

52周MD 不是简单的年度促销日历。它是以一年52周为周期，把季节、节日、顾客生活任务、商品波段、陈列资源、活动节奏、库存准备统一起来的商品经营计划。

它要回答：什么时候卖什么、提前多久备货、哪些商品做形象/走量/毛利、哪些品类在什么时间放大、哪些陈列资源给哪些主题、如何把年度经营节奏拆成月度/周度/门店动作。

适合关联目录：
- 01_科学零售方法论/52周MD.md
- 05_品类管理与商品规划/年度品类规划.md
- 07_营销活动策划与实施/52周活动日历.md
- 08_经营分析与数据看板/周度经营复盘.md
- 10_咨询交付模板/52周商品经营计划模板.md

社区超市52周MD重点关注：春节、元宵、清明、五一、端午、暑期、中秋、国庆、冬季、开学季、周末家庭消费、早餐场景、夜宵场景、亲子零食、社区团购冲击、生鲜带动食品。

## 13. 反幻觉规则

必须严格遵守以下规则：

1. 任何真实门店、真实商品、真实数据相关内容，必须区分事实、推断、建议、待确认。
2. 不得把推测写成事实。
3. 不得编造门店数据、销售数据、库存数据、顾客反馈、竞品信息。
4. 不得只凭销量判断商品力。
5. 商品诊断必须同时检查活动、陈列、缺货、价格、库存、客群影响。
6. 陈列方案必须考虑包装尺寸、层板高度、货架结构、顾客拿取便利性和真实门店条件。
7. 活动分析必须区分自然动销和促销拉动。
8. 外部案例必须经过本土化判断，不能直接套用到社区超市。
9. 不确定信息必须标注"待确认"。
10. 生成正式结论前，应列出依据文件或依据事实。
11. 现场照片、Excel、门店反馈、销售系统数据之间若存在矛盾，必须标注冲突，不能强行合并。
12. 对 DisplayMap、陈列图、货架方案的建议，必须考虑真实货架结构、商品包装尺寸、品类边界和门店动线。
13. 对 SKU 的淘汰、保留、新增建议，必须同时考虑销售额、毛利额、库存、动销、陈列、活动和客群。
14. 对营销活动效果的判断，必须区分活动期间、活动前、活动后，不能只看单点销量。
15. 对系统 PRD 或工具规则的设计，必须标注数据字段依赖和待确认项。

### 13.16-19 新硬约束（2026-05-09 新增 / vault 连环错误事件触发 / 4 层治理底盘）⭐

详细见 [[反幻觉检查清单]] v1.1 §I-L + [[文档工程化标准]] v1.3 frontmatter fact_layer + [[feedback_数据治理与反幻觉机制升级]] memory。

**16. 客户视角防火墙（§13.16）**：vault 不写"客户内部具体谁操作 X"作为事实。除非客户亲口告知（视频会议 / 访谈 / 微信文字记录有据可查），否则一律用门店主体表述（"客户方提供 / 门店 POS 导出 / 客户协调 / 客户内部分工"）。**反例**：错把"5/8 文件 mtime"推断为"启明导出 89 .xls"。

**17. 推断标注强制（§13.17）**：任何"推断 / 假设 / 估计"的表述必须 `[推断]` / `[假设]` / `[待确认]` 前缀。含"应该 / 可能 / 推测 / 估计 / 大概 / 大约 / 约"等词的段落必须明示推断状态。数据估算必须标注精度限制（"±X%"）。frontmatter 必填 `fact_layer` 4 元字段：`observed` / `client_told` / `inferred` / `pending`。

**18. 类比边界（§13.18）**：使用类比框架（沃尔玛 / 海底捞 / Apple / 等）必须明示"这是类比 / 保留结构同构 / 删戏剧化"。中国家族企业类比时默认协同+信任（参 [[feedback_客户家族关系协同信任]]）/ 不代入西方 Succession / Walton 接班 / Murdoch 等"权力斗争"剧本。**反例**：错把 Sam → Rob 接班张力套到启明身上。

**19. 用户订正优先（§13.19）**：当用户最早表述与后续订正冲突时，**优先订正版**（不要"我记得用户最早说过 X"）。旧版表述作 audit-log 留存（不删 / 标 ❌ 或 ~~~~~~ / 加版本说明）。feedback memory 必须沉淀此订正 / 让 vault 跨会话不再犯。**反例**：用户最早说"启明=店长股东" / 订正"启明=老板" / vault 仍用最早版本。

**强制范围**：涉及客户事实 / 客户内部操作 / 客户家族关系 / 类比框架 / 数据推断的所有文档，§13.16-19 一律必查。frontmatter `fact_layer` 4 元字段是数据契约 / 不可空挂。

### 13.20 数据一致性追溯机制（2026-05-09 v2.2 数据矛盾事件触发）⭐⭐

**触发场景**：vault 在不同数据源 / 不同时段 / 不同维度遇到同一指标的数字不一致。

**典型例子（5/9 v2.2 事件）**：
- 客流 145,490 × 客单 42.14 = 613 万 vs 类别销售 300 万（差 2 倍）
- 客流报表时段 4/7-5/8 vs 销售报表时段 4/8-5/8（错 1 天）
- 滞销 SKU 数 = 文件总行数（含 header）vs 实际数据行（去 header）

**硬约束（不可绕过）**：

1. **遇到数据不一致 / 立即停止分析**（不假定 / 不"圆"数据）
2. **追溯每个数字的 evidence**：文件名 + sheet + 行号 + 时段 + 字段定义
3. **列 ≥ 3 种解释假设**（如双倍统计 / 时段错开 / 字段定义不同 / 单位混淆）/ 每个假设带数学验证
4. **找内部一致性最高的解释**（与其他独立数据交叉验证）
5. **标 [推断 / 推理链]**：不是结论 / 是当前最可能假设
6. **必须用户 confirm 才能用作分析输入**
7. **沉淀为 vault 历史教训**：在 [[数据一致性追溯案例库]] 累积

**反例（v2.2 错误做法）**：
- ❌ 凭"差 2 倍"猜测"双倍统计"作为结论
- ❌ 用"30 天"代替实际"32 天"的时段
- ❌ 把"7,140 行 = 7,140 SKU"（实际 7,139 / 含 header 6 行 / 误差小但不严谨）

**正例（5/9 用户驱动）**：
- ✅ 用户问"7,139 怎么来的"→ 我重读文件 → 发现实际 7,139 SKU（不是 7,140 行）
- ✅ 时段 4/7-5/8（32 天）vs 4/8-5/8（31 天）/ 错 1 天
- ✅ 客单×客流 ≠ 销售 / 列 3 种解释 / 等用户 confirm（用户告知 "客流 = 成交客户数"）

**关联**：
- [[反幻觉检查清单]] §M（v1.2 加 / 数据一致性追溯专项）
- [[文档工程化标准]] frontmatter `data_source` 字段强制（标时段+维度）
- [[feedback_数据治理与反幻觉机制升级]] 4 层治理 + §13.20

### 17.11 类 1 — 目录与路径纪律（2026-05-09 新增 / "致命错误：不是 Git 仓库" 事件触发）⭐

> **触发**：5/9 W19 Python 脚本读 /tmp 数据时用 `cd /tmp && python3 ...` 切走 Bash cwd / 后续 `git add -A && git commit` 在 /tmp 跑 / 报"致命错误：不是 Git 仓库"。

**根因**：Bash 工具的工作目录在命令之间**持久化**。一次 `cd <path>` 会影响所有后续 Bash 命令 / 直到下一次 `cd` 切回。

**硬约束**：

1. **绝不在 Bash 层用 `cd /tmp` 等切走 cwd**（vault 是 cwd / 切走后 git / Read / Write 全部失效）
2. **Python / Shell 脚本读外部目录** → 在脚本内部 `os.chdir()` 或用绝对路径 / **不在 Bash 层 cd**
3. **git 命令必要时用 `git -C <repo> ...`** 而不是 `cd <repo> && git ...`（避免 cwd 副作用）
4. **如必须 cd**（极少数 / 用户明确要求）→ 命令末尾必须 `cd -` 或 `cd <vault>` 切回
5. **执行可疑的目录敏感命令前**（git / Read / Write）→ 先 `pwd` 验证 cwd 在 vault

**反例**（5/9 错误做法）：
```bash
# ❌ 这条命令副作用：把 Bash cwd 永久切到 /tmp
cd /tmp && python3 << 'EOF'
import openpyxl
wb = openpyxl.load_workbook("/tmp/01_门店数据材料_xlsx/...")
EOF
# 后续 git 在 /tmp 跑 → "致命错误：不是 Git 仓库"
```

**正例**：
```bash
# ✅ 方式 1：Python 脚本内部 chdir / Bash cwd 不变
python3 << 'EOF'
import os
os.chdir("/tmp/01_门店数据材料_xlsx")
import openpyxl
wb = openpyxl.load_workbook("滞销商品.xlsx")
EOF

# ✅ 方式 2：直接用绝对路径 / 不切 cwd
python3 << 'EOF'
import openpyxl
wb = openpyxl.load_workbook("/tmp/01_门店数据材料_xlsx/滞销商品.xlsx")
EOF

# ✅ 方式 3：git 用 -C 选项
git -C /Users/davidliu/KnowledgeBase/retail-knowledge-vault add -A
```

**关联**：[[CLAUDE.md]] §13.20 数据一致性追溯（同样是"细节差异叠加 = 系统性错误"的反例 / 但维度不同：§13.20 是数据维 / §17.11 是工程操作维）。

### 18.2 Query（G02）— 知识被调用

- **触发**：重要探索 / 大决策 / 跨多文件分析 / 客户问答前的依据查证。
- **流程**：1) 明确问题 → 2) 列依据文件清单 → 3) LLM 跨文件检索并整合 → 4) 输出标注依据链 → 5) 探索结果如有沉淀价值，归档为综合主页（不是堆 90_）。

- **SOP 详见**：[[G02_Query]]（D4 占位 / Phase 1 充实）。

### 18.3 Lint（G03）— 每周一次健康巡检

- **触发**：每周一周仪式（30 分钟）。
- **流程**：1) 扫孤立页 → 2) 扫概念矛盾 → 3) 扫 status: draft 超 30 天文件 → 4) 扫 [[99_待Ingest]] 超 14 天素材（D5 落地）+ 90_/03_临时灵感 超 30 天素材 → 5) 更新 [[Vault健康仪表]] 8 维度（D2 落地）。

## 19. 四象限治理（顶层最高纲领）

vault 中**任何元素**必须归入四象限之一，frontmatter 强制 `quadrant` 字段（参 [[文档工程化标准]] v1.1）。这是 v0.2 重构中**优先级最高的设计**——它定义了"什么能动 / 谁能动 / 怎么动"，在它之下才有 SOP、PM、伙伴层。

| 象限 | 名称 | 内容举例 | 治理要求 | 变更门槛 |
|---|---|---|---|---|
| **I** | 内核稳固（不变 + 可复制）| 反幻觉规则 / 逻辑一致性 / 文档工程化 / 三大优先级判据 / 三回归原则 / 三力骨架 / 品牌内核 | 永不轻动；变更须经核心团队议事 | 最高，进入 [[变更影响检查清单]] 顶级条目 |
| **II** | 内核灵活（不变 + 权变）| 客户老板访谈话术 / 现场危机判断 / 报告对老板的表达色调 / 团队信任与节奏 / "这次客户特殊"通道 | 原则不动，表达权变；沉淀金句库 / 话术库 / 红线集 | 中，每次实战权变后必须复盘回写 |
| **III** | 外延复制（变化 + 可复制）| SOP 套件（S01-S08 + G01-G06）/ 销售包模板 / 客户报告模板 / 数据清洗规范 / DisplayMap 模板 / 经营看板模板 / Claude Code 配置 | 必须 SOP 化、模板化、可教学 | 低，标准 v0.x 流程升级 |
| **IV** | 外延权变（变化 + 权变）| 创造性方案设计 / 区域文化差异 / 客户家族 / 关系 / 行业趋势 / 业态变化 / 工具栈迭代 | 守住价值观下自由发挥 | 低，但每次发挥后必须强制 AAR |

**双向防火墙**：
- 防"用 SOP 杀死创造性"：保留 II + IV 象限的合法性，让顾问敢于权变。
- 防"用创造性瓦解标准"：I + III 象限是必须遵守的稳定基线，不允许"我特殊我例外"。

完整论述见 [[2026-05-04_知识库重构方案_v0.2_完整作战方案]] §2。

## 20. 三回归仪表（事业级指南针）

vault 是工具，不是目的；战役是路径，不是终点。每周一短扫（10 分钟）+ 月初深扫（30 分钟），用三回归反"假忙碌"。详见 [[三回归仪表]]（Phase 0 D2 落地）。

1. **回归到人**：本期有几个真人因为我们的工作变得更好？（客户老板 / 店长 / 店员 / 顾客 / 伙伴 / 自己）
2. **回归到价值**：客户能"自己说出来"的价值是什么？如果客户说不出来，就还没产生价值。
3. **回归到结果**：这周离回款近了几步？离花厅坊样板对外可讲近了几步？离第一个新客户近了几步？

**假忙碌扫描**：如果三个回归都没动，但 vault 推进了 5 个版本——这是假忙碌，必须停。

**三个永远的拷问**（每次有疑虑时问，参 v0.2 §10.2）：
- 这件事跟客户交付、签约、保留有什么直接关系？
- 哪个真人因为这件事变得更好？
- 这件事属于哪个象限——内核稳固 / 内核灵活 / 外延复制 / 外延权变？

**答得清，干；答不清，停。**

### 23.4 与 vault 反幻觉规则的边界

1. gstack 任何输出（含 `/learn` 记忆、`/browse` 抓取内容、`/qa` 报告）进入本知识库前，**必须经 §11 知识沉淀分类规则**，不得直接落入 99_ 之外的目录。
2. `/browse` `/scrape` 抓取的外部资料默认进入 `90_素材暂存与待整理/` → 走 §18.1 Ingest（G01）流程。
3. `/learn` 的"项目级模式 / 偏好"沉淀属于**外延复制（象限 III）**，**不允许**自动改写 §6 核心方法论 / §13 反幻觉规则 / §19 四象限 / §21 客户验证 24h 等内核条目。
4. 任何 gstack skill 的执行结果，如要写回正式方法论 / 模板 / PRD，必须保留来源（哪个 skill / 哪次会话 / 时间），符合 §17.6 真实项目内容来源追溯要求。

### 23.5 安全闸默认开启建议

修改本知识库重要文件时，建议默认运行 `/guard`（= `/careful` + `/freeze`）：

- `/careful`：在 `rm -rf` / `git reset --hard` / `force-push` / `DROP TABLE` 等破坏性操作前强制确认。
- `/freeze`：把 edit 范围锁死在当前任务的目录，防止 Claude "顺手修复"无关文件（特别防止误改 `01_科学零售方法论/` 等内核文件）。
- `/investigate` 自动 freeze 到被排查模块。
- 完成后 `/unfreeze` 解除。
