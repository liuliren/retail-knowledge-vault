## §2. P1-10：AI 协作准确率预期调整

### 2.1 v1.0 §2.1.2 假设（过高）

```
"AI 写代码准确率 > 90%"
```

### 2.2 v1.1 修订（实测预期 / Anthropic Honest）

```
═══════════════════════════════════════════════════════════════
              AI 协作准确率（实测预期 / 不夸大）
═══════════════════════════════════════════════════════════════

层 1 - Boilerplate（路由 / 配置 / 模板）：90%
  - Cursor / Claude 写得很好
  - 你只需检查 + 微调
  - 节省 80% 时间

层 2 - UI 组件（shadcn / Tailwind）：85%
  - 流行库训练数据丰富
  - 略需调整样式 / 交互
  - 节省 70% 时间

层 3 - 业务逻辑（API / 数据流）：60%
  - 需要自己设计架构
  - AI 帮写实现细节
  - 节省 40% 时间

层 4 - 复杂集成（PDF 中文 / 算法包 / WeasyPrint）：40%
  - 中文字体 / 编码问题 AI 容易出错
  - 算法包 v0.1.3 对接需手动验证
  - **关键代码自己写**
  - 节省 < 30% 时间

═══════════════════════════════════════════════════════════════

实操建议：
  ✅ Boilerplate / UI → 让 AI 全权
  ✅ 业务逻辑 → AI 提供草稿 / 你严审 + 调试
  ⚠️ 复杂集成 → 自己设计 + AI 协助 / 不依赖 AI
  ❌ 不要"100% 信任 AI" / 容易踩坑
  
关键代码自测原则：
  - 算法包集成（4 级匹配）：自己写 + AI 辅助
  - PDF 中文字体配置：自己验证 + AI 给方案
  - 数据库 schema：自己设计 + AI 实现
  - SQLite 分片（P2-15）：自己设计 + AI 实现
═══════════════════════════════════════════════════════════════
```

## §3. P2-14：F-07 改"多店对比 N 店动态"

### 3.1 v1.0 §3.6 4 店硬编码

```
"4 店多店对比" — 硬编码 4 店
```

### 3.2 v1.1 修订（兼容肇庆 10 店）

```
═══════════════════════════════════════════════════════════════
              F-07 多店对比 N 店动态（v1.1 修订）
═══════════════════════════════════════════════════════════════

支持范围：N 店动态（2-20 店灵活）

数据库 schema 变更：
  - stores 表（store_id / store_name / client_id）
  - 不限制 store 数量
  - 前端动态渲染 N 列

前端 UI 变更：
  - 横向滚动表格（> 5 店时）
  - 折叠 / 展开（默认显示 5 店）
  - 点击单店下钻

适配场景：
  ✅ 乐易购 4 店：4 列对比
  ✅ 肇庆 10 店：10 列对比（横向滚动）
  ✅ 第三客户 N 店：动态适配

10 项核心对比指标（保持）：
  - 销售额 / 毛利率 / 客单价 / SKU 数
  - ABCD 分布 / D1 占比 / 库存周转
  - 大湾区品类贡献 / TOP 10 SKU / 异常 SKU 数

自动洞察生成（适配 N 店）：
  - 找最高 / 最低店（不限定 ABCD 4 店）
  - 排名分布（10 店时给 TOP 3 + Bottom 3）
  - 大湾区品类对比（> 5 店时区域对比）
═══════════════════════════════════════════════════════════════
```

### 3.3 验收标准

```
- ✅ 4 店：原 v1.0 模式（兼容）
- ✅ 10 店：横向滚动 / 折叠 / 性能 < 3 秒
- ✅ 自动洞察 ≥ 5 条
- ✅ 支持下钻（N 店任一店点击进入详情）
```

### 4.2 v1.1 修订（SQLite 分片 / 不上云）

```
═══════════════════════════════════════════════════════════════
              多客户支持架构（v1.1 / SQLite 分片）
═══════════════════════════════════════════════════════════════

  ✅ 仍本地运行（你的笔记本）

数据库分片（v1.0 末期 / Week 14 起）：
```

## §5. P3-16：加 Anthropic HHH 验收标准

### 5.1 v1.1 §5 验收标准新增 AC-A-01

```
═══════════════════════════════════════════════════════════════
              AC-A-01：Anthropic HHH（v1.1 新增）
═══════════════════════════════════════════════════════════════

Honest（诚实）：
  ✅ 报告所有数字必须有数据依据
  ✅ 推断 vs 事实必须区分（标"分析" / "事实" / "建议" / "待确认"）
  ✅ 客户原话**一字不改**记入（不允许润色）
  ✅ 缺失数据标"待确认"（不允许补全或编造）
  
  自动化检查（v1.5 加 lint）：
    - PDF 报告所有数字字段必须有 source 注释
    - 推断字段必须前缀"分析:" / "建议:" / "估算:"
    - 客户原话必须 quote 标注
    - "待确认"字段必须 highlight

Helpful（有用）：
  ✅ 报告对客户有真价值（不是堆砌数字）
  ✅ 老板能看懂 / 店长能执行
  ✅ 一线员工能用（Apple 简单语言）
  ✅ 验收方式：客户认可度 ≥ 8/10

Harmless（无害）：
  ✅ 客户敏感数据保护（client_safety 三级）
  ✅ 不外发未经授权数据
  ✅ 反幻觉规则全程 enforce
  ✅ 错误情况下安全降级（不崩溃 / 不丢数据）

═══════════════════════════════════════════════════════════════

实施检查（每次发布前）：
  □ 数字依据可追溯（≥ 95%）
  □ 推断 vs 事实清晰区分
  □ 客户原话准确率（≥ 100% / 一字不改）
  □ 客户敏感字段加密（client_safety: confidential）
  □ 错误处理友好（不暴露内部信息）
═══════════════════════════════════════════════════════════════
```
